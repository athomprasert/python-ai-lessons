Handling Timeouts & Network Errors

Stop your script from hanging forever. Catch the right exceptions and fail loudly.

Lesson 1 ✓ Lesson 2 ✓ Lesson 3 — Timeouts & Errors Lesson 4 — Calling Claude

Why this matters

By default, requests waits forever for a response. If the server hangs, your script hangs. In a RAG system or AI agent — where you might be calling many APIs in sequence — one slow endpoint can freeze the entire pipeline.

Connection to your mission
Every external call in an agent (vector DB lookup, document fetch, LLM call) needs a timeout. A robust agent never hangs — it fails, logs, and either retries or moves on.

1 — Set a timeout on every call

Pass timeout= to any requests call. The number is seconds.

import requests

# Single number = total max wait (in seconds)
response = requests.get(
    "https://api.example.com/products",
    timeout=5      # raise an exception if no response in 5 seconds
)
Better: tuple form
A tuple (connect_timeout, read_timeout) separates "how long to wait to start talking to the server" from "how long to wait for data". 
timeout=(3, 10)   # 3s to connect, 10s for response body
Use this for slow APIs (like LLMs) where the server is fast to accept but slow to generate.
Common mistake
timeout=0 doesn't mean "no timeout" — it means "fail immediately". To disable timeouts (don't), use timeout=None. Always prefer a real number.

2 — The exceptions you'll actually catch

All requests exceptions inherit from requests.exceptions.RequestException. The four you'll meet most:

ExceptionWhat it means
Timeout Either connect or read took too long. Server may be slow or down.
ConnectionError Couldn't reach the server at all — DNS, network, firewall, or server offline.
HTTPError Server returned 4xx/5xx. Only raised by raise_for_status().
RequestException Parent of all the above — catch this if you don't care about the specific cause.
If you know Java
Same pattern as SocketTimeoutException, ConnectException, and HttpClientErrorException in Spring/Java. RequestException is the analog of catching IOException as a general fallback.

3 — try / except: Python's try/catch

Python uses try / except instead of try / catch. Same idea, slightly different syntax.

import requests
from requests.exceptions import Timeout, ConnectionError, HTTPError

try:
    response = requests.get(
        "https://api.example.com/products",
        timeout=(3, 10)
    )
    response.raise_for_status()
    data = response.json()

except Timeout:
    print("⏱  The server took too long to respond.")

except ConnectionError:
    print("🔌 Could not reach the server. Check the URL or network.")

except HTTPError as e:
    print(f"❌ Server returned {e.response.status_code}: {e.response.text}")

except requests.exceptions.RequestException as e:
    print(f"💥 Unexpected request error: {e}")
Order matters
Catch specific exceptions first, then more general ones. Python checks except blocks top-to-bottom — if you put RequestException first, it would swallow everything before Timeout ever ran.

4 — Put it together: a robust API call

This is a small but production-shaped function. You'll see this exact shape in agent code.

# robust_call.py
import requests
from requests.exceptions import Timeout, ConnectionError, HTTPError

def fetch_posts(user_id):
    """Fetch posts for a user. Returns [] on failure, never crashes."""
    try:
        response = requests.get(
            "https://jsonplaceholder.typicode.com/posts",
            params={"userId": user_id},
            timeout=(3, 10)
        )
        response.raise_for_status()
        return response.json()

    except Timeout:
        print(f"Timeout fetching posts for user {user_id}")
    except ConnectionError:
        print(f"Network error fetching posts for user {user_id}")
    except HTTPError as e:
        print(f"API returned {e.response.status_code}")

    return []   # safe fallback

# Test the happy path
print(len(fetch_posts(1)))   # 10

# Test the failure path — bogus host triggers ConnectionError fast
def fetch_from_bad_host():
    try:
        return requests.get(
            "https://this-host-does-not-exist-xyz.invalid",
            timeout=(2, 2)
        )
    except ConnectionError:
        print("Caught it — host is unreachable.")

fetch_from_bad_host()

5 — Simulate a real timeout

Hard to test a timeout without a slow server. httpbin.org has a /delay/<seconds> endpoint that holds the response for the number of seconds you ask for. Set your timeout shorter than the delay to force a Timeout exception.

import requests
from requests.exceptions import Timeout

try:
    # Server holds for 5s, we'll only wait 2s
    requests.get("https://httpbin.org/delay/5", timeout=2)
except Timeout:
    print("Got a timeout, exactly as expected!")
Try it yourself
  1. Save the snippet above as test_timeout.py and run it.
  2. Change timeout=2 to timeout=10 — does the script wait, then succeed?
  3. Combine with fetch_posts() from section 4: change the URL to https://httpbin.org/delay/5 and confirm your except Timeout: branch fires.
  4. Challenge: Add a retry loop — if a Timeout happens, try up to 3 times before giving up. (Hint: a for loop with range(3) and break on success.)

Quick check

Q1. What does requests.get(url) do by default if the server never responds?

  • Times out after 30 seconds
  • Times out after 60 seconds
  • Waits forever
  • Raises a Timeout exception immediately

Q2. What's the difference between timeout=10 and timeout=(3, 10)?

  • No difference — both wait 10 seconds total
  • The tuple sets connect and read timeouts separately
  • The tuple retries 3 times with a 10-second timeout
  • The tuple is invalid syntax

Q3. You wrote this code, but the Timeout branch never runs — even when timeouts happen. Why?

try:
    requests.get(url, timeout=2)
except requests.exceptions.RequestException:
    print("Generic error")
except Timeout:
    print("Timeout!")
  • The general RequestException catches the timeout first — specific exceptions must come before general ones
  • Timeout isn't a subclass of RequestException
  • requests.get() doesn't raise Timeout — only HTTPError
  • You can't have two except blocks in Python

Q4. Which exception fires when DNS can't resolve the hostname (e.g. you typed foo.invalid)?

  • Timeout
  • ConnectionError
  • HTTPError
  • ValueError

Cheat sheet

The minimum every external call should have:

try:
    r = requests.get(url, timeout=(3, 10))
    r.raise_for_status()
    data = r.json()
except requests.exceptions.RequestException as e:
    # log and decide: retry, fall back, or re-raise
    print(f"call failed: {e}")
