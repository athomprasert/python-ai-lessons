Gemini SDK vs Google ADK

Two ways to build with Gemini. When to pick which — and why it matters for your RAG and agent work.

Lesson 1 ✓ Lesson 2 ✓ Lesson 3 ✓ Lesson 4 ✓ Lesson 5 — SDK vs ADK Lesson 6 — TBD

The short answer

You're choosing between two layers of abstraction on top of Gemini:

  • SDK google-genai — a thin client to call the model. You manage everything else.
  • ADK google-adk — a framework to build agents. It manages tools, memory, sessions, deployment.

Rule of thumb: SDK for a single LLM call inside your own code. ADK when you're building a system where the LLM repeatedly decides what to do next.

MuleSoft analogy
The SDK is like calling an HTTP connector inside a flow you wrote yourself. ADK is like Anypoint Platform — it gives you the runtime, deployment, observability, and orchestration so you don't have to build them yourself.

What each one actually is

SDK google-genai
  • A Python client wrapping the Gemini REST API
  • You write the prompt, send it, read the response
  • Conversation memory? You build it (a list of messages)
  • Tool calls? You implement the loop yourself
  • Deployment? Whatever you want — script, Cloud Function, container
  • Switching to Claude? Switch SDKs
ADK google-adk
  • A full agent framework — open source, model-agnostic
  • You declare an agent + tools; the framework runs the loop
  • Built-in Sessions, Memory, Tools, Runners
  • Supports MCP tools, OpenAPI tools, function tools
  • Deploys via container — GCP one-click, or any Docker/Podman host (AWS, Azure, on-prem)
  • Works with Gemini, Claude, Ollama, LiteLLM and more

Source: adk.dev — ADK is an open-source agent framework that "lets you build, debug, and deploy reliable AI agents at enterprise scale."

Same task — both ways

Suppose you want a model to answer a question, and to call a tool (e.g. look up a MuleSoft API in Anypoint Exchange) when needed.

SDK The bare-metal version

from google import genai
from google.genai import types

def lookup_api(name):
    # your real implementation
    return {"name": name, "version": "3.1"}

client = genai.Client()

# 1. Describe the tool to the model
tool_decl = { "name": "lookup_api", "parameters": {...} }

# 2. First call
response = client.models.generate_content(
    model="gemini-3.5-flash",
    contents="Look up orders-api",
    config=types.GenerateContentConfig(tools=[tool_decl])
)

# 3. YOU inspect the response. Did model want the tool?
#    YOU run the tool. YOU send the result back.
#    YOU manage history if there's another turn.
#    YOU loop until the model is done.
...

Working — but you're writing the agent loop by hand.

ADK The framework version

from google.adk import Agent

def lookup_api(name: str) -> dict:
    """Look up a MuleSoft API by name."""
    return {"name": name, "version": "3.1"}

agent = Agent(
    name="mule_lookup",
    model="gemini-flash-latest",
    instruction="Help users look up MuleSoft APIs.",
    tools=[lookup_api]          # pass the function directly
)

# ADK runs the loop, manages session state, retries, etc.

Same task, less boilerplate. ADK turns the function's docstring + type hints into the tool spec automatically.

How to choose: a decision table

Your situationPickWhy
A script that summarises a document SDK One LLM call. No loop. ADK is overkill.
A simple chatbot in a notebook SDK You can manage a list-of-messages yourself. Stay light.
First time learning LLMs SDK Understand the raw call shape before adopting an opinionated framework.
A RAG pipeline you control end-to-end SDK + libs RAG = retrieve → prompt → call. The "call" is one SDK line. You'll add a vector DB (Chroma / Vertex AI Search) yourself.
Agent that uses many tools, multi-turn ADK The agent loop, tool routing, session memory are the hard parts. Don't build them.
Multiple agents collaborating ADK Sequential / parallel / loop workflow agents are first-class.
Need to deploy to GCP with traces, auth ADK One-command deploy + Cloud Trace + IAM is built in (smoothest path).
Deploying to AWS / Azure / on-prem ADK Package as a Docker container and run anywhere containers run. You write the Dockerfile and wire up your own logging/auth.
Calling tools through MCP / OpenAPI specs ADK First-class MCP tool and OpenAPI tool integrations.
Need to switch models often (Gemini, Claude, local) ADK ADK is model-agnostic via LiteLLM and direct integrations.

What about RAG specifically?

RAG (Retrieval Augmented Generation) is three steps: retrieve documents → build a prompt with them → call the model. Neither tool gives you all three out of the box — you compose them.

StepSDK approachADK approach
Retrieve Use a vector DB lib (e.g. chromadb, Vertex AI Search SDK) Wrap the retrieval as a Tool
Build prompt String formatting in Python Agent instruction + tool result automatically injected
Call model client.models.generate_content(...) Agent runner does it
Best fit Single retrieval, one shot answer Agent decides whether to retrieve, then maybe again
My recommendation for your mission
Build your first RAG with the SDK so you understand all three pieces. Then graduate to ADK when you want the agent itself to decide what to look up — or when you want to deploy it as a service.

Where ADK can actually run

A common misconception: "ADK = Google Cloud only." Not true. ADK has four deployment paths according to the official deploy docs:

TargetWhat it gives youGCP-only?
Agent Runtime Fully managed, auto-scaling service purpose-built for agents Yes
Cloud Run Container-based, serverless, scale to zero Yes
GKE Kubernetes — full control, can run open / self-hosted models Yes
Any container host Docker / Podman / ECS / AKS / on-prem K8s — anywhere containers run. Explicitly designed for "offline or disconnected" use. No
Why this matters for your work
Many Salesforce / MuleSoft customers run on AWS, Azure, or on-prem and can't (or won't) deploy to GCP. ADK's container path means you're not locked in — but you'll write your own Dockerfile and wire up logging/auth yourself.

The trade-off:

  • On GCP — one command deploy, Cloud Trace observability, IAM auth all wired up.
  • Off GCP — you get the same agent runtime, but you supply the surrounding plumbing (logging stack, secrets management, ingress, scaling).

Common pitfalls

Pitfall 1 — adopting ADK too early
Frameworks make easy things easier and hard things harder when you don't yet know which is which. Build something with the SDK first; you'll appreciate ADK for what it removes, not just what it adds.
Pitfall 2 — confusing ADK with the old "google-generativeai"
The SDK landscape recently consolidated: google-generativeai (legacy) → google-genai (current SDK). ADK (google-adk) is a separate project on top of it. Older blog posts mix these freely — verify install commands against the official sites.
Pitfall 3 — assuming ADK locks you to Gemini
It doesn't. ADK supports Claude, Ollama, LiteLLM, and others. The "Google" in the name refers to the maintainer, not the model lock-in.

Quick check

Q1. You want a one-off Python script that takes a PDF, extracts text, and asks Gemini to summarise it. Which is the right tool?

  • SDK (google-genai) — single call, no agent loop needed
  • ADK — easier to deploy
  • Both — start with ADK, fall back to SDK
  • Neither — use plain HTTP requests

Q2. You're building an assistant for your team that should be able to look up MuleSoft assets, fetch API specs, and run a few admin tools — calling each as needed. Which fits best?

  • SDK — write the tool-calling loop by hand
  • ADK — the agent loop and tool routing are exactly its job
  • Both equally good
  • Build it as a MuleSoft flow only — no Python needed

Q3. True or false: ADK only works with Gemini.

  • True
  • False — it's model-agnostic; supports Gemini, Claude, Ollama, LiteLLM and more

Q4. What's the main risk of jumping straight into ADK without first learning the SDK?

  • ADK is more expensive
  • ADK requires GCP billing
  • You learn the framework's abstractions before understanding the underlying call shape — debugging gets harder
  • ADK is closed source
