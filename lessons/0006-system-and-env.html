System Instructions, Multi-turn & .env

Give Gemini a personality, hold a real conversation, and store your API key like a professional.

Why this lesson

Four small upgrades take your Gemini calls from "toy" to "production-shaped":

  1. System instructions — tell the model who it is and how to answer
  2. Multi-turn — keep context across messages
  3. .env file — the right way to store your API key
  4. Trimming history — keep cost from spiralling as the chat grows
Connection to your mission
A RAG system is a system instruction ("Answer using only the provided context") + a list of past turns + a retrieved chunk. You're learning the scaffolding now.

1 — System instructions: giving Gemini a role

A system instruction is a separate prompt that sets the model's behaviour for the whole conversation. The user's question goes in contents; the persona/rules go in the system instruction.

from google import genai
from google.genai import types

client = genai.Client()

response = client.models.generate_content(
    model="gemini-3.5-flash",
    contents="What is OAuth?",
    config=types.GenerateContentConfig(
        system_instruction=(
            "You are a senior MuleSoft architect. "
            "Answer in 2 sentences. "
            "Always relate your answer to API security in the Anypoint Platform."
        )
    )
)

print(response.text)
Tip — what to put there
Good system instructions tell the model:
  • Who it is ("You are a senior MuleSoft architect")
  • How to answer ("In 2 sentences", "In bullet points", "JSON only")
  • What to avoid ("Don't speculate", "Don't invent API names")
MuleSoft analogy
System instructions are like a global error handler or a connector configuration in a Mule project — set once, applied to every message that flows through.

2 — Multi-turn: holding a conversation

The Gemini SDK has a built-in chats helper that tracks history for you. Source: Gemini API docs.

chat = client.chats.create(
    model="gemini-3.5-flash",
    config=types.GenerateContentConfig(
        system_instruction="You are a MuleSoft architect. Be concise."
    )
)

r1 = chat.send_message("What's a System API?")
print("A1:", r1.text)

# The chat object remembers turn 1 — no need to re-send it
r2 = chat.send_message("How does it differ from a Process API?")
print("A2:", r2.text)

# Inspect the history any time
for m in chat.get_history():
    print(m.role, "→", m.parts[0].text[:60])
Cost consciousness
Each turn re-sends the full history under the hood — so a 20-message chat costs ~20× a single call. Trim or summarise old turns when conversations grow long. (We'll cover this when we build the RAG system.)

What if you want manual control?

If you want full control (logging, custom trimming, persistence), drop chats and pass a list of messages to generate_content yourself:

history = [
    {"role": "user",  "parts": [{"text": "What's a System API?"}]},
    {"role": "model", "parts": [{"text": "A System API exposes core data..."}]},
    {"role": "user",  "parts": [{"text": "How is it different from a Process API?"}]},
]

response = client.models.generate_content(
    model="gemini-3.5-flash",
    contents=history
)
Roles you'll see
Only two: user (you) and model (Gemini). The system instruction is separate — it doesn't have a role.

3 — .env: the proper home for your API key

export GEMINI_API_KEY=... works for one terminal session. For real projects, use a .env file that's git-ignored.

Step 1: install python-dotenv

pip install python-dotenv

Step 2: create .env in your project folder

# ~/ai/claude/projects/python/.env
GEMINI_API_KEY=AIza...your-key-here

Step 3: add it to .gitignore

# ~/ai/claude/projects/python/.gitignore
.env
.venv/
__pycache__/
Critical — don't skip this
Adding .env to .gitignore is what stops you from accidentally pushing your key to GitHub. Many leaks happen this way. Treat your API key like a password.

Step 4: load it in your script

from dotenv import load_dotenv
from google import genai

load_dotenv()                    # reads .env into os.environ

client = genai.Client()        # picks up GEMINI_API_KEY automatically

response = client.models.generate_content(
    model="gemini-3.5-flash",
    contents="Say hi."
)
print(response.text)
Bonus — share a template safely
Create a .env.example with placeholder values and commit that. Teammates copy it to .env and fill in their own keys. 
# .env.example  (safe to commit)
GEMINI_API_KEY=your-key-here

4 — Trimming history to control cost

Remember the warning in section 2: every turn re-sends the full history. Concrete numbers make this real.

The problem in numbers

Suppose each user message is ~50 tokens and each model reply is ~150 tokens (200 tokens per round). After 20 rounds you're sending ~4,000 tokens of history with every new question — even if your new question is 10 tokens long. The bill grows linearly with conversation length.

# A quick way to see the growth yourself
for i, turn in enumerate(chat.get_history()):
    print(i, turn.role, len(turn.parts[0].text))
New syntax — list slicing
The trimming code uses Python's slice notation list[start:stop] to take a portion of a list. Quick reference:
  • lst[-10:] → the last 10 items (negative = count from the end)
  • lst[2:] → everything except the first 2 items
  • lst[:3] → just the first 3 items
  • lst[1:4] → items at indexes 1, 2, 3 (stop is exclusive)

Slicing always returns a new list; the original is unchanged unless you reassign it (history = history[2:]).

MuleSoft analogy: the same idea as payload[2..-1] in DataWeave — extract a sub-array without mutating the original.

Strategy 1 — Sliding window (start here)

Keep only the last N messages. Drop the oldest. Simple, predictable, works for most chatbots.

# sliding_window_chat.py
from dotenv import load_dotenv
from google import genai
from google.genai import types

load_dotenv()
client = genai.Client()

PERSONA     = "You are a concise MuleSoft architect."
MAX_MSGS    = 10          # keep last 10 messages = 5 user/model rounds

history = []           # list of {"role", "parts"} dicts (we manage it)

def say(user_text):
    # 1. Append the user's message
    history.append({"role": "user", "parts": [{"text": user_text}]})

    # 2. Take only the most recent MAX_MSGS messages
    trimmed = history[-MAX_MSGS:]

    # 3. Call the model — system_instruction is separate, never trimmed
    response = client.models.generate_content(
        model="gemini-3.5-flash",
        contents=trimmed,
        config=types.GenerateContentConfig(system_instruction=PERSONA)
    )

    # 4. Append the model's reply so future turns can see it
    history.append({"role": "model", "parts": [{"text": response.text}]})
    return response.text, response.usage_metadata.total_token_count

for q in ["What's a System API?", "Give me 3 examples.", "Pick one and elaborate."]:
    answer, tokens = say(q)
    print(f"\nQ: {q}\nA: {answer}\n(tokens: {tokens})")
Why we manage history ourselves
The built-in chats helper grows forever. The moment you care about cost or context limits, switch to the manual list-of-messages pattern (section 2) — that's the only place you can trim.

Strategy 2 — Token budget (more precise)

A sliding window of "last N messages" is rough — one long message can blow the budget. A smarter approach drops the oldest message until the total token count fits a budget. Use client.models.count_tokens() to measure.

TOKEN_BUDGET = 2000

def trim_to_budget(history):
    while True:
        count = client.models.count_tokens(
            model="gemini-3.5-flash",
            contents=history
        ).total_tokens
        if count <= TOKEN_BUDGET or len(history) <= 2:
            return history
        # Drop the oldest pair (user + model) to keep turns balanced
        history = history[2:]
Drop in pairs
Always remove a user+model pair together. If you leave an orphan model reply at the start of history, Gemini gets confused — the conversation must start with a user turn.

Strategy 3 — Summarisation (advanced, mention only)

Once a chat gets long, ask Gemini itself to compress old turns into a 1-paragraph summary, then prepend that summary as the first user message. You preserve context at a fraction of the token cost. We'll use this technique inside the RAG system later.

Quick decision guide

Use case Strategy
Short chats, predictable messagesSliding window
Variable-length messages, strict cost capToken budget
Long-running assistants, important early contextSummarisation

Put it all together

One file that uses all three patterns. Save as chat_with_persona.py:

# chat_with_persona.py
from dotenv import load_dotenv
from google import genai
from google.genai import types

load_dotenv()
client = genai.Client()

PERSONA = (
    "You are a senior MuleSoft architect helping a colleague. "
    "Be concise (max 3 sentences). "
    "If the question is ambiguous, ask one clarifying question instead of guessing."
)

chat = client.chats.create(
    model="gemini-3.5-flash",
    config=types.GenerateContentConfig(system_instruction=PERSONA)
)

print("Chat ready. Type your question (or 'quit'):")
while True:
    q = input("\nYou: ").strip()
    if q.lower() in {"quit", "exit"}:
        break
    if not q:
        continue
    resp = chat.send_message(q)
    print(f"\nGemini: {resp.text}")
Try it yourself
  1. Set up your .env file and .gitignore as shown above.
  2. Run pip install python-dotenv (in your venv if you have one).
  3. Save the script and run python3 chat_with_persona.py.
  4. Ask a follow-up question that depends on your previous one (e.g. "What's a System API?" then "Give me 3 examples"). Confirm the model remembers the topic.
  5. Modify: Change PERSONA to make Gemini behave like a presales engineer — short, benefit-focused answers. Notice the tone shift.
  6. Stretch: After each answer, print resp.usage_metadata.total_token_count. Watch the number grow as the chat history grows — that's why you'll need to trim later.
