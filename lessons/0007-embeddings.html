Embeddings: The Foundation of RAG

Turn text into numbers, find what's "similar in meaning", and feel why RAG works.

L1 ✓ L2 ✓ L3 ✓ L4 ✓ L5 ✓ L6 ✓ L7 — Embeddings L8 — Chunking

The big idea in one paragraph

An embedding turns a piece of text into a list of numbers — a vector — that represents its meaning. Texts about similar topics produce similar vectors, even when they share no words. Computers can compare numbers fast, so once your documents are vectors you can ask "which document is closest in meaning to this question?" and get an answer in milliseconds. That's the retrieval in Retrieval-Augmented Generation.

Connection to your mission
By the end of this lesson you'll have done the core operation of RAG by hand: embed a corpus, embed a question, find the closest match. Lessons 8–9 just dress it up with chunking and a vector DB.

1 — Intuition: meaning as coordinates

Imagine plotting words on a 2D map where similar words end up close to each other:

Made-up visualisation — real embeddings have hundreds of dimensions, but the principle is identical.

Real embeddings don't use 2 dimensions — they use 768 to 3072. Each dimension captures some abstract "feature" the model has learned (formality, topic, tense, sentiment, syntax). You never inspect them individually; you just compare whole vectors with each other.

MuleSoft analogy
Think of an embedding as a richer hash. A normal hash (MD5, SHA) maps text → a unique fingerprint where any change scrambles the output. An embedding maps text → a fingerprint where similar meaning produces similar fingerprints. That's what makes it useful for search.

2 — Calling Gemini's embedding API

Same SDK as before. Different method: embed_content.

from dotenv import load_dotenv
from google import genai

load_dotenv()
client = genai.Client()

result = client.models.embed_content(
    model="gemini-embedding-001",
    contents="What is OAuth 2.0?"
)

vector = result.embeddings[0].values
print(len(vector))      # 3072 by default
print(vector[:5])      # first 5 dimensions

Output (yours will differ slightly):

3072 [0.0123, -0.0871, 0.0245, -0.0339, 0.0712]

That's it — text in, list of floats out. Source: official embeddings docs.

Tip — control vector size
Default is 3072 dimensions. Pass config=types.EmbedContentConfig(output_dimensionality=768) for smaller, faster vectors. 768 is the sweet spot for most RAG systems — quality is nearly identical to 3072 but storage is 4× smaller.

Aside — gemini-embedding-001 vs gemini-embedding-2

Google ships two embedding models in 2026. They produce incompatible vector spaces — a vector from one is meaningless when compared to a vector from the other. Picking the right one up front saves a full re-indexing job later.

Why this lesson uses 001
  1. The clean "one string in → one vector out" behaviour makes the core concepts easier to feel.
  2. Your immediate target is text-based MuleSoft Q&A.
  3. You'll understand task_type better by using it explicitly first; the prompt-prefix style of -2 will then make sense.

In a later lesson we'll port the same RAG to -2 so you feel both APIs back to back.

MuleSoft analogy
Choosing between 001 and -2 is like choosing between an HTTP connector and a generic API connector with image/audio support. Same shape, but the broader connector can carry richer payloads — and you can't mix flows that use different message-type contracts.

Aside 2 — what is "the batching gotcha"?

I've used the phrase "batching gotcha" a few times. Let me unpack it properly because it's the single most common gemini-embedding-2 bug.

What "batching" means

Batching = sending several pieces of work to an API in a single call instead of one call per item. Cheaper and faster — you avoid the overhead of N HTTP round-trips.

You'd expect batching to work like this:

texts = ["doc 1", "doc 2", "doc 3"]
vectors = embed(texts)
# Expected: 3 vectors, one per text

That's exactly how gemini-embedding-001 works.

The gotcha — gemini-embedding-2 does it differently

If you pass a list of strings to gemini-embedding-2, you get back one vector that represents the aggregated meaning of all the inputs — not one vector per input.

# gemini-embedding-001 — what you'd expect
result = client.models.embed_content(
    model="gemini-embedding-001",
    contents=["OAuth", "DataWeave", "CloudHub"]
)
# → 3 vectors

# gemini-embedding-2 — the gotcha
result = client.models.embed_content(
    model="gemini-embedding-2",
    contents=["OAuth", "DataWeave", "CloudHub"]
)
# → 1 vector — an aggregated blob of all 3 meanings
Why it's a "gotcha"
The API doesn't error. You think you've embedded 3 documents — you've actually embedded one average. Your RAG retrieval silently returns garbage. That's the footgun: wrong, not crashed.

But wait — when would you want aggregation?

It exists by design. There are real use cases where "merge these inputs into one vector" is what you want.

1. Multimodal items (the big one)

This is probably why the default exists. -2 is multimodal — a single "item" in your corpus might be text + image + audio together.

# Embed a product as ONE vector — image + description combined
result = client.models.embed_content(
    model="gemini-embedding-2",
    contents=[
        product_image,                              # the photo
        "Wireless headphones, 30hr battery, ANC"     # the description
    ]
)
# → 1 product-vector capturing both modalities together
MuleSoft-relevant analog
An Anypoint Exchange asset = name + description + RAML spec + screenshots. With -2 you pass all of these in one call and get a single asset-vector capturing the whole thing.

2. Multi-field records as one entity

# Embed a resume as ONE vector — find a person, not a paragraph
contents = [
    "Name: Attapol Thomprasert",
    "Role: MuleSoft Architect",
    "Summary: 10+ years of integration architecture...",
    "Skills: Anypoint, DataWeave, OAuth 2.0..."
]

3. Query expansion / robust queries

One user intent often has many phrasings. Aggregating them gives a sturdier query vector — less sensitive to wording.

contents = [
    "How do I secure my API?",
    "What's the best authentication for APIs?",
    "API security best practices"
]
# → 1 robust query vector less brittle to user phrasing

4. Session-level memory

# Embed a 10-turn conversation as ONE vector — for routing or cache lookup
contents = [turn.text for turn in chat_history]

The decision rule

Are the inputs… Use
Parts of ONE thing (multimodal item, multi-field record, query variants)Aggregation — the default -2 behaviour
Separate things (classic RAG corpus, distinct docs)Per-input vectors — use 001, or wrap each in Content

The three ways to force per-input vectors with -2

# Fix 1 — call the API once per string (slowest, simplest)
vectors = [embed_one(text) for text in texts]

# Fix 2 — wrap each string in its own Content object
from google.genai import types
result = client.models.embed_content(
    model="gemini-embedding-2",
    contents=[types.Content(parts=[types.Part(text=t)]) for t in texts]
)
# → N vectors, as expected

# Fix 3 — use the Batch API (best for large corpora; 50% cheaper)
#       Submit a job, results come back asynchronously
Reframe
The "gotcha" isn't a bug — it's that the default helps the multimodal-item use case but surprises the classic-RAG use case. Once you know which side you're on, the choice is obvious.

3 — Comparing two vectors with cosine similarity

"Closeness in meaning" = "closeness of vectors". The standard measure is cosine similarity — the cosine of the angle between two vectors. The math sounds scary; the code is one line.

What it returns

  • 1.0 → vectors point in the same direction (very similar)
  • 0.0 → vectors are perpendicular (unrelated)
  • -1.0 → vectors point in opposite directions (rare with embeddings; you mostly see 0–1)

The formula

cos(A, B) = dot(A, B) / (||A|| × ||B||)

In English: divide the dot product by the product of the lengths. Don't memorise — just remember it's a one-liner.

New syntax — zip() and tuple unpacking
The cosine code below uses two new things. Quick tour first:

zip(a, b) — pair items from two lists

zip walks two (or more) lists in lockstep, yielding pairs:

for pair in zip([1, 2, 3], ["a", "b", "c"]):
    print(pair)
# (1, 'a')
# (2, 'b')
# (3, 'c')

Imagine a zipper joining two rows of items — that's the mental model.

Tuple unpacking — for x, y in ...

Instead of getting one tuple pair, you can split it into two named variables on the spot:

for x, y in zip([1, 2, 3], ["a", "b", "c"]):
    print(x, y)
# 1 a
# 2 b
# 3 c

Generator expression inside sum(...)

sum(x * y for x, y in zip(a, b)) looks like a list comprehension without the brackets. That's a generator expression — it produces values one at a time instead of building a full list in memory. sum() walks them and adds them up. Use it whenever you'd build a list just to sum or min or max it — it's the same syntax, more efficient.

MuleSoft analogy: zip is like joining two arrays element-by-element in DataWeave (a zip b exists too); the generator expression is like a streaming transform that doesn't materialise the whole result.

import math

def cosine_similarity(a, b):
    """Return cosine similarity between two equal-length vectors."""
    dot     = sum(x * y for x, y in zip(a, b))
    norm_a  = math.sqrt(sum(x * x for x in a))
    norm_b  = math.sqrt(sum(y * y for y in b))
    return dot / (norm_a * norm_b)
In real projects
Use numpy: np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)). Same math, much faster on large arrays. We're using pure Python for clarity.

4 — Hands-on: RAG retrieval, by hand

This is the lesson's main exercise. Save it as tiny_rag.py in your activated venv.

# tiny_rag.py — the retrieval half of RAG, no vector DB
import math
from dotenv import load_dotenv
from google import genai
from google.genai import types

load_dotenv()
client = genai.Client()
MODEL = "gemini-embedding-001"

# ─── 1. Our tiny "knowledge base" ───────────────────────────
DOCS = [
    "OAuth 2.0 is the industry-standard protocol for API authorization "
    "on the Anypoint Platform; it issues short-lived access tokens.",

    "DataWeave is MuleSoft's expression language for transforming data "
    "between formats like JSON, XML and CSV inside a Mule flow.",

    "A System API exposes raw data from a system of record. "
    "A Process API orchestrates one or more System APIs to model a business process.",

    "CloudHub 2.0 is MuleSoft's managed runtime that deploys Mule applications "
    "as containers, with auto-scaling and replicas.",

    "To make chocolate cake, mix flour, sugar, cocoa, eggs and butter, "
    "then bake at 180°C for about 30 minutes."
]

# ─── 2. Helpers ─────────────────────────────────────────────
def embed(text, task_type):
    """Embed a single string. task_type tells the model how this text will be used."""
    result = client.models.embed_content(
        model=MODEL,
        contents=text,
        config=types.EmbedContentConfig(task_type=task_type)
    )
    return result.embeddings[0].values

def cosine(a, b):
    dot = sum(x * y for x, y in zip(a, b))
    return dot / (math.sqrt(sum(x*x for x in a)) * math.sqrt(sum(y*y for y in b)))

# ─── 3. Index the corpus (do this once) ─────────────────────
print("Embedding corpus...")
doc_vectors = [embed(doc, "RETRIEVAL_DOCUMENT") for doc in DOCS]

# ─── 4. Search ───────────────────────────────────────────────
def search(question, top_k=2):
    q_vec = embed(question, "RETRIEVAL_QUERY")
    scored = [(cosine(q_vec, d), doc) for d, doc in zip(doc_vectors, DOCS)]
    scored.sort(reverse=True)
    return scored[:top_k]

# ─── 5. Try it ──────────────────────────────────────────────
for q in [
    "How do I secure my API on Anypoint?",
    "What's the difference between System and Process APIs?",
    "How do I deploy a Mule app?"
]:
    print(f"\nQ: {q}")
    for score, doc in search(q):
        print(f"  [{score:.3f}] {doc[:80]}...")

Expected output (scores will vary slightly):

Q: How do I secure my API on Anypoint?
  [0.78] OAuth 2.0 is the industry-standard protocol for API authorization...
  [0.51] CloudHub 2.0 is MuleSoft's managed runtime...

Q: What's the difference between System and Process APIs?
  [0.81] A System API exposes raw data from a system of record...
  [0.49] OAuth 2.0 is the industry-standard protocol...

Q: How do I deploy a Mule app?
  [0.74] CloudHub 2.0 is MuleSoft's managed runtime...
  [0.46] A System API exposes raw data...

Notice three things:

  1. The cake recipe never wins — it's not relevant to any MuleSoft question.
  2. "How do I secure my API?" matches OAuth, even though the words don't overlap. That's the magic.
  3. The right doc has a noticeably higher score than the runner-up — that gap is your confidence signal.

5 — A subtle but important parameter: task_type

You may have noticed two different task_type values in the script: RETRIEVAL_DOCUMENT for the corpus and RETRIEVAL_QUERY for the question. Why two?

The embedding model produces slightly different vectors for the same text depending on what you'll use it for. A long document and a short question are written in different styles, so the model "tilts" the embedding to make them match better when the right pairing is used.

Use case Corpus side Query side
Search / RAGRETRIEVAL_DOCUMENTRETRIEVAL_QUERY
Q&A over docsRETRIEVAL_DOCUMENTQUESTION_ANSWERING
Find duplicates / similaritySEMANTIC_SIMILARITYSEMANTIC_SIMILARITY
Group similar docsCLUSTERINGN/A
Heads-up
task_type is a parameter on gemini-embedding-001. On the newer gemini-embedding-2, you bake the task type into the prompt text (e.g. "task: search result | query: ..."). Different shape, same idea.

Worked example — adding a relevance threshold

Stretch challenge #5 above asks you to filter out weak matches. Here's the answer in full, with the reasoning. This is a real production pattern, not just an exercise.

The problem

Right now search() always returns the top K docs — even if every one of them is a poor match. If a user asks "what's the weather in Tokyo?", your MuleSoft corpus has nothing relevant, but the function will still hand back its least bad docs. Those will then be fed to the LLM and produce a confident-sounding hallucination.

Why this matters
Refusing to retrieve is a feature, not a failure. A RAG system that says "I don't have information about that" is more trustworthy than one that always answers. The threshold check is what turns retrieval from "best of a bad bunch" into "good answer or none".

The plan

  1. Score every doc as before.
  2. Filter out anything with cosine ≤ THRESHOLD.
  3. Return the top K of what's left (which might be nothing).
  4. In the caller, check for an empty result and print the friendly message.
New syntax — list comprehension with if
Lesson 1 introduced list comprehensions [x for x in items]. Adding an if at the end turns it into a filter: 
# Lesson 1 reminder — transform every item
[x * 2 for x in [1, 2, 3]]           # [2, 4, 6]

# New — keep only items that pass a test
[x for x in [1, 2, 3, 4] if x > 2]     # [3, 4]

# Combine — transform AND filter
[x * 2 for x in [1, 2, 3, 4] if x > 2] # [6, 8]

Order is fixed: [expression for x in items if condition]. The if always comes last.

The code

Replace the search() function and the print loop in tiny_rag.py with these:

THRESHOLD = 0.5     # cosine below this = "not relevant"

def search(question, top_k=2):
    q_vec = embed(question, "RETRIEVAL_QUERY")
    scored = [(cosine(q_vec, d), doc) for d, doc in zip(doc_vectors, DOCS)]
    scored.sort(reverse=True)

    # Filter — keep only pairs whose score beats the threshold
    relevant = [(score, doc) for score, doc in scored if score > THRESHOLD]

    return relevant[:top_k]     # may be empty if nothing passed the filter

# Updated print loop with a "no answer" branch
for q in [
    "How do I secure my API on Anypoint?",
    "What's the difference between System and Process APIs?",
    "How do I deploy a Mule app?",
    "What's the weather in Tokyo?"     # nothing in the corpus answers this
]:
    print(f"\nQ: {q}")
    results = search(q)
    if not results:
        print("  no relevant document found.")
    else:
        for score, doc in results:
            print(f"  [{score:.3f}] {doc[:80]}...")

Expected output:

Q: How do I secure my API on Anypoint?
  [0.78] OAuth 2.0 is the industry-standard protocol...

Q: What's the difference between System and Process APIs?
  [0.81] A System API exposes raw data from a system of record...

Q: How do I deploy a Mule app?
  [0.74] CloudHub 2.0 is MuleSoft's managed runtime...

Q: What's the weather in Tokyo?
  no relevant document found.

How to pick a good threshold

Threshold Behaviour Best for
> 0.7Very strict — misses borderline matchesCompliance Q&A, "answer or admit you don't know"
> 0.5Balanced — picks up reasonable matches, filters obvious junkGeneral-purpose RAG (good default)
> 0.3Loose — most things passExploratory tools, "give me anything related"

You'll usually tune the threshold by running real questions through your real corpus and watching the scores. Threshold values are model-specific too — vectors from gemini-embedding-2 may cluster at different magnitudes, so what's "0.5" on 001 isn't necessarily "0.5" on -2.

Why > not >=?
A cosine of exactly 0.5 in a 768+ dimension space is essentially noise — basically a coincidence. Strict > makes the threshold a real bar. Use >= only if you genuinely need to keep ties.
