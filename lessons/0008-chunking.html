Chunking Strategies for RAG

Eight ways to slice a document — and how to pick the right one for your data.

L1 ✓ L2 ✓ L3 ✓ L4 ✓ L5 ✓ L6 ✓ L7 ✓ L8 — Chunking L9 — Build RAG

Why chunking is the make-or-break step

You can't just embed a 50-page MuleSoft white paper as one giant blob. Embedding models have token limits (~2,048 for gemini-embedding-001, 8,192 for gemini-embedding-2). Even if you could, retrieval would always return the entire document — defeating the point of finding the relevant section.

So you split the document into chunks, embed each one, and retrieve only the chunks that match a question. Chunking is where most RAG systems quietly fail. Bad chunks → bad retrieval → bad answers, no matter how good your model is.

Connection to your mission
The doc list in Lesson 7 was already chunked — you just didn't notice because each "doc" was 1–2 sentences. Real corpora come as full PDFs, RAML specs, or Confluence pages. This lesson is how you turn those into the small text pieces you've already learned to embed.

Three principles before any strategy

1 — Chunks must be self-contained

A chunk is going to be retrieved alone — without its surroundings — and shown to the LLM. If a chunk says "Revenue grew 3% that quarter", the LLM has no idea which company or which quarter. Useless context.

Rule of thumb (Pinecone): If a chunk makes sense to a human reading it on its own, it'll make sense to the model too.

2 — Size: small enough to be precise, big enough to be meaningful

Common ranges: 128–1024 tokens. Smaller = more precise retrieval but loses context. Bigger = retains context but retrieval gets fuzzy and the LLM hits the "lost in the middle" problem (relevant info buried in long inputs gets ignored).

3 — Overlap prevents seam-loss

If you split a document at character 500, a sentence might be cut in half — and the meaning lives across the split. Overlap means each chunk shares its last ~10–20% with the next chunk's start. A common default is chunk_size=500, overlap=50.

Original text: AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA

Without overlap:
[chunk 1: AAAAAAAAAAAAAAAAAAAAAAAA][chunk 2: AAAAAAAAAAAAAAAAAAAAAAAA][chunk 3: AAAAAAAAAAAAAAAAAAAAAAAA]

With 4-char overlap:
[chunk 1: AAAAAAAAAAAAAAAAAAAAAAAA]
←AAAAchunk 2: AAAAAAAAAAAAAAAAAAAA]
←AAAAchunk 3: AAAAAAAAAAAAAAAAAAAA]

The eight strategies, ordered simplest → smartest

Each one builds on the previous. Sourced from Pinecone's chunking guide + Anthropic's Contextual Retrieval.

1 Fixed-size chunking

How: Pick a chunk size (e.g. 500 chars or 200 tokens) and overlap (e.g. 50 chars). Slice from start to end at fixed intervals. No regard for content.

def chunk_fixed(text, size=500, overlap=50):
    chunks = []
    step = size - overlap
    for i in range(0, len(text), step):
        chunks.append(text[i:i + size])
    return chunks
Verdict: The default. Start here and only change if you have evidence it's not enough. Trade-off: cuts mid-sentence often.
New syntax — .strip() and the empty-string filter

You'll see .strip() all over the chunking code. It returns a new string with leading and trailing whitespace (spaces, tabs, \n, \r) removed. The original string is unchanged.

"   Hello world   \n".strip()
# → "Hello world"

"  hello  world  ".strip()
# → "hello  world"   (the middle double-space stays — only the ends are touched)

The pattern [p.strip() for p in parts if p.strip()] does two things at once:

  • Clean each piece (p.strip() on the left)
  • Filter out empty pieces (if p.strip() on the right) — splits often produce empty strings between consecutive separators, and we don't want to embed those.

In Python, an empty string "" is falsy — it evaluates to False in a boolean context. So if p.strip() is shorthand for "if the stripped string isn't empty".

The strip family

.strip()Whitespace from both ends
.lstrip()Left only
.rstrip()Right only
.strip(",.")Any of the listed characters from both ends, e.g. "...hi.." → "hi"

MuleSoft analogy: exactly like DataWeave's trim / Java's String.trim(). The empty-string filter is the equivalent of a filter step that drops empty payloads after a split.

2 Sentence-level chunking

How: Split on sentence boundaries (periods, question marks, exclamation marks). Tools: regex for naïve splits, NLTK or spaCy for accurate ones (handles "Mr. Smith" correctly).

import re

def chunk_sentences(text):
    # Naïve split — fine for clean text, bad for abbreviations
    return [s.strip() for s in re.split(r"(?<=[.!?])\s+", text) if s.strip()]
Verdict: Good for FAQs, classification, short-answer search. Loses broader context for multi-sentence concepts.

3 Paragraph-level chunking

How: Split on double newlines (\n\n) — the natural paragraph delimiter in most prose. Each paragraph becomes a chunk.

def chunk_paragraphs(text):
    return [p.strip() for p in text.split("\n\n") if p.strip()]
Verdict: Good for prose where paragraphs are meaningful units (blog posts, articles). Risk: paragraphs vary wildly — some 30 words, some 800.

4 Recursive character chunking (the practical default)

How: Try to split on big separators first (\n\n), then smaller ones (\n), then spaces, then characters — until each chunk is under your size limit. LangChain's RecursiveCharacterTextSplitter popularised this pattern.

This is the "great middle ground" — fixed-size predictability with respect for natural boundaries.

def chunk_recursive(text, size=500, separators=["\n\n", "\n", ". ", " ", ""]):
    # Try the biggest separator first; if pieces still too long, recurse with the next.
    if len(text) <= size:
        return [text] if text.strip() else []
    sep, *rest = separators
    parts = text.split(sep) if sep else [text[:size], text[size:]]
    out = []
    for p in parts:
        if len(p) <= size:
            out.append(p)
        else:
            out.extend(chunk_recursive(p, size, rest))
    return [x.strip() for x in out if x.strip()]
Verdict: Use this 80% of the time. It's what LangChain defaults to and what most production RAG systems use.
New syntax — re module & splat unpacking

import re — regular expressions

Python's built-in regex library. re.split(pattern, text) splits on a pattern instead of a fixed string. The r"..." prefix makes it a raw string so backslashes don't need doubling.

r"(?<=[.!?])\s+" means "split on whitespace that comes after a ., ! or ?" — the (?<=...) is a lookbehind, used to keep the punctuation attached to the previous sentence.

Splat unpacking — sep, *rest = separators

*rest grabs "everything else" from a list. So sep, *rest = ["\n\n", "\n", ". ", " ", ""] sets sep = "\n\n" and rest = ["\n", ". ", " ", ""]. Useful for "head + tail" splits without slicing.

MuleSoft analogy: regex with lookbehind is like a DataWeave matches expression with a positive look-around.

5 Document-structure-aware chunking

How: Use the document's own structure as the splitter — Markdown headings (#, ##), HTML tags (<h1>, <p>), PDF sections, or LaTeX environments. Each section becomes a chunk, often with the heading prepended for context.

Example: a RAML spec where each /resource: block becomes one chunk; or a Markdown doc where each ## section becomes one chunk.

# LangChain ships specialist splitters for many formats:
from langchain.text_splitter import MarkdownHeaderTextSplitter

splitter = MarkdownHeaderTextSplitter(
    headers_to_split_on=[("#", "H1"), ("##", "H2"), ("###", "H3")]
)
docs = splitter.split_text(markdown_text)
# Each result includes both the section text and the heading hierarchy as metadata
Verdict: Best for structured content (docs, specs, papers). The headings often answer the "what is this chunk about?" question for free.
Direct MuleSoft application
A RAML spec is highly structured. Splitting it by /resource + method blocks gives you chunks that each describe one operation — perfect for "find me the API that does X" queries.

6 Semantic chunking

How: Don't split on syntax — split where the meaning changes. Embed every sentence, group it with its neighbours, and start a new chunk whenever the cosine distance to the previous group spikes (i.e., topic shifted).

Greg Kamradt popularised this. It's compute-intensive — you embed the document twice (once for chunking, again per chunk) — but produces chunks that align with actual themes.

Verdict: Use when fixed/recursive splitters mangle your content (e.g. transcripts, Q&A logs, mixed-topic docs). Skip if recursive is good enough.

7 Late chunking (emerging technique)

How: Embed the whole document first, then take chunk-aligned slices of that whole-document embedding. Reverses the order: instead of "chunk → embed", you "embed → chunk". The chunk vectors carry the document's full context.

Requires an embedding model that exposes token-level outputs. Newer text-embedding models (Jina v3 family etc.) support this; Gemini doesn't expose it as of 2026.

Verdict: Promising for chunks that depend heavily on full-document context. Limited support — file under "watch this space".

8 Contextual chunking (Anthropic, 2024)

How: Before embedding each chunk, ask an LLM to write a 50–100 token contextual blurb situating the chunk inside its parent document. Prepend the blurb to the chunk text, then embed.

Example transformation, from Anthropic's announcement:

# Original chunk (loses context when retrieved alone)
"The company's revenue grew by 3% over the previous quarter."

# Contextual chunk (the LLM has prepended a situating blurb)
"This chunk is from an SEC filing on ACME Corp's Q2 2023 performance; "
"the previous quarter's revenue was $314M. The company's revenue "
"grew by 3% over the previous quarter."

Measured impact:

  • Contextual embeddings alone: 35% reduction in retrieval failures
  • Plus contextual BM25 (lexical search): 49% reduction
  • Plus reranking: 67% reduction

Cost trick: Use prompt caching — cache the full document once, then run cheap "describe this chunk" calls against it. Anthropic estimates ~$1.02 per million document tokens processed this way.

Verdict: Best-in-class as of late 2025. Use when you have long documents and high-stakes retrieval (compliance, legal, technical specs). Cost is non-zero but reasonable with caching.

How to choose

Your data Start with Upgrade to (if recall isn't good enough)
Plain prose, single topic Fixed-size or paragraph (#1, #3) Recursive (#4)
Markdown docs, RAML specs, API references Structure-aware (#5) Add contextual chunking (#8)
FAQ / Q&A pairs Sentence-level (#2) — each Q&A is one chunk Pair-level metadata + recursive (#4)
Long PDFs, mixed topics Recursive (#4) Semantic (#6) or contextual (#8)
Compliance / legal / high-stakes Structure-aware + reranking Contextual chunking (#8)
Conversation logs / transcripts Semantic (#6) — topics drift Contextual (#8)
Pragmatic order of operations
  1. Start with recursive (#4), chunk_size=500, overlap=50. Measure retrieval quality on real questions.
  2. If your data has structure, switch to structure-aware (#5).
  3. If retrieval still misses things, layer in contextual chunking (#8).
  4. Only consider semantic (#6) or late chunking (#7) if you have specific evidence the above aren't working.

Hands-on — chunk a real MuleSoft-style doc

Three strategies on the same input so you can feel the difference. Save as chunk_demo.py.

# chunk_demo.py
import re

DOC = """
# Anypoint Platform Authentication

## OAuth 2.0
OAuth 2.0 is the industry-standard authorization protocol. It issues short-lived access tokens that clients send with each request. Anypoint's Identity Provider can issue and rotate these tokens, and API Manager validates them at the edge.

## Mutual TLS
For machine-to-machine integrations, mTLS provides bilateral authentication via X.509 certificates. Both client and server present certificates; both are validated against a trusted CA chain.

## Rate Limiting
Independent of authentication, you should always apply rate limits. Anypoint's Rate Limiting policy supports per-client and per-method limits with configurable response codes.
"""

# Strategy 1 — Fixed-size
def chunk_fixed(text, size=200, overlap=40):
    step = size - overlap
    return [text[i:i+size] for i in range(0, len(text), step)]

# Strategy 3 — Paragraph
def chunk_paragraphs(text):
    return [p.strip() for p in text.split("\n\n") if p.strip()]

# Strategy 5 — Markdown structure-aware (heading + body)
def chunk_markdown(text):
    # split before each "## " heading; keep the heading attached
    parts = re.split(r"(?=^## )", text, flags=re.MULTILINE)
    return [p.strip() for p in parts if p.strip()]

for name, fn in [("FIXED", chunk_fixed), ("PARAGRAPH", chunk_paragraphs), ("MARKDOWN", chunk_markdown)]:
    chunks = fn(DOC)
    print(f"\n=== {name} → {len(chunks)} chunks ===")
    for i, c in enumerate(chunks):
        print(f"--- chunk {i} ({len(c)} chars) ---")
        print(c[:100], "..." if len(c) > 100 else "")

Multilingual considerations

All eight strategies above were demonstrated on English text. The good news: most of them work for any language — but each chunker has language-specific gotchas worth knowing.

Embeddings — multilingual by default

Modern embedding models (gemini-embedding-001, gemini-embedding-2, bge-m3) support 100+ languages. A Thai sentence and its English translation land near each other in vector space — which means a user can ask in Thai and retrieve from English docs. No code changes needed for this; it's a property of the model.

Quality is uneven though:

  • Tier 1 — English, Spanish, French, German, Chinese, Japanese, Korean → excellent
  • Tier 2 — Thai, Vietnamese, Indonesian, Arabic, Hindi, Portuguese → good
  • Tier 3 — Swahili, Khmer, Burmese, etc. → workable but noisier; test before trusting

Chunking — language-specific gotchas per strategy

StrategyMultilingual safe?What to watch for
1. Fixed-size character ✓ Mostly Python strings are Unicode-aware, so it runs. But 500 chars of English ≈ 80 words; 500 chars of Thai ≈ 250 words. Use token-based size for mixed-language corpora.
2. Sentence-level ✗ Needs adapting split(". ") is English-only. Japanese/Chinese use 。; Hindi uses ।; Thai often has no sentence delimiter at all.
3. Paragraph-level ✓ Yes \n\n is whitespace — language-agnostic.
4. Recursive character ✗ Adjust ladder Swap the sentence/word separators per language. See snippet below.
5. Document-structure (Markdown ##) ✓ Yes Markup is language-neutral. Safest chunker for multilingual content.
6. Semantic ✓ Yes Relies on embeddings, which are multilingual. Boundaries follow meaning, not punctuation.
7. Late chunking ✓ Yes Works in any language the embedding model supports.
8. Contextual retrieval ✓ Yes Keep the prompt in English even for non-English docs — the LLM will write the blurb in the doc's language.

The big trap — sentence and word splitters

Strategy #2 (sentence-level) and any "word-level" splitter need real attention. The naive Lesson 8 versions (text.split(". "), text.split(" ")) silently misbehave on these languages:

LanguageSentence terminatorWord boundary?
English / French / Spanish. Spaces ✓
Japanese。 (no space)No spaces — needs segmenter
Chinese。 (no space)No spaces — needs segmenter
Thai(often unmarked)No spaces — needs segmenter
Hindi। (devanagari danda)Spaces ✓
Arabic. (right-to-left)Spaces ✓

# Japanese — fugashi (wraps MeCab)

Multilingual sentence splitting:

# Option A — pysbd: rule-based, supports ~20 languages
from pysbd import Segmenter
seg = Segmenter(language="ja", clean=False)
sentences = seg.segment(japanese_text)

# Option B — spaCy: model-based, more accurate
import spacy
nlp = spacy.load("ja_core_news_sm")
sentences = [sent.text for sent in nlp(japanese_text).sents]

Recursive chunking — adjust the separator ladder

The Lesson 8 ladder (["\n\n", "\n", ". ", " ", ""]) is English. Swap the sentence and word separators for the target language: 

# Japanese / Chinese — note: NO " " in the list (no word spaces)
separators = ["\n\n", "\n", "。", "、", ""]

# Thai — paragraph and line only; no reliable sentence delimiter
separators = ["\n\n", "\n", " ", ""]

# Hindi
separators = ["\n\n", "\n", "। ", " ", ""]

Or skip the manual work — LangChain ships a helper: 

from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter.from_language(
    language="ja", chunk_size=500
)
chunks = splitter.split_text(japanese_text)

Mixed-language documents (e.g. RAML with Japanese comments)

Real-world enterprise docs often mix languages — an English MuleSoft RAML with Japanese annotations, AWS docs with localised examples. Don't split them apart. Modern multilingual embedding models handle code-switching well; just embed the chunk as-is and let the model figure it out.

The only reason to split per language is if you specifically need to filter by language at query time (e.g. "search Japanese chunks only"). In that case, store language as metadata on the chunk:

from langdetect import detect

collection.add(
    ids=ids,
    documents=chunks,
    embeddings=vectors,
    metadatas=[{"lang": detect(c)} for c in chunks],
)
Practical recommendation for your rag_chroma.py
Your existing pipeline (markdown ## chunker → multilingual embedding → Chroma) works on Japanese / Thai / Chinese / Spanish docs with zero code changes. MuleSoft analogue
Think of language as a content-type header on a Mule message. The transport (Chroma, Gemini, retrieval logic) doesn't care. But the processors — chunkers — need language-specific rules, like a transformer that knows whether to use application/json or application/xml. Same shape, different processor.
