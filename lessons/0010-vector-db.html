From JSON to a Real Vector Database

Survey the vector DB market, then upgrade your RAG to use Chroma — the gentle starting point for production.

Why this lesson

In Lesson 9 you stored every chunk's embedding in a single index.json file. That works — for ten chunks. The moment your corpus grows, the cracks show:

  • Linear scan. Every query compares against every chunk. 10 chunks is fine. 10,000 is slow. 1,000,000 is impossible.
  • Full reload. One byte changes → you load the entire JSON into memory.
  • No metadata filter. Can't say "search only chunks tagged auth".
  • No concurrency. Two processes writing at once = corrupted file.

Vector databases solve all four. They store vectors, run nearest-neighbour search with proper indexes (HNSW, IVF, etc.), let you attach metadata, and handle reads/writes safely. This is the single biggest upgrade you can make to a RAG system.

Part 1 — The vector DB market in 2026

There are dozens. You only need to know seven. Here's what each is good for:

ToolTypeSweet spot
Chroma Open-source, embedded Prototypes, single-machine apps. Zero setup — runs in your Python process. This lesson uses it.
Pinecone Managed SaaS The default "I want it to just work in production". No servers to run. Pay per usage.
Weaviate Open-source + managed Strong hybrid search (vector + keyword) and built-in modules for embeddings, generation, multi-tenancy.
Qdrant Open-source, Rust core Fast metadata filtering at scale. Self-host or managed cloud. Good when you query by tags often.
pgvector Postgres extension You already run Postgres and want vectors next to your business data. One DB to back up, one to monitor.
Milvus Open-source, distributed Billion-scale. Built for serious scale-out clusters. Heavier ops cost — overkill for most.
Vertex AI Vector Search Google managed You're all-in on GCP and using Gemini. Tightest integration with the Google stack.
Decision shortcut
  • Learning / prototype: Chroma
  • Already on Postgres: pgvector
  • Want it managed, no ops: Pinecone
  • All-in on Google Cloud: Vertex AI Vector Search
  • Need hybrid search (vector + keyword): Weaviate
  • Heavy metadata filtering: Qdrant
  • Billions of vectors, big team: Milvus
MuleSoft analogue
Think of it like picking a system of record. Chroma = SQLite (embedded, easy). Pinecone = managed SaaS API like Anypoint Cloud. pgvector = "we already have Postgres, just put it there". Vertex Vector Search = the GCP-native option, like using CloudHub when your platform is already Anypoint.
Don't get distracted by the leaderboard
Benchmarks change quarterly. The boring truth: for < 1 million vectors and a small team, any of these works. Pick by operational fit (where will it run? who maintains it?), not raw QPS numbers.

Part 2 — Hands-on: upgrade rag.py to use Chroma

We'll keep everything from Lesson 9 — chunking, system instruction, the "I don't know" guard — and only swap out the storage and retrieval pieces.

Concepts you'll meet

  • Client — connects to the database. PersistentClient(path=...) stores data on disk.
  • Collection — like a table. Holds documents + their vectors + metadata. You name them.
  • Add — insert chunks. You provide your own IDs and (optionally) embeddings.
  • Query — give it an embedding (or text), get the top-K nearest neighbours back. The DB does the search.

Step 1 — Install Chroma

From your project's virtual environment (the one from Lesson 4):

# Activate venv first if not already active

Step 2 — Create a client and a collection

import chromadb

# Persistent = saves to disk; survives restarts
client = chromadb.PersistentClient(path="./chroma_store")

# get_or_create_collection avoids "already exists" errors on re-runs
collection = client.get_or_create_collection(name="anypoint_docs")
New syntax: keyword arguments with =
path="./chroma_store" is a keyword argument. Chroma's PersistentClient accepts several optional parameters; using the name path= lets you pass just the one you want without worrying about position. You've seen this pattern already (e.g. config= in Gemini calls), but worth naming it.

Step 3 — Add chunks (with their embeddings)

Chroma can embed text for you, but we'll keep using Gemini's embedding model so we stay consistent with Lesson 9. We pass the vectors ourselves. 

def build_index(collection):
    text = DOC_PATH.read_text()
    chunks = chunk_markdown(text)
    vectors = [embed(c, "RETRIEVAL_DOCUMENT") for c in chunks]
    ids = [str(i) for i in range(len(chunks))]

    collection.add(
        documents=chunks,
        embeddings=vectors,
        ids=ids,
    )
New syntax: str(i) for i in range(len(chunks))
Three small things stacked together:
  • len(chunks) — number of chunks (e.g. 12).
  • range(12) — produces 0, 1, 2, ... 11.
  • str(i) — converts each integer to a string. Chroma requires IDs to be strings.
Wrapped in [...], it's a list comprehension that builds ["0", "1", "2", ..., "11"].

Step 4 — Query it

def search(question, collection):
    q_vec = embed(question, "RETRIEVAL_QUERY")
    results = collection.query(
        query_embeddings=[q_vec],
        n_results=TOP_K,
    )

    # Chroma returns lists-of-lists (one inner list per query).
    # We sent one query, so take index [0] of each.
    docs = results["documents"][0]
    distances = results["distances"][0]

    # Chroma returns DISTANCE (lower = closer), not similarity.
    # Convert to similarity so our threshold logic stays the same.
    scored = [(1 - d, doc) for d, doc in zip(distances, docs)]
    return [(s, c) for s, c in scored if s > THRESHOLD]
Gotcha — distance vs similarity
Chroma returns distance (smaller = more similar). Our Lesson 9 threshold logic was written for cosine similarity (larger = more similar). The line 1 - d converts Chroma's default cosine distance back to similarity, so all our existing thresholds (0.5, etc.) keep working.

Step 5 — Wire it together

def load_or_build():
    client = chromadb.PersistentClient(path=CHROMA_PATH)
    collection = client.get_or_create_collection(name="anypoint_docs")

    # If empty, populate it. Otherwise it's already on disk.
    if collection.count() == 0:
        build_index(collection)

    return collection

collection.count() replaces the INDEX_PATH.exists() check from Lesson 9. The data is in the ./chroma_store/ directory, but you don't manage the files — Chroma does.

The complete rag_chroma.py

One file you can copy and run. Save next to your anypoint-auth.md.

# rag_chroma.py — RAG backed by Chroma vector database
import re
from pathlib import Path
from dotenv import load_dotenv
from google import genai
from google.genai import types
import chromadb

load_dotenv()
client = genai.Client()

# ─── Config ───────────────────────────────
EMBED_MODEL  = "gemini-embedding-001"
CHAT_MODEL   = "gemini-3.5-flash"
DOC_PATH     = Path("anypoint-auth.md")
CHROMA_PATH  = "./chroma_store"
COLLECTION   = "anypoint_docs"
THRESHOLD    = 0.5
TOP_K        = 3

SYSTEM_INSTRUCTION = """
You are an expert assistant answering questions from a provided document.
RULES (must follow):
1. Answer ONLY from the CONTEXT below. Do not use outside knowledge.
2. If the CONTEXT does not contain the answer, reply exactly:
   "I don't have that information in the provided document."
3. Cite the chunk number(s) you used at the end of your answer, e.g. [1] [3].
4. Be concise. Maximum 4 sentences.
"""

# ─── Helpers ──────────────────────────────
def chunk_markdown(text):
    parts = re.split(r"(?=^## )", text, flags=re.MULTILINE)
    return [p.strip() for p in parts if p.strip()]

def embed(text, task_type):
    result = client.models.embed_content(
        model=EMBED_MODEL,
        contents=text,
        config=types.EmbedContentConfig(task_type=task_type)
    )
    return result.embeddings[0].values

# ─── Index build / load ───────────────────
def build_index(collection):
    text = DOC_PATH.read_text()
    chunks = chunk_markdown(text)
    vectors = [embed(c, "RETRIEVAL_DOCUMENT") for c in chunks]
    ids = [str(i) for i in range(len(chunks))]
    collection.add(documents=chunks, embeddings=vectors, ids=ids)
    print(f"Indexed {len(chunks)} chunks into Chroma.")

def get_collection():
    db = chromadb.PersistentClient(path=CHROMA_PATH)
    collection = db.get_or_create_collection(name=COLLECTION)
    if collection.count() == 0:
        build_index(collection)
    return collection

# ─── Retrieval ────────────────────────────
def search(question, collection):
    q_vec = embed(question, "RETRIEVAL_QUERY")
    results = collection.query(query_embeddings=[q_vec], n_results=TOP_K)
    docs      = results["documents"][0]
    distances = results["distances"][0]
    scored = [(1 - d, doc) for d, doc in zip(distances, docs)]
    return [(s, c) for s, c in scored if s > THRESHOLD]

# ─── Prompt + ask ─────────────────────────
def build_user_message(question, retrieved):
    blocks = []
    for i, (score, chunk) in enumerate(retrieved, start=1):
        blocks.append(f"[{i}] (relevance {score:.2f})\n{chunk}")
    context = "\n\n".join(blocks)
    return f"CONTEXT:\n{context}\n\nQUESTION: {question}"

def ask(question, collection):
    retrieved = search(question, collection)
    if not retrieved:
        return "I don't have that information in the provided document."
    user_msg = build_user_message(question, retrieved)
    response = client.models.generate_content(
        model=CHAT_MODEL,
        contents=user_msg,
        config=types.GenerateContentConfig(system_instruction=SYSTEM_INSTRUCTION),
    )
    return response.text

# ─── Entry point ──────────────────────────
if __name__ == "__main__":
    coll = get_collection()
    for q in [
        "How do I rotate a Connected App secret?",
        "What is the capital of France?",  # should fall back
    ]:
        print(f"\nQ: {q}\nA: {ask(q, coll)}")
First run vs subsequent runs
First run: Chroma is empty → build_index() embeds your doc and writes to ./chroma_store/. Every later run: collection.count() is non-zero → no re-embedding, instant startup.

Run it

source .venv/bin/activate
python rag_chroma.py

You should see "Indexed N chunks into Chroma." once, then your Q/A output. On the second run, no indexing message — it loaded from disk.

What you gained — and what's still missing

Gained

  • Real index. Chroma uses HNSW under the hood — sub-linear search.
  • Concurrency-safe. Multiple readers OK; no JSON corruption.
  • Metadata-ready. You can add metadatas=[{"section": "auth"}, ...] on add(), then filter on query (we'll do this in a later lesson).
  • Incremental updates. collection.add() with new IDs adds without rebuilding everything.

Still missing (for production)

  • Hybrid search (vector + keyword) — Weaviate does this best out of the box. Lesson 12.
  • Multi-machine. Embedded Chroma runs in one Python process. For a real service, run Chroma in server mode or move to Pinecone/Qdrant/pgvector.

Updating chunks without rebuilding

The collection.count() == 0 trick from Step 5 only handles the first run. Once your source doc changes — a new section, an updated paragraph, a deleted FAQ entry — you need a way to sync the index without re-embedding everything.

The naive approach: rm -rf chroma_store/ and rebuild. That works, but you pay the full embedding bill (slow, and it costs money) every time. Let's do better.

The four CRUD operations

Chroma exposes the same four basics you'd find in any database:

collection.add(...)      # insert new records
collection.update(...)   # replace by ID (record must exist)
collection.upsert(...)   # update if exists, else insert
collection.delete(...)   # remove by ID

One direct example — you edited chunk "3" and want only that chunk re-embedded:

new_text = "## Rotating secrets\nUse the new self-service portal..."
new_vec  = embed(new_text, "RETRIEVAL_DOCUMENT")

collection.update(
    ids=["3"],
    documents=[new_text],
    embeddings=[new_vec],
)

One API call, one re-embedding. The other 99 chunks aren't touched.

Gotcha — update doesn't auto-re-embed
If you call collection.update(ids=["3"], documents=[new_text]) without passing embeddings=..., Chroma keeps the old vector. We told it we'd supply our own embeddings, so it won't re-call Gemini behind your back. Always pass the new vector explicitly.

The harder problem — knowing which chunk to update

The example above assumes you already know the ID. In real life, you re-process a Markdown file and need to figure out: which chunks are new, changed, deleted, or unchanged?

The clean trick: use a hash of the chunk text as its ID. Then a chunk's ID changes if and only if its content changes.

import hashlib

def chunk_id(text):
    # SHA-256 of the chunk text → first 16 hex chars
    return hashlib.sha256(text.encode()).hexdigest()[:16]
New syntax: hashlib, .encode(), .hexdigest()
Three things at once:
  • hashlib — Python's built-in module for cryptographic hashes (SHA-256, MD5, etc.).
  • text.encode() — converts a Python str into raw bytes. Hash functions only work on bytes, not strings, so this step is mandatory.
  • .hexdigest() — returns the hash as a hex string (e.g. "a3f2c1b9...") instead of raw bytes. The [:16] slice keeps the first 16 characters — plenty unique for a small corpus and shorter to log.

Visualise what content-hash IDs do for you when the source doc changes: 

BEFORE EDIT                       AFTER EDIT (chunk 3 edited, chunk 4 deleted)

  IDs in DB:                        New IDs from doc:
    a3f2  ← chunk 1                   a3f2  ← unchanged ✓
    9b4e  ← chunk 2                   9b4e  ← unchanged ✓
    77c1  ← chunk 3 (edit me)         6d8a  ← edited (new hash!)
    bb09  ← chunk 4 (delete me)       e5ff  ← unchanged ✓
    e5ff  ← chunk 5

  Diff:
    delete: {77c1, bb09}
    add:    {6d8a}
    keep:   {a3f2, 9b4e, e5ff}     ← no re-embedding for these

The sync function

Drop-in replacement for the get_collection() from Step 5. Computes the diff, then only embeds what's new:

import hashlib

def chunk_id(text):
    return hashlib.sha256(text.encode()).hexdigest()[:16]

def sync_index(collection):
    # 1. What's in the doc right now?
    text       = DOC_PATH.read_text()
    chunks     = chunk_markdown(text)
    new_ids    = [chunk_id(c) for c in chunks]
    new_id_set = set(new_ids)

    # 2. What's already in the DB?
    existing_ids = set(collection.get()["ids"])

    # 3. Diff: what to delete, what to add
    to_delete = existing_ids - new_id_set
    to_add    = [(cid, c) for cid, c in zip(new_ids, chunks)
                 if cid not in existing_ids]

    # 4. Apply changes
    if to_delete:
        collection.delete(ids=list(to_delete))
        print(f"Deleted {len(to_delete)} stale chunks")

    if to_add:
        ids   = [cid for cid, _ in to_add]
        docs  = [c   for _, c in to_add]
        vecs  = [embed(c, "RETRIEVAL_DOCUMENT") for c in docs]
        collection.add(ids=ids, documents=docs, embeddings=vecs)
        print(f"Added {len(to_add)} new/changed chunks")

    if not to_delete and not to_add:
        print("Index already up to date.")

def get_collection():
    db = chromadb.PersistentClient(path=CHROMA_PATH)
    collection = db.get_or_create_collection(name=COLLECTION)
    sync_index(collection)   # ← always run sync; it's a no-op if nothing changed
    return collection
New syntax: set() and set difference (-)
A set is like a list, but with two superpowers: items are unique, and membership checks (x in s) are very fast — constant time, not linear.
  • set([1, 2, 2, 3]) → {1, 2, 3} (duplicates dropped)
  • {1, 2, 3} - {2, 4} → {1, 3} (everything in the left set that isn't in the right)
That's exactly what existing_ids - new_id_set does: "give me the IDs in the DB that are no longer in the doc — those are the ones to delete."

Why this matters operationally
Embeddings are the slow, paid step. The whole point of incremental sync is to only pay the embedding bill for what actually changed. A 500-chunk doc with one edit = 1 embedding call, not 500.
Vector DB ≠ source of truth
Treat your source markdown (or wherever the docs really live) as the source of truth. The vector DB is a derived index — like a search index over a database. Don't try to edit chunks directly inside Chroma; edit the source and re-run sync.
MuleSoft analogue
Content-hash IDs are basically git's content-addressable model: a blob's identity is the hash of its bytes. Same idea applied to chunks — unchanged content keeps its identity across rebuilds, so derived indexes (like vector embeddings) can be reused safely.

Quick check

Q1. Why does Lesson 9's JSON cache fall over at scale?

  • JSON files can't store floats accurately
  • Python can't read files larger than 100 MB
  • Every query scans every chunk linearly — no index
  • JSON loses the order of keys

Q2. You already run Postgres for your business data. Cheapest way to add vectors?

  • Stand up a separate Pinecone account
  • Install the pgvector extension in your existing Postgres
  • Migrate everything to Milvus
  • Run Chroma in a sidecar container

Q3. Why do we convert Chroma's distance with 1 - d?

  • Chroma returns distance (smaller = closer); our threshold logic uses similarity (larger = closer)
  • It's required by the Gemini API
  • It normalises the vectors
  • It's a typo and should be removed

Q4. What does collection.count() == 0 guard against?

  • Network errors when calling Gemini
  • Bad embeddings being stored
  • Two processes writing at once
  • Re-embedding the whole document on every run after the first

Q5. Which vector DB choice fits "I want managed, no servers, easiest path to production"?

  • Milvus
  • Chroma (embedded mode)
  • Pinecone
  • pgvector

Q6. Why use a content hash (e.g. SHA-256 of the chunk text) as the chunk's ID?

  • Hashes are shorter than UUIDs, so the DB is faster
  • A chunk's ID changes only when its content changes — unchanged chunks keep their ID and don't get re-embedded
  • SHA-256 is required by Chroma
  • It encrypts the chunk so it can't be read back
