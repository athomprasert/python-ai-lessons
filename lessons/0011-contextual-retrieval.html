Contextual Retrieval

Cure the "they expire" problem with a tiny LLM-generated context blurb prepended to each chunk before embedding. 35–67% fewer retrieval misses, no special model required.

L1 ✓ L2 ✓ L3 ✓ L4 ✓ L5 ✓ L6 ✓ L7 ✓ L8 ✓ L9 ✓ L10 ✓ L11 — Contextual Retrieval L12 — Hybrid Search

Why this lesson

In Lesson 8 you met the pronoun problem: a chunk that says "They expire after 90 days" embeds with no idea what "they" refers to. Search for "how often do Connected Apps expire" and the relevant chunk gets missed.

Late chunking (Lesson 8) was one fix — but it required a special embedding model that exposes per-token vectors. Gemini's embedding API doesn't.

Contextual Retrieval (Anthropic, 2024) is a different fix for the same disease. It works with any embedding model — including Gemini — because it solves the problem in the text layer, not the model layer:

The core idea (one sentence)
Before embedding each chunk, ask an LLM to write a short "what is this chunk about, given the whole document?" blurb, then prepend it to the chunk text. The chunk's vector now encodes both the chunk and its document context.

Anthropic's published eval showed retrieval-failure rates dropping 35–67% across multiple datasets. That's the kind of number that justifies the extra indexing cost — read on for the trade-off.

The context-generation prompt

This is the prompt Anthropic published, lightly adapted for our markdown chunks:

"""
<document>
{whole_document}
</document>

Here is a chunk we want to situate within the document:
<chunk>
{chunk_text}
</chunk>

Provide a short, succinct context (1-2 sentences) that situates
this chunk within the overall document for the purposes of
improving search retrieval. Answer ONLY with the context — no
preamble, no explanation."
"""

Three things to notice in the prompt:

  • The whole document is included. The LLM needs full context to write a meaningful blurb. Yes, that means re-sending the doc with every chunk's request.
  • "1–2 sentences." Anything longer dilutes the chunk's own signal in the embedding.
  • "Answer ONLY with the context." Without this, models love to add "Here's the context:..." prefixes that pollute every chunk.
Use a cheap, fast model for context generation
Use gemini-3.5-flash (or smaller) — not your top-tier reasoning model. The task is summarisation with full doc grounding, which the small models nail. Saves money and time at indexing.

Hands-on — extend rag_chroma.py

We're keeping everything from Lesson 10 (Chroma storage, sync logic, content-hash IDs) and inserting one new step into the indexing pipeline: generate the context blurb, prepend, then embed.

Step 1 — Add a generate_context() function

CONTEXT_PROMPT = """
<document>
{whole_document}
</document>

Here is a chunk we want to situate within the document:
<chunk>
{chunk_text}
</chunk>

Provide a short, succinct context (1-2 sentences) that situates
this chunk within the overall document for the purposes of
improving search retrieval. Answer ONLY with the context — no
preamble, no explanation.
"""

def generate_context(whole_document, chunk_text):
    prompt = CONTEXT_PROMPT.format(
        whole_document=whole_document,
        chunk_text=chunk_text,
    )
    response = client.models.generate_content(
        model=CHAT_MODEL,
        contents=prompt,
    )
    return response.text.strip()
New syntax: str.format() with named placeholders
"...{name}...".format(name=value) substitutes {name} in the string with value. It's the same thing as f-strings, but with the substitution happening later (when you call .format()), not at the moment the string is written. We use it here because CONTEXT_PROMPT is defined once at the top of the file as a template, and filled in many times — once per chunk.

Equivalent f-string: f"...{whole_document}...{chunk_text}..." — but that only works if the variables are in scope when the string literal is written.

Step 2 — Modify build_index / sync_index to prepend context

def contextualise(whole_document, chunks):
    # Returns a parallel list: each chunk preceded by its blurb
    out = []
    for i, chunk in enumerate(chunks):
        ctx = generate_context(whole_document, chunk)
        print(f"  [{i+1}/{len(chunks)}] context: {ctx[:60]}...")
        out.append(f"{ctx}\n\n{chunk}")   # prepend, blank line, then chunk
    return out

Now update the sync-loop's add branch to embed the contextualised text — but keep the original chunk text in documents so retrieval still returns the human-readable chunk:

if to_add:
    ids       = [cid for cid, _ in to_add]
    docs      = [c   for _, c in to_add]

    # NEW — generate context blurbs, prepend before embedding
    whole_doc = DOC_PATH.read_text()
    enriched  = contextualise(whole_doc, docs)
    vecs      = [embed(t, "RETRIEVAL_DOCUMENT") for t in enriched]

    # Store ORIGINAL chunk text as the document, but with the CONTEXT-AWARE vector
    collection.add(ids=ids, documents=docs, embeddings=vecs)
Why store the original chunk, not the enriched one?
The blurb helps the embedding find the chunk. But once retrieved, you want to show Gemini (and your user, for citations) the actual source text — not an LLM-generated summary stapled to the front. Embed the enriched version; store the original.

Step 3 — Query side: nothing changes

The query pipeline is identical to Lesson 10:

q_vec = embed(question, "RETRIEVAL_QUERY")
results = collection.query(query_embeddings=[q_vec], n_results=TOP_K)

This is the elegance of the technique: only indexing changes. Querying, retrieval scoring, prompt construction, the LLM call — all unchanged.

The cost / benefit honestly

PhaseCost changeNotes
Indexing time +1 LLM call per chunk 500 chunks → 500 Flash calls. A few cents on Gemini Flash. Slower than embedding alone.
Indexing $$ Up Each call sends the whole document as context. With prompt caching (most providers support it), the doc is cached once and reused — much cheaper.
Storage ~Same You store the original chunks; only vectors change.
Query time Unchanged Same Chroma query, same Gemini answer call.
Retrieval quality +35–67% fewer misses Per Anthropic's eval. Bigger gains on docs heavy with pronouns, cross-references, and section-relative phrasing.
When NOT to use it
  • Tiny corpora (a handful of chunks). The indexing cost outweighs the recall gain on data this small.
  • Self-contained chunks. If your chunks are FAQ-style — each one a complete Q&A with no dependence on neighbours — there's nothing for the blurb to add.
  • Doc bigger than the LLM context window. The prompt sends the whole doc per chunk. For a 500-page PDF, you'd split into "macro sections" first and contextualise within each.

Late chunking vs Contextual Retrieval — same problem, different fix

Late chunking (L8)Contextual Retrieval (this lesson)
Where context is added Inside the embedding model (attention) Before the embedding model (in the text)
Special embedding model required? Yes — needs per-token output (Jina v3, nomic, bge-m3) No — works with Gemini, OpenAI, anything
Indexing cost Single embed call per doc +1 LLM call per chunk
Maximum doc size Embedding model's context window Chat model's context window (typically larger)
What ends up in the DB Raw chunks + contextual vectors Raw chunks + vectors of context-prepended chunks

You can also combine them — late chunking with a model that supports it, applied to context-prepended chunks. Anthropic's paper notes additive gains. Diminishing returns kick in fast though; pick one technique first, measure, then add the other only if you need it.

Make this cheaper — prompt caching

Look at what contextualise() does for a 100-chunk doc: 

For each of 100 chunks:
  → send the WHOLE document + this chunk to Gemini
  → get back a 1-2 sentence blurb

Total: 100 LLM calls
       Each call ships the full document
       → 99% of input tokens are repeated

For a 50,000-token document, you've sent 5,000,000 tokens of input — but most are the same document, just resent. Prompt caching fixes this by letting Gemini remember the front of your prompt across calls.

How prompt caching works

FIRST CALL
  ┌─────────────────────────────────┐  ┌──────────┐
  │  WHOLE DOCUMENT (50k tokens)    │  │ chunk 1  │
  └─────────────────────────────────┘  └──────────┘
   ↑ full price                         ↑ full price
   ↑ Gemini stores a CACHE of this prefix

SUBSEQUENT CALLS (within the cache TTL)
  ┌─────────────────────────────────┐  ┌──────────┐
  │  WHOLE DOCUMENT (cache hit)     │  │ chunk 2  │
  └─────────────────────────────────┘  └──────────┘
   ↑ ~75-90% discount                  ↑ full price
                                         (chunk text only — small)

Cached input tokens cost roughly ¼ the price of fresh ones. For a 100-chunk indexing run, that typically cuts the contextualisation bill by 70–85% and roughly halves the wall-clock time.

Two flavours — pick the right one

Implicit cachingExplicit caching
How it works Auto-cache when you reuse identical prefixes You create a cache object, then reference it
Code change Zero Two extra lines
Control Gemini decides You set the TTL, you reuse it deliberately
Best for Ad-hoc workloads Batch indexing — you know the doc upfront

For Contextual Retrieval — looping through every chunk in one go — explicit caching is the right tool. You know you'll reuse the document immediately and many times. No guessing.

The code change — 3 small edits

A — Create the cache once, before the chunk loop

def make_doc_cache(whole_document, ttl_seconds=600):
    """Cache the document so it's not re-billed for every chunk."""
    cache = client.caches.create(
        model=CHAT_MODEL,
        config=types.CreateCachedContentConfig(
            contents=[whole_document],
            system_instruction=(
                "You write short context blurbs that situate a chunk "
                "within a larger document. Answer ONLY with the blurb."
            ),
            ttl=f"{ttl_seconds}s",
        ),
    )
    return cache

The cache holds the parts that are identical across all 100 chunk requests: the document and the system instruction.

New syntax: typed config builder
types.CreateCachedContentConfig(...) is a typed builder object. The Gemini SDK uses these instead of plain dicts so editors can autocomplete and catch typos. You've seen the pattern with GenerateContentConfig and EmbedContentConfig — same idea.
New syntax: implicit string concatenation
Look at the system instruction: two adjacent strings inside the parentheses with no +: 
system_instruction=(
    "You write short context blurbs that situate a chunk "
    "within a larger document. Answer ONLY with the blurb."
),
Python automatically concatenates adjacent string literals. It's how you write a long string across multiple lines without using + or a triple-quoted block. The two strings become one: "You write short context blurbs that situate a chunk within a larger document. Answer ONLY with the blurb."

B — Modify generate_context() to reference the cache

def generate_context(cache, chunk_text):
    response = client.models.generate_content(
        model=CHAT_MODEL,
        contents=(
            f"Here is a chunk we want to situate:\n"
            f"<chunk>\n{chunk_text}\n</chunk>\n\n"
            f"Provide a 1-2 sentence context that situates this chunk "
            f"within the document for retrieval. Answer ONLY with the context."
        ),
        config=types.GenerateContentConfig(
            cached_content=cache.name,   # ← the magic line
        ),
    )
    return response.text.strip()

What changed compared to the original:

  • We don't send the document in the prompt anymore — only the chunk-specific bit
  • cached_content=cache.name tells Gemini "prepend the cached prefix to this prompt"
  • The system instruction also lives in the cache, so it's not in the prompt either

C — Wire it into the contextualise loop

def contextualise(whole_document, chunks):
    cache = make_doc_cache(whole_document)
    print(f"Cache created: {cache.name} (TTL 10 min)")

    out = []
    for i, chunk in enumerate(chunks):
        ctx = generate_context(cache, chunk)
        print(f"  [{i+1}/{len(chunks)}] context: {ctx[:60]}...")
        out.append(f"{ctx}\n\n{chunk}")

    # Polite cleanup — caches expire on TTL, but deleting stops billing sooner
    client.caches.delete(name=cache.name)
    return out

That's it. Same logic; the document ships to Gemini once, not 100 times.

Important gotchas

Minimum cache size
Gemini requires the cached content to be at least ~32k tokens to qualify (this number changes; check current docs). A 5k-token README won't be eligible. If your doc is small, prompt caching is pointless anyway — re-sending 5k tokens 100 times is cheap.
TTL — don't overpay for storage
Gemini bills you a small per-minute fee for the cache being alive, even if you don't use it. Match the TTL to your workload:
  • One-off batch indexing → 10–30 min is plenty
  • Hourly re-index service → don't bother caching; re-create per run
  • Long-running streaming workload → manage cache lifecycle explicitly
A 2-minute indexing run with a 24-hour TTL = paying for 22 hours of unused storage.
Cache key = exact-match prefix
Caches are keyed on the exact bytes of the prefix. Change one character in the document → new cache. This is fine for our pattern (cache → process all chunks → delete), but matters if you try to share caches across separate runs.
Caching also works for query-time RAG
The same trick applies to your ask() function. If you have a long, fixed system instruction (rules, examples, persona), cache it once at startup. Each user question becomes a small, cheap call against the cached prefix. Same pattern, different stage of the pipeline.

Q1. What does Contextual Retrieval prepend to each chunk before embedding?

  • The chunk's content hash
  • A list of keywords extracted by TF-IDF
  • A 1–2 sentence LLM-generated blurb describing the chunk's role in the whole document

Q2. Why store the original chunk text in the DB even though we embed the enriched version?

  • Chroma rejects duplicate text
  • The blurb helps retrieval find the chunk, but downstream you want to feed the LLM (and cite) the actual source — not an LLM-generated summary
  • Storage is cheaper that way
  • It avoids licensing issues

Q3. Compared to Late Chunking, what's the main advantage of Contextual Retrieval?

  • It works with any embedding model — no need for one that exposes per-token vectors
  • It's free
  • It uses fewer tokens
  • It eliminates the need for chunking

Q4. The phrase "Answer ONLY with the context — no preamble, no explanation" in the prompt exists because…

  • It's required by the Gemini API
  • It prevents the model from refusing the request
  • It saves tokens by half
  • Without it, models tend to add "Here's the context:..." preambles that pollute every chunk's embedding

Q5. Which corpus would benefit LEAST from Contextual Retrieval?

  • Long technical specs heavy with "the above", "this method", "they"
  • Multi-section runbooks where chunks reference earlier setup
  • A flat FAQ file where every Q&A is independent and self-contained
  • Legal documents where definitions appear once and are referenced throughout
