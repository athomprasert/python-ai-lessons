Hybrid Search — BM25 + Vectors

Vectors miss exact IDs. Keywords miss paraphrases. Combine them with Reciprocal Rank Fusion and you stop missing both.

Why hybrid wins

Vectors and keywords have opposite failure modes. Consider one document chunk: 

"Rotate the Connected App secret quarterly. Reference ticket ABC-12345."
QueryVector searchKeyword (BM25)
"rotate connected app secret" ✓ exact match ✓ words overlap
"how do I cycle my client credentials" ✓ understands paraphrase ✗ zero word overlap
"OAuth token expiry policy" ✓ semantically related ✗ different vocabulary
"ABC-12345 ticket reference" ✗ embeddings smear specific tokens ✓ exact ID match
"v2.3.1-rc4 changelog" ✗ version strings get mangled ✓ tokenises cleanly

Each method covers the other's blind spots. Hybrid search = run both and merge the results. Industry standard since 2023; Pinecone, Weaviate, Qdrant, OpenSearch all ship it natively. Chroma doesn't (yet) — so you build it.

Where exact matching matters most
Identifiers, error codes, version strings, SKUs, file paths, function names. Anything where the exact characters carry meaning. Vector models trained on natural-language semantics treat "ABC-12345" and "ABC-12346" as nearly identical neighbours — keyword search keeps them apart.

BM25 in 60 seconds

BM25 is the keyword-search algorithm Elasticsearch, Lucene, OpenSearch, and Solr have all used as their default since the early 2000s. It scores how relevant a document is to a query based on three things:

IngredientPlain EnglishWhy it matters
TF (term frequency) How often the query word appears in this doc Doc that mentions "OAuth" 5 times beats one that mentions it once
IDF (inverse document frequency) How rare the word is across the corpus "the" appears everywhere → worthless. "ABC-12345" is unique → very informative.
Length normalisation Penalty for long documents A long doc has more chances to mention the word by accident — discount that

You don't need the formula. You need the intuition: BM25 rewards rare-word matches in short documents. That's exactly the recipe for matching specific identifiers.

Why not just count keywords?
Naive counting weights "the" and "ABC-12345" equally. IDF fixes that. Length normalisation prevents a 10-page doc from beating a 1-page doc just by being longer. BM25 is the smallest set of fixes that makes naive counting actually work.

The combining trick — Reciprocal Rank Fusion (RRF)

Now you have two ranked lists. The naive idea: "add up the scores." But:

  • Cosine similarity is bounded between 0 and 1
  • BM25 scores are unbounded raw numbers (could be 0.3, could be 18.7)
  • Adding them = whichever has the bigger numbers always wins

Reciprocal Rank Fusion sidesteps the whole problem. It throws away the scores and uses only the rank (1st place, 2nd place, 3rd place) from each list. The formula: 

RRF_score(doc) = Σ over each ranker:  1 / (k + rank_in_that_ranker)

  where k is a constant (usually 60)

That's it. No score calibration, no tuning weights, no normalising distributions. Read the original paper if you're curious — it's two pages and the algorithm is in one equation.

Visualised on three docs

VECTOR RANKING                BM25 RANKING

  1. doc_A  (rank 1)             1. doc_C  (rank 1)
  2. doc_B  (rank 2)             2. doc_A  (rank 2)
  3. doc_C  (rank 3)             3. doc_D  (rank 3)

RRF (k = 60):

  doc_A:  1/(60+1) + 1/(60+2)  = 0.01639 + 0.01613 = 0.03252  ← winner
  doc_B:  1/(60+2)             = 0.01613
  doc_C:  1/(60+3) + 1/(60+1)  = 0.01587 + 0.01639 = 0.03226
  doc_D:                1/(60+3) = 0.01587

Final ranking:  doc_A → doc_C → doc_B → doc_D

Notice doc_A wins because it ranked highly in both — even though no single ranker put it first. That's the whole virtue of RRF: it rewards agreement between the rankers, regardless of their score scales.

Why k = 60?
The constant k controls how much top ranks dominate. Small k (e.g. 1) → rank 1 vs rank 2 differs hugely. Large k (60+) → all ranks contribute meaningfully. 60 is the value from the original paper and what every search engine uses out of the box. Don't tune it on day one.

Step 2 — Build a BM25 index from the same chunks

from rank_bm25 import BM25Okapi

def tokenise(text):
    # Lowercase + split on non-word characters. Simple, language-leaning to English.
    return re.findall(r"\w+", text.lower())

def build_bm25(chunks):
    tokenised = [tokenise(c) for c in chunks]
    return BM25Okapi(tokenised), chunks
New syntax: re.findall(r"\w+", text)
re is Python's regex module (you've seen it in re.split from L9).
  • r"\w+" — a raw string holding the regex pattern. \w matches word characters (letters, digits, underscore). + means "one or more in a row".
  • re.findall(...) returns all non-overlapping matches as a list.
Result: "ABC-12345 expires!" → ["abc", "12345", "expires"]. The hyphen and exclamation are dropped — they're not \w characters. This is a simple, English-leaning tokeniser; for Chinese/Japanese/Thai use the segmenters from Lesson 8's multilingual section.
Why an extra index, not replace Chroma?
The vector index in Chroma is staying. We're adding BM25 alongside it. At query time we hit both, fuse the rankings, and return the merged top-K. Two indexes, one search function.

Step 3 — Implement RRF

def rrf_fuse(vector_ranked, bm25_ranked, k=60):
    # Both inputs are lists of chunk-ids in ranked order (best first).
    scores = {}
    for rank, doc_id in enumerate(vector_ranked, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    for rank, doc_id in enumerate(bm25_ranked, start=1):
        scores[doc_id] = scores.get(doc_id, 0) + 1 / (k + rank)
    # Sort by combined score, descending
    return sorted(scores.items(), key=lambda x: x[1], reverse=True)
New syntax: dict.get(key, default)
scores.get(doc_id, 0) means: "give me scores[doc_id] if it exists; otherwise return 0". Avoids a KeyError when the doc hasn't been seen yet. Common idiom for "running totals" in dicts.
New syntax: lambda functions
lambda x: x[1] is a tiny anonymous function that takes x and returns x[1]. Used here to tell sorted() "sort by the second element of each tuple" — i.e. by the score, not the doc_id. Equivalent to writing: 
def get_score(x): return x[1]
sorted(scores.items(), key=get_score, reverse=True)
lambda is the inline shortcut when the helper is one line and used once.

Step 4 — The hybrid search function

def hybrid_search(question, collection, bm25, chunk_ids, k_each=10, top_k=3):
    # A) Vector side
    q_vec = embed(question, "RETRIEVAL_QUERY")
    vec_results = collection.query(query_embeddings=[q_vec], n_results=k_each)
    vector_ranked = vec_results["ids"][0]   # list of chunk-ids, best first

    # B) BM25 side
    q_tokens = tokenise(question)
    bm25_scores = bm25.get_scores(q_tokens)
    # Pair each chunk-id with its score, sort descending, take top k_each
    bm25_ranked = [cid for cid, _ in
                   sorted(zip(chunk_ids, bm25_scores), key=lambda x: x[1], reverse=True)
                   [:k_each]]

    # C) Fuse
    fused = rrf_fuse(vector_ranked, bm25_ranked)[:top_k]

    # D) Look up the actual chunk text from Chroma using the IDs
    ids_only = [doc_id for doc_id, _ in fused]
    fetched = collection.get(ids=ids_only)
    id_to_text = dict(zip(fetched["ids"], fetched["documents"]))

    return [(score, id_to_text[doc_id]) for doc_id, score in fused]

Notice the structure: both rankers vote, RRF merges, Chroma supplies the final text. The vector store remains the source of truth for chunk content; BM25 is just a parallel ranker.

Step 5 — Wire it into get_collection()

def get_collection():
    db = chromadb.PersistentClient(path=CHROMA_PATH)
    collection = db.get_or_create_collection(name=COLLECTION)
    sync_index(collection)

    # Build BM25 from whatever's in Chroma right now
    fetched = collection.get()
    bm25, _ = build_bm25(fetched["documents"])
    chunk_ids = fetched["ids"]

    return collection, bm25, chunk_ids

# In your main loop:
if __name__ == "__main__":
    coll, bm25, chunk_ids = get_collection()
    for q in [
        "How do I rotate a Connected App secret?",
        "What does ticket ABC-12345 say?",
    ]:
        results = hybrid_search(q, coll, bm25, chunk_ids)
        print(f"\nQ: {q}")
        for score, chunk in results:
            print(f"  [{score:.4f}] {chunk[:80]}...")

BM25 is rebuilt from Chroma's contents on startup. Cheap (BM25 indexing is ~milliseconds even for thousands of chunks), and guarantees the two indexes never drift apart.

The native alternative

You wrote BM25 + RRF by hand because Chroma doesn't ship it. Most production vector DBs do. One call gets you hybrid: 

# Weaviate
client.query.get("Doc").with_hybrid(query=q, alpha=0.5).do()

# Qdrant (with sparse vectors)
client.query_points(collection_name=...,
    prefetch=[Prefetch(query=vec), Prefetch(query=sparse)],
    query=FusionQuery(fusion=Fusion.RRF))

# Pinecone (sparse-dense vectors)
index.query(vector=vec, sparse_vector=sparse, top_k=10)

# pgvector + Postgres full-text
# SELECT id FROM docs ORDER BY (embedding <=> q_vec) LIMIT 10
# UNION the result with a tsvector @@ to_tsquery(...) lookup, then RRF

The algorithm you just learned is the same one each of these tools runs internally. You'll know what's happening when you see alpha=0.5 (a vector/keyword weight) or fusion=RRF in their docs.

When NOT to bother
  • Pure prose corpus with no identifiers (literary text, blog posts) — vector alone is enough.
  • Chunks that have already been heavily contextualised (L11) — the blurb's vocabulary often matches what users type, narrowing BM25's edge.
  • Tiny corpora (<100 chunks) — measure first; the gain may be invisible.

Quick check

Q1. Why does pure vector search struggle with queries like "ABC-12345 ticket reference"?

  • Vector models can't process numbers
  • The query is too short
  • Embedding models trained on natural-language semantics treat similar-looking IDs as close neighbours; the exact characters lose specificity
  • BM25 is required by Chroma

Q2. What does Reciprocal Rank Fusion combine?

  • The raw scores from each ranker
  • The position (rank) each doc holds in each ranker's list
  • The vector embeddings of the documents
  • The user query and the document text

Q3. Why throw away the raw scores instead of adding them?

  • Cosine similarity (0–1) and BM25 (unbounded) are on different scales — adding them lets one dominate
  • Score addition is patented
  • Raw scores leak private information
  • Python doesn't support adding floats

Q4. Why do we keep Chroma (the vector DB) when we add BM25?

  • BM25 doesn't store the chunk text
  • Chroma is required for embedding
  • It's faster that way
  • Each ranker covers the other's weakness — vector for paraphrase, BM25 for exact identifiers — and Chroma stays the source-of-truth chunk store

Q5. When does hybrid search add the LEAST value?

  • Tech docs full of error codes and version strings
  • Customer support tickets with product SKUs
  • A literary corpus of essays — pure prose, no identifiers
  • Code documentation referencing function names
