Advanced Retrieval Patterns

Nine architectural patterns that build on chunking + embeddings + hybrid search. Recognise them in any framework; reach for the right one when your eval points at a specific gap.

L1 ✓ L2 ✓ L3 ✓ L4 ✓ L5 ✓ L6 ✓ L7 ✓ L8 ✓ L9 ✓ L10 ✓ L11 ✓ L12 ✓ L13 — Advanced Retrieval L14 — Evaluating RAG

Why this lesson

L8–L12 covered foundation retrieval: chunking, embedding, vector storage, contextual blurbs, hybrid (vector + BM25). Those answer the question "how do I find the right chunks?"

This lesson covers the architectural patterns built on top — orchestration designs that combine retrieval steps in smarter ways. Sub-question decomposition, recursive retrieval, auto-merging, knowledge-graph RAG, and so on.

Architectural pattern, not a library
Each pattern below is a design you could hand-code in any framework. LlamaIndex packages most of them as ready-made classes — that's its main value-add. LangChain has its own equivalents. The patterns themselves are framework-agnostic.

You don't need all nine. The eval (L14) tells you which one to add — each fixes a specific failure mode. This lesson is the menu; future eval results pick the dish.

1. Sub-question decomposition

The pattern

Break a complex question into smaller sub-questions, retrieve and answer each separately, then combine.

"Compare the auth model between MuleSoft and Salesforce"
                          │
                          ▼
                  [decomposition LLM]
                  │              │
                  ▼              ▼
   "What is MuleSoft auth?"   "What is Salesforce auth?"
                  │              │
                  ▼              ▼
            [retrieve]      [retrieve]
                  │              │
                  └──── combine ─┘
                          │
                          ▼
                  Final answer

When to reach for it: users ask comparison or multi-faceted questions and you see "answer covers half the question" in your eval failures.

LlamaIndex implementation: SubQuestionQueryEngine — wires up the decomposition LLM, the per-sub-question retrieval, and the final combine step.

2. Recursive retrieval

The pattern

Indexes can reference other indexes. A retrieved node can be a summary that points to a deeper sub-index of the actual content.

Top-level index — one node per document (summaries only)
       │
       │ retrieved: "MuleSoft auth doc summary"
       ▼
Sub-index of THAT document — chunks of the auth doc
       │
       ▼
Specific chunks → LLM

The summary layer routes retrieval to the right document first, then drills into chunks. Solves the "100 docs of 50 chunks each = 5000 chunks competing on cosine" problem.

When to reach for it: large multi-document corpora where the right document is obvious but flat chunking buries it under noise from 99 other docs.

LlamaIndex: RecursiveRetriever + IndexNode. Each IndexNode in the top-level index references a sub-retriever that knows its parent doc's chunks.

3. Multi-document agents

The pattern

Each document gets its own agent with its own tools (typically a vector tool + a summary tool). A top-level agent routes queries to the right document-agent.

Top-level routing agent
        │
        │ "user is asking about MuleSoft auth specifically"
        ▼
   MuleSoft-doc agent ──→ [vector tool] [summary tool]
        │
        ▼
   Response

Step beyond recursive retrieval: each document gets its own retrieval policy, not just its own chunks. Useful when different documents need different strategies (one is structured, one is prose, one is tabular).

When to reach for it: 50+ heterogeneous documents where each warrants its own retrieval strategy.

LlamaIndex: part of AgentWorkflow + multi-agent patterns. FunctionAgent per doc, top-level FunctionAgent for routing.

4. Hybrid search — RECAP from L12

The pattern

Vector search + BM25 keyword search, fused with Reciprocal Rank Fusion.

You built this by hand in L12. Restated here because it sits at the same level as the other patterns — and so the menu is complete.

LlamaIndex: QueryFusionRetriever with mode="reciprocal_rerank". Same algorithm as your L12 implementation, packaged.

5. Query rewriting and HyDE

The pattern

Transform the query before retrieval to improve recall. Two flavours:

  • Query rewriting — LLM cleans up a vague/short query before embedding
  • HyDE — LLM writes a fake answer to the question, embed THAT, retrieve docs near it (covered as a force-multiplier in L16)
Query rewriting:
   "rotate secrets?"
        ↓ [LLM]
   "How do I rotate Connected App OAuth client secrets?"
        ↓ [embed + retrieve]

HyDE:
   "How do I rotate secrets?"
        ↓ [LLM writes a hypothetical answer]
   "To rotate Connected App secrets, navigate to..."
        ↓ [embed THIS, retrieve real chunks near it]

When to reach for it: short or ambiguous user queries; eval shows recall climbs when you manually rewrite queries.

LlamaIndex: HyDEQueryTransform, MultiStepQueryEngine.

6. Auto-merging retrieval

The pattern

Retrieve small chunks. If multiple small chunks come from the same parent paragraph, "merge up" and return the parent paragraph instead of the fragments.

Retrieved:  chunk_A (subsection 1)
            chunk_B (subsection 1)
            chunk_C (subsection 1)   ← 3 of the same parent
                            ▼
                    Return: subsection 1 (parent)   ← merged up

Why it works: small chunks give precision in matching, but the LLM needs a paragraph or section's worth of context to compose a real answer. Auto-merging gives you both.

When to reach for it: chunks are small enough that you often retrieve several siblings from the same section. Eval shows answers feel "thin" or "ungrounded".

LlamaIndex: AutoMergingRetriever + HierarchicalNodeParser at indexing time.

7. Sentence-window retrieval

The pattern

Embed individual sentences for precision. At retrieval time, return a window of surrounding sentences for context.

Indexing:
   doc → split into sentences → embed each individually

   sentence 12: "Rotate the Connected App secret quarterly."
                          │
                          ▼ embedding (precise)

Retrieval:
   query embedding matches sentence 12
                          │
                          ▼
   Return sentences 9–15  ← window around the match
                          │
                          ▼
   LLM has both the matched line AND its surrounding context

This is the inverse of auto-merging — small embedding granularity, big retrieval granularity.

When to reach for it: precision-recall trade-off on chunk size feels stuck. You want sentence-level matching without losing paragraph-level context.

LlamaIndex: SentenceWindowNodeParser at indexing + MetadataReplacementPostProcessor at retrieval time.

8. Knowledge graph RAG

The pattern

Build an entity-relationship graph from the corpus at indexing time. Retrieve via graph traversal, alongside (or instead of) vector similarity.

Build phase (one-time, expensive):
   docs → [LLM extraction] → entities & relations →

   (Connected App) ──[expires in]──→ (90 days)
        │
        [authenticated by]
        ▼
   (OAuth client) ──[contains]──→ (client secret)


Query phase:
   "What expires after 90 days?"
        ↓
   Graph traversal: nodes connected to "90 days" via "expires in"
        ↓
   Returns: Connected App  (relational answer, not similarity-based)

When to reach for it: heavily relational corpora — corporate org charts, biomedical entities, product hierarchies. Questions that contain "show me everything connected to X" or "what's the relationship between A and B".

LlamaIndex: KnowledgeGraphIndex (older), PropertyGraphIndex (newer, recommended).
Note: graph build is the most expensive of any pattern here — 1 LLM extraction call per chunk minimum. Worth it only when relational queries dominate.
Don't reach here first
Knowledge graph RAG is the most expensive and most niche pattern. Try hybrid search and contextual retrieval before graphs. The cost-to-benefit only justifies when "show all entities connected to X" is the primary query shape.

9. Router query engine

The pattern

A "selector" decides at query time which query engine to use — vector, summary, SQL, web search, graph — based on the question.

"How many customers do we have?"
            ↓ [selector LLM]
       SQL query engine

"What does the auth doc say?"
            ↓ [selector LLM]
       Vector query engine

"Show me everything connected to OAuth"
            ↓ [selector LLM]
       Graph query engine

When to reach for it: mixed corpora — structured database + prose docs + graph data — where different question shapes need different retrieval engines. The router answers the meta-question "which engine?"

LlamaIndex: RouterQueryEngine + LLMSingleSelector or PydanticSingleSelector.

Architectural pattern vs library — clarified

Every pattern above is a design pattern. You could hand-code each one in plain Python, in LangChain, in your rag_chroma.py, or in any framework. They're recipes, not products.

LlamaIndex's value is that it ships ready-made classes for all of them: SubQuestionQueryEngine, RecursiveRetriever, AutoMergingRetriever, etc. You import and use; you don't reimplement.

LangChain has equivalents (often less mature for retrieval-specific patterns). Haystack has its own (pipeline-DAG-shaped). Native SDK has none — you write everything yourself. The pattern is portable; the implementation isn't.

Why this matters for L17 (framework choice)
LlamaIndex's "advanced RAG patterns out of the box" claim is real — and these are the patterns. If your eval reveals you need three of them, that's a strong signal toward LlamaIndex. If you need none of them, the simpler frameworks (Pydantic AI, native SDK) are fine.

Don't reach for patterns speculatively

The cardinal rule: let the eval pick the pattern. Adding sub-question decomposition because it sounds clever, before you've shown your eval set has comparison-question failures, just adds cost and latency for no gain.

The diagnostic loop
Run eval (L14) → identify failure mode → match to pattern below → apply ONE → re-run eval

   Failure mode                       Pattern to try
   ──────────────────                 ──────────────
   "missed half the question"      →  Sub-question decomposition
   "right doc, wrong chunks"       →  Recursive retrieval
   "exact ID misses"               →  Hybrid search (you have it)
   "vague queries miss"            →  Query rewriting / HyDE
   "answers feel under-context'd"  →  Auto-merging or sentence-window
   "show everything connected"     →  Knowledge graph
   "mixed source types"            →  Router query engine

What to use for your MuleSoft RAG bot

Given the corpus (Anypoint docs, RAML specs, runbooks, ticket archives) and likely query shape:

  • Sub-question decomposition — for "compare connector A vs B" or "what's the difference between OAuth and basic auth". Likely worthwhile.
  • Auto-merging retrieval — if your eval shows answers feel under-context'd. One chunking-time change.
  • Router query engine — when you eventually have multiple corpora (Anypoint docs + RAML + runbooks + ticket history) needing different retrieval strategies.

The rest are useful elsewhere but probably overkill for your shape. Eval-driven, as always.

Q3. Auto-merging retrieval and sentence-window retrieval solve opposite-feeling problems. The relationship is…

  • Auto-merging is a typo for sentence-window
  • They are identical
  • Auto-merging is for graph data; sentence-window is for prose
  • Both decouple embedding granularity from retrieval granularity — auto-merging escalates UP from small chunks to parents; sentence-window expands a small match to its surrounding window

Q4. Why is knowledge graph RAG flagged as "don't reach here first"?

  • Graph build at indexing time is the most expensive of any pattern (1+ LLM call per chunk), so the cost only justifies when relational queries dominate
  • It doesn't work with Gemini
  • It produces less accurate answers
  • LlamaIndex deprecated it
