Evaluating RAG Quality

Stop guessing whether your changes help. A 50-line eval harness turns vibes into numbers — and tells you which K to use.

L1 ✓ L2 ✓ L3 ✓ L4 ✓ L5 ✓ L6 ✓ L7 ✓ L8 ✓ L9 ✓ L10 ✓ L11 ✓ L12 ✓ L13 ✓ L14 — Evaluating RAG L15 — Enterprise Eval

Why this lesson

You've made big upgrades since Lesson 9 — vector DB, contextual retrieval, prompt caching, hybrid search. Each was sold to you with a promise ("35–67% better recall"). But on your corpus, did each upgrade actually help, or did it cost money for no benefit?

Without measurement, you're flying blind. With a small evaluation harness, you can answer two questions every time you change something:

  • Did the change make retrieval better, worse, or the same?
  • What's the right value of K for my corpus?
The mindset shift
Production RAG teams treat the eval set like a unit test suite. You write it once, run it after every change, and only ship if numbers stay flat or improve. Without it, you're shipping on hope.

The three layers to measure

RAG has three places things can go wrong. Each needs its own metrics.

LayerThe questionMetrics we'll cover
1. Retrieval Did we surface the right chunks? Hit@K, Precision@K, Recall@K, MRR, (nDCG)
2. Generation Did the LLM use those chunks faithfully? Faithfulness, Answer Relevance
3. End-to-end Did the user actually get a correct answer? Correctness (LLM-as-a-judge)

The retrieval layer is where most issues live and the easiest to evaluate objectively. We'll spend most of the lesson there.

The eval set — your golden source of truth

Before any metric makes sense, you need a list of questions paired with the chunks that should answer them. This is your eval set (also called a golden set).

Format — one JSON file:

[
  {
    "question": "How do I rotate a Connected App secret?",
    "relevant_chunk_ids": ["chunk_5", "chunk_12"],
    "expected_answer": "Use the self-service portal at /admin/secrets..."
  },
  {
    "question": "What does ticket ABC-12345 say?",
    "relevant_chunk_ids": ["chunk_18"],
    "expected_answer": "It tracks the deprecation of the v1 token endpoint."
  },
  {
    "question": "What is the capital of France?",
    "relevant_chunk_ids": [],
    "expected_answer": "I don't have that information in the provided document."
  }
]
Build a balanced 20-question set
Mix three categories of question:
  • Paraphrase queries — different words from the doc (tests vector search)
  • Exact-ID queries — error codes, tickets, version strings (tests BM25 / hybrid)
  • Should-fall-back queries — questions not in the corpus (tests the threshold guard)
Anthropic's published Contextual Retrieval eval used as few as 50 questions per dataset. 20 is enough to start; grow it as you find real-user questions you missed.

The chunk IDs come from your existing Chroma collection (Lesson 10's content-hash IDs). Open Chroma, browse a few chunks, write the questions yourself the first time. Yes, this is manual work — once.

Retrieval metrics — five ways to score the same retrieval

Worked example for every metric below. Eval question:

"How do I rotate a Connected App secret?"
relevant_chunk_ids = ["chunk_5", "chunk_12"]    ← ground truth: 2 chunks are relevant

Your retrieval returns top 5:

Position 1:  chunk_8     ← not relevant
Position 2:  chunk_5     ← relevant ✓
Position 3:  chunk_3     ← not relevant
Position 4:  chunk_12    ← relevant ✓
Position 5:  chunk_99    ← not relevant

1. Hit@K — "did we get at least one right?"

Hit@K = 1 if any relevant chunk appears in top K, else 0

The simplest metric. Binary yes/no.

  • Hit@1 = 0 (top 1 was chunk_8, not relevant)
  • Hit@3 = 1 (chunk_5 is in the top 3 ✓)
  • Hit@5 = 1

Use when: coarse "did retrieval work at all?" check across many queries. Average it across the eval set to get an overall hit rate.

2. Precision@K — "of what we returned, how much was useful?"

Precision@K = (relevant chunks in top K) / K
  • Precision@3 = 1/3 = 0.33 (1 relevant out of 3 returned)
  • Precision@5 = 2/5 = 0.40 (2 relevant out of 5 returned)

Use when: you care about prompt pollution. Every irrelevant chunk you pass to Gemini wastes tokens, costs money, and dilutes the signal.

3. Recall@K — "of what was relevant, how much did we find?"

Recall@K = (relevant chunks in top K) / (total relevant chunks)
  • Recall@3 = 1/2 = 0.50 (found 1 of 2 relevant chunks)
  • Recall@5 = 2/2 = 1.00 (found both)

Use when: some questions need multiple chunks to answer correctly (think runbooks, multi-step procedures). Missing one means an incomplete answer.

4. MRR — Mean Reciprocal Rank

RR = 1 / (rank of first relevant chunk) MRR = average of RR across the eval set
  • First relevant chunk (chunk_5) appeared at rank 2.
  • RR = 1/2 = 0.50
  • If retrieval put it at rank 1, RR = 1.0. If at rank 5, RR = 0.20. If never found, RR = 0.

Use when: position matters. LLMs weight earlier chunks in the prompt more heavily — a relevant chunk at rank 1 is more useful than the same chunk at rank 5.

5. nDCG@K — for graded relevance (sidebar — recognise it, don't implement it yet)

normalised Discounted Cumulative Gain. Like Precision@K, but treats relevance as a score (e.g. 0 = irrelevant, 1 = somewhat, 2 = highly relevant) and discounts it by position. Used in academic IR papers and in tools like RAGAS.

For practical RAG with binary relevance (relevant or not), Precision@K + Recall@K + MRR cover what nDCG would. Skip it on day one; come back if you start labelling chunks with graded relevance.

The classic precision/recall trade-off

The same retrieval scores differently at different K. This is the most important fact about K:

K = 3                                  K = 10
─────────────                          ────────────────
Returned: 3 chunks                     Returned: 10 chunks
Relevant in returned: 2                Relevant in returned: 4
Total relevant in DB: 4                Total relevant in DB: 4

Precision@3 = 2/3  = 0.67              Precision@10 = 4/10 = 0.40   ↓
Recall@3    = 2/4  = 0.50              Recall@10    = 4/4  = 1.00   ↑

Bigger K = better recall, worse precision. The eval shows you the right K for your corpus.

Heuristic for K
  • Self-contained chunks (FAQ): K = 3
  • Multi-step runbooks: K = 5–8
  • Long repetitive corpora (legal): K = 8–15
  • Code / cross-referenced refs: K = 5 + reranking
Use the eval to confirm. "My recall stops improving past K=5" is the kind of finding that justifies the choice.

Hands-on — build eval.py

Step 1 — Save your eval set

Create eval_set.json next to your rag_chroma.py. Use real chunk IDs from your Chroma collection — open it once with collection.get() to find them.

Step 2 — Implement the four core retrieval metrics

import json
from statistics import mean

def hit_at_k(retrieved_ids, relevant_ids, k):
    top_k = retrieved_ids[:k]
    # Set intersection: shared elements between two sets
    return 1 if set(top_k) & set(relevant_ids) else 0

def precision_at_k(retrieved_ids, relevant_ids, k):
    top_k = retrieved_ids[:k]
    if not top_k:
        return 0.0
    hits = len(set(top_k) & set(relevant_ids))
    return hits / len(top_k)

def recall_at_k(retrieved_ids, relevant_ids, k):
    if not relevant_ids:
        return 1.0   # nothing to find = trivially perfect
    top_k = retrieved_ids[:k]
    hits = len(set(top_k) & set(relevant_ids))
    return hits / len(relevant_ids)

def reciprocal_rank(retrieved_ids, relevant_ids):
    for i, doc_id in enumerate(retrieved_ids, start=1):
        if doc_id in relevant_ids:
            return 1.0 / i
    return 0.0
New syntax: set intersection (&)
set(a) & set(b) returns the elements present in both sets. Just like the set difference (-) you saw in Lesson 10, but for "what's common" instead of "what's missing". 
set(["a", "b", "c"]) & set(["b", "c", "d"])
# → {"b", "c"}
Wrap in len(...) and you have "how many elements appear in both" — exactly what Precision and Recall need.
New syntax: statistics.mean()
from statistics import mean imports Python's built-in average function. mean([0.5, 1.0, 0.0]) → 0.5. You could write sum(xs)/len(xs) instead, but mean handles edge cases (returns a clean error on empty input) and signals intent at a glance.

Step 3 — The eval runner

def eval_retrieval(eval_set_path, k_values=(3, 5, 10)):
    eval_set = json.loads(Path(eval_set_path).read_text())
    coll, bm25, chunk_ids = get_collection()   # from your rag_chroma.py

    # Collect per-question scores at every K
    rows = []
    for entry in eval_set:
        q = entry["question"]
        relevant = entry["relevant_chunk_ids"]

        # Retrieve a generous number — up to max(k_values)
        max_k = max(k_values)
        results = hybrid_search(q, coll, bm25, chunk_ids, k_each=20, top_k=max_k)
        # hybrid_search returns [(score, chunk_text)] — we need IDs
        # Quick adapter: re-fetch IDs from Chroma using the chunk text
        retrieved_ids = retrieve_ids_for_query(q, coll, bm25, chunk_ids, max_k)

        row = {"question": q[:50]}
        for k in k_values:
            row[f"hit@{k}"]       = hit_at_k(retrieved_ids, relevant, k)
            row[f"precision@{k}"] = precision_at_k(retrieved_ids, relevant, k)
            row[f"recall@{k}"]    = recall_at_k(retrieved_ids, relevant, k)
        row["rr"] = reciprocal_rank(retrieved_ids, relevant)
        rows.append(row)

    # Aggregate (averages across the eval set)
    summary = {}
    for k in k_values:
        summary[f"Hit@{k}"]       = mean(r[f"hit@{k}"] for r in rows)
        summary[f"Precision@{k}"] = mean(r[f"precision@{k}"] for r in rows)
        summary[f"Recall@{k}"]    = mean(r[f"recall@{k}"] for r in rows)
    summary["MRR"] = mean(r["rr"] for r in rows)
    return summary, rows

And a small printer for the table:

def print_summary(summary):
    print("\n=== Eval summary ===")
    for name, val in summary.items():
        print(f"  {name:<15} {val:.3f}")
New syntax: {name:<15} in f-strings
The colon-and-stuff inside an f-string is format spec. {name:<15} means "left-align in a 15-character-wide field". {val:.3f} means "float with 3 decimal places". Useful for printing aligned tables.

Step 4 — Run it

if __name__ == "__main__":
    summary, _ = eval_retrieval("eval_set.json")
    print_summary(summary)

Don't just trust marketing claims
Anthropic's "35–67% improvement" was measured on their datasets. On a clean MuleSoft RAML corpus, contextual retrieval might only buy 5%. On a pronoun-heavy legal corpus, 80%. The eval tells you which world you're in.

Generation metrics — when retrieval isn't the whole story

Retrieval can be perfect and the LLM can still produce a bad answer — by hallucinating, ignoring the context, or summarising poorly. Two metrics catch this.

Faithfulness

"Is every claim in the answer supported by the retrieved chunks?"

You can't compute this with set arithmetic — you need a judge. The standard trick is LLM-as-a-judge: ask another LLM call to grade the answer.

FAITHFULNESS_PROMPT = """
Given the following CONTEXT and ANSWER, identify whether every
factual claim in the ANSWER is supported by the CONTEXT.

CONTEXT:
{context}

ANSWER:
{answer}

Respond with ONLY a JSON object:
{{"faithful": true|false, "unsupported_claims": [...]}}
"""

def faithfulness(context, answer):
    response = client.models.generate_content(
        model="gemini-3.5-flash",
        contents=FAITHFULNESS_PROMPT.format(context=context, answer=answer),
    )
    parsed = json.loads(response.text)
    return 1.0 if parsed["faithful"] else 0.0

Answer relevance

"Does the answer actually address the question?" Same pattern, different prompt:

RELEVANCE_PROMPT = """
Question: {question}
Answer: {answer}

Score how well the ANSWER addresses the QUESTION on a 0–1 scale.
Respond with ONLY a JSON object: {{"score": 0.0-1.0, "reason": "..."}}
"""

End-to-end correctness

Compare the produced answer to the expected_answer field in your eval set. Same LLM-as-a-judge pattern: "Is the produced answer semantically equivalent to the expected answer?"

LLM-as-a-judge has known biases
  • Verbosity bias — judges prefer longer answers, even if shorter would do
  • Self-preference — Gemini judging Gemini-produced answers tends to be lenient. Use a different family of model as the judge when stakes are high
  • Ordering bias — when asked to compare A vs B, judges often pick the first one
Treat LLM-as-a-judge scores as signals, not ground truth. Always sample a few by hand to sanity-check.

The tools landscape

You wrote your eval harness from scratch because the concepts are simple enough to grok. In production you'll typically use a library — both for richer metrics and to integrate with dashboards.

ToolWhat it gives you
RAGAS The de facto standard for RAG eval. Implements faithfulness, context precision/recall, answer relevance, etc., with built-in LLM-judges. Plays nicely with LangChain & LlamaIndex.
TruLens Feedback-functions framework with a rich UI. Strong on observability — track eval scores over time as you ship changes.
Phoenix / Arize Open-source observability for LLM apps. Captures traces, metrics, and lets you replay queries. Strongest of the bunch on production monitoring.
DeepEval Pytest-style "unit tests for LLMs." Best fit if you want eval to live in your existing test suite.

Your hand-rolled eval.py is enough for now. Switch to RAGAS or DeepEval the day you find yourself reimplementing nDCG or wanting eval-runs to show up in CI.

Quick check

Q1. Your retrieval returns 5 chunks. 2 are relevant. There are 4 relevant chunks total in the corpus. What is Recall@5?

  • 2/5 = 0.40
  • 2/4 = 0.50, then divided by 5
  • 2/4 = 0.50
  • 5/4 = 1.25

Q2. Same scenario as Q1. What is Precision@5?

  • 2/5 = 0.40
  • 2/4 = 0.50
  • 5/4 = 1.25
  • It cannot be calculated

Q3. What happens to Precision@K and Recall@K as you increase K?

  • Both go up
  • Both go down
  • Precision goes up, recall goes down
  • Precision goes down, recall goes up (or stays the same)

Q4. When does Recall@K matter MORE than Hit@K?

  • When you have a small eval set
  • When the question requires multiple chunks to answer correctly — finding only one isn't enough
  • When K is small
  • Only when using BM25

Q5. Why use a different model family as the judge in LLM-as-a-judge eval?

  • Different models are cheaper
  • It's required by the OpenAI Terms of Service
  • A model judging answers from its own family tends to be lenient (self-preference bias) — using a different family makes the score more honest
  • Different models produce different embeddings

Q6. Anthropic's blog claimed Contextual Retrieval gives 35–67% recall improvement. After running your eval, you only see a 5% improvement on your corpus. What's the correct conclusion?

  • Anthropic was lying
  • Their measured improvement was on their datasets; yours is what counts for your corpus. The eval is doing its job.
  • You implemented it wrong; redo it until you see 35%
  • 5% improvement is unacceptable; remove contextual retrieval
