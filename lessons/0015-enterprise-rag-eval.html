Enterprise RAG Eval — Beyond the Basics

L14 taught you the foundation. This lesson covers the metrics enterprise teams actually track when RAG goes to production: per-query depth, safety, operations, and system-level health.

Why this lesson

Lesson 14 gave you per-query metrics — Hit@K, Precision@K, Recall@K, MRR — and a generation-side judge. Those are necessary but not sufficient for production.

Enterprise teams track three additional layers:

LayerThe question it answersExamples
Per-query depth How well did this answer perform across multiple quality dimensions? Context Precision, Context Recall, Groundedness, Citation Accuracy
Safety & compliance Are we exposing the org to hallucination, toxic output, or PII leakage? Hallucination Rate, Toxicity, PII Leakage
Operations Can we afford to run it? Is it fast enough? Latency p50/p95/p99, Cost/query, Coverage
System health Is the system stable and improving over time? Drift, Robustness, Refusal Correctness, Failure Clustering

Each layer answers a different stakeholder. Per-query is for engineers. Safety is for legal/compliance. Operations is for finance. System health is for SREs and product managers. Production means satisfying all four.

Tag legend used below: DETERMINISTIC = pure math, no LLM. LLM-JUDGE = uses an LLM to grade. EMBEDDING = uses an embedding-cosine, no LLM-judge.

Layer 1 — Per-query depth metrics

Context Precision LLM-JUDGE

Like Precision@K from L14, but position-aware. A relevant chunk at rank 1 scores higher than the same chunk at rank 5 — because LLMs weight earlier context more heavily.

CP = Σ (Precision@k × relevance_k) / total_relevant

RAGAS uses an LLM judge to decide each chunk's relevance, then weights by position. The intuition: "reward retrievers that rank good stuff higher."

You'd reach for this when you've already optimised Precision@K and are now tuning where good chunks land in the list — typical right before adding a reranker.

Context Recall LLM-JUDGE

L14's Recall@K asks "did we get the chunk IDs marked relevant?" — but the chunk IDs were labelled by you, by hand. Context Recall is different — it asks an LLM:

"Given the expected answer, can each claim in it be traced back to one of the retrieved chunks?"

So it doesn't depend on having labelled chunk IDs — only on having a ground-truth answer. Useful when annotating chunks is expensive but writing target answers isn't.

CONTEXT_RECALL_PROMPT = """
Given the EXPECTED ANSWER and the RETRIEVED CONTEXT, identify which
sentences in the answer are supported by the context.

EXPECTED ANSWER: {answer}
CONTEXT: {context}

Respond with JSON: {{"supported_count": N, "total_sentences": M}}
"""

def context_recall(retrieved_chunks, expected_answer):
    response = client.models.generate_content(
        model="gemini-3.5-flash",
        contents=CONTEXT_RECALL_PROMPT.format(
            answer=expected_answer,
            context="\n\n".join(retrieved_chunks),
        ),
    )
    parsed = json.loads(response.text)
    return parsed["supported_count"] / max(parsed["total_sentences"], 1)
New syntax: max(x, 1) guard
parsed["total_sentences"] / max(parsed["total_sentences"], 1) protects against divide-by-zero. max(x, 1) returns x if it's at least 1, otherwise 1. Common defensive idiom for ratios. Cleaner than an explicit if branch.

Context Entities Recall LLM-JUDGE

Of the named entities (companies, error codes, ticket IDs, version strings) in the expected answer, how many appear verbatim in retrieved chunks?

CER = |entities in retrieved ∩ entities in expected| / |entities in expected|

This is the metric that explicitly catches hybrid-search wins. If your eval shows CER jumped after adding BM25 (Lesson 12), you've proven the upgrade fixed the exact-ID weakness — not just hand-waved it.

Implementation: ask an LLM to extract entities from both, then set-intersect.

Groundedness LLM-JUDGE

Stricter cousin of Faithfulness from L14. Faithfulness asks "is the answer broadly supported?" Groundedness asks "is every individual sentence supported, with a citation?" 

def groundedness(answer, context):
    sentences = re.split(r"(?<=[.!?])\s+", answer.strip())
    supported = 0
    for sentence in sentences:
        prompt = f"""
Is the SENTENCE supported by the CONTEXT? Answer JSON only.
CONTEXT: {context}
SENTENCE: {sentence}
Respond: {{"supported": true|false}}
"""
        result = client.models.generate_content(
            model="gemini-3.5-flash", contents=prompt
        )
        if json.loads(result.text)["supported"]:
            supported += 1
    return supported / max(len(sentences), 1)
New syntax: regex with lookbehind (?<=...)
r"(?<=[.!?])\s+" splits on whitespace, but only when preceded by ., !, or ?. The (?<=...) is a lookbehind assertion — it requires a pattern to be there but doesn't consume it, so the punctuation stays attached to the previous sentence. Standard trick for sentence splitting in English.

Vertex AI's eval API ships a managed Groundedness metric with citations baked in — useful if you don't want to roll your own.

Citation Accuracy DETERMINISTIC + LLM-JUDGE

Your RAG (from L9 onward) instructs Gemini to cite chunks with [1] [3]. Citation accuracy verifies the citations are correct:

  1. Parse the answer for citation markers like [3] (deterministic regex)
  2. For each cited chunk, ask the LLM: "Does this sentence's claim actually come from chunk N?"
  3. Score = correctly cited / total citations

Critical for any user-facing system that shows citations — fake citations destroy trust faster than no citations.

Answer Semantic Similarity EMBEDDING

Cosine similarity between the embedding of the produced answer and the embedding of the expected answer. No LLM judge needed — pure math, fast, cheap. 

def answer_similarity(produced, expected):
    p_vec = embed(produced, "SEMANTIC_SIMILARITY")
    e_vec = embed(expected, "SEMANTIC_SIMILARITY")
    return cosine(p_vec, e_vec)   # your cosine helper from L9
Note the new task_type
For comparing two pieces of output text (not query↔doc retrieval), use task_type="SEMANTIC_SIMILARITY". Gemini's embedding API has a separate optimisation path for this case. Reference

Pros: deterministic, free of LLM-judge bias, runs in milliseconds. Cons: can't catch subtle factual errors — two factually different answers can still be semantically similar.

Refusal Correctness DETERMINISTIC

For eval entries where the right answer is "I don't have that information", did the system actually refuse? Or did it confidently hallucinate? 

def refusal_correct(produced_answer, expected_is_refusal):
    refusal_phrases = [
        "i don't have", "i don't know",
        "not in the provided", "no information",
    ]
    refused = any(p in produced_answer.lower() for p in refusal_phrases)
    return 1.0 if refused == expected_is_refusal else 0.0
New syntax: any(...) with a generator
any(p in answer for p in phrases) returns True as soon as any phrase is found. any() short-circuits — it stops as soon as it finds a hit. Cousin: all(...) returns True only when every element is true.

This is the #1 metric regulators care about. A confidently-wrong answer to "is this drug safe?" is far worse than "I don't know." Refusal Correctness measures the system's epistemic humility.

BLEU / ROUGE / BERTScore — mostly deprecated for RAG DETERMINISTIC

Classic NLP metrics that score surface-level word overlap between produced and expected text. Why they're falling out of favour for RAG: a correct paraphrase scores low (different words), and a wrong-but-similar-words answer scores high. Answer Semantic Similarity does the same job better.

You'll still see them in compliance reports (regulators trust standardised, deterministic numbers) and academic IR papers. Recognise them; don't optimise for them.

MAP — Mean Average Precision DETERMINISTIC

Average of Precision@K computed at each position where a relevant chunk is found, averaged across the eval set. Standard in academic IR (TREC papers, BEIR benchmark). For most RAG work, MRR + Recall@K is enough — but MAP is what you'll see in research papers.

Layer 2 — Safety & compliance metrics

If your RAG handles regulated content (financial advice, medical info, customer PII, internal HR data), the next four metrics are not optional.

Hallucination Rate LLM-JUDGE

The headline number for compliance and trust teams. Inverse of Faithfulness, but tracked separately because it's the metric executives ask about.

Hallucination Rate = % of answers containing claims not supported by retrieved context

You compute it the same way as Faithfulness from L14, but report it as "X% of answers had at least one unsupported claim." A 1% hallucination rate sounds different from a 99% faithfulness rate — even though they're identical.

Toxicity / Safety LLM-JUDGE

Did the answer contain harmful, biased, or policy-violating content? Three standard tools:

  • Perspective API (Google) — scores toxicity, threat, insult, profanity. Free up to 1 QPS.
  • OpenAI Moderation API — categories include hate, harassment, self-harm, sexual, violence. Free.
  • Internal policy classifier — for org-specific rules ("don't recommend competitor products", "don't mention deprecated APIs").

Pattern: every produced answer goes through a moderation check. Failures get flagged for review and don't ship to users.

PII Leakage Detection DETERMINISTIC

Did personally-identifiable information from the retrieved chunks end up in the answer when it shouldn't? Tools:

  • Microsoft Presidio — open-source PII detection (names, emails, SSNs, credit cards). Mature.
  • Google DLP API — managed equivalent.
  • spaCy NER + custom rules — for domain-specific identifiers (case numbers, employee IDs).

The pattern: detect PII in the answer, compare to the user's authorisation level, redact or refuse if mismatch. Critical when the retrieval corpus contains documents users aren't supposed to see directly.

Bias & Fairness LLM-JUDGE

Does the system answer differently for protected attributes (gender, race, location) when the question is otherwise identical? Sample technique:

  1. Take a query like "Is this candidate qualified?" with a chunk about candidate Alex.
  2. Run the same query 5x with the candidate's name varied across demographics.
  3. Score the answers for consistency (Answer Semantic Similarity across runs).

Low similarity = the system is biased on the swap variable. This is the technique used in published LLM bias evaluations (e.g. MT-Bench style).

Layer 3 — Operational metrics

This is the layer most tutorials skip. Real production cares as much about cost and speed as about quality.

Latency — p50 / p95 / p99 DETERMINISTIC

Average latency lies. A pipeline with mean = 800ms but p99 = 12s is unusable — 1 in 100 users waits 12 seconds.

  • p50 = median. Half of users wait less than this.
  • p95 = 95th percentile. The "typical bad day" experience.
  • p99 = 99th percentile. Tail latency — what your worst-case users get.

Track latency broken down by stage: embed, retrieve, generate. The bottleneck is rarely where you think.

import time
from functools import wraps

def measure(stage_name):
    # A decorator that records latency for the wrapped function.
    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            start = time.perf_counter()
            result = func(*args, **kwargs)
            elapsed_ms = (time.perf_counter() - start) * 1000
            METRICS[stage_name].append(elapsed_ms)
            return result
        return wrapper
    return decorator

METRICS = {"embed": [], "retrieve": [], "generate": []}

@measure("embed")
def embed(text, task_type): ...

@measure("retrieve")
def hybrid_search(...): ...

@measure("generate")
def ask(...): ...
New syntax: decorators (@measure(...))
A decorator is a function that wraps another function, transparently adding behaviour. Reading the example top-down:
  • @measure("embed") on top of def embed(...) means: "before defining embed, pass it through measure("embed")."
  • measure(stage_name) returns a decorator function.
  • decorator(func) returns a wrapper that times the call and records it.
  • From the rest of your code, embed(...) still looks like a normal function — it just secretly logs latency.
The @wraps(func) from functools preserves the original function's name and docstring (otherwise embed.__name__ becomes "wrapper").
New syntax: *args, **kwargs
*args = "accept any number of positional arguments as a tuple." **kwargs = "accept any number of keyword arguments as a dict." Together: "accept whatever arguments the wrapped function accepts, without needing to know in advance." Standard pattern in decorators.
New syntax: time.perf_counter()
The right way to measure elapsed time in Python. Use it instead of time.time() — perf_counter uses the highest-resolution monotonic clock available, which is what you want for timing code. time.time() can jump backwards if the system clock is adjusted.

Then computing percentiles:

from statistics import quantiles

def latency_summary():
    for stage, samples in METRICS.items():
        if not samples: continue
        q = quantiles(samples, n=100)
        print(f"{stage:<10} p50={q[49]:.0f}ms  p95={q[94]:.0f}ms  p99={q[98]:.0f}ms")

statistics.quantiles(samples, n=100) returns the 99 cut points for percentiles 1, 2, ..., 99. Index [49] is p50, [94] is p95, [98] is p99.

Cost Per Query DETERMINISTIC

Break it down by stage. Real RAG cost typically lives in three buckets:

BucketWhat drives itHow to reduce
Embedding (query)1 call per queryCheaper embedding model; batching
Embedding (indexing)1 call per chunkIncremental sync (L10), prompt caching for context generation (L11)
GenerationK chunks × prompt overhead × output tokensLower K (eval-tuned!), prompt caching, smaller chat model when possible
 
# Capture token counts from each Gemini response
COST_LOG = []

def log_cost(stage, response):
    usage = response.usage_metadata
    COST_LOG.append({
        "stage": stage,
        "input_tokens": usage.prompt_token_count,
        "output_tokens": usage.candidates_token_count,
        "cached_tokens": getattr(usage, "cached_content_token_count", 0),
    })
New syntax: getattr(obj, "name", default)
Like obj.name but with a fallback if the attribute doesn't exist. Useful here because cached_content_token_count only appears when prompt caching was used. Saves you from AttributeError branches.

Then multiply by current per-token prices to get $/query. Track this as a metric over time — sudden cost spikes often signal regressions before quality metrics do.

Eval Set Coverage EMBEDDING

How representative is your eval set of the queries real users actually send? If your eval has 20 questions about authentication but real users mostly ask about deployment, your scores are misleading.

The technique:

  1. Embed every real user query (or a sample) from production logs
  2. Embed every eval question
  3. For each user query, compute distance to the nearest eval question
  4. If the distance is large → that query is uncovered

Report % of user queries within a similarity threshold of some eval question. Low coverage = your eval is misleading; expand it. Most teams discover their eval set is severely under-representative the first time they measure this.

Layer 4 — System health metrics

Drift Detection

Are eval scores degrading over time? Causes:

  • Corpus drift — docs change, chunks shift, new content unindexed
  • Model drift — vendor updates the chat or embedding model under the hood
  • Query drift — users start asking new categories of questions

The pattern: run the full eval suite on a schedule (nightly), store scores in a time-series DB, alert on regressions > X%. Treat eval scores like any other SLO.

A/B Traffic-Split & Live Feedback

Your offline eval is a sample. Live signals from production are the real test:

SignalWhat it captures
👍 / 👎 thumbsExplicit user satisfaction
Click-through on cited chunksDid the answer + citations help users find the source?
Follow-up question rateHigh rate = answers are incomplete
Escalation to humanThe "I gave up on the bot" signal
Edit distance on retryIf users heavily reword to retry, retrieval is missing

A/B test new pipeline versions on traffic splits, watch these signals, ship the winner.

Robustness EMBEDDING

Same question, slightly reworded — does the answer change? 

variants = [
    "How do I rotate a Connected App secret?",
    "What's the procedure for rotating Connected App credentials?",
    "Can you walk me through Connected App secret rotation?",
]
answers = [ask(v) for v in variants]
similarities = [answer_similarity(answers[0], a) for a in answers[1:]]
robustness_score = mean(similarities)

Production teams generate paraphrase variants automatically (prompt an LLM to rewrite each eval question 3 ways) and track the average similarity. A drop in robustness flags pipelines that are overfit to specific phrasing.

Failure Clustering EMBEDDING

When the system fails, what patterns emerge? The technique:

  1. Embed every failing query (low score, thumbs-down, escalation)
  2. Run K-means or HDBSCAN clustering on the embeddings
  3. Sample a few queries from each cluster to label the pattern by hand

Typical clusters that emerge:

  • "Misspelled product names" — fix with fuzzy matching at ingestion
  • "Multi-step questions" — fix with query decomposition (a future lesson)
  • "Time-sensitive queries" — fix by adding date metadata to chunks
  • "Out-of-corpus questions" — eval coverage is too narrow

Each cluster suggests a specific upgrade. Without clustering, you'd be guessing.

Human Review Sampling

Cheap insurance for high-stakes domains. Pattern:

  • For every answer, compute a confidence proxy (e.g. retrieval score of top chunk × generation logprobs if available)
  • Bottom 1–5% by confidence get sent to a human reviewer queue before reaching the user (or for post-hoc audit)
  • Reviewer feedback feeds back into the eval set as new ground truth

This is how compliance-heavy systems (medical, legal, financial advice) get to ship.

No single tool covers all four layers. Production teams typically combine 2–3: one for per-query quality (RAGAS), one for ops/observability (Phoenix or Arize), one for safety (Presidio + Perspective).

Quick check

Q1. Why report Hallucination Rate separately from Faithfulness?

  • They measure different things mathematically
  • Faithfulness can't be computed automatically
  • They are mathematical inverses, but Hallucination Rate is the framing executives, regulators, and trust teams actually ask about
  • RAGAS doesn't support Faithfulness

Q2. Your average latency is 800ms but p99 is 12 seconds. What does this tell you?

  • The system is fast enough for production
  • 800ms is the only number that matters
  • p99 isn't a real metric
  • 1 in 100 users waits 12 seconds — average lies; track tail latency

Q3. What does Context Recall measure that Recall@K from L14 doesn't?

  • Latency
  • Whether the retrieved chunks cover the information needed for the expected answer — judged by an LLM, doesn't require pre-labelled chunk IDs
  • The cosine similarity between answer and context
  • The position of relevant chunks

Q4. Why is Refusal Correctness considered a top-priority metric for regulated domains?

  • A confidently-wrong answer to a high-stakes question is worse than "I don't know"
  • It's the cheapest metric to compute
  • RAGAS scores it highest
  • It's required by GDPR

Q5. Eval Set Coverage measures…

  • How many chunks are in your corpus
  • How fast retrieval runs
  • How representative your eval questions are of the queries real users actually send
  • The percentage of correct answers

Q6. Which metric explicitly catches the value of adding BM25 / hybrid search?

  • Hit@K alone
  • Latency p50
  • Cost per query
  • Context Entities Recall — measures whether named entities (IDs, codes, version strings) made it into retrieved chunks

Q7. Failure clustering helps you find…

  • The fastest way to embed
  • Patterns in failed queries (e.g. "Spanish queries", "deprecated API references") so you know which fix to prioritise
  • The cheapest model
  • Bias in the eval set
