The Agent Framework Landscape

Ten frameworks in 2026, split into two tiers — vendor-agnostic (community-run) and cloud-vendor-native (Amazon, Google, Microsoft) — plus native SDK as the foundation. No agent code — just decisions.

Why this lesson before any framework

Picking an agent framework is like picking a build tool — the wrong choice forces a rewrite later. You're a MuleSoft architect; you already think this way about platforms. Same instinct here.

This lesson is decision-making, not coding. Read it once. Pick the framework you'd reach for given your context. Then in L18 we build an agent from scratch with the native SDK (no framework — to feel what frameworks abstract away), and in L19 we re-implement the same agent in five frameworks side-by-side.

A note on honesty
Star counts and ecosystem rankings move weekly. I'll use ranges ("tens of thousands of stars"), not exact numbers. The relative positions in this lesson are accurate as of mid-2026; absolute numbers may have moved.

Two tiers, then the foundation

Every agent framework belongs in one of two tiers, and picking the wrong tier is a much bigger mistake than picking the wrong framework within a tier.

TierWho runs itRuns whereBest when the customer…

Below: seven vendor-agnostic frameworks, then three cloud-vendor frameworks, then native SDK as the foundation every framework sits on. Each card has a one-line summary, sweet spot, weakest case, a short style snippet, and notable users where verifiable. Style snippets all show the same trivial pattern: "call a single tool, return the answer."

Tier 1 — Vendor-agnostic frameworks

These frameworks are backed by independent teams or open-source communities. They work with any LLM provider (Gemini, Claude, GPT, Llama, etc.) and can be deployed anywhere. Choose from this tier when you want the framework decision to be independent of the cloud decision.

1. LangChain / LangGraph

2022 · multi-LLM · Python & JS

LangGraph is the modern agent layer on top of LangChain. Think of LangChain as the integrations bag (loaders, vector stores, tools) and LangGraph as the orchestration layer (state machines for agents).

Sweet spotMaximum flexibility, huge ecosystem, complex multi-step flows. Anything custom.
Weakest atSteep learning curve, lots of magic, sometimes called "framework theatre" — abstractions can outpace your need.
Style
from langgraph.prebuilt import create_react_agent
from langchain_google_genai import ChatGoogleGenerativeAI

llm = ChatGoogleGenerativeAI(model="gemini-3.5-flash")
agent = create_react_agent(llm, tools=[my_rag_tool])
result = agent.invoke({"messages": [("user", "How do I rotate secrets?")]})
Notable usersKlarna, Replit, LinkedIn, Uber, Elastic — most named LLM production deployments use LangChain somewhere.
ResourcesLangGraph docs · LangChain docs

2. LlamaIndex

2022 · multi-LLM · Python & TS

RAG-first lineage. Many of the chunking, retrieval, and indexing patterns you've learned (L7–L12) were popularised here. Their agent layer (Workflows, FunctionAgent) sits on top of that data foundation.

Sweet spotAgents whose primary job is reasoning over a document corpus. Strong data loaders for hundreds of formats.
Weakest atGeneric agent orchestration where retrieval isn't the centre of gravity.
Style
from llama_index.core.agent.workflow import FunctionAgent
from llama_index.llms.google_genai import GoogleGenAI

agent = FunctionAgent(
    tools=[my_rag_tool],
    llm=GoogleGenAI(model="gemini-3.5-flash"),
)
response = await agent.run("How do I rotate secrets?")
Notable usersWorkato, Notion, KPMG case studies — strong in enterprise RAG-shaped deployments.
ResourcesLlamaIndex docs

3. CrewAI

2024 · multi-LLM · Python

Multi-agent collaboration. You define roles ("Researcher", "Writer", "Critic"), each gets a goal and tools, and a "crew" coordinates them. Opinionated and very readable.

Sweet spotWorkflows where multiple specialised agents collaborate — content pipelines, research-and-summarise patterns.
Weakest atSingle-agent or tightly-controlled flows — the role-playing abstraction adds friction when you don't need it.
Style
from crewai import Agent, Task, Crew

researcher = Agent(
    role="Documentation Searcher",
    goal="Find accurate answers in our internal docs",
    tools=[my_rag_tool],
)
task = Task(description="How do I rotate secrets?", agent=researcher)
result = Crew(agents=[researcher], tasks=[task]).kickoff()
Notable usersSalesforce, Oracle, IBM, Deloitte — fast adoption in 2024–25 enterprise PoCs.
ResourcesCrewAI docs

4. Pydantic AI

2024 · multi-LLM · Python

Type-safe, minimal, Python-native. Built by the Pydantic team — uses Pydantic models for tool schemas and structured outputs. Feels like FastAPI for agents.

Sweet spotProduction code where you want type safety, IDE autocomplete on tool inputs/outputs, and minimal magic.
Weakest atMassive ecosystem features — fewer pre-built integrations than LangChain.
Style
from pydantic_ai import Agent

agent = Agent("google-gla:gemini-3.5-flash", tools=[my_rag_tool])
result = agent.run_sync("How do I rotate secrets?")
print(result.output)
Notable usersNewer — adoption growing in Python-shop production teams that already use Pydantic.
ResourcesPydantic AI docs

5. Haystack (deepset)

2018 · multi-LLM · Python

The most mature framework here. Started as an open-source NLP/QA pipeline tool from deepset (Berlin), evolved into a production-grade RAG and agent framework. Pipeline-DAG-first; explicit, testable nodes.

Sweet spotProduction NLP/RAG systems where reliability, evaluation, and deployment matter more than rapid prototyping. Strong on built-in eval and deployment tooling.
Weakest atCutting-edge agent patterns — the agent layer is solid but newer than the pipeline foundation.
Style
from haystack.components.agents import Agent
from haystack.components.generators.chat import GoogleAIGeminiChatGenerator

agent = Agent(
    chat_generator=GoogleAIGeminiChatGenerator(model="gemini-3.5-flash"),
    tools=[my_rag_tool],
)
result = agent.run(messages=[{"role": "user", "content": "How do I rotate secrets?"}])
Notable usersAirbus, NASA JPL, Deutsche Telekom, plus widespread enterprise NLP/QA deployments.
ResourcesHaystack docs

6. Smolagents (HuggingFace)

2024 · multi-LLM · Python

Lightweight, code-first. The agent's "thoughts" are Python code blocks it writes and executes — instead of JSON tool calls, the model writes tool.search("query") directly. Tiny core (~1k LOC).

Sweet spotAgents that primarily generate and run code, ML-flavoured workflows, anyone who wants minimum framework footprint.
Weakest atConservative production environments — code-execution agents have specific safety constraints.
Style
from smolagents import CodeAgent, LiteLLMModel

agent = CodeAgent(
    tools=[my_rag_tool],
    model=LiteLLMModel(model_id="gemini/gemini-3.5-flash"),
)
result = agent.run("How do I rotate secrets?")
Notable usersHuggingFace Spaces ecosystem, ML research teams.
ResourcesSmolagents docs

Tier 2 — Cloud-vendor-native frameworks

Each hyperscaler now has its own agent framework, backed by a managed runtime in that cloud. All three (ADK, Strands, AutoGen) are open-source Python libraries — but their strongest story is "deploy to our managed runtime in three commands", and that story only works inside the parent cloud.

Choose from this tier when the cloud decision is already made. If it isn't, choose from Tier 1 first and cross the deploy bridge later.

7. Google ADK

2025 · Gemini-first (works with other models via LiteLLM) · Python

Google's Agent Development Kit. Model-agnostic in API but Gemini-native in tooling. Includes session memory, deployment helpers, and tight integration with Vertex AI. Covered in depth in the L26–L31 mini-series.

Sweet spotYou're on Google Cloud, using Gemini, and want a managed deployment path (Vertex Agent Engine) with first-party tooling.
Weakest atNewer — smaller community than LangChain. Off-GCP deployment loses the managed-runtime story.
Style
from google.adk.agents import Agent

agent = Agent(
    model="gemini-3.5-flash",
    tools=[my_rag_tool],
    instruction="You answer from internal docs.",
)
result = agent.run("How do I rotate secrets?")
Notable usersEarly adopters in GCP-native shops; tight Vertex AI integrations.
ResourcesADK docs · GitHub

8. AWS Strands Agents

2025 · multi-LLM · Python

AWS's answer to ADK. Open-source (Apache 2.0), model-driven agent loop, tool decorators, multi-agent primitives (Swarm, Graph). Runs anywhere but pairs with Amazon Bedrock AgentCore — a managed serverless runtime for agents with built-in session management, memory, identity, and observability. AgentCore is also framework-agnostic: it can host a Strands agent, a LangGraph graph, or a CrewAI crew.

Sweet spotYou're on AWS, using Bedrock models (Claude, Nova, Llama). Strands gives you the SDK; AgentCore gives you the managed runtime.
Weakest atNewest of the three vendor frameworks (GA mid-2025). Smaller ecosystem than ADK today, though its model-agnostic design travels better.
Style
from strands import Agent, tool
from strands.models import BedrockModel

@tool
def my_rag_tool(question: str) -> str:
    """Search internal docs."""
    return "..."

agent = Agent(
    model=BedrockModel(model_id="anthropic.claude-sonnet-4-5-20250929-v1:0"),
    tools=[my_rag_tool],
)
result = agent("How do I rotate secrets?")
Notable usersAWS internal (Q Developer, others), early AWS-native enterprise adopters.
ResourcesStrands docs · GitHub · Bedrock AgentCore

9. AutoGen (Microsoft)

2023 · multi-LLM · Python & .NET

Microsoft Research's multi-agent framework. Strong on autonomous agents that can plan, delegate, and execute code. The conversation-driven model (agents talk to each other) is the signature. The managed-runtime story is Azure AI Foundry Agent Service — Azure's answer to Agent Engine and AgentCore.

Sweet spotYou're on Azure. Research-grade multi-agent systems, code-execution agents, autonomous coding workflows.
Weakest atSimple production agents — the conversational abstraction adds complexity for routine tool-calling. The v0.4 rewrite in 2025 also invalidated a lot of pre-existing tutorials.
Style
from autogen_agentchat.agents import AssistantAgent
from autogen_ext.models.google import GoogleAIChatCompletionClient

agent = AssistantAgent(
    name="helper",
    model_client=GoogleAIChatCompletionClient(model="gemini-3.5-flash"),
    tools=[my_rag_tool],
)
result = await agent.run(task="How do I rotate secrets?")
Notable usersMicrosoft internal, NASA JPL research, several university labs.
ResourcesAutoGen docs · Azure AI Foundry

Foundation — what everything above is built on

Below every framework in both tiers is the LLM provider's native function-calling API. Every framework abstracts this same loop. Knowing it means you can drop down into it when a framework gets in your way.

10. Native SDK function-calling

always available · vendor-specific (google-genai, openai, anthropic, bedrock)

No framework. You implement the agent loop yourself with the LLM provider's native function-calling API. The basis every framework above is built on.

Sweet spotLearning the concept (we'll do this in L18), tiny projects, situations where any framework dependency is unwelcome.
Weakest atAnything complex — you reimplement memory, retries, state, observability yourself.
Style
from google import genai
from google.genai import types

client = genai.Client()
chat = client.chats.create(
    model="gemini-3.5-flash",
    config=types.GenerateContentConfig(tools=[my_rag_tool]),
)
response = chat.send_message("How do I rotate secrets?")
# YOU implement the loop: check for function_calls, execute, send results back
ResourcesGemini function-calling guide

Picking a framework — three honest questions

  1. Is the framework single-vendor or multi-vendor? Frameworks tied to one LLM provider (ADK to Gemini, partially) are great when you're committed; risky when you might switch. Multi-LLM frameworks (LangChain, LlamaIndex, Haystack, Pydantic AI, CrewAI, AutoGen) protect you from provider lock-in.
  2. Can you drop down to native SDK calls when needed? Mature frameworks let you escape into raw provider APIs for the 5% of cases the abstraction doesn't cover. The newer ones often don't (yet). Test this before committing.
  3. Is the ecosystem where you need it? Vector DBs, observability (Phoenix, LangSmith), eval (RAGAS), deployment. Count the integrations your stack needs. LangChain wins on raw integration count; specialist frameworks (Haystack, LlamaIndex) often win on quality within their domain.
My personal take after building this curriculum
  • For most generalist agent work → LangGraph. The ecosystem and community size compounds.
  • If your system is RAG-shaped (your case) → LlamaIndex or Haystack. Both fit the L7–L16 work cleanly. Haystack edges ahead if you weight production maturity and built-in eval.
  • If you want type safety → Pydantic AI.
  • If you're committed to Vertex AI deployment → ADK, with the caveat that the API is evolving.
  • Always learn native SDK first. You'll see what every framework abstracts away — and you'll know when to drop down.

LangChain or LlamaIndex when you need both retrieval AND tools?

This is the most common "I'm stuck" question because both can technically do both. The honest answer:

The decision rule
  • LangChain / LangGraph if your agent is mostly a tool-using assistant that happens to retrieve sometimes
  • LlamaIndex if your agent is mostly a RAG-over-data system that happens to call tools sometimes

In other words: where is the centre of gravity?

When LangChain / LangGraph wins

  • Many heterogeneous tools — APIs, calculators, code-execution, browser, RAG, etc.
  • Complex flow control — branches, retries, loops, human-in-the-loop
  • You'll want LangSmith for observability (best-in-class)
  • The LLM is the orchestrator, retrieval is one capability among many
  • You expect to grow the integration list — LangChain has the largest ecosystem

When LlamaIndex wins

  • Corpus is the centre of gravity — hundreds of files, multiple formats, ingestion pipelines matter
  • Advanced RAG patterns out of the box — multi-doc agents, sub-question decomposition, recursive retrieval
  • Document loaders matter — LlamaIndex has 300+ data connectors (Confluence, Notion, S3, GDrive, etc.)
  • You want retrieval-quality views in observability (Phoenix integration is tightest here)
  • The agent's job is mostly "answer from this corpus, occasionally do something else"

The decision tree

1. Will retrieval be > 70% of what the agent does?
   YES → LlamaIndex
   NO  → continue

2. Do you have many heterogeneous tools (5+)?
   YES → LangGraph
   NO  → continue

3. Will you need complex flow control (branches, loops, human-in-the-loop)?
   YES → LangGraph
   NO  → continue

4. Do you need exotic data loaders (Confluence, Notion, S3, GDrive)?
   YES → LlamaIndex
   NO  → either; pick by team familiarity

The "use both" pattern that production teams use

A surprisingly common production setup: LlamaIndex for retrieval, LangGraph for orchestration. They interoperate cleanly because both treat each other as black boxes via OpenAI-style tool schemas:

# LlamaIndex builds the retrieval layer (where it shines)
query_engine = index.as_query_engine(...)

# Wrap it as a LangChain tool
from langchain.tools import Tool
docs_tool = Tool(
    name="anypoint_docs",
    func=lambda q: str(query_engine.query(q)),
    description="...",
)

# LangGraph orchestrates everything
agent = create_react_agent(llm, tools=[docs_tool, other_tools])

Trade-off: two dependencies to maintain, two paradigms to learn. Worth it for non-trivial systems; overkill for small ones.

For your specific MuleSoft RAG bot
Centre of gravity is the corpus (you've spent 7 lessons on RAG). Likely 2–4 tools at most (search docs, query Anypoint, fetch ticket, post to Slack). Production-conscious. → LlamaIndex with FunctionAgent is the better fit today.

If the agent's responsibilities later expand beyond docs Q&A — say, an integration-lifecycle assistant — the gravity shifts to orchestration and LangGraph becomes the better backbone. The eval (L14–L16) tells you when that shift has happened in practice.

Observability per framework

Once you ship an agent to production, the first thing you'll want is observability — the ability to see why the agent did what it did. Agent traces are harder to debug than regular API logs because:

  • They're deeply nested — agent calls tool, tool calls another agent, that agent calls the LLM 3 times
  • Each step's input/output is a multi-page prompt, not a clean JSON payload
  • Failures are often semantic (LLM picked wrong tool) not technical (HTTP 500) — logs alone don't tell you what's wrong

Three levels of observability

LevelWhat you seeWhy you need it
LoggingEach LLM call's prompt, response, tokens, latencyBasic debugging, cost tracking
TracingHierarchical view of all calls in one user request — nested tool calls, retries, sub-agent runsUnderstanding why the agent behaved as it did
Eval-in-productionMetrics from L14/L15 computed on live trafficCatching regressions and drift

The honest production pattern: most teams pick one observability backend and instrument all their agents to it via OpenTelemetry, regardless of which framework each agent uses.

Three decision dimensions when picking observability

  1. Where does the trace data go? Vendor cloud (LangSmith, Logfire, AgentOps) = easy but vendor lock-in. Self-hosted (Phoenix local, Langfuse self-host) = more setup, more control. Already-existing infra (Datadog, Grafana via OTel) = reuse what you have.
  2. Are you single-framework or multi-framework? Single → use the framework's native partner. Multi → use OTel + a multi-framework backend (Phoenix, Langfuse).
  3. Is observability privacy-sensitive? Healthcare/finance data → self-host is often mandatory. Internal MuleSoft docs → vendor SaaS is fine for most teams.
My personal take for your context
  • Start with Phoenix — free, runs locally, supports every framework you'll try in L19, gives genuine production-grade tracing without vendor commitment.
  • If you commit to LangChain later, LangSmith pays for itself — the Cadillac of agent observability.
  • If you commit to Pydantic AI, Logfire is the obvious match.
  • Avoid rolling your own observability past basic logging — it's a deceptively deep rabbit hole.

MuleSoft tie-in

The same instinct, different domain
The agent-framework choice maps directly onto how you already think about platforms:
  • LangGraph = MuleSoft Anypoint. Generalist, huge ecosystem, lots of integrations.
  • LlamaIndex / Haystack = a specialised iPaaS focused on document/data flows.
  • CrewAI = orchestration tool where the unit is a workflow of collaborating actors.
  • ADK / Strands / AutoGen = a vendor-managed platform with deep integration into one ecosystem (think CloudHub-only, but for each hyperscaler). Same shape, different cloud.
  • Native SDK = writing the integration in raw Java/Python without any iPaaS — full control, all the work yours.
The lock-in vs ergonomics, vendor-neutrality vs managed-convenience trade-offs are isomorphic. You already have the architect's instincts for this; just apply them.

Customer conversation shortcut: ask "which cloud is the customer already committed to?" That single question narrows Tier 2 to one option. If they say "we're multi-cloud" or "we haven't decided," start from Tier 1 instead.

Quick check

Q1. Your agent's main job is answering questions from a 10,000-document corpus. Which framework should you reach for first?

  • CrewAI
  • AutoGen
  • LlamaIndex (or Haystack) — RAG/data-heavy lineages with strong loaders and indexes
  • Native SDK

Q2. Why are we building L18 in native SDK before any framework?

  • Native SDK is faster than any framework
  • You only understand a framework after feeling the work it removes — native SDK exposes the agent loop, tool registration, and error handling that frameworks hide
  • Frameworks aren't production-ready yet
  • Google requires it

Q3. Multi-vendor framework matters when…

  • You only ever use one model
  • You're a Microsoft shop
  • It improves latency
  • You may switch LLM providers later (cost, capability, regional rollout, or vendor outage) — multi-vendor frameworks insulate you from that switch

Q4. CrewAI's strongest fit is…

  • Workflows where multiple specialised agents collaborate (e.g. researcher → writer → critic)
  • Single-agent type-safe tool use
  • Maximum flexibility on graph-based flows
  • Vertex AI managed deployment

Q5. Why did this lesson refuse to declare a "best framework"?

  • All frameworks are identical
  • Each framework is a position on a set of trade-off axes (magic vs control, single vs multi-agent, generalist vs specialist) — "best" only makes sense given a specific job
  • Anthropic forbids it
  • It would offend the LangChain team

Q6. A MuleSoft customer is deeply invested in AWS. Bedrock is their default LLM provider. Which framework do you reach for first?

  • Google ADK — it's the most polished vendor framework
  • LangGraph — biggest ecosystem
  • AWS Strands — Tier-2 framework designed for AWS, with Bedrock AgentCore as the managed runtime
  • Native SDK — no framework needed

Q7. What's the honest reason to prefer a Tier-1 (vendor-agnostic) framework over a Tier-2 (cloud-vendor-native) one?

  • Tier-1 frameworks always have better performance
  • Tier-1 frameworks are always cheaper to run
  • Tier-1 protects you when the cloud choice isn't made, might change, or is multi-cloud. Tier-2's biggest value (managed runtime) is only realised inside the parent cloud.
  • Tier-2 frameworks are not production-ready
