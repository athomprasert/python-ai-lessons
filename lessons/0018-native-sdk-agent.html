What "tool use" actually means

The single most important misconception to clear: the LLM doesn't actually call your code.

It returns a structured request — "I'd like to call search_docs with these arguments" — and your code chooses to execute it. Then your code returns the result to the LLM, which decides what to do next.

USER: "How do I rotate a Connected App secret?"
        │
        ▼
   ┌────────────────────────────────────────┐
   │ LLM (sees question + list of tools)    │
   │                                        │
   │ Output: function_call(                 │
   │   name="search_docs",                  │
   │   args={"query": "rotate connected   "│
   │             " app secret"})            │
   └─────────────────┬──────────────────────┘
                     │ NOT a real call —
                     │ just a request
                     ▼
   ┌────────────────────────────────────────┐
   │ YOUR CODE                              │
   │  - parses the function_call            │
   │  - dispatches to your search_docs()    │
   │  - captures the return value           │
   └─────────────────┬──────────────────────┘
                     │ result string
                     ▼
   ┌────────────────────────────────────────┐
   │ LLM (now sees the function_response)   │
   │                                        │
   │ Output: text                           │
   │  "Rotate it via the self-service       │
   │   portal at /admin/secrets [chunk 5]"  │
   └─────────────────┬──────────────────────┘
                     ▼
   FINAL ANSWER to user

That round-trip — request → execute → response → next decision — is the agent loop. Some questions need no tool call (the LLM answers directly). Some need one. Some need many in sequence. Your code's job is to keep the loop running until the LLM stops calling tools.

Why this design?
LLMs are hosted services. They can't reach into your laptop and run Python. So they describe what they'd like to do; you decide whether to do it. This separation is the safety boundary — your code stays in control of which tools exist, which arguments are accepted, and what data goes back.

The four ingredients of any agent

IngredientWhat it does
ToolsPlain Python functions you wrote — the agent's "actions"
Tool schemasDescriptions the LLM reads to know what each tool does and what arguments it takes
Agent loopThe orchestration: send → check function_calls → execute → send results back → repeat until done
System instructionTells the agent when to use which tool, when to answer directly, and when to refuse

Every agent framework in L17 — LangGraph, CrewAI, Pydantic AI, LlamaIndex — implements these four. They differ in how much they hide and how much control they give you. We'll write all four ourselves, then in L19 see how each framework shrinks the code.

Hands-on — build agent.py

Step 1 — Wrap your RAG pipeline as a tool

Your rag_chroma.py already has an ask() function. We'll wrap it as a regular Python function with a clear docstring. The docstring matters — Gemini reads it to decide when to call this tool.

# agent.py
from google import genai
from google.genai import types
from dotenv import load_dotenv
from rag_chroma import ask as rag_ask, get_collection

load_dotenv()
client = genai.Client()

# The tool: a regular Python function. Docstring is the LLM-facing description.
def search_docs(query: str) -> str:
    """Search the internal MuleSoft Anypoint documentation for an answer.
    Use this for any question about MuleSoft, Anypoint Platform, Connected
    Apps, OAuth, RAML, or related platform concepts.
    """
    coll, bm25, chunk_ids = get_collection()
    return rag_ask(query, coll, bm25, chunk_ids)
New syntax: type hints (query: str -> str)
query: str declares the parameter is a string. -> str declares the return type. Python doesn't enforce these at runtime — they're hints for tools and humans. The Gemini SDK reads them to build the tool schema, so they're load-bearing here, not decoration.

Step 3 — Register the tools and start a chat

The Gemini SDK can build the tool schema directly from a Python function — type hints + docstring is enough. Pass the functions in a list and the SDK handles the JSON-schema part.

SYSTEM_INSTRUCTION = """
You are a MuleSoft assistant. You have two tools:

- search_docs: for questions about MuleSoft, Anypoint Platform,
  Connected Apps, OAuth, RAML, or platform concepts.
- get_anypoint_org_status: for live operational status of a
  specific Anypoint org by ID.

Rules:
1. For documentation questions → use search_docs.
2. For "what's the status of org X" → use get_anypoint_org_status.
3. For unrelated questions (math, general knowledge) → answer directly
   without calling tools.
4. If you cannot help, say so — don't fabricate.
"""

TOOLS = [search_docs, get_anypoint_org_status]

chat = client.chats.create(
    model="gemini-3.5-flash",
    config=types.GenerateContentConfig(
        system_instruction=SYSTEM_INSTRUCTION,
        tools=TOOLS,
    ),
)
Automatic vs manual function calling
The Gemini SDK has an "automatic" mode where it executes tools for you — but doing that hides the agent loop. We're staying in manual mode (default when you handle function_calls yourself) precisely so you see what's happening. Frameworks pick automatic mode and add their own loop on top.

Step 4 — Write the agent loop

This is the heart of every agent ever written. Send the user's question. Check whether the LLM wants to call tools. If yes, execute and send results back. Repeat until the LLM is done calling tools and just returns text.

def run_agent(user_message: str, max_steps: int = 10) -> str:
    # Tool name → Python function dispatch table
    tool_map = {tool.__name__: tool for tool in TOOLS}

    response = chat.send_message(user_message)

    for step in range(max_steps):
        # Did the LLM ask to call any tools?
        function_calls = response.function_calls

        if not function_calls:
            # No tool calls → the LLM gave a final text answer
            return response.text

        # Execute every requested tool call and collect responses
        responses = []
        for call in function_calls:
            print(f"  → tool: {call.name}({dict(call.args)})")
            if call.name not in tool_map:
                result = f"unknown tool: {call.name}"
            else:
                result = tool_map[call.name](**dict(call.args))
            responses.append(types.Part.from_function_response(
                name=call.name,
                response={"result": result},
            ))

        # Send the tool results back to the LLM and let it continue
        response = chat.send_message(responses)

    return "agent stopped: hit max_steps without final answer"
New syntax: dict comprehension {x: y for ...}
{tool.__name__: tool for tool in TOOLS} builds a dict mapping each tool's function name to the function object. Same shape as a list comprehension, but with key: value pairs and curly braces instead of square brackets. Result: {"search_docs": <func>, "get_anypoint_org_status": <func>}.
New syntax: **dict(call.args) — splat unpacking on a call
**{"query": "rotate"} spreads the dict into keyword arguments — the call becomes tool_map["search_docs"](query="rotate"). The mirror of **kwargs from L15. Used here because call.args is a dict-like object and you want to pass each entry as a named argument.
Pattern: the bounded loop
for step in range(max_steps) with an early return inside is the safe way to write an agent loop. while True would also work but risks runaway: a buggy tool that always errors could keep the LLM calling tools forever. max_steps=10 is a hard ceiling — production agents typically use 10–30.

Step 5 — Run it on three queries

if __name__ == "__main__":
    questions = [
        "How do I rotate a Connected App secret?",
        "What's the status of org 12345?",
        "What's 2 + 2?",
    ]
    for q in questions:
        print(f"\nQ: {q}")
        answer = run_agent(q)
        print(f"A: {answer}")

Expected output (yours will vary slightly — LLMs are stochastic):

Q: How do I rotate a Connected App secret?
  → tool: search_docs({'query': 'rotate Connected App secret'})
A: Rotate it via the self-service portal at /admin/secrets [chunk 5].

Q: What's the status of org 12345?
  → tool: get_anypoint_org_status({'org_id': '12345'})
A: Org 12345 is healthy — all systems operational.

Q: What's 2 + 2?
A: 4.

Look at the third query — no tool call printed. The agent answered directly from its own knowledge because the system instruction told it to.

That's the win this lesson exists for: the agent chooses when to use tools.

What you just felt that frameworks remove

Look back at run_agent. Honest list of what's painful or missing:

Pain point in the native loopWhat frameworks add
Manual function_calls parsing and dispatch tableAuto tool routing (LangGraph, LlamaIndex)
Manual Part.from_function_response packagingHidden behind decorators (Pydantic AI's @agent.tool)
No retry logic — if a tool errors, you'll send a string error to the LLMBuilt-in retry, fallbacks, error policies (LangGraph)

System instruction craft — the hidden skill

The agent's quality is almost entirely shaped by the system instruction. Look back at ours:

You are a MuleSoft assistant. You have two tools:

- search_docs: for questions about MuleSoft, Anypoint Platform,
  Connected Apps, OAuth, RAML, or platform concepts.
- get_anypoint_org_status: for live operational status of a
  specific Anypoint org by ID.

Rules:
1. For documentation questions → use search_docs.
2. For "what's the status of org X" → use get_anypoint_org_status.
3. For unrelated questions (math, general knowledge) → answer directly.
4. If you cannot help, say so — don't fabricate.

Three principles that consistently work:

  1. Name each tool and what it's for — the docstring on the function is also seen, but reinforcing it in the system instruction reduces routing mistakes
  2. Explicit "when to use which" rules — vague guidance leads to the LLM guessing
  3. Refusal rule — without it the agent will fabricate; with it, it says "I cannot help with that" cleanly
When tool routing goes wrong
If the agent calls the wrong tool, the fix is almost always the system instruction or the tool docstring — not more code. Add a sentence: "For X, use Y. Do NOT use Z." Re-test.

Productionising — what to add next

The native loop above is a real working agent. To take it to production you'd want:

AdditionHow
Tool schema validationValidate call.args against a Pydantic model before dispatching
Per-tool error handlingWrap the tool dispatch in try/except; return a structured error string to the LLM so it can retry or refuse
Latency / cost captureApply the @measure decorator from L15 to each tool
Streaming responsesSwitch send_message → send_message_stream and yield chunks
Persistent memorySerialise chat.get_history() to disk/Redis between sessions. Gemini's chat object only lives in-process — see SDK comparison reference for how OpenAI/Anthropic handle this differently.
Provider portabilityIf you ever need to swap Gemini for another model (local Ollama for dev, Groq for latency tests, a different cloud), use the OpenAI-compatible API reference — same client code, only base_url changes. Trade-off: you lose Gemini's auto-schema-from-Python-fn ergonomics that this lesson relies on.
ObservabilityPipe traces to Phoenix or Logfire via OpenTelemetry (see L17)
Timeout per toolWrap tool calls in asyncio.wait_for(...) or use a thread-pool with timeout

Each of these is a small addition — but together they're 200 lines of boilerplate. That's the framework value proposition. When the cost of that boilerplate exceeds the cost of a framework dependency, switch frameworks. The eval (L14–L16) and your operational pain tell you when.

Quick check

Q1. When the LLM "calls a tool", what is actually happening?

  • The LLM reaches into your Python process and runs the function
  • Gemini's servers run your function for you
  • The LLM returns a structured request describing which tool and what arguments — your code chooses to execute it and returns the result
  • The function gets compiled to JavaScript and runs in the browser

Q2. Why use for step in range(max_steps) instead of while True for the agent loop?

  • while True doesn't work in Python
  • It's faster
  • Range loops are required by the Gemini SDK
  • Bounded steps prevent runaway loops — a buggy tool that always errors could otherwise keep the LLM calling tools forever

Q3. The LLM saw "What's 2 + 2?" and answered without calling any tool. Why?

  • The tools were broken
  • The system instruction told it to answer unrelated questions directly without tools, and arithmetic doesn't need search_docs or org_status
  • Gemini disabled tool calling for math
  • The agent is broken — it should always call a tool

Q4. Why are the type hints (query: str -> str) and the docstring on each tool function load-bearing, not decoration?

  • The Gemini SDK reads them to build the tool schema and tell the LLM what each tool does and what arguments it expects
  • Python enforces type hints at runtime
  • They speed up function calls
  • They're required for OAuth

Q5. The agent calls the wrong tool for a question. What's the first fix to try?

  • Add a try/except around the loop
  • Switch to a different LLM
  • Tighten the system instruction or tool docstring with explicit "when to use this / not this" rules
  • Rewrite in LangGraph
