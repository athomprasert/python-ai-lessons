The mission moment

MISSION.md says: build AI agents that interact with integration systems. In L18 you built one — bare native loop, no framework. It works. So why would you ever use a framework?

This lesson is the fair comparison. Five frameworks. Same agent. Same two tools. Same system instruction. You'll see exactly which boilerplate each one removes — and exactly which constraints it adds in return.

Source honesty
The code blocks below are realistic shapes based on each framework's official quickstarts, not a working repo I've personally run end-to-end for this lesson. The mental model is what to take away. Before adopting any of these in production, verify against the current docs — agent frameworks move fast and APIs shift between minor versions.

Quick reminder — what L18 looks like

Your baseline (~30 lines for the loop, plus 2 tool functions):

# L18 native — no framework
from google import genai
from google.genai import types

def search_docs(query: str) -> str: # wraps rag_chroma.ask() ...
def get_anypoint_org_status(org_id: str) -> dict: ...

TOOLS = [search_docs, get_anypoint_org_status]
client = genai.Client()
chat = client.chats.create(
    model="gemini-2.0-flash",
    config=types.GenerateContentConfig(
        system_instruction=SYSTEM,
        tools=TOOLS,
    ),
)

def run_agent(msg, max_steps=10):
    tool_map = {t.__name__: t for t in TOOLS}
    response = chat.send_message(msg)
    for _ in range(max_steps):
        if not response.function_calls:
            return response.text
        results = [
            types.Part.from_function_response(
                name=c.name,
                response={"result": tool_map[c.name](**dict(c.args))},
            )
            for c in response.function_calls
        ]
        response = chat.send_message(results)
    return "max_steps reached"

Keep this in mind. Every framework below collapses some of these 30 lines into "magic" — and surfaces them again somewhere else as configuration.

The five contenders, in order of "distance from L18"

  1. Google ADK — same SDK, more sugar. Smallest jump.
  2. LangChain + LangGraph — graph-based, explicit state machine.
  3. CrewAI — role/task/crew abstraction. Biggest reframing.
  4. LlamaIndex — retrieval-native; the agent is a special index.
  5. Haystack — pipeline-based; agent is a component in a flow.
FRAMEWORK 1 Google ADK Agent Development Kit · by Google

First-party Gemini agent kit. Looks like L18 with the loop hidden and sessions/eval/deploy bolted on.

# pip install google-adk
from google.adk.agents import Agent
from google.adk.runners import Runner
from google.adk.sessions import InMemorySessionService

def search_docs(query: str) -> dict:
    """Search MuleSoft docs and return an answer with citations."""
    return {"answer": rag_chroma.ask(query)}

def get_anypoint_org_status(org_id: str) -> dict:
    """Live Anypoint org status."""
    return {"org_id": org_id, "status": "healthy", "apps": 42}

agent = Agent(
    name="mulesoft_helper",
    model="gemini-2.0-flash",
    instruction=SYSTEM,
    tools=[search_docs, get_anypoint_org_status],
)
runner = Runner(agent=agent, session_service=InMemorySessionService())

async for event in runner.run_async(
    user_id="att", session_id="s1",
    new_message="How do I rotate a Connected App secret?",
):
    if event.is_final_response():
        print(event.content.parts[0].text)
What it removes
  • The manual for step in range(max_steps) loop
  • Session lifecycle (history, user, multi-session — pluggable storage)
  • Built-in evals + tracing + Vertex Agent Engine deploy path
What it costs
  • Async-first API — async for isn't optional
  • Tied to Google ecosystem (best with Vertex AI / Gemini)
  • Newer (2024) — smaller community than LangChain/LlamaIndex
Verdict: Pick this if you're already all-in on Gemini/Vertex and want the loop hidden without leaving Google's runtime. Best "smallest jump from L18."
FRAMEWORK 2 LangChain + LangGraph by LangChain Inc.

Two sibling projects from the same team. LangChain = components library (models, retrievers, prompts). LangGraph = the modern agent runtime (graph-based state machine).

Read this first
LangChain's own agent classes (AgentExecutor, create_react_agent) are deprecated in favour of LangGraph. The LangChain docs themselves redirect users to LangGraph for new agents. We show LangGraph as the apples-to-apples comparison; the LangChain legacy shape is below for reading old code. 
# pip install langgraph langchain-google-genai
from langchain_core.tools import tool
from langchain_google_genai import ChatGoogleGenerativeAI
from langgraph.prebuilt import create_react_agent

@tool
def search_docs(query: str) -> str:
    """Search MuleSoft docs and return an answer."""
    return rag_chroma.ask(query)

@tool
def get_anypoint_org_status(org_id: str) -> dict:
    """Live Anypoint org status."""
    return {"org_id": org_id, "status": "healthy"}

llm = ChatGoogleGenerativeAI(model="gemini-2.0-flash")
agent = create_react_agent(
    llm,
    tools=[search_docs, get_anypoint_org_status],
    state_modifier=SYSTEM,
)

result = agent.invoke({"messages": [("user",
    "How do I rotate a Connected App secret?")]})
print(result["messages"][-1].content)
New syntax · @tool decorator
@tool is a decorator (we covered @decorator in L15) that turns a plain Python function into a LangChain Tool object. The decorator reads the function's name, docstring, and type hints to build the schema — exactly what Gemini's native SDK does for you automatically. The difference: LangChain makes it explicit and portable across all the LLM providers LangChain supports.

If you want to write the graph yourself (for branching/loops/checkpointing):

from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import ToolNode, tools_condition

graph = StateGraph(MessagesState)
graph.add_node("llm", lambda s: {"messages": [llm_with_tools.invoke(s["messages"])]})
graph.add_node("tools", ToolNode([search_docs, get_anypoint_org_status]))
graph.add_edge(START, "llm")
graph.add_conditional_edges("llm", tools_condition)  # llm → tools or END
graph.add_edge("tools", "llm")
agent = graph.compile(checkpointer=MemorySaver())

Legacy LangChain shape (deprecated, for reading old code only):

from langchain.agents import AgentExecutor, create_tool_calling_agent
agent = create_tool_calling_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools)
executor.invoke({"input": "..."})  # works but on the way out
What it removes
  • Provider lock-in — same code talks to OpenAI, Anthropic, etc.
  • Checkpointing / persistence (MemorySaver, Postgres, Redis)
  • Massive integration library (200+ retrievers, vector stores, loaders)
What it costs
  • Two-library mental model (LangChain ⊕ LangGraph) — easy to mix up
  • Frequent breaking changes; tutorials go stale quickly
  • Stack traces deep into LangChain internals when things go wrong
Verdict: Pick this if you need provider portability or want production-grade persistence/checkpointing baked in. Use LangGraph for any new agent — not the deprecated AgentExecutor.
FRAMEWORK 3 CrewAI by CrewAI Inc.

Models the world as roles doing tasks, organised into a crew. Best when you genuinely have multiple agents collaborating.

# pip install crewai
from crewai import Agent, Task, Crew
from crewai.tools import tool

@tool("search_docs")
def search_docs(query: str) -> str:
    """Search MuleSoft docs and return an answer."""
    return rag_chroma.ask(query)

@tool("get_anypoint_org_status")
def get_anypoint_org_status(org_id: str) -> dict:
    """Live Anypoint org status."""
    return {"org_id": org_id, "status": "healthy"}

helper = Agent(
    role="MuleSoft platform assistant",
    goal="Answer customer questions about MuleSoft accurately",
    backstory="Senior MuleSoft architect with deep product knowledge.",
    tools=[search_docs, get_anypoint_org_status],
    llm="gemini/gemini-2.0-flash",
)

task = Task(
    description="Answer: How do I rotate a Connected App secret?",
    expected_output="A clear step-by-step answer with a doc link.",
    agent=helper,
)

crew = Crew(agents=[helper], tasks=[task])
result = crew.kickoff()
print(result.raw)
Notice the prompt is split
In L18 you wrote one system_instruction. In CrewAI it's distributed across role, goal, backstory, and the task's description + expected_output. This is great for multi-agent crews where each agent has a distinct persona — and overkill for a single helper bot. See CrewAI production notes for the gotchas.
What it removes
  • Multi-agent orchestration boilerplate (sequential, hierarchical, parallel)
  • Role-based prompting structure — nudges you to think about who does what
  • Built-in delegation between agents
What it costs
  • The "role/task/crew" frame is heavy for a single agent
  • Less control over the loop — harder to inject custom retry/fallback logic
  • Production maturity still catching up to LangGraph (see ref doc)
Verdict: Pick this when you genuinely have multiple specialised agents collaborating (researcher + writer + reviewer). Overkill for a single helper. CrewAI production notes →
FRAMEWORK 4 LlamaIndex by LlamaIndex Inc.

Retrieval-native. Built around the idea that agents query indexes. Best when your agent's primary job is querying lots of data.

# pip install llama-index llama-index-llms-google-genai
from llama_index.core.agent.workflow import FunctionAgent
from llama_index.core.tools import FunctionTool
from llama_index.llms.google_genai import GoogleGenAI

def search_docs(query: str) -> str:
    """Search MuleSoft docs and return an answer."""
    return rag_chroma.ask(query)

def get_anypoint_org_status(org_id: str) -> dict:
    """Live Anypoint org status."""
    return {"org_id": org_id, "status": "healthy"}

tools = [
    FunctionTool.from_defaults(fn=search_docs),
    FunctionTool.from_defaults(fn=get_anypoint_org_status),
]

agent = FunctionAgent(
    tools=tools,
    llm=GoogleGenAI(model="gemini-2.0-flash"),
    system_prompt=SYSTEM,
)

response = await agent.run(
    user_msg="How do I rotate a Connected App secret?"
)
print(str(response))
What it removes
  • Best-in-class RAG primitives (your search_docs can be replaced by a native LlamaIndex query engine)
  • Advanced retrieval patterns (sub-question, recursive, auto-merging) — see L13
  • Async-by-default and built-in workflow visualisation
What it costs
  • "llama-index-core" vs satellite packages — dependency tree is sprawling
  • If your agent isn't retrieval-heavy, you're paying for unused machinery
  • Two agent APIs (legacy ReActAgent + new FunctionAgent) — pick the new one
Verdict: Pick this if your agent's main job is querying lots of documents/indexes and you want their retrieval research baked in. LlamaIndex production notes →
FRAMEWORK 5 Haystack by deepset

Everything is a pipeline of components. Agent is one component you plug into a flow with retrievers, rankers, validators.

# pip install haystack-ai google-genai-haystack
from haystack.components.agents import Agent
from haystack.tools import tool
from haystack_integrations.components.generators.google_genai import GoogleGenAIChatGenerator
from haystack.dataclasses import ChatMessage

@tool
def search_docs(query: str) -> str:
    """Search MuleSoft docs and return an answer."""
    return rag_chroma.ask(query)

@tool
def get_anypoint_org_status(org_id: str) -> dict:
    """Live Anypoint org status."""
    return {"org_id": org_id, "status": "healthy"}

agent = Agent(
    chat_generator=GoogleGenAIChatGenerator(model="gemini-2.0-flash"),
    tools=[search_docs, get_anypoint_org_status],
    system_prompt=SYSTEM,
)
agent.warm_up()

result = agent.run(messages=[
    ChatMessage.from_user("How do I rotate a Connected App secret?"),
])
print(result["messages"][-1].text)

In real Haystack code, the Agent rarely stands alone — you wrap it in a Pipeline next to a retriever, ranker, and validator:

from haystack import Pipeline
pipe = Pipeline()
pipe.add_component("retriever", ChromaEmbeddingRetriever(...))
pipe.add_component("agent", agent)
pipe.connect("retriever.documents", "agent.documents")
What it removes
  • Composing search → rerank → agent → validate as one declarative pipeline
  • Strong typing on component inputs/outputs — surfaces wiring errors early
  • Mature on document workflows (deepset's heritage is enterprise search)
What it costs
  • Pipeline-first thinking is heavy if you're not also doing search/rerank
  • Smaller agent-specific community than LangGraph / LlamaIndex
  • Component-input wiring (pipe.connect(...)) has a learning curve
Verdict: Pick this if you're building an enterprise document workflow — searches, rankers, agents, validators all wired as one production pipeline. Less compelling for "just an agent".

The MuleSoft-integration angle

For your day job
  • Customer demo / Salesforce stack alignment — Google ADK fits because Salesforce+MuleSoft is increasingly Gemini-aligned via Agentforce. One Google story.
  • Internal "ask MuleSoft docs" assistant — LangGraph or LlamaIndex. Both have battle-tested retrieval. LangGraph wins on persistence.
  • Multi-step integration ops bot (look up org → check apps → propose fix → ask approval) — LangGraph for graph control, or CrewAI if the steps map to distinct personas.
  • Document-extraction-style enterprise pipeline (like the IDP frontend you're building) — Haystack is genuinely the strongest fit here.

The honest answer — when to stay native (no framework)

None of these frameworks is automatically the right choice. Stay on the L18 native loop when:

  • You're one developer, single agent, < 100 lines today. The framework lift will outweigh the boilerplate it removes.
  • You only ever target Gemini. Provider portability is the #1 framework win; if you don't need it, you don't need most of LangChain.
  • You're still figuring out the right shape. Frameworks shape your design. Lock-in too early and you'll fight the framework when the design needs to change.
  • Debugging matters more than features. A 30-line loop you wrote is easier to step through than a framework's call stack.
  • You're learning. The L18 loop teaches you what every framework is doing under the hood. Frameworks hide what's still worth knowing.
The trigger to switch: when you find yourself building one of the framework features by hand — persistence, checkpointing, retries, multi-agent handoff, schema validation, eval harness — that's the moment. Not before.
