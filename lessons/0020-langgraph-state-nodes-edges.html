The mission moment

MISSION.md says: build AI agents that interact with integration systems. You already did that once in L18 with the bare Gemini loop. LangGraph is the framework you picked in L19 to take that agent into production — checkpointing, multi-provider portability, graph-based control flow.

But before any LLM appears: LangGraph is a graph library that happens to run LLMs at its nodes. If you don't understand the graph part, the LLM part will feel like magic and you won't be able to debug it. So this lesson is pure graph theory — three primitives, two exercises, zero AI.

The three primitives

PrimitiveWhat it isMuleSoft analogy
State A typed dictionary that flows through the graph. Every node reads from it and returns updates to it. The Payload object that flows through a Mule flow — every component reads and writes to it.
Node A Python function that takes the current state and returns a partial update to it. A Mule processor (Transform, HTTP request, Logger) — does one thing to the payload.
Edge A connection saying "after node A runs, go to node B." Can be unconditional, conditional, or terminal. The arrow between components in Anypoint Studio — including choice routers for conditional flow.

The lifecycle

START │ ▼ ┌─────────┐ reads state, │ Node A │ ─→ returns {"key": new_value} └────┬────┘ LangGraph merges into state │ ▼ ┌─────────┐ │ Node B │ ─→ returns more updates └────┬────┘ │ ▼ END
Why a graph and not just functions called in sequence?
Three reasons: (1) the graph lets you branch based on state — e.g., "if the LLM produced a tool call, go to the tool node; otherwise end" — without if/else littered through your code. (2) The graph lets you checkpoint at every node automatically (L22). (3) The graph lets the runtime stream events from every node for observability (L23). All three become free once you express your program as a graph.

The smallest possible LangGraph

Before you write code, read this carefully — every line introduces something new:

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

# 1. Define the state shape
class MyState(TypedDict):
    counter: int

# 2. Define nodes — functions that take state, return partial updates
def increment(state: MyState) -> dict:
    return {"counter": state["counter"] + 1}

# 3. Wire the graph
graph = StateGraph(MyState)
graph.add_node("inc", increment)
graph.add_edge(START, "inc")
graph.add_edge("inc", END)

# 4. Compile and run
app = graph.compile()
result = app.invoke({"counter": 10})
print(result)  # {'counter': 11}
New syntax · TypedDict
TypedDict is a way to declare what keys a dictionary should have and what types those values should be. It's not enforced at runtime (Python won't raise an error if you pass the wrong type), but type checkers (mypy, Pyright) and editors (VSCode) will warn you. Looks like a class, behaves like a dict. 
from typing import TypedDict

class User(TypedDict):
    name: str
    age: int

u: User = {"name": "Att", "age": 40}  # a normal dict
print(u["name"])  # "Att" — exactly like a dict
LangGraph requires TypedDict (or a Pydantic BaseModel) for state so it can validate node returns and generate type-aware tracing.

Three rules of LangGraph nodes

  1. A node takes the entire state as input, even if it only reads one field.
  2. A node returns a partial dict — only the keys it wants to update. LangGraph merges (shallow) into the full state.
  3. If a node returns None or {}, state is unchanged but the graph still advances along its edges.
Common beginner mistake
Do not mutate the state dict in-place inside a node. Always return a new dict. Mutation will sometimes work and sometimes silently fail (depending on which reducer is configured for that field — more on reducers in L21). Treat state as immutable from the node's perspective.

Exercise 2 — stretch

EXERCISE 2 · STRETCH

Branch on even or odd

Goal

A graph that takes a number, and:

  • If even: divides by 2 and ends.
  • If odd: adds 1 to make it even, divides by 2, ends.

So 10 → 5. 7 → 4 (7+1=8, 8/2=4).

The new primitive you need

A conditional edge. Instead of "always go to node B", a conditional edge runs a router function that returns the name of the next node.

def route_by_parity(state: NumberState) -> str:
    # Must return the string name of the next node
    return "even_branch" if state["value"] % 2 == 0 else "odd_branch"

graph.add_conditional_edges("check_parity", route_by_parity)

The router runs after the named node ("check_parity" here) finishes. Whatever string the router returns is the next node LangGraph runs.

Your task

Create l20_ex2_branching.py. Design a graph that uses one conditional edge to fork between an even branch (divide by 2) and an odd branch (add 1, then divide by 2). Test with at least two inputs — one even, one odd.

No skeleton this time. You decide the node names, you decide the shape. The only requirements:

  • At least 3 nodes (you can use more)
  • Exactly one add_conditional_edges call
  • Both branches must converge somehow (either both go to END, or both go to a final shared node before END)
Verify

app.invoke({"value": 10}) should give {'value': 5}. app.invoke({"value": 7}) should give {'value': 4}.

Mental models to take into L21

"A node is a pure function from state to partial state." Same idea as a Mule Transform component: read payload, return payload. The graph framework does the wiring; you only write the transforms.
"Edges are choices, not arrows." An unconditional edge says "always go here next." A conditional edge says "ask this router where to go." This is what makes LangGraph a graph rather than a sequence — and it's exactly what L21 will use for tool calling ("if the LLM produced a tool call, go to the tool node; otherwise end").
MuleSoft analogue you've seen before
add_conditional_edges is the Mule <choice> router. The function inside is the condition expression; the strings it can return are the route labels. Once you see this, every LangGraph diagram on the internet stops looking like magic.

Quick check

Q1. A LangGraph node returns {"counter": 5} when the current state is {"counter": 1, "name": "Att"}. What does the state become after the node runs?

  • {"counter": 5} — the return value replaces the state
  • {"counter": 1, "name": "Att"} — state is immutable
  • {"counter": 5, "name": "Att"} — return is merged into state, only updated keys overwritten
  • The graph crashes because the node didn't return name

Q2. What does a function passed to add_conditional_edges return?

  • A partial state update, same as a node
  • A string — the name of the next node to run
  • A boolean — True to continue, False to stop
  • A new graph object

Q3. Why does LangGraph require state to be a TypedDict (or Pydantic model) rather than a plain dict?

  • Plain dicts are slower
  • Python forbids passing plain dicts to graph libraries
  • It enforces the types at runtime — wrong types will raise TypeError
  • It declares the schema so LangGraph can validate node returns, generate type-aware tracing, and editors can autocomplete the keys. (TypedDict is not enforced at runtime — type checkers warn, the program runs.)
