The LLM node and tools — making the graph think

We re-build the L18 agent as a LangGraph state machine: MessagesState, ToolNode, tools_condition, and the agent loop.

Pedagogy reminder. You write the code in your editor. I'll show a skeleton with TODO markers and let you struggle a bit. Paste your attempt back; I review; then we see the canonical answer.

Where we are

In L20 you wrote graphs with no LLM — pure state, nodes, edges. You now know:

  • State is a TypedDict; nodes return partial dicts that get merged
  • Edges are unconditional (add_edge) or conditional (add_conditional_edges + a router function)
  • Multiple unconditional edges from one node = fan-out (parallel execution)
  • Without a reducer, parallel writes to the same key race and clobber each other

L21 puts an LLM inside a node. The LLM decides whether to answer directly or call a tool. The graph routes accordingly. When the tool returns, control comes back to the LLM. That loop — agent ↔ tools ↔ agent ↔ ... ↔ END — is the heart of every modern agentic system.

The agent loop in one picture

┌─────────┐ │ START │ └────┬────┘ ▼ ┌──────────────┐ │ agent │◄────────┐ │ (LLM call) │ │ └──────┬───────┘ │ │ │ tools_condition │ │ │ ┌───────┴────────┐ │ ▼ ▼ │ ┌─────────┐ ┌─────────┐ │ │ END │ │ tools │───┘ └─────────┘ └─────────┘ (final answer) (run tool, return result)

Two nodes (agent, tools) and one conditional edge. That's it. The LangGraph team ships prebuilt components for both — you just wire them.

Why MessagesState exists

In L20 your state was {"value": int}. For a chat agent, state is a list of messages that grows on every turn. You could write it yourself:

class ChatState(TypedDict):
    messages: list  # human msgs + AI msgs + tool calls + tool results

But the L20 fan-out problem bites hard here. If two tools run in parallel and each writes {"messages": [tool_result]}, the second write would replace the list — first tool result lost.

The fix is a reducer — a function LangGraph uses to combine writes instead of overwriting. For messages, the right reducer is "append the new ones to the list," which LangGraph ships as add_messages:

New syntax — Annotated[type, metadata]

From typing. It attaches metadata to a type without changing the type itself. LangGraph reads the metadata to find your reducer.

from typing import Annotated
from langgraph.graph.message import add_messages

class ChatState(TypedDict):
    messages: Annotated[list, add_messages]
    #         ^^^^         ^^^^^^^^^^^^
    #         actual type   reducer (metadata)

Translation: "messages is a list. When two nodes both write to it, run add_messages to merge their writes instead of last-write-wins."

The shortcut: LangGraph ships this exact TypedDict as MessagesState. Use it directly:

from langgraph.graph import MessagesState

# MessagesState IS: {"messages": Annotated[list, add_messages]}
# Use as-is for chat agents, or subclass to add fields.

The LLM node — what goes inside

A node is just a function that takes state and returns a partial dict. For an LLM node, the body is "send the messages so far to the model, return whatever it produced":

from langchain_google_genai import ChatGoogleGenerativeAI

llm = ChatGoogleGenerativeAI(model="gemini-2.5-flash")

def agent_node(state: MessagesState) -> dict:
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

Note the return shape: {"messages": [response]}, not {"messages": response}. The reducer expects a list of messages to append. A single message wrapped in a list is one of those.

Heads-up — LangChain vs native Gemini SDK

In L18 you used google.genai directly. Here we use ChatGoogleGenerativeAI from langchain-google-genai — a thin wrapper that gives Gemini a LangChain message interface so it fits in graphs. Same underlying model, different API surface. See SDK comparison reference for the cheat sheet.

Tools — defined with @tool

New syntax — the @tool decorator

A decorator is a function applied to another function with @name above the def. It wraps your function with extra behaviour. @tool turns a plain Python function into a LangChain-callable tool by inspecting its signature and docstring.

from langchain_core.tools import tool

@tool
def get_anypoint_org_status(org_id: str) -> str:
    """Return health status for a MuleSoft Anypoint org.

    Args:
        org_id: the organization identifier.
    """
    return f"Org {org_id}: 12 APIs healthy, 0 alerts."

The model sees: function name, parameter names + types, the docstring. It uses all three to decide whether and how to call the tool. Write a good docstring — that's the prompt the model reads.

The two prebuilts that wire it together

PrebuiltWhat it does
ToolNode([tools...]) A node that reads the last AI message, sees what tool(s) it asked for, runs them in parallel, returns {"messages": [tool_results...]}. Saves you ~30 lines of dispatch logic.
tools_condition A router function. Returns "tools" if the last AI message contains tool calls, otherwise END. Plug it into add_conditional_edges.

The full wiring looks like this (we're going to leave one piece out for you to fill in):

from langgraph.graph import StateGraph, START, END, MessagesState
from langgraph.prebuilt import ToolNode, tools_condition

graph = StateGraph(MessagesState)
graph.add_node("agent", agent_node)
graph.add_node("tools", ToolNode([tool1, tool2]))

graph.add_edge(START, "agent")
graph.add_conditional_edges("agent", tools_condition)  # "tools" or END
graph.add_edge("tools", "agent")                       # loop back

app = graph.compile()

That add_edge("tools", "agent") is what makes it a loop. After tools run, control returns to the LLM, which sees the tool results in state["messages"] and decides what to do next — call another tool or answer.

Hint 1 — what does .bind_tools(tools) do?

It returns a new LLM object that knows about the tool schemas. When you invoke it, the LLM can produce tool-call messages. Without .bind_tools(), the model has no idea the tools exist.

llm = ChatGoogleGenerativeAI(model="gemini-2.5-flash")
llm_with_tools = llm.bind_tools(tools)
Hint 2 — what does the agent node return?

llm_with_tools.invoke(state["messages"]) returns a single AIMessage. Wrap it in a list so the reducer can append it:

def agent_node(state):
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}
Hint 3 — the conditional edge

tools_condition is imported from langgraph.prebuilt. Pass it as the second argument to add_conditional_edges. It returns either "tools" or END, so no path_map is needed — it already uses the right names.

graph.add_conditional_edges("agent", tools_condition)

Question to chew on: if add_messages weren't the reducer — if MessagesState used last-write-wins — what would go wrong when both tool results land?

Quiz

1. Why does MessagesState use Annotated[list, add_messages] instead of plain list?

Answer

The add_messages reducer makes parallel writes append instead of overwrite. Without it, two tools running in parallel would each emit {"messages": [result]} and the second write would clobber the first. With the reducer, both results are appended to the existing list.

2. The model sees three things about each tool. What are they?

Answer

Function name, parameter names with type hints, and the docstring. That's the full prompt the LLM uses to decide whether and how to call the tool. Bad docstring = bad tool use.

3. Why is the edge add_edge("tools", "agent") not add_edge("tools", END)?

Answer

After a tool runs, the LLM needs to see the result and decide what to do next — answer, call another tool, or both. Going straight to END would leave the user with a tool result instead of a natural-language reply. The loop back to agent is what turns this into an agentic system.

What you now know

  • MessagesState is just TypedDict + the add_messages reducer
  • An LLM node calls llm.invoke(state["messages"]) and returns {"messages": [response]}
  • Tools are plain Python functions decorated with @tool — docstring and type hints are the prompt
  • .bind_tools(tools) tells the LLM the tools exist; ToolNode dispatches them
  • tools_condition is the prebuilt router for the agent loop
  • The loop back tools → agent is what makes this agentic
