Checkpointing & time travel — the graph remembers

A checkpointer turns your graph from a one-shot function into a resumable, debuggable, branchable conversation engine.

Pedagogy reminder. Same rules as L20 and L21 — TODO skeleton, you write, paste back, then canonical.

The problem this solves

Your L21 agent works, but it has amnesia. Run this:

app.invoke({"messages": [("user", "Is org acme-prod healthy?")]})
app.invoke({"messages": [("user", "What about prod-eu?")]})

The second call has no idea the first one happened. You'd have to manually accumulate messages and pass them in every time. Worse:

  • If the agent crashes mid-conversation, the state is gone
  • You can't say "rerun from before that bad tool call" — there's nothing to rerun from
  • You can't fork into "what if I'd given a different system prompt?" without starting over

A checkpointer fixes all three. It saves the full graph state after every super-step. Same graph, new superpower.

The mental model

Think of it as Git for graph state.

Git conceptLangGraph equivalent
RepositoryCheckpointer (where states are stored)
Branch namethread_id — names a conversation
Commit SHAcheckpoint_id — names a specific point in time
git logapp.get_state_history(config)
git checkout <sha>Invoke with that checkpoint_id in config

Every node execution becomes a commit. The thread is the linear (or branching) history.

Two checkpointers you'll meet

CheckpointerWhere state livesWhen to use
InMemorySaver Python dict in process memory Dev, tests, notebooks. State dies when the process dies.
SqliteSaver SQLite file on disk Real persistence with zero ops. Survives restarts. Perfect for single-user agents.
PostgresSaver (not in this lesson) Postgres database Production, multi-user. Same API as the others.
Heads-up — the import paths split into separate packages.
  • from langgraph.checkpoint.memory import InMemorySaver — ships with langgraph
  • from langgraph.checkpoint.sqlite import SqliteSaver — separate package: pip install langgraph-checkpoint-sqlite

(Older docs you'll find online use MemorySaver — that name still works as an alias, but InMemorySaver is the current preferred name.)

Plugging it in

Two changes to L21's code. The graph itself doesn't change at all — only how you compile and invoke it.

Change 1 — pass the checkpointer at compile time

from langgraph.checkpoint.memory import InMemorySaver

memory = InMemorySaver()
app = graph.compile(checkpointer=memory)

Change 2 — pass a thread_id on every invoke

config = {"configurable": {"thread_id": "conversation-1"}}

app.invoke({"messages": [("user", "hello")]}, config=config)
app.invoke({"messages": [("user", "do you remember me?")]}, config=config)
# ↑ Same thread_id → second call sees the first call's messages.
New syntax — the nested-dict config

That {"configurable": {"thread_id": ...}} shape looks weird, but it's a deliberate convention. LangGraph (actually LangChain underneath) accepts a RunnableConfig dict with several reserved keys:

  • "configurable" — your runtime settings (thread_id, checkpoint_id, user-id, model overrides, etc.)
  • "tags" — labels for observability tools (you'll see these in L23)
  • "callbacks" — hooks for tracing

Everything app-specific goes inside configurable so it doesn't collide with framework keys.

What gets saved — inspect it

After invoking, you can ask the app what's in memory:

state = app.get_state(config)
print(state.values)        # the full state dict ({"messages": [...]})
print(state.next)          # tuple of nodes that WOULD run next (empty if done)
print(state.config)        # includes the current checkpoint_id

And every step in the conversation is a separate checkpoint:

for snapshot in app.get_state_history(config):
    print(snapshot.config["configurable"]["checkpoint_id"], "|", len(snapshot.values["messages"]), "messages")

Newest first. Each one has a unique checkpoint_id you can use to time-travel.

The two killer moves

Move 1 — Resume after restart

With SqliteSaver, state lives on disk. You can quit Python entirely, start a new process, point it at the same SQLite file with the same thread_id, and continue the conversation. This is the move that takes you from "demo" to "real agent."

Move 2 — Time travel + branching

Pick any past checkpoint_id and pass it in config. The graph rewinds to that state. If you then call app.invoke() with a new input, you've forked — a new branch grows from that point.

old_config = {"configurable": {
    "thread_id": "conversation-1",
    "checkpoint_id": "",
}}
# Run from there with different input — creates a new branch
app.invoke({"messages": [("user", "alternate question")]}, config=old_config)

Why this matters for debugging: your agent did something weird at message 7. Instead of replaying the whole conversation, rewind to message 6, change the system prompt or the tool's response, and watch what the agent does differently. Like a debugger that rewinds.

Resuming a paused graph — invoking with None

One subtle but important move: if a graph paused mid-way (we'll do this on purpose in L23 with interrupts), you resume it by invoking with None as the input:

app.invoke(None, config=config)  # continue from the last checkpoint, no new input

This is different from app.invoke({"messages": [...]}, config=config), which adds new input to the conversation. Knowing the difference will save you an hour someday.

Predict before you run — what value does run 2 of thread A print?

If you predicted {"value": 1} (each invoke is fresh because you passed {"value": 0}): wrong.

If you predicted {"value": 2} (state persists across invokes on the same thread, the new input merges into existing state): right.

Here's the subtle bit: when you pass {"value": 0} to a thread that already has {"value": 1}, the new input is treated as a state update from START. With no reducer on value, it's last-write-wins, so value gets reset to 0 momentarily — then increment runs and produces {"value": 1}. So run 2 prints {"value": 1}.

Confused? That's the lesson: without a reducer, "fresh input + existing state" is murky. With MessagesState (and its add_messages reducer), new messages append to existing ones — which is exactly what you want for chat. That's why Exercise 2 uses messages, not counters.

Hint — the config dict 
config_a = {"configurable": {"thread_id": "thread-A"}}
config_b = {"configurable": {"thread_id": "thread-B"}}

Verification

  • Thread A run 1 → {"value": 1}
  • Thread A runs 2 and 3 → also {"value": 1} each (see prediction above)
  • Thread B run 1 → {"value": 1} (fresh state — different thread)
  • History of thread A → 3+ snapshots, each with its own checkpoint_id

Exercise 2 · guided · the real win

Multi-turn memory for the L21 agent — with SqliteSaver

Take your l21_ex1_mulesoft_agent.py from L21, copy it to l22_ex2_persistent_agent.py, and add persistent checkpointing. Skeleton below.

Goals

  1. Use SqliteSaver instead of InMemorySaver. Save state to ./agent_state.db.
  2. Have a 2-turn conversation on the same thread_id:
    • Turn 1: "Is org acme-prod healthy?"
    • Turn 2: "What about prod-eu?" — the agent should infer "what about" refers to org status from turn 1's context
  3. Print the message count after each turn. Expect ~4 after turn 1, ~7–8 after turn 2 (memory worked = total grew).

Skeleton — fill in the TODOs

Hint 1 — opening the SqliteSaver context 
with SqliteSaver.from_conn_string("agent_state.db") as memory:
    # everything goes here

The with syntax is Python's context manager — it auto-opens the DB connection at the start of the block and closes it cleanly at the end (even on a crash). Indentation matters: everything indented under with runs inside the context.

Hint 2 — invoking with the same thread_id

Reuse the same config dict on both calls. That's what makes them the same thread.

result1 = app.invoke(
    {"messages": [("user", "Is org acme-prod healthy?")]},
    config=config,
)
Hint 3 — verifying memory worked

Watch the message count. After turn 1 expect ~4 messages (human, AI-with-tool-call, tool-result, AI-final). After turn 2 you should see ~7–8 total — meaning turn 2 added to the existing list, not replaced it.

If turn 2 shows Message count: 4 (instead of 7–8), memory isn't working — the checkpointer or thread_id is misconfigured.

Cross-restart persistence test — the real win

After the script works:

  1. Comment out turn 1 entirely
  2. Quit Python, restart, run the script again (turn 2 only)
  3. If turn 2 still answers contextually — knowing "what about" refers to org status — the SQLite file remembered across processes. That's production-grade.

Security question to chew on

If thread_id is just a string the caller chooses, what stops a malicious user from passing someone else's thread_id and reading their conversation? (Hint: think about what thread_id is — and isn't.)

Quiz

1. Why is thread_id not the same as user_id?

Answer

A thread_id identifies one conversation. One user can have many threads (parallel conversations, different chat windows). A user_id identifies a person. In production you typically derive thread_id from user_id + some scope (project, session, chat tab), and you validate that the requester owns the thread before serving it. thread_id alone is not auth.

2. app.invoke(None, config=config) vs app.invoke({"messages": [...]}, config=config) — what's the difference?

Answer

None means "resume from the last checkpoint, no new input." Used after a graph paused (e.g., waiting for human approval).

Passing an input dict means "apply this as a state update from START, then run." For chat, this adds the new user message and the agent processes it.

3. Why does the graph topology stay identical when you add a checkpointer?

Answer

Checkpointing is orthogonal to graph structure — it's a side effect attached at compile time. Every super-step, LangGraph serialises the state and saves it. The nodes and edges don't know or care. This is why you can develop a graph without a checkpointer, then add one in production without touching graph code.

What you now know

  • A checkpointer + thread_id = persistent, resumable conversation
  • InMemorySaver for dev, SqliteSaver for real persistence
  • State, config, and history are inspectable: get_state, get_state_history
  • Time travel: pass a past checkpoint_id to rewind; invoke from there to fork
  • Graph topology doesn't change when you add a checkpointer — it's orthogonal
