Streaming & observability — making the agent visible

Your agent works. Now make it watchable. Stream output as it happens; trace every node call in a UI; debug in seconds instead of minutes.

Part 1 — Streaming

Until now you've called app.invoke(...), which blocks until the whole graph finishes. app.stream(...) instead returns an iterator that yields intermediate updates as they happen. You loop over it.

New syntax — generators / iterators

An iterator is an object you walk through with for ... in. Each iteration produces one value. Unlike a list, the iterator computes values on demand — perfect for "give me the next event when it's ready."

for event in app.stream(input, config=config):
    print(event)
    # prints each update AS the graph runs, not after

You'll see this same shape elsewhere: open(file), range-loops, file-reading-line-by-line. Python generators are first-class.

The four stream_mode options

The same stream() call can emit different shapes of event depending on the stream_mode argument. Each is useful for a different job.

stream_modeWhat it yieldsUse when
"updates" The dict each node returns, one event per node execution Debugging — see the agent loop unfold step by step
"values" The full state after each node, every time You want to see the cumulative state at each step
"messages" LLM tokens as the model generates them + tool call/result events End-user UX — the typewriter effect
"debug" Everything: node entry, exit, errors, internal events Deep debugging — you want it all

What "updates" looks like

for event in app.stream(input, config=config, stream_mode="updates"):
    print(event)

For your L22 agent answering "Is org acme-prod healthy?", you'd see roughly:

{"agent": {"messages": [AIMessage(content="", tool_calls=[...])]}}   # agent decided to call a tool
{"tools": {"messages": [ToolMessage(content="[mock] Org acme-prod: ...")]}}  # tool ran
{"agent": {"messages": [AIMessage(content="Yes, acme-prod is healthy...")]}}   # agent answered

Each dict's top-level key is the node name. The value is what that node returned. Three events = three super-steps. You can see the loop.

What "messages" looks like — the typewriter effect

for msg_chunk, metadata in app.stream(input, config=config, stream_mode="messages"):
    if msg_chunk.content:
        print(msg_chunk.content, end="", flush=True)

Each iteration is a tuple of (message_chunk, metadata). The chunk is one fragment of an AI message — usually a few tokens. The metadata tells you which node produced it. end="" + flush=True together create the streaming-typing-into-terminal effect.

Common gotcha. stream_mode="messages" yields chunks per token, but tool-call chunks are empty strings until the model finishes constructing the tool call. So if msg_chunk.content: filters out the empty noise.

Part 2 — Observability

Streaming shows you what's happening right now. Tracing records what happened across every run so you can debug retrospectively, compare runs side by side, and measure cost.

Two production-grade options. Both let you click into any past run and see:

  • Every node that executed
  • Each LLM call's exact prompt, response, latency, token count, $ cost
  • Each tool call's input and output
  • The full message tree
  • Errors with full stack traces

LangSmith — the 3-env-var trick

# Add to your .env:
LANGSMITH_TRACING=true
LANGSMITH_API_KEY=your_key_from_smith.langchain.com
LANGSMITH_PROJECT=mulesoft-agent-dev

That's the entire integration. LangGraph reads those env vars at startup; every app.invoke() and app.stream() from then on streams a trace to LangSmith. Open smith.langchain.com and your runs appear in real time.

No code changes. No SDK imports. Just env vars. This is what "instrumentation as a cross-cutting concern" looks like done right.

Phoenix — the local Docker move

# Run Phoenix locally (one command):
docker run -p 6006:6006 arizephoenix/phoenix:latest
# In your Python code:
from phoenix.otel import register

tracer_provider = register(
    project_name="mulesoft-agent-dev",
    endpoint="http://localhost:6006/v1/traces",
    auto_instrument=True,  # auto-detects LangChain / LangGraph
)

Open http://localhost:6006 in your browser. Same UI shape as LangSmith but you own the data.

Quiz

1. What's the difference between stream_mode="updates" and stream_mode="values"?

Answer

"updates" yields only the diff each node returned — useful for "what did this step contribute?". "values" yields the full state after each step — useful for "what does the world look like now?".

For a graph with 3 super-steps, "updates" emits 3 small dicts; "values" emits 3 cumulatively larger ones.

2. Why does app.stream() not require any change to the graph topology?

Answer

Streaming is orthogonal to graph structure — it's just a different way to iterate the same execution. Same nodes, same edges, same state. The only difference is the caller's API: you loop over results instead of getting one final dict. Same orthogonality principle as checkpointing in L22.

3. Your customer has a strict no-data-leaves-network policy. Which tracer do you reach for?

Answer

Phoenix. You run it in their VPC; traces never leave their network. LangSmith is hosted by LangChain Inc. — fine for your own dev, problematic for a regulated customer's production data.

Real-world MuleSoft customers in finance/healthcare almost always hit this constraint. Knowing the answer up front saves a procurement cycle.

What you now know

  • app.stream() turns invocation into a generator — events arrive as they happen
  • Four stream_mode options: updates, values, messages, debug — pick the right shape for the job
  • Tracing attaches via env vars (LangSmith) or one register() call (Phoenix) — zero graph code changes
  • Streaming = end-user UX; tracing = developer UX. Both orthogonal to graph topology.
  • For MuleSoft customer demos, default to Phoenix; for dev velocity, LangSmith
