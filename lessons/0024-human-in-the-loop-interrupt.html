Human-in-the-loop with interrupt()

Pause the graph mid-execution, ask a human, resume with their answer. The pattern that turns an autonomous agent into one enterprise will actually deploy.

Pedagogy reminder. Same rules — TODO skeleton, you write, paste back, then canonical.

The problem this solves

Your L22 agent will happily call any tool the LLM asks for. Fine for read-only tools like search_docs. Catastrophic for tools like these:

@tool
def deploy_to_cloudhub(app_name: str, env: str) -> str: ...

@tool
def delete_api_instance(api_id: str) -> str: ...

@tool
def rotate_secret(connected_app_id: str) -> str: ...

You don't want an LLM autonomously deploying to production at 3 AM because it hallucinated a customer request. You want a human to see the tool call, approve it (or edit, or reject), then let the graph proceed.

That's interrupt() — and it's the feature your enterprise customers will ask about within five minutes of seeing your demo.

The mental model — three things happen

interrupt() is a function you call inside a node. When the graph hits it:

  1. The node pauses. Execution stops mid-way.
  2. The checkpointer saves state. This is why L22 was a prerequisite — no checkpointer, no interrupts.
  3. Control returns to the caller with a "paused" status. Your application then collects human input (CLI prompt, web form, Slack message — anything).

You wake the graph back up by calling app.invoke(Command(resume=<value>), config=config). The graph picks up exactly where it left off, and interrupt(...) inside the node returns the value the human supplied.

Run 1: invoke(input) ──► graph runs ──► hits interrupt() ──► PAUSES │ (state saved to checkpointer) │ Your code: asks human, collects answer │ ▼ Run 2: invoke(Command(resume=answer)) ──► graph WAKES UP ──► interrupt() returns the answer │ ▼ graph continues

The two flavours

Static interruptsDynamic interrupts
Defined whereAt compile() timeInside a node, conditionally
Howinterrupt_before=["tools"]interrupt(payload) call
GranularityAlways pause before a given nodePause only when logic says so
Can return data?NoYes — returns the human's resume value
Use caseBlanket safety: "always confirm before tools"Targeted: "only pause for destructive tools"

Static is the older / simpler form. Dynamic (the interrupt() function) is the modern recommendation — same pattern preference as the L20 conditional vs unconditional edge distinction. We focus on dynamic.

The three things a human can do

Resuming isn't just yes/no. The human can:

ActionWhat it doesResume value example
Approve Tool call proceeds as-is "approve"
Edit Human modifies the tool's args before it runs (agent wanted env="production"; human changes to env="staging") {"action": "edit", "args": {"env": "staging"}}
Reject Tool call replaced with a message; agent decides what to do next {"action": "reject", "reason": "wrong env"}

Your interrupted node reads the resume value and branches accordingly.

The smallest possible interrupt() — no LLM

from typing import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from langgraph.types import interrupt, Command

class State(TypedDict):
    proposed: int
    approved: int

def propose(state: State) -> dict:
    return {"proposed": 42}

def human_approval(state: State) -> dict:
    # PAUSE HERE — the value passed to interrupt() is what the human sees
    answer = interrupt({"question": "approve?", "value": state["proposed"]})
    # After resume, `answer` holds the value passed via Command(resume=...)
    return {"approved": state["proposed"] if answer == "yes" else 0}

graph = StateGraph(State)
graph.add_node("propose", propose)
graph.add_node("human_approval", human_approval)
graph.add_edge(START, "propose")
graph.add_edge("propose", "human_approval")
graph.add_edge("human_approval", END)

memory = InMemorySaver()
app = graph.compile(checkpointer=memory)
config = {"configurable": {"thread_id": "demo-1"}}

# RUN 1 — graph pauses at interrupt()
result = app.invoke({"proposed": 0, "approved": 0}, config=config)
print("After pause:", result)
# {"proposed": 42, "approved": 0, "__interrupt__": [Interrupt(value={"question": "approve?", "value": 42})]}

# RUN 2 — resume with the human's answer
result = app.invoke(Command(resume="yes"), config=config)
print("After resume:", result)
# {"proposed": 42, "approved": 42}
The "no checkpointer, no interrupts" rule. If you forget compile(checkpointer=...), you'll get an error like "Interrupts require a checkpointer." The pause must be saved somewhere. This is why L22 was a hard prerequisite.

How to detect the pause from outside

You won't always know if a graph paused. Inspect state to check:

state = app.get_state(config)
if state.next:
    # Graph is paused; state.next is the tuple of nodes that would run next
    # state.tasks[0].interrupts gives the payload sent to interrupt()
    pending = state.tasks[0].interrupts[0].value
    print("Waiting on:", pending)
else:
    print("Done.")

This is how a web app's POST /chat handler would know to render an "approve this action?" UI vs. just returning the final answer.

Quiz

1. Why does interrupt() require a checkpointer?

Answer

The pause point must be saved somewhere so the graph can resume from it — possibly minutes, hours, or days later, possibly after a process restart. The checkpointer is that storage. Without one, "pause and resume" has nowhere to live.

2. Static interrupt_before=["tools"] vs dynamic interrupt(...) inside a node — when is each right?

Answer

Static: blunt safety net during dev — "always confirm before any tool runs." Quick to add, useful for early prototypes.

Dynamic: production. You can gate only destructive tools, you can show a richer payload to the human (not just "approve this node"), and you can capture the human's reply as a typed value the node uses to branch.

3. After an interrupt() pause, you call app.invoke({"messages": [("user", "new question")]}, config=config) instead of Command(resume=...). What happens?

Answer

The graph treats this as new input rather than a resume. The original interrupt is still pending; you've added a user message but the paused node never gets unblocked. The graph state ends up in a confusing half-pending state.

Always use Command(resume=...) to wake a paused graph. Use plain dict input only to start fresh or add a turn on a non-paused thread.

What you now know

  • interrupt(payload) pauses inside a node; the checkpointer holds the pause
  • Command(resume=value) wakes the graph; interrupt() returns that value
  • Three resume modes: approve, edit (change args), reject
  • Detect pauses externally with app.get_state(config).next and state.tasks[0].interrupts
  • Static interrupts for dev safety; dynamic interrupts for production logic
  • The enterprise pattern: read-only tools auto-run, write tools gate through HITL

The MuleSoft angle — why this matters for you

Your customers will not deploy an autonomous AI agent that can touch their integration runtime. They will deploy a supervised agent:

  • Read-only operations (status, metrics, doc lookups) run autonomously
  • Destructive operations (deploys, restarts, secret rotations) always stop for human approval
  • Every approval/rejection captured to an audit log (for SOX, GDPR, SOC2)
  • The "edit" path lets a human downgrade production → staging on the fly

This is the exact pattern a sales engineer at your level should be able to demo end-to-end. It's also the answer to the inevitable "but what about hallucinations?" objection. Hallucinations are unsafe only if the agent has authority — gate the authority, and the hallucination becomes a rejected suggestion.
