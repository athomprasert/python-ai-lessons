Lesson 25 · Python for AI/ML

Multi-agent: supervisor, swarm, hierarchical

Three ways to coordinate multiple agents in one graph. You learn them all so you can pick the right shape for the job — not the first one you saw on a blog.

Heads-up. This is the longest lesson in the series — three patterns means three exercises. Pace yourself; do them in order. The supervisor exercise is the foundation; swarm and hierarchical build on it.

Why bother with multi-agent at all?

Your L21 agent is one LLM in a loop with tools. It works. So why complicate it?

A single agent breaks down when:

  • Too many tools. Past ~20–30 tools the LLM gets lost picking the right one. Specialists shrink each agent's tool list.
  • Conflicting personas. "Researcher" wants to explore broadly; "Critic" wants to find flaws; "Planner" wants to commit. Cramming all three into one system prompt produces mediocre work in all three modes.
  • Cost-tier separation. Routing decisions are simple — use a cheap fast model. Synthesis is hard — use a smart expensive model. One agent can only have one model.
  • Permission boundaries. Read-only tools can run autonomously. Destructive tools need HITL. Different agents = different security postures.

Multi-agent is the answer. The question is which shape.

The three patterns — at a glance

PatternTopologyRouting decided byBest for
Supervisor Star — one boss, N specialists The supervisor (LLM or Python) Most cases; clear audit trail; easy HITL
Swarm Mesh — agents hand off peer-to-peer Each agent decides who to call next When the "next agent" depends on what the current agent just learned
Hierarchical Tree — supervisors of supervisors Each supervisor at its level Large systems (10+ agents); domain separation

Pattern 1 — Supervisor

┌──► researcher ──┐ │ │ user ──► supervisor ──► executor ┼──► supervisor (loop) ──► END │ │ └──► critic ──────┘

The supervisor receives the message history, picks the next specialist, sends the work, gets the result back, decides what to do next. Each specialist is a full agent with its own prompt and (usually) its own tools.

Why it's the default starting point

  • One place to inspect routing logic — easy debugging
  • One place to enforce policy ("always run researcher before executor")
  • One place to insert HITL — gate the supervisor's tool calls, or interrupt before the supervisor routes to executor
  • One audit trail — every routing decision flows through one node

Code shape

def supervisor(state: MessagesState) -> dict:
    # Reads state. Returns a "routing decision" — either a normal LLM message,
    # or a Command telling the graph which node to go to next.
    ...

graph.add_node("supervisor", supervisor)
graph.add_node("researcher", researcher)
graph.add_node("executor", executor)

graph.add_edge(START, "supervisor")
graph.add_conditional_edges("supervisor", route_from_supervisor)
graph.add_edge("researcher", "supervisor")  # specialists report back
graph.add_edge("executor",   "supervisor")

Specialists always return to the supervisor. The supervisor decides when to stop (route to END).

Pattern 2 — Swarm

user ──► researcher ──hands off──► executor ──hands off──► critic ──► END ↑ │ │ └──────────can call back to either────────────────┘

No central boss. Each agent decides, at the end of its turn, who to hand off to. The handoff is done via Command(goto="agent_name") — the agent returns a Command instead of a normal state update, and LangGraph routes accordingly.

New syntax — Command(goto=...)
You already saw Command(resume=...) in L24. Command is a general control-flow primitive: goto jumps to a specific node, update writes state, resume wakes an interrupted graph. You can mix them: Command(goto="executor", update={"plan": "..."}) hands off and updates state in one step. 
from langgraph.types import Command

def researcher(state: MessagesState) -> Command:
    findings = do_research(state)
    # Researcher itself decides: do I have enough? If yes, hand off to executor.
    if findings.complete:
        return Command(goto="executor", update={"messages": [findings]})
    return Command(goto="critic", update={"messages": [findings]})

When swarm beats supervisor

When the next agent depends on information the current agent just discovered. A supervisor would have to read the result, decide, then dispatch — two LLM calls. A swarm agent decides in the same step it produces the result — one LLM call.

Why it's not the default

  • Harder to audit — routing logic scattered across N agents
  • Easier to build infinite loops (A → B → A → B...)
  • Harder to enforce policy globally

Pattern 3 — Hierarchical

┌── top-level supervisor ──┐ │ │ research-supervisor execution-supervisor ┌──┴──┐ ┌──┴──┐ google- code- deploy- monitor- search- search- agent agent agent agent

Supervisors of supervisors. Each sub-supervisor is itself a compiled subgraph and is added as a single node to the top-level graph.

New syntax — subgraphs as nodes.
Any compiled graph can be added as a node to another graph. The outer graph just sees one node; internally it's a whole graph with its own state and logic. 
research_subgraph = build_research_team().compile()  # its own multi-agent system
parent_graph.add_node("research_team", research_subgraph)  # now one node
Keep the state shapes compatible (both use MessagesState) or write an adapter function.

When hierarchical is worth it

  • 10+ specialists — one supervisor managing all of them gets prompt-context-bloated and confused
  • Clean domain separation — a research team should not see deployment tools, and vice versa
  • Different ownership — different sub-supervisors can be maintained by different teams

Why you should resist using it early

Two layers of LLM routing means two layers of latency and cost per turn. Most projects never need hierarchical — they just think they do because it looks impressive on a slide. Start with supervisor. Add a layer only when supervisor is genuinely overwhelmed.

Cost reality — read this before you write code

PatternLLM calls per user turnToken usageFailure modes
Single agent1 (+1 per tool round)1×Wrong tool, hallucination
Supervisor2–4× (router + specialists)3–5×Router routes wrong
Swarm2–6×3–6×Infinite handoff loops
Hierarchical4–10×6–15×Layered confusion; debug hell

This is why a single-agent system that works beats a multi-agent system that's "more architecturally pure." Reach for multi-agent only when single-agent is genuinely failing.

Exercise 3 · stretch · hierarchical pattern

Build a hierarchical multi-agent system

No skeleton. You have the supervisor and swarm patterns down — combine them.

The scenario — Anypoint copilot

Build this topology:

top_supervisor ┌──────┴──────┐ research_team execution_team (subgraph) (subgraph) ├ doc_search ├ deployer └ log_search └ rotator

Requirements

  • Two subgraphs, each is its own supervisor + 2 specialists from Ex1's pattern
  • Top supervisor routes between the two subgraphs based on the task
  • Use deterministic routing for all supervisors (no LLM) — focus on topology

Quiz

1. You're building an agent that searches MuleSoft docs and answers questions. Single agent or multi-agent?

Answer

Single agent. One tool (search_docs), one persona (assistant), one model. Multi-agent here adds latency and cost with zero quality gain. Default to single agent.

2. Supervisor vs swarm — which is harder to debug, and why?

Answer

Swarm. In supervisor, all routing logic is in one place — open the supervisor's code, see the whole policy. In swarm, the routing is distributed across N agents — to understand "why did agent C run next," you have to read whichever agent called it, and the trace can ping-pong.

This is why supervisor is the default. Reach for swarm only when the per-turn cost saving (one LLM call instead of two) actually matters for your throughput.

3. You add a research-team subgraph as a node in your top-level graph. The subgraph uses MessagesState; the parent graph has a custom state with extra keys. What happens?

Answer

The subgraph only sees the keys it knows about (the MessagesState keys, i.e. messages). Other parent-state keys flow around it untouched. State updates from the subgraph merge back into parent state via the normal node-return mechanism.

If the schemas conflict (same key, different reducer or type), you need an adapter function as the node — read parent state, build the subgraph input, invoke the subgraph, translate the output back.

4. Where does HITL (interrupt() from L24) fit cleanest in a supervisor architecture?

Answer

Two good places:

  • Inside the executor agent, before any destructive tool — same as L24 Ex2, scoped to the agent that actually has the dangerous tools
  • In the supervisor's router, conditional on the route — "if the next agent is executor AND the task is destructive, pause for approval before dispatching"

The first is simpler and keeps the safety logic next to the danger. The second gives the supervisor a chance to refine the plan before HITL — useful when the human's approval is for a strategy, not a single tool call.

What you now know

  • Supervisor — central router, easiest to debug, the default starting point
  • Swarm — peer-to-peer handoffs via Command(goto=...); saves an LLM call when the next agent depends on what was just discovered
  • Hierarchical — subgraphs as nodes; use only when supervisor is overwhelmed by 10+ specialists
  • Command is the multi-tool primitive: goto for routing, update for state, resume for HITL — mix as needed
  • Each LLM call multiplies cost — multi-agent must justify itself with quality, not architectural beauty
  • HITL plugs cleanly into supervisor; harder to enforce uniformly in swarm
