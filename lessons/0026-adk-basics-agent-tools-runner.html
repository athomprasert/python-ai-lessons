Google ADK basics: Agent, tools, adk run / adk web

Spin up a working ADK agent in under 30 lines of code and visualise it in Google's built-in dev UI. The fastest path from zero to "show this to a customer".

Pedagogy reminder. Same as LangGraph: skeleton + TODOs + paste-back review. Don't read the canonical until you've tried.

What ADK actually is

ADK = Agent Development Kit. Google's open-source agent framework, released April 2025. It's the same kind of thing as LangGraph — a Python library for building LLM agents — but it has a very different philosophy.

LangGraphADK
Philosophy Build the graph yourself (unopinionated) Use our conventions (opinionated)
Core primitive StateGraph + nodes + edges Agent object (model + tools + instructions)
Where state lives TypedDict you design Session.state (managed dict)
How you run it app.invoke(...) — you write the harness adk run CLI + adk web UI — built-in
Dev UI None — bring your own (Phoenix / LangSmith) ADK Web ships with the package
Multi-agent You wire it (L25) Declarative: SequentialAgent, ParallelAgent, LoopAgent
Production deployment Bring your own infra Vertex Agent Engine — one-command deploy
Which to pick. LangGraph wins when you need fine control over the graph topology or you're framework-agnostic. ADK wins when you want fast iteration, Google's dev UI for free, and a managed deployment story on GCP. Your customers using Google Cloud will ask about ADK — they need to hear an informed answer.

The three pieces of an ADK app

┌─────────────┐ user msg ───► Runner ──► Agent ──┤ tool calls │──► tool results ──► response │ (LLM) └─────────────┘ │ │ │ └──── reads/writes ────► Session ◄───────────┘ (per-user state)
  1. Agent — the configuration: which model, what instructions, which tools. You define this.
  2. Runner — the executor: takes a user message, walks the agent through its loop, calls tools, returns the answer. Provided by ADK; you rarely customise it.
  3. Session — the per-conversation state: messages so far, scratchpad, anything you want to remember across turns. Managed by a SessionService (in-memory by default).

Compared to LangGraph: Agent ≈ a compiled graph, Runner ≈ app.invoke, Session ≈ thread + checkpointer.

The folder convention — ADK is opinionated

This is the biggest mental shift from LangGraph. ADK expects each agent to live in its own directory, with these three files:

my_agent/ ├── agent.py ← defines a variable named exactly "root_agent" ├── __init__.py ← exports the agent (usually one line) └── .env ← API keys, etc.

The CLI tools (adk run my_agent, adk web) look for this structure. They scan for directories, import each one's agent.py, and find the root_agent variable.

New convention — folder-as-agent.
There's no LangGraph equivalent. In LangGraph, an agent is whatever script you write. In ADK, an agent is a directory following a contract. Fight it and the tooling won't find your work. Embrace it and adk web auto-discovers every agent in your folder.

Minimal hello-world — read once, then write

File: my_agent/agent.py

from google.adk.agents.llm_agent import Agent

def get_current_time(city: str) -> dict:
    """Returns the current time in a specified city."""
    return {"status": "success", "city": city, "time": "10:30 AM"}

root_agent = Agent(
    model='gemini-flash-latest',
    name='root_agent',
    description="Tells the current time in a specified city.",
    instruction="You are a helpful assistant that tells the current time.",
    tools=[get_current_time],
)

File: my_agent/__init__.py

from . import agent

File: my_agent/.env

GOOGLE_API_KEY="your-key-from-aistudio.google.com"

Run it:

# CLI mode — chat in your terminal
adk run my_agent

# Web mode — opens a browser-based chat + trace UI
adk web --port 8000

The five things to notice

  1. Tool functions return dicts, not strings. ADK convention: {"status": "...", "...": "..."}. The LLM sees the dict and reasons about each field. (LangChain's @tool defaults to strings — different ergonomic.)
  2. The function's docstring becomes the tool description the LLM sees. Same as LangChain @tool.
  3. No @tool decorator. You just pass the function. ADK inspects the signature directly.
  4. Variable name matters. The CLI looks for root_agent — not agent, not my_root. Rename it and adk run can't find it.
  5. No Runner code. You don't write the run loop — adk run does it for you. (If you wanted to embed the agent in your own Python script, you'd write Runner code by hand — covered in L29.)
Auth — two paths, pick one.
  • AI Studio (dev/learning): GOOGLE_API_KEY from aistudio.google.com/app/apikey — fastest to set up.
  • Vertex AI (prod): Set GOOGLE_GENAI_USE_VERTEXAI=TRUE, GOOGLE_CLOUD_PROJECT=..., GOOGLE_CLOUD_LOCATION=..., and authenticate with gcloud auth application-default login. Covered properly in L29.
For this lesson, use AI Studio. Don't commit .env.

Hint 1 — the tool function shape 
def lookup_anypoint_status(org_id: str) -> dict:
    """Look up the operational status of a MuleSoft Anypoint organization.

    Args:
        org_id: the organization identifier, e.g. "acme-prod".

    Returns:
        Dict with keys: status (success|error), org_id, healthy_apis, alerts.
    """
    mock_db = {
        "acme-prod": {"healthy_apis": 12, "alerts": 0},
        "acme-dev":  {"healthy_apis": 8,  "alerts": 2},
    }
    if org_id not in mock_db:
        return {"status": "error", "message": f"Unknown org: {org_id}"}
    return {"status": "success", "org_id": org_id, **mock_db[org_id]}
Hint 2 — the Agent definition 
root_agent = Agent(
    model='gemini-flash-latest',
    name='mulesoft_helper',
    description="Answers questions about MuleSoft Anypoint org health.",
    instruction=(
        "You are a MuleSoft Anypoint Platform assistant. "
        "When a user asks about the health or status of an org, "
        "call lookup_anypoint_status with the org_id. "
        "Explain the result in plain English, including the number of "
        "healthy APIs and any alerts."
    ),
    tools=[lookup_anypoint_status],
)
Hint 3 — common gotchas
  • "No agent found" — your variable isn't named root_agent, or __init__.py doesn't import agent.
  • "GOOGLE_API_KEY not set" — you ran adk run from the wrong directory; the .env must be inside mulesoft_helper/.
  • "Model not found" — try gemini-2.5-flash if gemini-flash-latest isn't recognised. ADK model aliases evolve.
  • ADK Web shows no agents — run adk web from the parent of mulesoft_helper/, not from inside it.

Verification

In adk run, asking "Is acme-prod healthy?" should:

  1. Show a tool call to lookup_anypoint_status(org_id="acme-prod")
  2. Show the tool result (the dict)
  3. Print the LLM's plain-English summary (e.g. "Yes — acme-prod has 12 healthy APIs and 0 alerts.")

In adk web, you should see the same flow visually — including a trace view with the tool call expanded. This is the killer demo feature.

Quiz

1. You name your variable my_agent instead of root_agent. You run adk run mulesoft_helper. What happens?

Answer

ADK can't find an agent to run — error like "No agent named 'root_agent' found in agent.py." The CLI's auto-discovery looks for the exact name root_agent. This is the convention you signed up for by using ADK. (You can override it with explicit Runner code, but for CLI/Web use, just name it root_agent.)

2. Your tool function returns f"Org {org_id} is healthy" (a string) instead of a dict. Does it work?

Answer

Technically yes — ADK will wrap it. But you lose the structured-data benefit. By returning a dict like {"status": "success", "healthy_apis": 12, "alerts": 0}, the LLM can reason field-by-field, you can change one field without breaking others, and your trace view in ADK Web is far more readable. Stick to dicts — it's the ADK convention.

3. Why does ADK insist on the folder-per-agent structure?

Answer

Auto-discovery. adk web scans the current directory, treats every subdirectory with an agent.py as an agent, and shows them all in the picker. The convention is what lets the dev UI work with zero config. The cost is rigidity; the benefit is "drop in a folder, it appears in the UI" — huge for fast iteration.

You can think of it as ADK's equivalent of how Next.js makes every file under app/ a route, or how Rails makes every file under app/controllers/ a controller. Convention over configuration.

What you now know

  • ADK = Google's opinionated answer to LangGraph — fewer lines, more conventions, batteries included
  • Three primitives: Agent (config), Runner (executor — hidden behind CLI), Session (state — covered in L28)
  • Folder-as-agent convention — directory with agent.py, __init__.py, .env; variable must be root_agent
  • Tools are plain Python functions returning dicts — no decorator, signature + docstring drive the LLM
  • Two ways to run: adk run my_agent (CLI) and adk web (visual UI — the killer feature for demos)
  • Two auth paths: AI Studio key (dev) vs Vertex AI (prod)

The MuleSoft angle

Your Google Cloud customers will ask: "Why ADK instead of LangChain?" The honest answer for an enterprise audience is:

  • ADK Web — your engineers can show a working agent + trace view to stakeholders in minutes, not after Phoenix setup
  • Vertex Agent Engine — production deployment is a managed Google Cloud service, not a roll-your-own infra exercise (covered in L29)
  • Eval framework built in — ADK ships with agent evaluation tooling; LangGraph leaves this to third-party
  • Single vendor relationship — for GCP-committed customers, one throat to choke is operationally valuable

The honest cost: vendor lock-in. Your agent code is bound to Google's framework and (likely) Vertex deployment. For customers betting on a multi-cloud or framework-neutral future, LangGraph remains the safer bet.

Vertex preview

You'll see this in L29, but worth knowing now: any ADK agent you build with this folder structure can be deployed to Vertex Agent Engine with one Python script. No Dockerfile, no FastAPI wrapper. The opinionated structure is the price you pay for that deployment story.
