ADK Tools: function tools, built-ins, MCP

Three ways to give an agent capabilities — plain Python functions, Google's built-in tools, and any MCP server. Same agent, three toolboxes, wildly different implications for what you can ship.

Pedagogy reminder. Skeleton → TODO → run → paste back. If you don't run it, you didn't finish the lesson.

The three tool types

TypeWhat it isWhen to reach for it
Function tool Plain Python function you pass to tools=[...] Your own logic — DB queries, API calls, calculations. The default.
Built-in tool Ships with ADK — google_search, code execution, Vertex AI Search Common capabilities Google implemented for you. Zero code.
MCP toolset A whole set of tools imported from an MCP server Reusing tools built for the wider MCP ecosystem — no glue code.

You can mix function tools freely. But built-in tools have restrictions (see the warning below). MCP toolsets are the interesting one for your MuleSoft story.

Function tools — auto-wrap vs explicit wrap

In L26, you passed a plain function to tools=[lookup_anypoint_status]. ADK auto-wraps that in a FunctionTool for you. What if you want to wrap it yourself?

# Auto-wrap (what L26 did — the default)
def lookup_anypoint_status(org_id: str) -> dict:
    """..."""
    return {...}

root_agent = Agent(
    ...,
    tools=[lookup_anypoint_status],   # bare function — ADK wraps it
)

# Explicit wrap (same outcome, more control)
from google.adk.tools import FunctionTool

status_tool = FunctionTool(func=lookup_anypoint_status)

root_agent = Agent(
    ...,
    tools=[status_tool],
)

When to use explicit FunctionTool: when you need to override the auto-generated schema, mark a tool as long-running, or customise how it appears to the LLM. For 90% of cases, auto-wrap is fine.

New syntax — the schema comes from your function signature and docstring.
ADK inspects the function's parameter names, type hints, and the docstring to build the JSON schema the LLM sees. That's why L26 was strict about writing a proper docstring. No decorator, no manual schema — the Python signature IS the schema.

Built-in tools — one line, huge capability

ADK ships tools you'd otherwise have to build yourself. The one you'll use most:

from google.adk.tools import google_search
from google.adk.agents import Agent

root_agent = Agent(
    name="search_agent",
    model="gemini-flash-latest",
    instruction="Answer questions using Google Search. Always cite sources.",
    tools=[google_search],
)

That's it. Ask the agent "what's the current price of Gold?" and it'll grounds its answer in Google search results — with citations.

Gotcha — built-in "grounding" tools may not mix with function tools.
Historically, Gemini has enforced a rule where a "grounding tool" (like google_search) can't share an agent with regular function tools — the API returns an error if you try. ADK docs don't explicitly warn about this, but if you get an error like "Tool config incompatible" when combining google_search with your lookup_anypoint_status function, that's why.

The workaround: use two sub-agents. A search_specialist agent with just google_search, and a mulesoft_specialist agent with your function tools. A root agent routes between them. You'll build this pattern properly in L31 (multi-agent).

Other built-ins worth knowing

  • built_in_code_execution — LLM writes Python and runs it in a sandbox to compute answers. Great for math/data questions.
  • vertex_ai_search — searches your Vertex AI Search index (enterprise doc-search). Requires Vertex setup, not AI Studio.
  • load_memory — reads from ADK's long-term memory service. Covered in L28.

MCP toolsets — the killer feature for MuleSoft

MCP (Model Context Protocol) is Anthropic's open standard for exposing tools to LLMs. Any MCP server exposes a list of tools; ADK can consume that list and hand the tools to your agent — you write zero glue code.

┌──────────────────────┐ ADK Agent ─────►│ MCPToolset (config) │──stdio/HTTP──► MCP server │ │ │ │ discovers tools │◄────tools list──┤ │ makes them │ │ │ available to agent │ └── implements └──────────────────────┘ the actual work

Why this matters for you:

  • Reusability — build a tool once as an MCP server; use it from ADK, Claude Code, Cursor, any MCP-aware client
  • Language independence — MCP server can be in TypeScript, Python, Go — ADK doesn't care
  • MuleSoft strategy — Salesforce is investing in exposing Anypoint APIs as MCP servers. When that ships, an ADK agent (or any MCP client) gets Anypoint capabilities for free

Verified imports and shape from current ADK docs:

from google.adk.agents import LlmAgent
from google.adk.tools.mcp_tool import McpToolset
from google.adk.tools.mcp_tool.mcp_session_manager import StdioConnectionParams
from mcp import StdioServerParameters

root_agent = LlmAgent(
    model='gemini-flash-latest',
    name='fetch_agent',
    instruction='Fetch web pages and summarise them.',
    tools=[
        McpToolset(
            connection_params=StdioConnectionParams(
                server_params=StdioServerParameters(
                    command='uvx',
                    args=['mcp-server-fetch'],
                ),
            ),
        )
    ],
)
New syntax — the three-layer MCP config.
McpToolset holds the connection. StdioConnectionParams says "talk to it via stdin/stdout". StdioServerParameters says "here's the command to spawn". The nesting looks over-engineered, but it lets ADK swap in HTTP/SSE transports later with the same outer McpToolset.
Rule — the MCP toolset must be defined synchronously in agent.py.
ADK docs are explicit: "When deploying agents with MCP tools, the agent and its McpToolset must be defined synchronously in your agent.py file." You can't await or async def the setup at module scope. The example above is the pattern to copy.

def get_env_recommendation(org_type: str) -> dict:
    """Recommend an Anypoint environment topology for a given org type.

    Args:
        org_type: one of "startup", "enterprise", or "regulated".
    """
    recs = {
        "startup":    "CloudHub 2.0 shared workers, single sandbox + prod.",
        "enterprise": "CloudHub 2.0 private space, sandbox/staging/prod.",
        "regulated":  "Runtime Fabric on-prem, air-gapped, HITL deploys.",

    return {"status": "success", "org_type": org_type, "recommendation": recs[org_type]}

What to notice

  1. You wrote no tool functions. The fetch tool appeared in your agent because the MCP server exposed it. Look at adk web's tool list for fetch_agent — you'll see a tool you didn't define.
  2. The MCP server is a separate process. ADK spawns it, talks to it over stdin/stdout, and shuts it down when the agent exits. You never manage the process directly.
  3. If mcp-server-fetch gets updated, you get the new version by re-running uvx. No agent code changes.

Quiz

1. Your agent has both a bare function get_stock_price and an explicit FunctionTool(func=get_stock_price). What's the behavioural difference?

Answer

None for standard use — auto-wrap and explicit wrap produce equivalent tools. The explicit wrapper only matters when you need to override the auto-generated schema (custom parameter names, custom description), mark the tool as long-running, or configure other tool-level options that aren't exposed to bare functions.

Rule of thumb: start with auto-wrap. Reach for FunctionTool only when you have a specific option to set.

2. You try adding google_search to your existing mulesoft_helper agent alongside lookup_anypoint_status. It errors out. Why?

Answer

Built-in "grounding" tools like google_search historically can't share an agent with custom function tools — the underlying Gemini API rejects the combination. The docs are quiet on this, but you'll hit it in practice.

The fix is a two-agent pattern: one agent has google_search, another has your function tools, and a root agent delegates. Full pattern in L31 (multi-agent).

3. Why is "MCP toolset" a fundamentally different proposition from "function tool", even though they both end up as tools in tools=[...]?

Answer

Function tools couple the tool implementation to your agent code — the Python function lives inside agent.py. Change the tool, redeploy the agent.

MCP tools decouple: the tool lives in a separate process (possibly maintained by a different team, in a different language). Your agent just points at it. This is the enterprise integration pattern — one team ships the MCP server for their domain (e.g. MuleSoft ships mcp-server-anypoint), many agents consume it, no glue code.

For a MuleSoft sales engineer, this is the story: "our platform exposes hundreds of integration capabilities. Once we ship them as an MCP server, any AI agent — ADK, Claude Code, Cursor — can use them without custom code."

What you now know

  • Function tools: bare function auto-wrapped, or FunctionTool(func=...) for explicit control
  • Built-in tools: google_search, code execution, Vertex AI Search — one-line adds, but grounding tools don't mix with custom tools
  • MCP toolsets: McpToolset + StdioConnectionParams + StdioServerParameters; one line adds an entire external tool set with zero glue code
  • The MCP pattern is the strategic bet for enterprise integration platforms — MuleSoft included

The MuleSoft angle

The three-tool taxonomy maps to three tiers of MuleSoft go-to-market conversations:

  • Function tools — "Here's a custom agent for your CRM sync. Six functions, six use cases." Bespoke work; sold as a project.
  • Built-in tools — "Google's already solved search grounding for you. You don't need to build it." Anchors credibility for GCP-first customers.
  • MCP toolsets — "MuleSoft is exposing all of Anypoint as an MCP server. Point your ADK/Claude/Cursor agent at it, and you get every Anypoint capability out of the box." Sold as a platform.

The last one is your differentiator. Every customer builds function tools. Only customers on MuleSoft get the platform-scale MCP story.
