ADK state & memory: Session.state, output_key, SessionService

By default, every ADK agent forgets everything the second the response ends. Session state is how you fix that — and it's ADK's answer to LangGraph's checkpointer.

Pedagogy reminder. Skeleton + TODOs + paste-back review. Don't peek at the canonical until you've tried the code yourself and hit at least one error.

Why this lesson exists

Your L26/L27 agent works — you send a message, it uses tools, it answers. But watch what happens on turn 2:

# Turn 1
user: I am working on the acme-prod org.
assistant: Got it, acme-prod noted.

# Turn 2
user: What is the status of the org I mentioned?
assistant: I don't have any context about which org you are referring to.

The agent has no idea. Every turn is a fresh brain. To be useful in the real world — remembering the customer's org ID, tracking which ticket you're mid-way through, resuming a half-finished deployment — you need state that survives across turns.

That's what Session.state is for.

The three memory primitives in ADK

PrimitiveScopePersists?Analogy
Session.state One conversation (a "session") As long as the session lives — determined by SessionService LangGraph State dict
SessionService Storage backend for many sessions Depends on implementation (in-memory / DB / Vertex) LangGraph checkpointer
MemoryService Across sessions, cross-user long-term recall Yes — like a vector store of past conversations RAG over conversation history

This lesson covers the first two. MemoryService is a later topic — you can go far with just Session.state.

LangGraph vs ADK mental map. In LangGraph, you design the state TypedDict and pass it explicitly. In ADK, state is a plain dict that lives on the Session object — you just read and write keys. Less type safety, more flexibility.

The three ways to touch state

Way 1 — Read/write from inside a tool

Add a tool_context: ToolContext parameter to your tool function. ADK sees this and injects the context automatically. You then read and write tool_context.state like any dict.

from google.adk.tools import ToolContext

def remember_org(org_id: str, tool_context: ToolContext) -> dict:
    """Remember which Anypoint org the user is working on."""
    tool_context.state["current_org"] = org_id
    return {"status": "success", "org_id": org_id}

def recall_org(tool_context: ToolContext) -> dict:
    """Return the org the user is currently working on."""
    org = tool_context.state.get("current_org")
    if org is None:
        return {"status": "empty", "message": "No org remembered yet."}
    return {"status": "success", "org_id": org}
New syntax — ToolContext injection.
The parameter name matters. ADK inspects your function signature: if it sees a parameter typed as ToolContext, ADK provides it at call time — the LLM never sees it in the schema. Everything else in the signature is exposed to the LLM. This is why you can freely add tool_context: ToolContext without breaking your tool's shape.
Prefer .get() over [key] when reading state — the key may not exist yet on turn 1. state.get("current_org") returns None safely; state["current_org"] would raise KeyError and blow up your tool.

Way 2 — Auto-save the reply with output_key

Set output_key="foo" on an Agent, and after each turn ADK writes the agent's final text response into Session.state["foo"] automatically. No code needed.

from google.adk.agents import LlmAgent

root_agent = LlmAgent(
    name='greeter',
    model='gemini-flash-latest',
    instruction="Generate a warm greeting.",
    output_key="last_greeting",   # ← auto-saves to state["last_greeting"]
)

Way 3 — Interpolate state into the instruction

ADK's instruction strings support {key} placeholders. On each turn, ADK replaces them with the current value from Session.state.

root_agent = LlmAgent(
    name='mulesoft_helper',
    model='gemini-flash-latest',
    instruction="""You are a MuleSoft assistant.
The user is currently working on org: {current_org}.
Answer questions in the context of that org.""",
    tools=[remember_org, recall_org],
)

When the user has already run remember_org("acme-prod"), {current_org} gets substituted with "acme-prod" at every turn — the LLM never has to call recall_org. Instruction interpolation is the cheapest way to give the LLM stateful context because it costs zero extra tool calls.

New syntax — {key} in instruction.
Standard Python f-string-looking syntax, but it's not an f-string — the string is a plain str, and ADK does the substitution at runtime using the live Session.state. If {current_org} is not in state yet, you'll get a literal {current_org} in the prompt (annoying but not a crash). If you need literal curly braces (e.g., for JSON in the prompt), use an InstructionProvider function instead of a string.
Gotcha — instruction interpolation happens once per turn, at the top of the loop. If a tool inside the same turn writes to state, the change is visible on the next turn, not the current one. Chain via output_key if you need mid-conversation propagation.

Where sessions live — SessionService

Every adk web and adk run command uses a session service — the storage backend that decides whether your state survives a restart.

BackendWhen to usePersists across restart?
InMemorySessionService Local dev, unit tests, quick demos ❌ No — dict in RAM
DatabaseSessionService Anything real. SQLite for solo dev; Postgres/MySQL for prod ✅ Yes
VertexAiSessionService Deployed on Vertex Agent Engine (managed by Google) ✅ Yes — GCP-managed 
from google.adk.sessions import InMemorySessionService, DatabaseSessionService

# dev
session_service = InMemorySessionService()

# solo dev with persistence — sessions.db is a plain SQLite file on disk
session_service = DatabaseSessionService(db_url="sqlite:///./sessions.db")
How adk web uses this. By default, adk web uses InMemorySessionService. To make sessions survive a server restart, launch it with adk web --session-service-uri="sqlite:///./sessions.db". Kill the server, restart, and your previous conversation is still there.

The MuleSoft angle

Every enterprise agent you'll design for MuleSoft customers has to answer questions like:

  • Which environment is the user currently deploying to? (state["current_env"])
  • What ticket ID are we mid-way through resolving? (state["active_ticket"])
  • Which API did the last error come from? (state["last_failed_api"])
  • Has this user approved the destructive action yet? (state["deploy_approved_at"])

These are all natural fits for Session.state. Once you understand this pattern, you can walk into any enterprise agent design and answer the "how do you handle context" question fluently.

Hint 1 — What "the LLM won't use tools" means

LLMs decide which tool to call based on the tool's name, docstring, and the agent's instruction. If your instruction just says "you are a helpful assistant", the model has no reason to prefer remember_org over just… acknowledging the user in text. You need to say something like "When the user mentions an org they're working on, always call remember_org. When they ask about the org they mentioned, always call recall_org first."

Hint 2 — Why .get() matters here

On turn 2 in a fresh session, if the user asks "which org am I on?" without a prior remember_org call, state["current_org"] would raise KeyError. Using state.get("current_org") returns None, which lets you return a friendly "empty" response instead of crashing.

Exercise 2 — Guided

output_key + instruction interpolation

Goal. Skip the recall_org tool entirely by interpolating state directly into the instruction — this is faster and cheaper (one fewer LLM call per turn).

Steps:

  1. Keep remember_org from Exercise 1. Delete recall_org — you won't need it.
  2. Rewrite the instruction to include the placeholder {current_org}: 
    instruction="""You are a MuleSoft assistant.
Current org context: {current_org}
When the user mentions a new org, call remember_org.
Answer all other questions in the context of the current org."""
  3. Also add output_key="last_answer" to the Agent.
  4. Run the same two-turn conversation as Exercise 1.
  5. Open the session inspector and confirm state now contains two keys: current_org and last_answer.
Hint — what happens on the first turn?

On turn 1, Session.state is empty, so {current_org} won't have a value to interpolate. ADK will substitute an empty/placeholder — the LLM might see something like "Current org context: " with no value. That's fine. The user's turn-1 message says "I'm working on org acme-prod", and the LLM can still call remember_org. From turn 2 onwards, the interpolation kicks in properly.

Quick quiz

1. You add tool_context: ToolContext to a tool function. Does the LLM see this parameter in the tool's JSON schema?

Show answer

No. ADK inspects the signature, identifies ToolContext as a special dependency-injected parameter, and strips it from the schema exposed to the LLM. Only the "real" business parameters (like org_id: str) show up in the schema.

2. Your instruction is "Working on: {current_org}". On turn 1, state is empty. What does the LLM see?

Show answer

Roughly "Working on: " with the placeholder unresolved or blank. It's not a crash — but the LLM sees no useful context, so on the first turn you should still expect it to call remember_org based on what the user said. From turn 2 onwards, once state is populated, the interpolation is meaningful.

3. You want your agent's sessions to survive an adk web restart. Which SessionService do you use, and how do you configure adk web?

Show answer

DatabaseSessionService, configured via the CLI flag adk web --session-service-uri="sqlite:///./sessions.db" for solo dev, or a Postgres/MySQL URL for production. InMemorySessionService (the default) loses everything on restart.

4. What's the difference between saving state via output_key versus writing it in a tool function?

Show answer

output_key saves the agent's final text response — the message going back to the user. Tool state writes save whatever you put there (structured data like an org ID, a ticket number, a boolean flag). Use output_key when you want to chain outputs across agents; use tool state for structured facts the LLM has extracted.
