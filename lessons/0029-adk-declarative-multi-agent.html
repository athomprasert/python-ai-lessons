ADK declarative multi-agent: SequentialAgent, ParallelAgent, LoopAgent

Three first-class classes for the three most common multi-agent patterns. No graph wiring, no Command(goto=...), no supervisor nodes to author. Trade flexibility for authoring speed.

Pedagogy reminder. Skeleton + TODOs + paste-back review. Multi-agent orchestration looks simple in ADK because most of the wiring is hidden — that's exactly why writing it yourself matters.

Why declarative multi-agent

In L25 (LangGraph) you built three multi-agent patterns — supervisor, swarm, hierarchical — by wiring nodes and edges. Every routing decision was a Command(goto=...) or a conditional edge you designed.

ADK takes the opposite bet: most multi-agent workflows fit one of three shapes. Ship those shapes as classes. Let the developer say "run these three in sequence" or "run these three in parallel" and be done.

The trade. LangGraph = infinite flexibility, more code. ADK workflow agents = less flexibility, dramatically less code. For 80% of enterprise agents (research pipelines, refinement loops, fan-out/fan-in aggregation) the ADK shapes are enough. For the other 20% (dynamic routing, cyclic supervisor patterns, HITL branch selection) you'll still reach for LangGraph or write a CustomAgent in ADK.

The three shapes

ClassShapeUse forLangGraph equivalent
SequentialAgent A → B → C Multi-stage pipelines: research → draft → review A linear chain of nodes with fixed edges
ParallelAgent A + B + C simultaneously Independent research/enrichment: fetch API health + deploy history + alerts in parallel Fan-out via multiple edges to the same next node
LoopAgent A → B → A → B ... until "done" Refinement / critic loops: draft → critique → refine until acceptable A cycle in the graph with a conditional break

The one mechanism that ties them together

All three shapes use the same state-passing mechanism, and you've already seen it in L28:

  1. Each sub-agent has an output_key — its final text response is written to Session.state[output_key].
  2. Downstream sub-agents interpolate that value into their instruction with {key}.

That's it. No message passing, no queues, no explicit hand-offs. The workflow agent is just a small piece of orchestration on top of shared session state.

Mental model — think of workflow agents as loops around L28. SequentialAgent is a for-loop that runs each sub-agent once. ParallelAgent is an asyncio.gather. LoopAgent is a while-loop. All three just call sub-agents that read and write Session.state.

Shape 1 — SequentialAgent

from google.adk.agents import LlmAgent
from google.adk.agents.sequential_agent import SequentialAgent

researcher = LlmAgent(
    name='researcher',
    model='gemini-flash-latest',
    instruction="Research the user's question. Output bullet-point notes.",
    output_key="research_notes",   # ← writes final text to state["research_notes"]
)

writer = LlmAgent(
    name='writer',
    model='gemini-flash-latest',
    instruction="""Given these research notes, write a polished 3-sentence answer.
Notes: {research_notes}""",     # ← reads from state via interpolation
    output_key="final_answer",
)

root_agent = SequentialAgent(
    name='research_pipeline',
    sub_agents=[researcher, writer],
    description="Researches then writes.",
)

The SequentialAgent runs researcher, waits for it to write state["research_notes"], then runs writer — whose instruction interpolates that key. Two lines of wiring for a two-stage pipeline.

Shape 2 — ParallelAgent

from google.adk.agents.parallel_agent import ParallelAgent

api_health_researcher = LlmAgent(
    name='api_health_researcher',
    instruction="Look up API health for the org.",
    tools=[lookup_api_health],
    output_key="api_health_result",
)

deploy_history_researcher = LlmAgent(
    name='deploy_history_researcher',
    instruction="Look up recent deployments for the org.",
    tools=[lookup_deployments],
    output_key="deploy_history_result",
)

parallel_research = ParallelAgent(
    name='parallel_research',
    sub_agents=[api_health_researcher, deploy_history_researcher],
)

Both researchers start at ~the same time — ParallelAgent calls run_async on each and lets them run concurrently. When both finish, Session.state contains both api_health_result and deploy_history_result.

Important — parallel sub-agents don't see each other's conversation history during execution. From the docs: "no automatic sharing of conversation history or state between these branches during execution." Each parallel branch is essentially isolated until it completes. If branch A needs branch B's output, you're not in a parallel pattern — you're in a sequential one.

To do anything useful with the two results, you almost always wrap the whole thing in a SequentialAgent with a "merger" downstream:

merger = LlmAgent(
    name='merger',
    instruction="""Summarise both research findings into one health report.
API health: {api_health_result}
Deploy history: {deploy_history_result}""",
    output_key="health_report",
)

root_agent = SequentialAgent(
    name='health_check_pipeline',
    sub_agents=[parallel_research, merger],
)

Workflow agents compose. This is the ADK equivalent of "hierarchical" from L25 — subgraphs as nodes — but you nest workflow-agent classes instead.

Shape 3 — LoopAgent

Loops need two things: a way to iterate, and a way to stop.

Iterate: pass max_iterations=N to cap the loop. Prevents runaway costs.

Stop early: a sub-agent calls a tool that sets tool_context.actions.escalate = True. That's the signal-from-within pattern.

from google.adk.agents import LoopAgent, LlmAgent
from google.adk.tools import ToolContext

def exit_loop(tool_context: ToolContext):
    """Signal the loop to terminate — call this when the draft is good enough."""
    tool_context.actions.escalate = True
    return {}

drafter = LlmAgent(
    name='drafter',
    instruction="""Write or refine a draft answering the user's question.
Previous draft (if any): {draft}
Previous critique (if any): {critique}""",
    output_key="draft",
)

critic = LlmAgent(
    name='critic',
    instruction="""Critique the draft. If it fully answers the question, call exit_loop.
Otherwise output specific improvements needed.
Draft: {draft}""",
    tools=[exit_loop],
    output_key="critique",
)

root_agent = LoopAgent(
    name='refinement_loop',
    sub_agents=[drafter, critic],
    max_iterations=5,
)
New syntax — tool_context.actions.escalate = True.
You've seen tool_context.state (L28). tool_context.actions is a sibling — it's how a tool can send control signals back up to the framework. escalate=True is specifically for exiting a LoopAgent. Other action flags exist (like skip_summarization) but this is the one that matters.
Always set max_iterations. If your critic forgets to call exit_loop, you don't want the loop running forever charging you for tokens. max_iterations=5 is a good default for demos; production critics may need higher.

Compared to L25 (LangGraph multi-agent)

LangGraph L25 patternClosest ADK shapeVerdict
Supervisor (star) SequentialAgent where the first agent picks the next ADK's built-in isn't a perfect fit — for a true supervisor with routing, use ADK's agent transfer feature (later lesson)
Swarm (peer hand-off) Agent-to-agent transfer via transfer_to_agent ADK covers this via sub-agents + transfer tool — different mechanic (covered in L30)
Hierarchical (subgraphs) Workflow agents composed inside workflow agents Direct match — nest a ParallelAgent inside a SequentialAgent

The three ADK shapes are not a 1:1 mapping to L25's patterns — they're a different slicing of the same problem space. Once you internalise both slicings, you can walk into any customer's multi-agent design conversation with two vocabularies.

The MuleSoft angle

Enterprise integration workflows are overwhelmingly shaped like one of these three:

  • Sequential — the "onboarding pipeline". Validate customer data → enrich from CRM → format for the target system → post. This is 60% of MuleSoft flows in the wild.
  • Parallel — the "360-degree view". A single customer query fans out to CRM + billing + support ticket system + inventory, then a merger produces the "customer 360" answer. Perfect ADK ParallelAgent use case.
  • Loop — the "retry with correction". Send order to fulfillment. If rejected, LLM inspects the reason, corrects the payload, retries. Bounded by max_iterations so it can't loop forever.

These are exactly the enterprise stories you should be telling. ADK ships them as three lines each — the demo speed alone is a compelling narrative.

Hint 1 — Why order matters

The sub_agents list is ordered. SequentialAgent runs them left-to-right. If you put the writer first, it will try to interpolate {research_notes} from empty state — you'll get a literal {research_notes} in the prompt and the LLM will be very confused.

Hint 2 — What "instruction MUST include {research_notes}" means

Writing instruction="Write a 3-sentence answer." won't work — the writer has no idea what research was done. It needs instruction="Write a 3-sentence answer using these notes: {research_notes}". The interpolation is the only mechanism that ties the two agents together.

Exercise 2 — Guided

Fan out then merge — ParallelAgent inside a SequentialAgent

Goal. Build a "MuleSoft org health check" that fires two research agents in parallel, then a merger summarises both. This is composition — a workflow agent inside a workflow agent.

The shape:

SequentialAgent
├── ParallelAgent
│   ├── api_health_researcher   → state["api_health_result"]
│   └── deploy_history_researcher → state["deploy_history_result"]
└── merger  (reads both keys, produces state["health_report"])

Skeleton:

from google.adk.agents.sequential_agent import SequentialAgent
from google.adk.agents.parallel_agent import ParallelAgent

    return {"org_id": org_id, "healthy_apis": 18, "failing_apis": 2}

deploy_history_researcher = ...

#   - instruction MUST interpolate BOTH keys: {api_health_result} and {deploy_history_result}
#   - Ask for a single-paragraph health report

root_agent = ...
Hint — Why put ParallelAgent inside a SequentialAgent?

Because ParallelAgent doesn't produce a single output — it writes multiple state keys. To use those keys (aggregate, summarise, act on them), you need a downstream agent. The idiomatic pattern in ADK is: SequentialAgent(sub_agents=[ParallelAgent(...), merger]). This is the ADK equivalent of "fan-out then join" in traditional pipeline languages.

  3. The merger runs after both complete and produces a single paragraph combining both.
  4. State should contain three keys: api_health_result, deploy_history_result, health_report.

Exercise 3 — Stretch

Draft → critique → refine, terminating with exit_loop

Goal. Build a LoopAgent where a drafter writes an answer, a critic scores it, and the loop repeats until the critic says "good enough" by calling exit_loop.

You get less scaffolding on this one — that's the stretch. You have all the pieces from L28 (ToolContext) and this lesson's LoopAgent code snippet. Build it end-to-end.

Requirements:

  • Folder: refinement_loop/
  • A tool exit_loop(tool_context: ToolContext) that sets tool_context.actions.escalate = True
  • Two LlmAgents: drafter and critic. Only the critic gets exit_loop in its tools.
  • Both agents must read {draft} and/or {critique} from state so the loop actually refines instead of restarting each iteration.
  • root_agent = LoopAgent(sub_agents=[drafter, critic], max_iterations=5)
  • Critic's instruction must be explicit: "If the draft is high quality, CALL exit_loop. Otherwise output specific improvements."

Test prompt: "Write a one-sentence definition of MuleSoft Anypoint Platform." Watch the trace — you should see the loop run 2-3 times before the critic escalates.

Reflection questions:

  1. What happens if you forget max_iterations? What safety concern does this raise for production?
  2. Why does the drafter's instruction need {draft} in it? What happens without it?
  3. Compare this to a LangGraph cycle (L21/L25). Which is easier to read? Easier to debug? Easier to extend with a third participant?
Stuck? — big hint

The drafter's instruction should be something like: "Write or refine a draft answering the user's question. If a previous draft and critique exist, incorporate the feedback. Previous draft: {draft}. Previous critique: {critique}. Output only the new draft." On iteration 1 both interpolations will be empty — that's fine, the drafter treats it as "no previous work".

Quick quiz

1. In a SequentialAgent, how does agent B receive agent A's output?

Show answer

Not by function argument or message-passing. A writes to Session.state[A.output_key]. B's instruction interpolates the same key with {key}. The wiring is via shared session state, not direct hand-off. This is why every sub-agent in a workflow agent typically declares an output_key.

2. Two sub-agents in a ParallelAgent. Can one call the other's tool mid-execution?

Show answer

No. Parallel branches are isolated during execution — "no automatic sharing of conversation history or state between these branches" (from the ADK docs). If branch A needs branch B's output, you're not doing parallel — you're doing sequential. Refactor to SequentialAgent.

3. What are the two ways a LoopAgent can terminate?

Show answer

(1) It hits max_iterations — the hard cap. (2) A sub-agent calls a tool that sets tool_context.actions.escalate = True — the soft, semantic exit. Every well-designed loop should have both: max_iterations as the safety net, escalate as the "we're actually done" signal.

4. You want fan-out research + a summary. What's the idiomatic ADK composition?

Show answer

SequentialAgent(sub_agents=[ParallelAgent(sub_agents=[researcher_A, researcher_B]), merger]). The ParallelAgent writes multiple state keys concurrently. The merger interpolates them into a single instruction and produces the summary. Workflow agents compose — that's the point.
