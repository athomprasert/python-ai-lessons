ADK agent transfer & dynamic routing — supervisor and swarm patterns

Workflow agents are static. Real customer conversations aren't. This is how ADK does what LangGraph L25 called "supervisor" and "swarm" — with one parameter and one auto-injected tool.

Pedagogy reminder. Skeleton + TODOs + paste-back review. The magic in this lesson is invisible — ADK auto-injects a tool the LLM decides to call. That means when it doesn't work, you have to reason about why the LLM didn't pick it. Description-writing is the skill.

Why L29 isn't enough

L29's SequentialAgent, ParallelAgent, and LoopAgent all bake the flow into the code. You decide at author time: "run A, then B, then C." Great for pipelines.

Now consider this MuleSoft support scenario:

  • User: "Why is my invoice wrong?" → route to billing_agent
  • User: "My Anypoint deployment is failing" → route to technical_agent
  • User: "How do I cancel my subscription?" → route to billing_agent

You can't hard-code this in a SequentialAgent. You need the LLM to read the question and pick the right specialist. That's dynamic routing.

The mechanism — sub_agents on LlmAgent

When you set sub_agents=[...] on a regular LlmAgent (not a workflow agent), ADK does three things automatically:

  1. Auto-injects a transfer_to_agent(agent_name: str) tool into the parent agent's tool list.
  2. Constrains agent_name to a JSON-schema enum of the sub-agents' names — so the LLM can't hallucinate a non-existent target.
  3. When the LLM calls transfer_to_agent(agent_name="billing_agent"), ADK sets tool_context.actions.transfer_to_agent = "billing_agent", and the framework re-runs the loop with that agent as the new active agent.
New syntax — sub_agents=[...] on LlmAgent.
You've seen sub_agents before on SequentialAgent and ParallelAgent. The same parameter name on LlmAgent means something totally different: not "run them in a fixed order" but "let me choose to hand off to one of them at any time". Same list, different semantics — read the class name to know which behaviour applies.

Minimal hello-world — a triage agent with two specialists

from google.adk.agents import LlmAgent

# The parent agent doesn't do the answering — it does the routing.
# Its instruction is short: figure out which specialist should handle this.
billing_agent = LlmAgent(
    name='billing_agent',
    model='gemini-flash-latest',
    description="Handles all questions about invoices, subscriptions, payment methods, and refunds.",
    instruction="You are a billing specialist. Answer only billing questions.",
)

technical_agent = LlmAgent(
    name='technical_agent',
    model='gemini-flash-latest',
    description="Handles all technical questions about MuleSoft Anypoint deployments, APIs, and errors.",
    instruction="You are a MuleSoft technical support engineer.",
)

root_agent = LlmAgent(
    name='triage_agent',
    model='gemini-flash-latest',
    instruction="""You are a customer-support triage agent.
Look at the user's question and transfer to the correct specialist.
Do NOT try to answer the question yourself.""",
    sub_agents=[billing_agent, technical_agent],
)

That's it. No transfer_to_agent in the tools list — it's auto-injected. No routing logic in code — the LLM reads each sub-agent's description and picks.

The description field is now load-bearing.
In L26 you could put anything in description and the agent still worked. Not here. The parent LLM's only signal for "which specialist should I pick" is the child's description. Write it like you're writing router documentation: what the specialist covers, in one crisp sentence.

Peer transfer — the swarm pattern

By default, when two agents are peers (siblings under the same parent), either can transfer to the other. This is what L25 called the "swarm" pattern.

# billing_agent can transfer to technical_agent mid-conversation, and vice versa.
# You don't need to configure anything for this — it's the default.

You can turn it off per-agent:

billing_agent = LlmAgent(
    name='billing_agent',
    # ...
    disallow_transfer_to_peers=True,     # can't hand off sideways
    disallow_transfer_to_parent=False,   # can still bubble up (default)
)
When to disallow peer transfer. When you want a strict supervisor pattern where all routing goes through the parent. This makes the routing behaviour much easier to reason about — you have one router and N leaves, instead of N routers.

What "supervisor" and "swarm" mean in ADK vs LangGraph

LangGraph L25ADK L30
Supervisor You write a node that returns Command(goto="billing") Parent LlmAgent with sub_agents=[...] and disallow_transfer_to_peers=True on each child
Swarm Every specialist calls Command(goto="other_agent") when it wants to hand off Peers under a common parent with default disallow_transfer_to_peers=False
Who writes the routing prompt You write it — the LLM calls a function you define ADK writes it — you shape it via description fields
Explicit tool call visible in trace Command(goto=...) in state transfer_to_agent(agent_name=...) tool call event

Composition — putting L29 and L30 together

Nothing stops you from making a sub-agent a workflow agent. This is the full ADK vocabulary:

research_pipeline = SequentialAgent(sub_agents=[researcher, writer])

billing_agent = LlmAgent(name='billing_agent', description="...")

root_agent = LlmAgent(
    name='triage',
    instruction="Route the user to the right specialist.",
    sub_agents=[billing_agent, research_pipeline],   # mix leaf agent + workflow
)

Now the LLM can transfer control to either a simple leaf agent (billing) or a whole two-stage pipeline (research). The parent doesn't care what its children look like internally.

Two gotchas that will bite you in demos

Gotcha 1 — the parent tries to answer instead of transferring. If your parent's instruction says "you are a helpful assistant", the LLM will happily answer everything itself. You have to be explicit: "Do NOT try to answer. Always call transfer_to_agent." Otherwise the router becomes another responder.
Gotcha 2 — control doesn't automatically return to the parent. Once billing_agent takes over, the user's next message goes to billing_agent, not back to triage. To go back, the specialist has to call transfer_to_agent(agent_name="triage"). For most demos this is fine — the specialist handles the whole thread — but for multi-topic conversations, plan the return trip.

The MuleSoft angle

Enterprise customer-support agents almost always need this pattern. A single "MuleSoft copilot" is really 4–6 specialists behind a router:

  • Design specialist — API design, RAML/OAS, Anypoint Design Center
  • Runtime specialist — deployments, CloudHub, Runtime Fabric
  • Security specialist — policies, OAuth, JWT, mTLS
  • Data specialist — DataWeave transformations
  • Billing specialist — subscription, usage, invoices

Trying to shove all this into one system prompt makes the LLM mediocre at everything. Splitting into specialists with a router gets you focused, high-quality answers per topic. That's the story customers want to hear.

Quick quiz

1. You set sub_agents=[a, b] on an LlmAgent called parent. What tool does ADK auto-inject into parent?

Show answer

transfer_to_agent(agent_name: str). It's constrained by a JSON-schema enum to only allow "a" or "b" as valid values, so the LLM can't hallucinate a non-existent target. When the LLM calls it, ADK sets tool_context.actions.transfer_to_agent and re-runs the loop with the new active agent.

2. Your router keeps trying to answer questions itself instead of transferring. What's the fix?

Show answer

Rewrite the parent's instruction to explicitly say "Do NOT answer questions yourself. Always transfer to the correct specialist." Also make sure the parent has tools=[] (or the parameter omitted) — if it has any real tool of its own, it'll be tempted to use it.

3. What's the difference between "supervisor" and "swarm" in ADK terms?

Show answer

Supervisor = disallow_transfer_to_peers=True on all specialists. Every routing decision goes through the parent, giving you one point of control and one clear trace. Swarm = the default (False) — peers can hand off to each other directly, which is more flexible but harder to debug because routing decisions are distributed.

4. The parent LLM has three specialists to choose from. What is its only signal for which one to pick?

Show answer

The description field on each sub-agent. The parent LLM reads the descriptions (surfaced through the auto-injected transfer_to_agent tool's schema) and picks the best match against the user's question. Vague descriptions = bad routing. This is why description is load-bearing in this pattern but decorative in L26.
