Deploying ADK agents to Vertex Agent Engine

One CLI command turns your my_agent/ folder into a managed, auto-scaling, GCP-native service with built-in tracing, IAM, and session persistence. This is the pitch that closes the deal.

Pedagogy note — different from prior lessons. Exercises 1 and 2 need a real GCP project with billing enabled. If you don't have one right now, read them as walkthroughs — the concepts still land. Exercise 3 (cost + trace analysis) needs Exercise 1 to be actually deployed.

The three ADK deployment options

TargetBest forOps overheadManaged session storage?
Vertex Agent Engine Production ADK agents on GCP. The Google-recommended path. None. Fully managed. Yes — VertexAiSessionService auto-wired.
Cloud Run You already run everything else on Cloud Run and want the agent alongside it. Low. You get a container URL; you bring session storage. No — bring your own (e.g. DatabaseSessionService against Cloud SQL).
GKE Enterprise SRE teams that want full container-orchestration control. High. Cluster ops, autoscaling policies, networking. No — bring your own.

This lesson covers Agent Engine — the ADK-native option. It's the story that makes ADK compelling vs LangGraph: LangGraph gives you no equivalent one-command deploy, so you would have to build the Cloud Run wrapper, wire the session store, and expose the endpoint yourself.

What "Agent Engine" (formerly "Reasoning Engine") gives you for free.
  • A stable resource name: projects/{project}/locations/{region}/reasoningEngines/{id}
  • Managed autoscaling (0 → N instances)
  • Cloud Trace + Cloud Logging out of the box
  • IAM (only callers you authorise can query the agent)
  • VertexAiSessionService automatically wired — no extra DB to provision
What about on-premise? (The question every MuleSoft customer asks.)

Agent Engine is GCP-only — no on-prem install. But ADK itself is just a Python library, so you have three fallback paths for regulated customers who can't put agents in a public cloud:

PathHowWhat you lose
Container on your infra Wrap the agent in Docker (using adk api_server or a FastAPI shell) and run on your K8s / OpenShift / VMs. You now own scaling, session storage, and observability. Everything Agent Engine did for free.
GKE on-prem / Anthos Same container, Google's on-prem Kubernetes distribution. Not pure on-prem — still a GCP subscription, hybrid model.
Air-gapped (no internet) Same container plus swap the LLM. Gemini needs egress to AI Studio or Vertex — neither works air-gapped. Use a self-hosted model (Llama, Mistral, Gemma) via Ollama or vLLM, plugged in through the LiteLlm adapter (from google.adk.models.lite_llm import LiteLlm). Gemini quality. Plus you now run a GPU cluster.

What doesn't come along for the ride:

  • VertexAiSessionService → replace with DatabaseSessionService against on-prem Postgres (see L28 Exercise 3).
  • Cloud Trace + Cloud Logging → replace with OpenTelemetry → your existing stack (Datadog, Splunk, Grafana).

The middle-ground question you'll get: "Can we run the agent in our VPC but still use Gemini?" Yes — via Vertex AI Private Service Connect. The LLM call goes over a private GCP link instead of the public internet, but the model still runs in Google's data centres. Standard compromise for banks and healthcare.

The dev → prod switch

Every ADK agent you've built in L26–L30 used one of two auth modes. Which one you're on determines which env vars matter:

Dev — AI StudioProd — Vertex
Signal env var GOOGLE_API_KEY=... GOOGLE_GENAI_USE_VERTEXAI=TRUE
Required extras None GOOGLE_CLOUD_PROJECT, GOOGLE_CLOUD_LOCATION
Auth API key gcloud application-default credentials
Billing Free tier + pay-as-you-go per request Vertex AI billing tied to your GCP project
Zero code changes. The same agent.py from L26 runs in both modes. ADK reads the env vars at startup and picks the right backend. This is the "same agent, dev-to-prod without rewrites" promise.

The deploy command — adk deploy agent_engine

This is the single command that turns a folder into a running service:

# From the directory containing my_agent/
adk deploy agent_engine \
    --project=my-gcp-project \
    --region=us-central1 \
    --display_name="MuleSoft Helper" \
    --description="MuleSoft Anypoint copilot" \
    my_agent

What the CLI does under the hood:

  1. Reads my_agent/ — your agent folder from L26.
  2. Auto-generates a requirements.txt (or extends yours) with the correct google-cloud-aiplatform[adk,agent_engines] dependency.
  3. Packages the folder, uploads it to a Google-managed staging area.
  4. Provisions a Reasoning Engine resource in Vertex.
  5. Returns the resource name — copy this, you'll need it.

Typical output ends with something like:

Successfully created Agent Engine.
Resource name: projects/123456/locations/us-central1/reasoningEngines/7516195516779
The --staging_bucket flag is deprecated. Older tutorials mention it — you don't need it anymore. ADK manages staging automatically.
To update an existing agent, pass --agent_engine_id. Without it, every deploy creates a new Reasoning Engine resource and you'll end up with 20 unused agents in your project. Copy the ID from the first deploy's output and pass it back on subsequent deploys.

Calling a deployed agent from Python

Once deployed, the agent is a Vertex resource you can query from any Python client — a notebook, a Cloud Function, a MuleSoft flow calling out via HTTP.

import vertexai
from vertexai import agent_engines

vertexai.init(project="my-gcp-project", location="us-central1")

# Get a handle to the deployed agent by resource name
remote_agent = agent_engines.get(
    "projects/123456/locations/us-central1/reasoningEngines/7516195516779"
)

# Create a session (this is where VertexAiSessionService comes in — it's
# the backing service for these calls, but you don't instantiate it directly)
session = remote_agent.create_session(user_id="attapol")

# Send a query — streams events back
for event in remote_agent.stream_query(
    user_id="attapol",
    session_id=session["id"],
    message="What's the status of org acme-prod?",
):
    print(event)

Same conversational primitives you had on your laptop (user_id, session_id, message) — now hitting a managed GCP service.

REST is also available. If you want to call from something non-Python (e.g., a MuleSoft flow), the resource exposes a REST endpoint at https://{region}-aiplatform.googleapis.com/v1/{resource_name}:streamQuery. Auth is via a Google Cloud IAM identity token. This is the integration point for putting an ADK agent inside a MuleSoft flow — you're calling one Google API from another.

Session persistence in production

You saw VertexAiSessionService in L28's session-service table but never used it. In production on Agent Engine, it's already wired — every create_session call goes through it. No code change needed. Your L26/L28 agent, deployed via adk deploy agent_engine, automatically gets:

  • Sessions that survive process restarts
  • Sessions that survive autoscaling (any instance can pick up any session)
  • Per-user session isolation
  • Session lifecycle management via the SDK (list_sessions, delete_session)

This is the killer feature. In LangGraph you'd wire a Redis/Postgres checkpointer, provision the DB, handle failover, manage connection pools. Here — nothing to do.

Observability — Cloud Trace & Cloud Logging

Every deployed agent emits structured traces to Cloud Trace and structured logs to Cloud Logging automatically. Open the GCP console and you'll see:

  • Cloud Trace: a span tree per invocation showing every LLM call, tool call, sub-agent transfer with latency and token counts. This is your L27+L30 trace panel, but for the deployed instance.
  • Cloud Logging: raw request/response logs, with the query, session ID, and any exceptions from tool execution.
  • Cloud Monitoring: latency percentiles (p50, p95, p99), request rate, error rate — the SRE dashboard, free.

Your MuleSoft customers already have Cloud Trace enabled for their existing GCP workloads. The agent slots straight into their existing observability without any new dashboards to introduce.

Cost model — the back-of-envelope

Two things you pay for:

  1. LLM tokens. Gemini prices per input + output token, no different from calling the Gemini API directly. This dominates cost.
  2. Agent Engine runtime. Priced per vCPU-hour + GB-memory-hour of the underlying reasoning engine. Autoscales to zero when idle — so for a demo used sporadically, this is close to nothing.

Rough example — 10,000 requests/day, average 2 LLM calls per request, ~1 KB prompt + ~500 B response:

ComponentRough monthly cost (USD)
Gemini Flash tokens (60M input, 15M output)~$10
Agent Engine runtime~$30–80 depending on scaling
Cloud Trace + Logging~$5
Total~$45–100/mo
These numbers are illustrative. Check the current Gemini pricing at ai.google.dev/pricing and Vertex AI Reasoning Engine pricing at cloud.google.com/vertex-ai/pricing before quoting a customer.

For a MuleSoft customer already on GCP, ADK on Agent Engine is obviously the right choice for a support agent. This is the story you should be able to tell without slides.

Quick quiz

1. What's the single command that deploys an ADK agent folder to Vertex Agent Engine?

Show answer

adk deploy agent_engine --project=... --region=... my_agent_folder. Optional flags: --display_name, --description, and --agent_engine_id for updates. --staging_bucket exists but is deprecated — don't include it.

2. Your agent works locally with GOOGLE_API_KEY but fails on Agent Engine. What env vars does Vertex mode need instead?

Show answer

GOOGLE_GENAI_USE_VERTEXAI=TRUE, GOOGLE_CLOUD_PROJECT=your-project, GOOGLE_CLOUD_LOCATION=us-central1. Agent Engine mode uses application-default credentials from gcloud auth application-default login, not an API key.

3. Your first deploy created a resource. You want to push a code change. What flag prevents ADK from creating a second resource?

Show answer

--agent_engine_id=<the ID from the first deploy>. Without it, every adk deploy creates a fresh Reasoning Engine, which is expensive-to-orphan and cluttering your project.

4. Compared to running LangGraph on Cloud Run yourself, what are the three biggest things Agent Engine handles for you?

Show answer

(1) Session persistence — VertexAiSessionService is auto-wired, no DB to provision. (2) Autoscaling to zero — no infrastructure tuning. (3) Observability — Cloud Trace + Logging out of the box, no OpenTelemetry setup. Bonus fourth: native GCP IAM on the resource, so you don't build your own auth.
