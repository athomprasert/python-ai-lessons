Prerequisites — the AWS side

Three one-time setup steps. Do these first; the code won't work until they're done.

1. AWS credentials on your machine

Strands uses boto3 under the hood — same credential chain as any AWS SDK. Either:

  • Environment variables — AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION=us-east-1
  • Or a named profile in ~/.aws/credentials — set AWS_PROFILE=your-profile

Quick check from your shell:

aws sts get-caller-identity # should print your account ID + IAM ARN, no error

2. Enable Bedrock model access

Even with valid credentials, Bedrock refuses to serve a model until you explicitly grant your account access to it. This is a one-time click:

  1. Open the Bedrock console in us-east-1
  2. Left sidebar → Model access
  3. Click Enable specific models, tick Claude Haiku 4.5 (Anthropic), submit
  4. Wait ~30 seconds. Status should flip to Access granted.
The #1 gotcha
If you get AccessDeniedException: You don't have access to the model... at runtime, this is almost always it — you enabled the model in a different region, or you enabled a different Claude variant. Model access is per-region.

3. Install the library

pip install strands-agents

That's it — no separate SDK for Bedrock, no framework config file. strands-agents pulls boto3 transitively.

The three new syntax constructs you'll see

Before you write anything, know what to look for. Everything else is plain Python you've already seen.

New syntax · @tool decorator
@tool is a decorator (we covered @decorator in L15) imported from strands. Put it above a plain function and Strands reads the function's name, docstring, and type hints to build the JSON schema the LLM needs to call it. Zero-config schema — same idea as LangGraph's @tool (L19) and Pydantic AI's @agent.tool_plain.
New syntax · BedrockModel(...)
BedrockModel is Strands' provider adapter for Amazon Bedrock. You construct it with a model_id (Bedrock's opaque string ID for a specific model+version) and hand it to the Agent. Strands ships equivalent classes for other providers — OpenAIModel, AnthropicModel (direct API, not Bedrock), LiteLLMModel. Same Agent, swap the model= argument. That's the whole point of the abstraction, and it's what L33 demonstrates end-to-end.
New syntax · agent("prompt") — calling the Agent directly
Strands makes Agent instances callable (via Python's __call__ dunder method). So agent("How do I...?") runs one full agent turn — LLM call, any tool calls, LLM continuation, final text — and returns a result object. Compare this to L26's ADK, where you needed a Runner and async for event in runner.run_async(...). Strands is deliberately more minimal at the entry point.

What just happened under the hood

Same agent loop as L18, but Strands hides it:

  1. You called agent("..."). Strands built a Bedrock Converse API request — messages, tools, system prompt.
  2. Bedrock returned either text (done) or tool_use content blocks.
  3. If tool_use, Strands looked up your Python function by name, invoked it with the arguments the model produced, sent the result back as a tool_result block, and repeated.
  4. When the model returned only text, the loop ended and you got the AgentResult.

That's the entire agent loop from L18 — for _ in range(max_steps), if not response.function_calls, all of it — hidden inside one agent(...) call. The trade-off is the same as every framework you saw in L19: fewer lines to write, one more layer to debug through when things go wrong.

Quick check

Q1. You run python strands_hello.py and get AccessDeniedException: You don't have access to the model with the specified model ID. Most likely cause?

  • Your pip install didn't complete — reinstall strands-agents
  • You haven't enabled Claude Haiku 4.5 in the Bedrock Model access page for the region you're calling — model access is per-region, per-model
  • Your AWS credentials expired — rotate the keys
  • Strands doesn't support Bedrock yet

Q2. How does Strands build the JSON schema for a tool from a Python function?

  • You write it by hand as a dict argument to @tool
  • You define a Pydantic model and pass it to the tool
  • The @tool decorator reads the function's name, docstring, and type hints — same shape as Pydantic AI's @agent.tool_plain and LangGraph's @tool
  • Strands generates it from the function's source code using AST inspection at import time

Q3. Why is Strands' entry point agent("prompt") shorter than ADK's async for event in runner.run_async(...)?

  • Strands makes the Agent instance callable via Python's __call__ dunder, and returns one final result object — hiding the event stream behind the object's attributes (result.message, result.metrics)
  • Strands doesn't stream, so it's less capable
  • ADK is slower
  • Strands blocks the event loop, which is bad practice

Q4. Your agent answers "The status is healthy" for query (b) but result.metrics.tool_calls is empty — the model made up the answer. What's the first thing to fix?

  • Upgrade to Claude Sonnet — Haiku is too small
  • Add a retry loop that forces the tool call
  • Sharpen the docstring on get_anypoint_org_status — Strands sends it verbatim to the LLM, so a vague docstring means the model doesn't know when to call the tool
  • Switch region to us-west-2

Q5. In L33 (next lesson) we'll swap Bedrock Haiku for a different provider without touching the tools or the system prompt. Which line of your strands_hello.py will change?

  • The @tool decorators — each provider has its own decorator
  • Just the model= argument on Agent(...) — swap BedrockModel(...) for OpenAIModel(...), AnthropicModel(...), or LiteLLMModel(...). That's the whole abstraction.
  • The system prompt has to be rewritten for each provider
  • You'd move to a different framework entirely — Strands only supports Bedrock
