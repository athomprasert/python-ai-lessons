The mission moment

From MISSION.md: build demos your customers can trust. One of the first questions from any enterprise buyer is "aren't we locking ourselves into Bedrock / Anthropic / OpenAI if we go with this framework?" The honest answer, when the framework is Strands (or Pydantic AI, or LangGraph): no — the provider is a swap-in.

This lesson shows you exactly what "one-line swap" means, and where the abstraction leaks. Because there's always a leak.

The three new syntax constructs

New syntax · AnthropicModel(...)
Strands' direct-Anthropic provider adapter. Talks to api.anthropic.com, not Bedrock. Model IDs are different too — Bedrock prefixes them (anthropic.claude-...-v1:0), the direct Anthropic API doesn't. Why bother with direct when Bedrock also serves Claude? Latency (fewer hops), price differences, and access to new models the day they release (Bedrock lags by weeks). Requires ANTHROPIC_API_KEY in your environment.
New syntax · LiteLLMModel(...)
Strands' universal remote. LiteLLM is a third-party library that speaks a common protocol to ~100 model providers (OpenAI, Anthropic, Cohere, Groq, Ollama, Together, Perplexity, and many more). The model string carries the provider prefix — openai/gpt-4o-mini, anthropic/claude-sonnet-4-5, ollama/llama3. Great for "try five providers on the same benchmark." Costs a layer of indirection — if something goes wrong, you now have Strands ⊕ LiteLLM ⊕ the provider to debug.
New syntax · list of tuples for parametrised runs
You'll use a list[tuple[str, Agent]] to hold the configured agents — [("Bedrock Haiku", agent1), ("Bedrock Sonnet", agent2), ...]. Tuples with meaningful positions like this are a common Python pattern for labelled fixtures: the string is a display label, the object is what you're actually operating on. We iterate with for label, agent in configs: and get both variables in one line — this is called tuple unpacking, one of Python's small-but-daily conveniences. Nothing Strands-specific, but if you haven't used it yet, this is a good moment to see it.

Setup — the new env vars

Anthropic API key (for config #3)

  1. Log in to console.anthropic.com
  2. Settings → API keys → Create key. Copy it once — it's shown only once.
  3. Add to your .env: 
    ANTHROPIC_API_KEY=sk-ant-...
  4. Load it with python-dotenv (same pattern as L6).
Same rules as always
.env stays in .gitignore. Never paste an API key into a chat, a Slack thread, or a git commit. If a key leaks, revoke it in the Anthropic console immediately and rotate.

Bedrock Sonnet access (for config #2)

In the Bedrock console, Model access page, tick Claude Sonnet 4.5 the same way you ticked Haiku in L32. Per-region, still us-east-1.

OpenAI key (only if doing stretch config #4)

OPENAI_API_KEY from platform.openai.com. Plus pip install litellm.

Verification — how you know it worked

Checklist
  1. All four (or three) runs complete without exceptions. If one provider fails at construction, check its env var / model access step above.
  2. Every run calls search_docs exactly once. If one model doesn't call the tool, you've hit an "abstraction leak" — see the next section. Same query, same tools, different behaviour = the model made a different judgement, not a Strands bug.
  3. The final reply text mentions rotating secrets / Access Management / Connected Apps for every provider. Small phrasing differences are normal.
  4. Nothing in the code except the model= argument differed between runs. If you had to change the system prompt or the tool schemas to make a provider work, note it — that's a leak worth reporting to the customer.

Where the abstraction leaks — be honest with yourself

"One line change" is a great slide bullet. Here's what it hides:

LeakWhat actually happens
Model IDs are provider-specific Bedrock uses anthropic.claude-sonnet-4-5-20250929-v1:0, Anthropic direct uses claude-sonnet-4-5-20250929, LiteLLM wants anthropic/claude-sonnet-4-5. Same underlying model, three strings — you can't map them programmatically.
Tool-call format varies Under the hood, Bedrock Converse, direct Anthropic Messages, and OpenAI ChatCompletions each format tool calls differently. Strands normalises the surface, but a model trained on OpenAI-style tools may call a tool slightly differently than the same model on Bedrock. Usually invisible; occasionally not.
Rate limits + quotas Bedrock quotas live in the AWS account. Anthropic direct quotas live in the Anthropic console. OpenAI has its own. Swapping providers moves the bottleneck.
Pricing shape Anthropic direct and Bedrock don't charge the same for the same Claude model. Bedrock adds an AWS margin; direct is usually cheaper for high volume, but Bedrock bundles into your existing enterprise agreement.
Where your data goes Bedrock = data stays in the AWS region. Anthropic direct = data goes to Anthropic's infrastructure (still SOC-2 etc., but a different trust boundary). This one matters for the customer's security team. Always the first question in real deals.
MuleSoft angle — the story you actually tell
  • To an AWS-shop customer: "Your data stays in your Bedrock account — same VPC, same CloudWatch, same IAM story you already have."
  • To a multi-cloud customer: "The agent code doesn't care. If you decide to move models later, it's one line and a re-deploy — no rewrite."
  • To a cost-sensitive customer: "We can A/B two models behind the same Anypoint Flex Gateway policy — same tools, same prompts, different model=. Pick whichever wins on cost-per-successful-response after a week."
That last one is the killer demo. Same pipeline, real production traffic split 50/50, honest numbers at the end of the week.

Quick check

Q1. Which line(s) of your L32 strands_hello.py did you have to change to run against a different provider?

  • The @tool decorators — each provider has its own tool decorator
  • Only the model= argument on Agent(...). The tools, the system prompt, and the entry point (agent("...")) are provider-agnostic.
  • The SYSTEM string — each provider has its own prompting style you must adapt to
  • The whole file — Strands abstracts the model but not the agent structure

Q2. A customer asks: "If we build on Strands + Bedrock today and want to move to direct Anthropic API next year, what's the migration cost?" Which answer is most honest?

  • Zero — it's the same code, same everything
  • Enormous — you'd rewrite the agent from scratch
  • Small — one line of code changes per agent, but you also inherit new rate limits, new pricing, new data-residency posture, and a different model-ID format. It's a config migration, not a code migration.
  • Impossible — Strands is Bedrock-only

Q3. You run the same query on Bedrock Sonnet and Anthropic direct Sonnet. They give slightly different replies. Why?

  • One of them is broken — file a bug with Strands
  • Bedrock and Anthropic run different fine-tunes of the model
  • Your tools returned different values
  • LLM outputs are non-deterministic by default (temperature > 0). Also, the Bedrock Converse API and Anthropic Messages API package the request slightly differently even for the same underlying model. Small variation is expected; large variation warrants investigation.

Q4. What does LiteLLMModel buy you that BedrockModel and AnthropicModel don't?

  • Lower latency
  • Access to ~100 providers (OpenAI, Cohere, Groq, Ollama, Together, Perplexity, and more) through one adapter — at the cost of an extra library in the debugging path
  • Better tool-call quality
  • Automatic caching of LLM responses

Q5. Which of the following is not a leak in the "one-line swap" abstraction?

  • Model ID strings differ across providers for the same underlying model
  • The Python code for defining tools with @tool changes when you swap providers
  • Rate limits, quotas, and pricing shape change per provider
  • Data residency and security posture change — Bedrock keeps data in-region, direct-Anthropic sends it to Anthropic's infra

Ask follow-ups

Anything unclear — why Bedrock and direct-Anthropic have different pricing, when LiteLLM is the right choice vs a bad idea, how to A/B two providers behind a Flex Gateway — just ask.
