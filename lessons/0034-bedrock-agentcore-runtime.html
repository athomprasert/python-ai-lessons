Bedrock AgentCore Runtime: hand your agent to AWS

Your L32/L33 agent runs on your laptop. Fine for learning; not a story you tell a customer. This lesson introduces AgentCore Runtime — AWS's managed serverless container for any Python agent — and lets you deploy the same code without touching Kubernetes, ECS, or a Dockerfile.

The mission moment

From MISSION.md: agents you can actually demo to enterprise customers. "It runs on my laptop" is where every prototype starts and where every prototype dies. When the customer asks "OK, how do we operationalise this in our AWS account?" — that's what this lesson answers.

What AgentCore actually is — the mental model

The biggest mistake is treating AgentCore as one thing. It's seven managed primitives. You use only the ones you need. All are framework-agnostic — they work with Strands, LangGraph, CrewAI, ADK-in-a-container, or a hand-rolled Python script.

The two new syntax constructs

New syntax · BedrockAgentCoreApp()
BedrockAgentCoreApp is the runtime SDK's app object. Think of it the way you think of Flask's app = Flask(__name__) or FastAPI's app = FastAPI(): it's the container that AgentCore Runtime will serve. You instantiate one at module top level, decorate one or more functions as entrypoints, and end with app.run() for local testing. The agentcore CLI reads this object to know what to package up when you deploy.
New syntax · @app.entrypoint
@app.entrypoint is a decorator (L15 covered the @decorator pattern) that registers the function below it as the callable AgentCore invokes on each request. The function receives a payload dict (the HTTP request body, parsed as JSON) and returns whatever should go back to the caller (typically another dict or a string). Whatever your agent does inside — Strands, LangGraph, native SDK, doesn't matter — AgentCore doesn't care. Same shape as an AWS Lambda handler, if you've written one of those.

Prerequisites — what you need before pip install

  1. AWS creds working — from L32. aws sts get-caller-identity should still succeed.
  2. Bedrock model access for Haiku (from L32) and Sonnet (from L33) in us-east-1.
  3. IAM permissions for AgentCore. Your user/role needs to be able to create AgentCore Runtimes and pass a service role. Easiest for a demo: attach the AWS-managed policy BedrockAgentCoreFullAccess (name may vary in your region — check the current console).
  4. An AgentCore execution role — the role AWS assumes to run your container. AgentCore creates one on demand via the CLI; you don't need to hand-craft it for a first deploy.
  5. Install the runtime SDK: 
    pip install bedrock-agentcore
    This gives you the BedrockAgentCoreApp class and the agentcore CLI.
Cost heads-up
Runtime bills for container invocation time and Bedrock token usage. For a hello-world with one query, expect single-digit US cents. Set a billing alert before you deploy for the first time — same discipline as L32.

Verification — the four-step ladder

Test locally before touching the cloud. This is the single most useful habit for AgentCore work.

Rung 1 · Import test
python -c "import agentcore_hello" exits with no error. If it fails, the SDK isn't installed or a Strands import broke.
Rung 2 · Local server
Run python agentcore_hello.py. You should see a startup log mentioning port 8080. In another terminal: 
curl -X POST http://localhost:8080/invocations \
  -H "Content-Type: application/json" \
  -d '{"prompt": "How do I rotate a Connected App secret?"}'
You should get a JSON reply with the same content you got in L32. If the local server won't start, port 8080 is probably occupied — lsof -i :8080 will show you what's holding it.
Rung 3 · Deploy
From your project directory: 
agentcore configure --entrypoint agentcore_hello.py
agentcore launch
The CLI builds a container image, pushes to ECR, provisions a Runtime, and gives you back an ARN plus an invoke URL. Read every line of the output — the ARN is what you'll pass to callers later.

Exact CLI verbs and flags may differ in your version — agentcore --help is the source of truth.
Rung 4 · Cloud invoke
agentcore invoke '{"prompt": "How do I rotate a Connected App secret?"}'
Same reply as the local run. If not, look at CloudWatch logs in the AWS console — AgentCore streams your container's stdout/stderr there automatically. This is the payoff for using a managed runtime.

What just happened, architecturally

  1. The agentcore CLI built a container image from your Python file + requirements.
  2. Pushed the image to Amazon ECR under an AgentCore-managed repository.
  3. Created an AgentCore Runtime resource in the AWS control plane — pointing at that image, using the execution role, with autoscaling defaults.
  4. Exposed an invocation endpoint. Every call to agentcore invoke (or the SDK, or a signed HTTPS request) launches a container instance (or reuses a warm one), calls your @app.entrypoint function, streams the response back.
  5. Each session gets an isolated microVM. That's the built-in security model — sessions can't leak state to each other.

Compare that to what you'd write yourself: Dockerfile, ECR repo, ECS task definition, ALB listener, IAM role, CloudWatch log group, autoscaling config, session isolation. AgentCore is that whole stack, but you filled in one Python file.

Where the abstraction leaks

Same discipline as L33 — name the leaks up front so you're not embarrassed by them in a customer meeting.

  • Cold-start latency. First invocation after idle spins up a microVM. Milliseconds-to-seconds range depending on image size. Warm invocations are fast.
  • Container size affects deploy time. A heavy dependencies file (LangChain + LlamaIndex + Haystack + Strands in one image) will make each agentcore launch slow. Keep the deployable minimal.
  • Runtime is not free-tier. Unlike EC2's free tier, AgentCore Runtime bills from the first invocation. Cheap, but not free.
  • Region availability. AgentCore rolls out region-by-region. If your customer needs a specific region, check the current availability page before promising anything.
  • The AWS docs are the source of truth. If a code sample here doesn't run, the API almost certainly moved — this is by far the fastest-evolving corner of AWS's AI stack.

Quick check

Q1. What is Bedrock AgentCore, in one sentence?

  • A specific agent framework — you rewrite your Strands agent into AgentCore syntax to use it
  • A family of framework-agnostic AWS-managed primitives (Runtime, Memory, Identity, Gateway, Observability, Browser, Code Interpreter) — you pick the ones you need, not the whole set
  • AWS's answer to LangGraph — a graph-based agent runtime
  • A managed vector database for agents

Q2. Which AgentCore primitive did this lesson cover, and what does it do?

  • Memory — stores conversation history for each session
  • Gateway — turns existing REST APIs into agent tools
  • Runtime — serverless container that hosts any Python agent, framework-agnostic, with per-session isolation
  • Browser — sandboxed headless browser the agent can drive

Q3. Your L32 strands_hello.py works locally. You copied the tools + system prompt into agentcore_hello.py and wrapped it in BedrockAgentCoreApp(). What's the smallest verification step you should run before agentcore launch?

  • Run python agentcore_hello.py locally and curl -X POST http://localhost:8080/invocations — same code path as production, no cloud costs, fast feedback
  • Skip local testing — agentcore launch is fast enough
  • Deploy to a staging AWS account first
  • Run the unit test suite for AgentCore itself

Q4. How is AgentCore Runtime different from Vertex Agent Engine (L31) from a framework point of view?

  • Both are framework-specific — Vertex only runs ADK, AgentCore only runs Strands
  • Vertex Agent Engine is designed around ADK (best fit; others via adapters), while AgentCore Runtime is framework-agnostic — Strands, LangGraph, CrewAI, ADK-in-a-container, or a raw script all deploy the same way
  • Both are framework-agnostic — no difference
  • Both only accept containers built from a specific base image published by the cloud

Q5. An AWS-committed customer asks: "If we build our agent on AgentCore, how portable is it if we want to run it on our own ECS cluster later?" Most honest answer?

  • Not portable — AgentCore locks you in
  • Fully portable — just docker pull the image and run it anywhere
  • The Python agent code (Strands + tools + prompt) is fully portable. The AgentCore wrapper (BedrockAgentCoreApp, @app.entrypoint) would need to be replaced with something like FastAPI. You'd also lose the managed session isolation, autoscaling, and observability wiring — but the core agent moves cleanly.
  • Portable only if you use LangGraph instead of Strands
