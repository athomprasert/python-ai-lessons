AgentCore Memory: your agent remembers who it's talking to

The L34 deployment forgets everything between invocations. Fine for one-shot Q&A, useless for a real assistant. This lesson wires managed AgentCore Memory into your Runtime — short-term session history and long-term facts, both on-tap from AWS.

The mission moment

Your L34 agent is live on AWS. Great — but try this: send it "I'm working on org 12345, any known issues?", then follow up with "what org did I say?". It has no idea. Every call to @app.entrypoint starts from a blank slate.

Every real assistant needs at least a working memory of the current conversation, and often a long-term memory of who this user is and what they care about. AWS's answer is AgentCore Memory — a managed store you can query on each invocation. Same problem L28 solved for ADK; this is the AWS side.

Source honesty
AgentCore Memory GA'd in late 2025. The concepts (short-term events, long-term strategies) are stable; the exact API method names may have drifted. Cross-check with the current AWS AgentCore docs before wiring this into anything a customer will see.

Two kinds of memory — the mental model

The mistake newcomers make is treating "memory" as one thing. AgentCore Memory offers two distinct capabilities that solve two distinct problems.

Short-term (event history)Long-term (extracted strategies)
What it stores Raw turn-by-turn conversation events for one session Extracted facts / preferences / summaries derived from the raw events
Scope One session (one session_id) One actor (user) — spans all their sessions
Write path Your code writes each turn via create_event AgentCore extracts asynchronously, on its schedule, using strategies you configured up front
Read path list_events(session_id=...) returns the recent turns retrieve_memories(actor_id=..., query=...) — semantic search over extracted memories
Latency Fast (list-events is bounded) Slower (semantic search over a vector-backed store)
Analogy The transcript sitting on your desk The notes you took after the meeting ended

For most first assistants, short-term is enough. Long-term becomes worth its cost once you have repeat users with recognisable preferences ("Attapol is on us-east-1, prefers concise answers, works with MuleSoft").

The three new syntax constructs

New syntax · MemoryClient
MemoryClient (from bedrock_agentcore.memory) is the SDK class you use for every memory operation — create the memory resource, save events, retrieve strategies. Think of it as the boto3-style client but for the Memory primitive. You typically construct one at module top-level and reuse it across invocations. It reads AWS creds from the standard chain (same as everything else in L32–L34).
New syntax · strategies at creation time
When you call client.create_memory(...), you pass a list of strategies — recipes AgentCore uses to build long-term memories from raw events. Three common built-ins are: SEMANTIC (extracts general facts), USER_PREFERENCE (preferences and settings), and SUMMARIZATION (rolling summaries of long conversations). You can also define custom strategies with your own extraction prompt. The strategies are declared once, then AgentCore runs them asynchronously against every event you write. No batch jobs to schedule — that's the managed part.
New syntax · actor_id vs session_id
Every memory event has an actor_id (who — usually a stable user ID like an email or IAM sub) and a session_id (which conversation). Short-term memory is scoped to session_id; long-term strategies are scoped to actor_id. This is the same two-level scoping ADK uses (user_id + session_id) — see L28. Get this right on day one, or you'll be migrating memory scopes later.

Prerequisites

  1. Everything from L34 still works. Your agentcore_hello.py deploys and responds.
  2. IAM permissions cover AgentCore Memory (attaching BedrockAgentCoreFullAccess as in L34 covers this too).
  3. No new pip install — bedrock-agentcore already includes the memory client.

Your turn — a two-file exercise

Memory works in two phases: set up once (create the memory resource), then read + write on every invocation. So we split into two files.

File 1 — setup_memory.py (run once)

You only create the memory resource once. Running this file twice would create duplicates — real production code checks-then-creates, but for learning, just remember to run it once.

Hint on prompt formatting
A minimal but effective preamble: 
Recent conversation:
user: I'm working on org 12345.
assistant: Got it — I can look that up.

Current message:
what org did I say?
You're just giving the model context it wouldn't otherwise have. Not magic. Strands doesn't need a special API for this — the whole conversation is text.

Verification — the four-part memory check

Test 1 · Same-session recall (the whole point)
Two invocations, same session_id: 
curl -X POST http://localhost:8080/invocations -H "Content-Type: application/json" \
  -d '{"session_id":"s1","actor_id":"att","prompt":"I am working on org 12345."}'

curl -X POST http://localhost:8080/invocations -H "Content-Type: application/json" \
  -d '{"session_id":"s1","actor_id":"att","prompt":"What org did I say?"}'
Second reply should include "12345". If it doesn't, your load_history is returning empty or your prompt-preamble formatting is wrong.
Test 2 · Different-session isolation
Same actor_id, different session_id. The second session should not know about org 12345 (short-term is per-session). If it does, your scoping is off — you probably passed the wrong ID to list_events.
Test 3 · Long-term extraction (async — be patient)
Run five turns in one session mentioning "MuleSoft" and "Anypoint". Wait 1–2 minutes. Then in a new session for the same actor_id, ask "what do you know about me?" and use mem.retrieve_memories(...) to surface strategies. The SEMANTIC strategy should have extracted something. If it hasn't, wait longer — extraction is deliberately async so it doesn't block invocations.
Test 4 · Cloud parity
Redeploy with agentcore launch and re-run tests 1 and 2 against the cloud endpoint. Same behaviour. If it works locally but not in the cloud, your Runtime's execution role is missing memory permissions.

Where the abstraction leaks

  • Long-term extraction is asynchronous. A memory you write now becomes a retrievable strategy later — seconds to minutes. Don't build UX that assumes immediate long-term recall.
  • Every invocation now makes 2+ extra API calls (list events, create event). Latency budget: usually low tens of milliseconds each, but they add up. For high-QPS agents, batch or skip on cache hits.
  • Storage bills. You pay per event stored and per strategy extraction. For a hello-world you'll never notice; for a 10k-user production agent you'd want a retention policy tighter than 30 days.
  • PII is your problem. AgentCore stores what you send. If users type credit card numbers into the prompt, that's in the memory store. Redact at the ingress, not after.
  • Right-to-delete is your problem too. AWS gives you delete APIs; wiring them to your app's account-deletion flow is up to you. Enterprise buyers will ask.

MuleSoft angle
Every enterprise buyer will ask three memory questions. Have answers ready:
  1. Retention — "How long do we keep conversation data?" Answer: whatever your event_expiry_days + long-term deletion policy says. Not "forever."
  2. Right-to-delete — "How do we handle a GDPR erasure request?" Answer: call the memory-delete API keyed by actor_id when the user's account is deleted. Show them your wiring.
  3. Data residency — "Where does this data live?" Answer: the AWS region your memory resource was created in. AgentCore doesn't replicate across regions unless you explicitly configure it.
Managed memory doesn't make these questions go away — it just gives you clean APIs to answer them with. Also: if the customer already has a source-of-truth for user preferences (Anypoint DataGraph, Salesforce Data Cloud), sometimes the right answer is "don't use long-term Memory at all — read from your existing system". Managed features aren't obligations.

Quick check

Q1. A user says "I'm working on org 12345" in turn 1. In turn 2 of the same session, they ask "what org did I say?" and the agent has no idea. Most likely cause?

  • The model is too small — upgrade to Sonnet
  • Your invoke function isn't loading events with list_events(session_id=...) before calling the agent, or the loaded history isn't being included in the prompt. Memory doesn't feed itself into the model — you have to do it.
  • AgentCore Memory doesn't support recall in the same session
  • The SEMANTIC strategy hasn't extracted anything yet

Q2. What's the difference between short-term and long-term memory in AgentCore?

  • Short-term is in-memory; long-term is on disk
  • Short-term is free; long-term costs money
  • Short-term stores raw turn-by-turn events scoped to one session_id. Long-term stores extracted facts/preferences/summaries scoped to one actor_id, built asynchronously by strategies you configured at memory-creation time.
  • Short-term is for text, long-term is for embeddings

Q3. Why can't you just use a Python dict at module scope to store conversation history in an AgentCore Runtime?

  • Because AgentCore Runtime uses per-session microVMs — different invocations may land on different containers, so an in-process dict isn't shared across them. Managed Memory sits outside the containers, which is exactly why it works.
  • Python dicts aren't thread-safe
  • You can, actually — that's the recommended approach
  • A dict can't hold conversation events

Q4. You wrote five turns to a session mentioning "MuleSoft" and immediately queried retrieve_memories(actor_id=..., query="MuleSoft") in a new session. Nothing comes back. Why?

  • You mis-typed the actor_id
  • The SEMANTIC strategy wasn't included in create_memory
  • Both of the above are possible — but the most common cause is that long-term extraction is asynchronous. Strategies run seconds-to-minutes after the events are written. Wait, then re-query.
  • AgentCore doesn't support cross-session retrieval

Q5. An enterprise buyer asks: "If a user in the EU exercises their GDPR right-to-erasure, how do we honour that in this agent?" Which answer is most professional?

  • "AgentCore handles that automatically — nothing to worry about."
  • "AWS gives us delete APIs on the Memory resource keyed by actor_id. We wire those into our account-deletion flow so that when a user is deleted from our system, their memory events and extracted strategies are removed. AgentCore gives us the tools; the wiring is our responsibility. Happy to walk your DPO through the retention policy and delete flow."
  • "We'd have to switch to a different platform for GDPR compliance."
  • "We keep everything forever — the audit trail is more valuable than compliance."

Ask follow-ups

Anything unclear — the difference between "event" and "strategy", when to use SUMMARIZATION vs SEMANTIC, how to handle a user who switches devices mid-conversation, how to redact PII before create_event — just ask.
