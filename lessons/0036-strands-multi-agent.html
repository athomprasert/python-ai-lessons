Multi-agent in Strands: three patterns, one problem

One agent doing everything ends up mediocre at all of it. Real integration work wants a coordinator plus specialists. Strands ships three multi-agent patterns natively — agent-as-tool, swarm, and graph. This lesson teaches you when to reach for each.

The mission moment

Your L32 agent has two tools: search_docs (retrieval) and get_anypoint_org_status (live API). A user asks "What's the status of org 12345, and what does the runbook say about degraded state?". One agent, one system prompt, one context window juggling both jobs.

That works — until you add a third specialist (billing lookups), then a fourth (log analysis). The system prompt balloons; the model picks the wrong tool; debugging becomes archaeology. The industry answer is the same as any org's answer: split into specialists, add a coordinator. Strands supports three flavours of that split, and picking the wrong flavour is the most common multi-agent mistake.

Rule of thumb: start with agent-as-tool. Move to graph when you can draw the workflow on paper. Reach for swarm only when you genuinely can't predict who should act next — and be prepared for harder debugging.

The three new syntax constructs

New syntax · agent-as-tool (the wrapper trick)
You already know @tool wraps a plain function. The trick: the function body can itself run an Agent. So the "tool" the outer LLM sees is actually a fully-formed specialist. The docstring on that wrapper function is what the outer agent uses to decide when to call it — treat it like a job description, not a comment. 
@tool
def docs_specialist(question: str) -> str:
    """Answer MuleSoft documentation questions. Use this when the user asks
    how something works, best practices, or configuration details."""
    return docs_agent(question).message
No new import, no new class — @tool from L32 does all the work.
New syntax · Swarm from strands.multiagent
Swarm(nodes=[agent_a, agent_b, ...]) creates a group where each agent can call a built-in hand-off tool to pass control to a named peer. You invoke the swarm with a prompt; the first agent runs, decides whether to answer or hand off, and so on. Termination is either "an agent produced a final answer" or a hop-count safety limit. 
from strands.multiagent import Swarm

swarm = Swarm(nodes=[triager, docs_expert, ops_expert])
result = swarm("What's the status of org 12345 and what does the runbook say?")
print(result.final_message)
The critical detail: each agent's system prompt must name its peers ("hand off to docs_expert for documentation questions") — otherwise the hand-off tool has no signal on where to route.
New syntax · GraphBuilder (the DAG)
GraphBuilder is a fluent builder for a directed acyclic graph. You .add_node(agent, "name") for each specialist, .add_edge("a", "b") for each transition, .set_entry_point("name"), then .build(). Executing the graph runs nodes in topological order, threading state between them. This is Strands' answer to the LangGraph StateGraph you saw in L25 — same concept, AWS-native. 
from strands.multiagent import GraphBuilder

graph = (GraphBuilder()
    .add_node(triager,    "triage")
    .add_node(docs_expert, "docs")
    .add_node(ops_expert,  "ops")
    .add_edge("triage", "docs")
    .add_edge("triage", "ops")
    .set_entry_point("triage")
    .build())

result = graph("What's the status of org 12345 and what does the runbook say?")

Source honesty
Strands' strands.multiagent package (Swarm + GraphBuilder) landed in mid-2025 and is still evolving. Method names (.add_node vs .node, Swarm(nodes=) vs Swarm(agents=)) may shift version-to-version. The concepts — coordinator+tool, peer hand-off, DAG — are stable across every agent framework. Verify names against the current Strands docs when it matters.

Verification — one query, three shapes

The same question, three traces
Run QUERY = "What's the status of org 12345, and what does the runbook say about degraded state?" against each pattern. Expect roughly:
  • Agent-as-tool — coordinator's LLM makes two tool calls (ops_specialist, then docs_specialist), then composes a merged answer. One outer trace, two inner traces nested inside it.
  • Swarm — triager hands off to ops_expert (or docs_expert), which answers part, then hands off to the other, which answers the rest. Trace shows a chain of hand-offs. Order can vary run-to-run — that's the swarm's job.
  • Graph — triage node runs first, then both docs and ops run (fan-out) because both edges exist. Trace is deterministic and matches the graph you drew.
The wrong-pattern test
Deliberately mis-choose the pattern for a task, then feel it:
  • Use swarm for a task where you already know the answer flow (say, "fetch org, then check runbook, always in that order"). You'll waste tokens on the routing overhead and get non-deterministic ordering.
  • Use graph for an open-ended task ("help me investigate this incident"). You'll either need every possible edge or the graph will dead-end.
Doing this once cements why the choice matters more than any table.

Where the abstraction leaks

  • Token bill multiplies. Every specialist has its own system prompt, its own context, its own tool descriptions. N agents ≈ N times the input tokens per turn. Don't split for splitting's sake.
  • Latency stacks serially. Agent-as-tool and swarm hand-offs are inherently sequential — call B waits for call A. Graph fan-out can parallelise, but only if you actually run parallel edges.
  • Swarms loop. A poorly-prompted swarm can hand off in circles. The hop-count safety limit rescues you, but a burnt-out user rarely comes back. Set the limit; monitor it; alert on hits.
  • Debugging swarms is hard. Emergent control flow means "why did it pick B?" has no source-of-truth. Prefer graph or agent-as-tool for anything a customer will run.
  • Docstrings are load-bearing. In agent-as-tool, the outer LLM's only signal for when to call a specialist is that specialist's docstring. Terse or vague docstrings = wrong specialist picked. Treat them like a routing spec.

MuleSoft angle
Multi-agent is the org-chart pattern applied to LLMs. Which sounds abstract until you notice you've been selling exactly this for years:
  • An omni-channel Experience API (the coordinator) that routes to
  • Process APIs (the specialists) that orchestrate against
  • System APIs (the tools) that touch systems of record.
A multi-agent architecture is the same three tiers, with LLMs where you used to have DataWeave. For customers considering their first agent build, that mapping is a strong opener: "you already know how to design this — you've done it in Anypoint. The layer names don't change; only what runs inside each box does." And when a customer asks "can we just make the coordinator smarter and skip the specialists?" — the answer is the same as in API-Led: yes, until scope grows. Then you split.

Quick check

Q1. Your customer wants an agent that always follows the same pipeline: extract intent → look up the customer → fetch the relevant policy → respond. Best Strands pattern?

  • Swarm — let the agents figure out the order
  • Graph — the workflow is fully predictable, so a DAG gives you deterministic execution, easy debugging, and the fastest onboarding for the next engineer
  • Agent-as-tool — always cheapest
  • A single monolithic agent — three steps isn't enough to warrant splitting

Q2. In the agent-as-tool pattern, what does the outer agent's LLM use to decide when to call a specialist?

  • The specialist agent's system prompt
  • The tools registered on the specialist
  • The docstring on the @tool-wrapped function that hosts the specialist call — that docstring is the specialist's job description, and treating it like a comment (or leaving it terse) is the #1 reason routing goes wrong
  • A special @specialist decorator

Q3. You built a Swarm with three agents and their system prompts describe their own job in detail but never mention the other two agents. Runs go in circles or fail to hand off. Why?

  • Because the built-in hand-off tool routes by name, and each agent's LLM has no signal on who to hand off to unless the system prompt names its peers and their roles
  • Because Swarm requires exactly two agents
  • Because the model is too small
  • Because you didn't set an entry point

Q4. Which is a valid criticism of the swarm pattern for production agents?

  • It only works with Bedrock models
  • It's not compatible with AgentCore Runtime
  • It doesn't support Python 3
  • Its emergent, LLM-driven control flow makes debugging genuinely hard — "why did it pick agent B here?" has no deterministic source-of-truth. That's an acceptable price for open-ended tasks, but often unacceptable for regulated or high-stakes workflows

Q5. A MuleSoft customer says: "We already have Experience/Process/System APIs. Do we need multi-agent at all?" Which framing is most useful?

  • "Yes, always — multi-agent is the future"
  • "You already have the multi-agent architecture. Your Experience APIs are coordinators, Process APIs are specialists, System APIs are tools. What we're deciding is whether the coordinator's routing brain should be an LLM (multi-agent) or DataWeave (classic API-Led). Same three tiers; different runtime inside each box. The answer depends on how much fuzzy natural-language routing the request actually needs."
  • "No — LLMs are a passing trend"
  • "Only if you use AWS — Strands doesn't run on GCP"
