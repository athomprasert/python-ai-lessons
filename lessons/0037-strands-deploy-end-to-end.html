The mission moment

L34 got you a deployed agent that talks to itself. L35 gave it memory. L36 split it into specialists. Each was fine in isolation. But every real agent hits the same three questions before it ships:

  1. Who's calling? — Your actor_id can't be trusted from a JSON payload. You need verified identity.
  2. What happened? — When it misbehaves at 2am, you need traces, not print statements.
  3. Does it still work? — Composing five primitives is where things quietly break. You need a verification checklist that touches each seam.

This lesson answers those three. Two new AgentCore primitives (Identity, Observability) join the four you already know (Runtime, Memory, the agent, the model).

Source honesty
AgentCore Identity and Observability configuration is the piece most likely to have drifted since this lesson was written — auth configuration in cloud SDKs turns over fast. The architecture (JWT authorizer verifies token → claims flow to the entrypoint → your code uses the verified sub as actor_id) is timeless. Verify exact API/CLI names against the current AWS AgentCore docs before wiring for a customer.

The three new syntax constructs

New syntax · request context in @app.entrypoint
Until now, invoke(payload) took only the JSON body. The Runtime SDK also gives you the incoming request's context — headers, verified JWT claims, request ID. The signature is invoke(payload, context) (or the SDK may expose it via a decorator argument or module helper — check your installed version). The critical bit: after the Identity authorizer verifies the JWT, its claims are attached to the context. That's where a trustworthy actor_id comes from. 
@app.entrypoint
def invoke(payload, context):
    # context.claims["sub"] came from a verified token, not the payload.
    actor_id = context.claims["sub"]
    ...
Rule of thumb: never read actor_id from payload in a production build. That's a classic auth-vs-authz mix-up.
New syntax · AgentCore Identity resource
Identity in AgentCore is a resource (not just an SDK call). You create an identity that references an OAuth provider — Cognito user pool, Okta issuer, or a generic OIDC provider (same shape as better-auth's genericOAuth, which you saw in idp-document-frontend). Then you tell your Runtime to require that identity for incoming requests. Typical CLI flow (exact commands may differ): 
agentcore identity create --oauth-provider cognito \
    --user-pool-arn arn:aws:cognito-idp:us-east-1:...:userpool/us-east-1_abc
agentcore configure --entrypoint agentcore_shippable.py --identity my-agent-identity
agentcore launch
Once launched, /invocations requires Authorization: Bearer <jwt>. No token → 401.
New syntax · Observability (managed traces)
Observability is turned on per Runtime, not per invocation. Once enabled, every invocation produces a trace visible in the AWS console — nested spans for coordinator → specialist → tool → model. Verbose logs go to CloudWatch; distributed traces go to X-Ray (or the AgentCore-integrated equivalent). Enable it either via agentcore configure --observability enabled or the equivalent flag in your SDK config. No code change in your agent. 
agentcore configure --entrypoint agentcore_shippable.py \
    --identity my-agent-identity \
    --observability enabled
This is the difference between "it broke somewhere" and "the ops_expert's get_anypoint_org_status tool call took 8.2s at 02:14:36 UTC".

Prerequisites

  1. Your L34 deploy pipeline still works (agent deploys, cloud endpoint responds).
  2. Your L35 Memory resource exists — you have the MEMORY_ID in your environment.
  3. Your L36 multi-agent code (agent-as-tool variant) is understood — you don't have to have it as a file, but you should remember the shape.
  4. An OAuth provider you can test against:
    • Easiest: create a Cognito user pool with one test user. AWS-native, single console.
    • Enterprise-realistic: an Okta dev tenant with an OIDC app (like the one wired up in idp-document-frontend/lib/auth.ts).
  5. A way to obtain a JWT for testing: the Cognito hosted UI, an Okta login flow, or the AWS CLI cognito-idp initiate-auth. Save the access token to a shell variable: export TOKEN=eyJ....

Deploy — three CLI steps outside the file

  1. Create the Identity resource (once): agentcore identity create --oauth-provider <cognito|okta> --...
  2. Configure with identity and observability: agentcore configure --entrypoint agentcore_shippable.py --identity my-agent-identity --observability enabled
  3. Launch: agentcore launch

Verification — the four-part shipping check

Check 1 · The 401 test (auth wiring)
Hit the deployed endpoint without a token. You should get 401 Unauthorized. If you get 200 with a response, Identity isn't wired — the Runtime is happily accepting anonymous calls and your code is now leaking a real answer to anyone with the URL. 
curl -X POST https://runtime.bedrock-agentcore.aws/.../invocations \
  -H "Content-Type: application/json" \
  -d '{"prompt": "hi"}'
# Expect: HTTP 401
Check 2 · Identity flows through (authz correctness)
Same request with Authorization: Bearer $TOKEN. Inspect the response's actor_id — it should match the sub claim of the JWT you sent, not something guessable from the payload. Try tampering: pass {"actor_id": "attacker"} in the body. It must be ignored. If it isn't, you're trusting the payload.
Check 3 · Composed behaviour still works (L35 + L36 didn't break)
Two authenticated calls, same session_id: first says "org 12345, degraded state, what does the runbook say?"; second says "what org did I ask about?". Second reply mentions 12345 (memory works) and, if you inspect the trace, the coordinator called both specialists on the first turn (multi-agent works). Both pieces alive after composition.
Check 4 · Read the trace like an SRE
Open the AgentCore Observability view in the AWS console. Find the request from Check 3 by request-id. You should see nested spans:
  • Top span: invoke (your entrypoint)
  • Child: coordinator model call
  • Child: docs_specialist tool call → docs_agent model call → search_docs call
  • Child: ops_specialist tool call → ops_agent model call → get_anypoint_org_status call
  • Child: coordinator's final model call (composing the merged answer)
  • Child: create_event for memory persistence
If you don't see nesting like this, observability isn't fully enabled, or your instrumentation is missing spans. Get this right before shipping — every future 2am incident will start here.

Where the abstraction leaks

  • Cold start got worse. Identity token verification + Memory lookup add tens to hundreds of milliseconds to the first cold invocation. Measure it, don't guess. If it matters to UX, provision concurrency or keep a warm pool.
  • Cognito is easy, enterprise is not. An Okta or Azure AD hookup adds SCIM sync, group-based authz, RBAC mappings, and probably a conversation about SAML vs OIDC. AgentCore Identity accepts the token; using its claims for real authorization is still your job.
  • Observability samples by default. To keep bills sane, most managed observability samples traces at less than 100%. Great for aggregate signals, painful when the exact failing request isn't sampled. Turn sampling to 100% during rollout and pilot; ease it later.
  • Verbose tool responses eat trace-storage limits. If your tools return large payloads (a full RAG result set, a big JSON blob from Anypoint), the trace will either be truncated or expensive. Keep tool responses lean; store the big stuff in S3 and pass an S3 URI.
  • "Shippable" is technical, not operational. You still need SLOs, alarms, an on-call rotation, a rollback playbook, load-test data, cost guardrails, and a red-team pass before this touches a customer. This lesson gets you to the starting line, not past it.

MuleSoft angle — the AWS-native "first agent" pitch
For an AWS-committed customer asking "how do we run our first production agent?", the reference architecture is now:
  • Bedrock — the model (Haiku for cost, Sonnet for hard turns)
  • Strands — the agent framework (Apache 2.0, no vendor lock at the code layer)
  • AgentCore Runtime — where it runs (framework-agnostic serverless)

  • AgentCore Identity — Cognito/Okta at the front door
  • AgentCore Observability — traces + CloudWatch metrics
  • Flex Gateway — rate limits, WAF-style policies, DLP at the edge

One AWS bill for the AI layer; existing Anypoint estate for integration. If AWS is already the destination, use their managed versions and spend the saved cycles on the actual business logic.

Quick check

Q1. Which is the safest source for actor_id in a production agent behind AgentCore Identity?

  • payload["actor_id"] — the caller tells us who they are
  • A UUID generated per invocation
  • A verified claim (typically sub) on the request context, populated by the Identity authorizer after it validated the JWT signature and expiry. The payload is untrusted; the context claim is trusted.
  • The client IP address

Q2. Same authenticated user, two different sessions. The user reads a note in session B that they only ever mentioned in session A. What went wrong?

  • This is expected — short-term memory is per-user
  • Your load_history call is scoped only by actor_id, not by both actor_id AND session_id. Short-term memory must be session-scoped or sessions bleed into each other. This is a real production bug pattern, easy to make when copy-pasting.
  • AgentCore Memory doesn't isolate sessions
  • You used the wrong JWT provider

Q3. A customer complains: "The agent gave a wrong answer at 02:14 last night." What do you look at first?

  • The AgentCore Observability trace for that request-id — you'll see the coordinator's decision, which specialist ran, what tool payload was returned, and where the reasoning went off. Traces are the SRE equivalent of a stack trace for agents.
  • Redeploy immediately with a stricter system prompt
  • Rotate the Cognito user pool
  • Add more RAG documents

Q4. Managed Observability samples traces at less than 100% by default. When would you turn sampling to 100%?

  • Never — 100% is always overkill
  • Always — anything less is negligence
  • Only when required by regulation
  • During rollout, pilot, and any active incident — the marginal cost is small, and losing "just the failing request" is the worst kind of missing data. Ease sampling later when the agent has been stable and every incident isn't the first of its kind.

Q5. A customer asks "why not just run our LangGraph agent in ECS instead of using AgentCore Runtime?" Best framing?

  • "You can't — LangGraph doesn't run on AWS."
  • "You absolutely can, and for some teams it's the right call — you keep the framework you already know. The tradeoff is that in ECS you build Runtime, Memory, Identity, and Observability yourself: session microVMs, memory store schema, JWT verification, distributed tracing. AgentCore gives you those managed. If AWS is already the destination and those primitives are on the roadmap, AgentCore saves engineering months. If you're multi-cloud or already have a battle-tested platform, ECS is fine. Not one right answer — a tradeoff to size against your team."
  • "AgentCore is always faster than ECS."
  • "AgentCore is free."
