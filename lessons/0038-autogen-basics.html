AutoGen basics: your first agent on Azure OpenAI

Third cloud, same journey. You've built the MuleSoft assistant in Google's ADK (L26–L31) and AWS's Strands (L32–L37). This lesson opens the Microsoft chapter — Microsoft's AutoGen agent framework, calling Azure OpenAI, following the exact teach-don't-build shape you know.

The mission moment

AutoGen is Microsoft Research's open-source (Apache 2.0) agent framework. Same idea as Strands or LangGraph — write agents in Python, wire tools, orchestrate multi-agent workflows — but shaped around Microsoft's assumptions: async-first, actor-model core, deep Azure OpenAI integration. Big v0.4 rewrite in late 2024 rearranged the packages; ignore any L3-year-old tutorial that imports autogen at the top level.

You already know the drill. Build the same MuleSoft assistant you built in L18 (native), L26 (ADK), L32 (Strands) — one system prompt, two tools (search_docs, get_anypoint_org_status), a handful of test queries. What changes is the framework's shape.

Source honesty
AutoGen v0.4 (the async-first rewrite) is the current major line as of 2025. Package names are autogen-core, autogen-agentchat, autogen-ext. If you find yourself typing from autogen import ... you're on the pre-v0.4 API and your tutorial is stale. Cross-check with the current AutoGen docs.

The three new syntax constructs

New syntax · AssistantAgent
The high-level building block is AssistantAgent from autogen_agentchat.agents. You give it a name, a model client, a list of tools (plain Python functions with type hints work directly — no decorator needed), and a system message. Unlike Strands' agent("prompt") shortcut, AutoGen is async-first: you call await agent.on_messages([...], cancellation_token). 
from autogen_agentchat.agents import AssistantAgent

agent = AssistantAgent(
    name="mulesoft_helper",
    model_client=client,
    tools=[search_docs, get_anypoint_org_status],
    system_message="You are a MuleSoft platform assistant...",
)
"Plain Python functions as tools" is the ergonomic surprise here — no @tool, no schema declaration. AutoGen introspects your type hints and docstring.
New syntax · AzureOpenAIChatCompletionClient
Model clients live in autogen_ext.models.openai. For Azure OpenAI, you use AzureOpenAIChatCompletionClient — takes your Azure endpoint, API version, and a deployment name (not a model ID). Deployment name is Azure's abstraction: you deploy gpt-4o under whatever name you want (e.g. my-gpt4o-prod), and code references that name. 
from autogen_ext.models.openai import AzureOpenAIChatCompletionClient

client = AzureOpenAIChatCompletionClient(
    azure_deployment="my-gpt4o-deployment",
    model="gpt-4o",                # informational, for token counting
    api_version="2024-06-01",
    azure_endpoint="https://my-resource.openai.azure.com/",
    api_key="...",                 # or use azure_ad_token_provider
)
Deployment name ≠ model ID is the #1 confusion when starting on Azure OpenAI. Get it wrong and you get a 404 that looks like an auth issue.
New syntax · async / await + on_messages
AutoGen is async-first. Every framework you've used so far had a synchronous shortcut (agent.run_sync, agent("prompt")). AutoGen doesn't — you write async code, or you wrap it in asyncio.run(...). Invocation is await agent.on_messages(messages, cancellation_token): 
import asyncio
from autogen_agentchat.messages import TextMessage
from autogen_core import CancellationToken

async def main():
    response = await agent.on_messages(
        [TextMessage(content="How do I rotate a Connected App secret?",
                     source="user")],
        cancellation_token=CancellationToken(),
    )
    print(response.chat_message.content)

asyncio.run(main())
New Python syntax if you've never seen it: async def declares a coroutine (a function whose body is paused when it hits await). await x pauses until x completes. asyncio.run(main()) starts an event loop and runs the top-level coroutine. If you've written Node.js or C#, this shape is familiar.

Prerequisites

  1. An Azure subscription and access to Azure OpenAI in your region (some regions require an application).
  2. Create an Azure OpenAI resource, then a deployment (e.g. deploy gpt-4o-mini, name it gpt4o-mini-dev).
  3. Get the endpoint URL, API version, and either an API key (for testing) or configure Entra ID auth (better for production; deferred to L43).
  4. Install packages: pip install autogen-agentchat autogen-ext[openai,azure]
  5. Environment variables recommended:
    • AZURE_OPENAI_ENDPOINT
    • AZURE_OPENAI_API_KEY
    • AZURE_OPENAI_DEPLOYMENT
    • AZURE_OPENAI_API_VERSION (e.g. 2024-06-01)

Verification

Five checks
  1. File runs without an import error → your v0.4 packages are installed correctly.
  2. First query returns a docs answer → search_docs was invoked; check the trace to confirm.
  3. Second query returns "healthy" for org 12345 → get_anypoint_org_status was invoked.
  4. Third query invokes both tools in one turn → the model is deciding, not hard-coded.
  5. Inspect response.chat_message — you'll see the final assistant message. To see the tool-call trace, look at response.inner_messages (v0.4 exposes intermediate messages).

Where the abstraction leaks

  • Deployment name ≠ model ID. Azure OpenAI's abstraction. Two devs will hit the same bug in the first hour. Set it via env var.
  • API version pins matter. New Azure OpenAI features (structured outputs, streaming variants) ship in specific API versions. If your call silently ignores a param, check the version.
  • Async everywhere. If your calling code is a Flask sync route, wrap with asyncio.run or use a sync web framework. Mixing sync/async without care leads to blocked event loops.
  • Tool introspection is best-effort. Skip the docstring, and the model sees a poorly-described tool. Type hints are load-bearing — str, int, dict, Pydantic models all fine; Any defeats the schema.
  • Regional model availability. gpt-4o might be in East US 2 but not Southeast Asia. Check availability before quoting timelines.

Quick check

Q1. You call AzureOpenAIChatCompletionClient(model="gpt-4o-mini", ...) with a valid endpoint and API key, but get a 404. Most likely cause?

  • gpt-4o-mini doesn't exist
  • You passed the model ID as azure_deployment. Azure OpenAI routes by deployment name, not model ID — the model ID is informational (mostly for token counting). Set azure_deployment to the name you gave your deployment in the Azure portal.
  • Your API key is expired
  • AutoGen doesn't support Azure

Q2. Why does AutoGen introspect your function signature instead of asking you to write a JSON schema?

  • Python is slow at parsing JSON
  • JSON schemas aren't compatible with OpenAI
  • Because a type-hinted Python function is a schema. Parameter names + types + docstring already describe everything an LLM needs to call the tool. Making the developer write both is redundancy that causes drift. Same reason FastAPI works.
  • To lock you into using AutoGen

Q3. A colleague copies your autogen_hello.py into a Flask sync route and it hangs the server. Why?

  • AutoGen is async-first and expects an event loop. Calling asyncio.run inside an already-running loop, or blocking sync code inside an async context, deadlocks. Fix: use an async web framework (FastAPI, Litestar) or run AutoGen work in a background task with proper loop management.
  • Flask isn't compatible with tools
  • Azure OpenAI blocks Flask
  • Nothing — that always works

Q4. You import from autogen import AssistantAgent. Your IDE finds it, but the API you're calling looks nothing like the docs. Why?

  • The docs are wrong
  • The package is corrupted
  • You need to reinstall Python
  • You have the pre-v0.4 package installed alongside v0.4. AutoGen split into autogen-core, autogen-agentchat, autogen-ext in v0.4. Uninstall the old top-level autogen or check pip list for the confusion.

Q5. An Azure-committed customer asks "what's the AutoGen advantage over just calling Azure OpenAI directly with the openai SDK?" Best framing?

  • "AutoGen is faster"
  • "AutoGen is required for Azure"
  • "You can absolutely call Azure OpenAI directly for single-turn tasks. AutoGen earns its keep when you need the agent loop (tool calls, retries), multi-agent orchestration, structured message passing, or clean handoffs between specialists. Below that complexity threshold, the raw SDK is fine and simpler."
  • "AutoGen replaces Azure OpenAI"
