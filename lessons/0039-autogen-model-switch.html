AutoGen model switch: Azure OpenAI, OpenAI direct, Anthropic on Foundry

Same agent, four backends. This is L33 for Azure — the "swap the model client, keep the agent" drill that proves the abstraction actually earns its keep.

The mission moment

Customers ask "can you use Claude on Azure?" and "do we have to be locked into OpenAI?" every single week. In AutoGen v0.4, the answer is a one-line client swap — but each provider has small differences that matter for real deployments. You'll build the same agent from L38 four times, swapping only the model client.

Why this lesson exists
In L33 you did this drill for Strands (Bedrock, Anthropic direct, LiteLLM). In L05 you saw the Vertex/direct split for Google. AutoGen's story is different again — its model-client abstraction is the cleanest of the three, and Azure AI Foundry's Model Catalog means Claude, Mistral, Llama, and Cohere are all first-class citizens in your Azure account.

New syntax · AzureAIChatCompletionClient
Not to be confused with AzureOpenAIChatCompletionClient. This one calls the Azure AI Foundry "Models as a Service" endpoint — Foundry's serverless Model Catalog where you deploy Claude, Mistral, Llama, Cohere behind an Azure-managed endpoint. 
from autogen_ext.models.azure import AzureAIChatCompletionClient
from azure.core.credentials import AzureKeyCredential

client = AzureAIChatCompletionClient(
    model="Claude-Sonnet-4",
    endpoint="https://my-foundry-project.services.ai.azure.com/models",
    credential=AzureKeyCredential("..."),
    model_info={
        "family": "unknown",
        "function_calling": True,
        "json_output": False,
        "vision": True,
    },
)
Note model_info: for non-OpenAI models Foundry needs you to declare capabilities so AutoGen knows whether to attempt tool calls or JSON mode. If you forget it for a model AutoGen doesn't recognise, you'll get a KeyError on the first call.
New syntax · OpenAI-compatible base_url
Ollama (local), vLLM (self-hosted), Groq, Together, DeepInfra, Fireworks all speak the OpenAI wire protocol. AutoGen supports them via OpenAIChatCompletionClient with a custom base_url: 
client = OpenAIChatCompletionClient(
    model="llama3.2",
    base_url="http://localhost:11434/v1",  # Ollama
    api_key="ollama",                      # dummy
    model_info={"family": "llama", "function_calling": True,
                "json_output": False, "vision": False},
)
This is the escape hatch for on-prem / sovereign / cost-sensitive customers. Same code as production — different base_url.

Verification

Four runs, one diff
  1. BACKEND=azure-openai python autogen_model_switch.py → baseline.
  2. BACKEND=openai python autogen_model_switch.py → should match closely.
  3. BACKEND=foundry-claude python autogen_model_switch.py → different voice, same tool calls.
  4. BACKEND=ollama python autogen_model_switch.py → if the local model is smart enough, tool calls fire; if not, it may hallucinate the answer instead. That's not a bug — that's why model choice matters.
Common failure modes
  • KeyError on model_info — Foundry client for non-OpenAI models. Add the dict.
  • Tool calls silently ignored on Ollama — model too small. Try llama3.1:70b or a proper tool-tuned model.
  • Higher latency on Foundry-Claude — first request warms the deployment. Second request is normal.
  • Different tool-call formats per provider — AutoGen normalises, but if you inspect inner_messages you'll see the raw shape differs. Rely on the normalised view.

Where the abstraction leaks

  • model_info is a manual declaration. AutoGen can't introspect an arbitrary Foundry endpoint. Get it wrong and either tool calls silently disable or JSON mode errors at runtime.
  • Region coverage differs. gpt-4o may be available in East US 2 but not Southeast Asia. Claude on Foundry has its own region matrix. Verify before promising anything.
  • Rate limits differ wildly. Azure OpenAI TPM quotas ≠ OpenAI direct usage tiers ≠ Foundry MaaS quotas. Load-testing on one provider does not qualify you on another.
  • Streaming implementations differ. The token-boundary behaviour of Azure OpenAI vs OpenAI vs Claude-on-Foundry is not identical. If your UX depends on it, test each one.
  • Auth stories diverge in production. Azure OpenAI can use Entra ID; OpenAI direct is API key only; Foundry MaaS is API key from the deployment; Ollama has no auth at all. Standardise via env vars, don't try to unify at the code layer.

Quick check

Q1. Your customer says "we want to run Claude but keep everything inside our Azure tenant." Best path in AutoGen?

  • Direct Anthropic API
  • Azure AI Foundry Model Catalog + AzureAIChatCompletionClient — Claude is served from an Azure-managed serverless endpoint, billed through Azure, staying inside their tenant. No Anthropic API key needed.
  • Impossible — Claude is OpenAI-only
  • Bedrock

Q2. You add a new Foundry model and get KeyError: 'function_calling'. Why?

  • The Foundry endpoint is misconfigured
  • The model doesn't support tools
  • You forgot the model_info dict. AutoGen keeps a registry of well-known OpenAI models with their capabilities, but for Foundry MaaS models you must declare capabilities yourself. Add model_info={"family": ..., "function_calling": True, "json_output": ..., "vision": ...}.
  • Python version mismatch

Q3. Same code works fine against Azure OpenAI. Against Ollama with llama3.2:1b the tool never fires. Why?

  • AutoGen doesn't support Ollama
  • Tool schema is wrong
  • Ollama endpoint is down
  • The model is too small to reliably emit tool-call syntax. Tool calling is an emergent behaviour of capable instruction-tuned models. Try a larger or explicitly tool-tuned model, or accept that some backends aren't fit for agentic workflows.

Q4. When would you pick OpenAI direct over Azure OpenAI?

  • The customer isn't on Azure, doesn't need enterprise SLAs, wants the newest model on day-0, or is prototyping. Azure OpenAI has enterprise features but lags on new model availability by weeks-to-months. Pick per-workload, not once for all workloads.
  • Always — it's cheaper
  • Never — Azure OpenAI is a superset
  • Only for images

Q5. Your code passes tests on OpenAI direct but fails intermittently on Azure OpenAI. Most likely single cause?

  • AutoGen bug
  • Different Python version
  • TPM (tokens-per-minute) quota on the Azure deployment. Azure OpenAI quotas are per-deployment, and hitting them returns 429s that look intermittent under load. Check the deployment's quota and increase, or spread traffic across deployments/regions.
  • Wrong region
