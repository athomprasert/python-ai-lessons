Azure AI Foundry Agents Service: the managed runtime

L34 was Bedrock AgentCore Runtime. L31 was Vertex Agent Engine. This is the Azure equivalent — Foundry Agents Service, the managed platform that owns identity, threads, tool execution, tracing, and content safety so you don't have to.

L38 ✓ L39 ✓ L40 — Foundry Agents L41 — Threads / memory L42 — Multi-agent L43 — Deploy capstone

The mission moment

You've been running AutoGen locally: your Python process is the agent. That's fine for a lesson. In production, "my laptop" is not the answer to "where does the agent run." Foundry Agents Service is Microsoft's answer — a managed agent runtime that lives inside your Azure AI Foundry project alongside your models, evaluations, and content-safety filters.

The mental model shift, same as L34: your Python code stops being the runtime and becomes the client to a runtime. You create an agent as an Azure resource. You invoke it via a thread. Azure owns state, retries, and tool execution.

Why this lesson exists
Every Azure-committed customer will ask "where does this thing run in prod?" Answering "AutoGen in a container on AKS" is a valid answer but not the Microsoft-preferred one. Foundry Agents is what Microsoft points customers at — content safety, evaluations, and tracing are wired in without extra code. This lesson gets you fluent in the alternative shape.
Source honesty
Foundry Agents Service is evolving rapidly. Package names azure-ai-agents and azure-ai-projects are what shipped GA in 2025; the earlier "Azure AI Assistants" surface is deprecated. Always cross-check against the current Microsoft Learn page — "Azure AI Foundry Agent Service."

Architecture

┌─────────────────────────────────────────────────────┐ │ Your Python client │ │ • project client (from AZURE_AI_PROJECT_CONN) │ │ • credential (DefaultAzureCredential preferred) │ └──────────────┬──────────────────────────────────────┘ │ HTTPS ▼ ┌─────────────────────────────────────────────────────┐ │ Azure AI Foundry Project │ │ Agents Service │ │ ├─ Agent (definition: model + tools + prompt) │ │ ├─ Thread (conversation state, server-managed) │ │ └─ Run (one turn — tool loop lives here) │ │ Model Catalog · Evaluations · Content Safety │ │ Traces → Application Insights │ └─────────────────────────────────────────────────────┘

Three server-side resources: Agent, Thread, Run. You never see a request-loop again — Foundry drives the tool loop for you and streams events back.

The four new syntax constructs

New syntax · AIProjectClient
Entry point for everything. Connects to your Foundry project via connection string or endpoint. 
from azure.ai.projects import AIProjectClient
from azure.identity import DefaultAzureCredential

project = AIProjectClient.from_connection_string(
    conn_str=os.environ["AZURE_AI_PROJECT_CONN"],
    credential=DefaultAzureCredential(),
)
DefaultAzureCredential is Azure's identity-negotiation chain — tries az login, managed identity, env vars, in order. In prod on Azure infra, this "just works." Locally, run az login first.
New syntax · agents.create_agent
Create an Agent resource. This is a persistent object in Foundry — same as an ADK Agent Engine deployment or AgentCore Runtime. 
agent = project.agents.create_agent(
    model="gpt-4o-mini",  # deployment name in your project
    name="mulesoft-helper",
    instructions="You are a MuleSoft platform assistant...",
    tools=[
        {"type": "function", "function": {
            "name": "get_anypoint_org_status",
            "description": "Look up the live status of an Anypoint org.",
            "parameters": {"type": "object",
                           "properties": {"org_id": {"type": "string"}},
                           "required": ["org_id"]}
        }}
    ],
)
Tool schemas here are declarative JSON — different shape from AutoGen's function-introspection. Foundry doesn't run your Python; it emits tool-call requests and expects you to run the function and submit the output.
New syntax · Thread + create_message + create_run
A thread is server-managed conversation state — the Foundry equivalent of Bedrock AgentCore's Memory session or ADK's Session.state. 
thread = project.agents.threads.create()
project.agents.messages.create(
    thread_id=thread.id, role="user",
    content="Status of org 12345?"
)
run = project.agents.runs.create_and_process(
    thread_id=thread.id, agent_id=agent.id
)
create_and_process is the "wait for completion" convenience. If you need to handle tool calls yourself (functions Foundry can't call, e.g. anything on your VNet), use create + poll and submit tool_outputs in the requires_action state.
New syntax · requires_action loop
For tools Foundry can't execute (private code), your run pauses in status requires_action with a list of tool calls. You run them, then submit outputs: 
while run.status in ("queued", "in_progress", "requires_action"):
    if run.status == "requires_action":
        outputs = []
        for call in run.required_action.submit_tool_outputs.tool_calls:
            args = json.loads(call.function.arguments)
            if call.function.name == "get_anypoint_org_status":
                result = get_anypoint_org_status(**args)
                outputs.append({"tool_call_id": call.id,
                                "output": json.dumps(result)})
        run = project.agents.runs.submit_tool_outputs(
            thread_id=thread.id, run_id=run.id, tool_outputs=outputs
        )
    else:
        run = project.agents.runs.get(thread_id=thread.id, run_id=run.id)
        time.sleep(0.5)
This is the biggest shape difference from AutoGen. AutoGen owns the agent loop in your process. Foundry owns the loop server-side and calls back for tools it can't run.

Prerequisites

  1. An Azure AI Foundry project (create one in Azure portal or via az CLI).
  2. A model deployed in that project (gpt-4o-mini is cheapest for testing).
  3. The project's connection string from Foundry → Project Settings. Set as AZURE_AI_PROJECT_CONN.
  4. Signed in via az login, or an assigned managed identity in prod.
  5. Install: pip install azure-ai-projects azure-ai-agents azure-identity

Verification

Five checks
  1. Foundry portal → your project → Agents shows a new agent while the run is executing.
  2. Threads view shows one thread with the user message and the assistant reply.
  3. Run status transitions queued → in_progress → requires_action → in_progress → completed.
  4. Tracing pane shows the tool call with args and output (if Application Insights is wired to the project).
  5. After delete_agent runs, the agent no longer appears in the portal. If it does, you left orphan agents from earlier runs — clean them up.

MuleSoft angle
Foundry Agents' requires_action pattern is a gift for MuleSoft positioning. Your Python client is the callback point — the natural place to sit an Anypoint call is right there. Foundry orchestrates, MuleSoft integrates. Two clean layers. For customers who want Foundry to manage everything with MCP built-in tools, MuleSoft MCP endpoints plug into Foundry's MCP tool support directly. Either shape works.

Where the abstraction leaks

  • Agent objects persist. If you create one every run and never delete, you accumulate orphans that show up in the portal and count against project quotas. Create once in real code, save the ID.
  • Tool JSON schemas are not Python function introspection. You lose AutoGen's "function is the schema" ergonomics. Consider generating the schema from your function via pydantic.TypeAdapter or similar.
  • Threads are billed storage. They persist until deleted. Have a cleanup policy — Foundry doesn't auto-expire threads.
  • DefaultAzureCredential fails silently in weird ways. If you're getting 401s, run az account show and verify. Confirm your identity has the "Azure AI Developer" (or higher) role on the project.
  • Region + model availability. Foundry Agents may be GA in some regions and preview in others. Check before quoting SLAs.
  • Foundry's tool loop ≠ AutoGen's tool loop. Behaviour under retries, timeouts, and content-safety blocks differs. Don't assume identical semantics.

Quick check

Q1. What's the practical difference between AutoGen locally and Foundry Agents Service?

  • Foundry is faster
  • AutoGen's agent loop runs in your Python process. Foundry runs the loop server-side inside your Azure project — it manages agent definitions, threads, and runs as Azure resources, and calls back to your code only for tools it can't execute. Different runtime model, not different capability.
  • Foundry doesn't support tools
  • They're the same thing

Q2. Your run gets stuck in requires_action. Why?

  • Foundry outage
  • Wrong model
  • Your client hasn't submitted tool outputs. Foundry has paused the run and is waiting for you to run the function and call submit_tool_outputs. Either your loop doesn't handle that status, or the tool call threw before submission.
  • Content safety blocked it

Q3. Why doesn't Foundry just introspect your Python function like AutoGen does?

  • Microsoft doesn't like Python
  • Type hints aren't reliable
  • Performance
  • Foundry never sees your function — the agent lives server-side, your function lives in your client. It only sees the JSON schema you registered. That's why you have to declare the schema explicitly (or generate it from your function via a helper).

Q4. When would you keep AutoGen local instead of moving to Foundry Agents?

  • When you need heavy multi-agent orchestration with tight in-process communication (Foundry's model is single-agent-per-thread; complex handoffs across agents live better in the framework), or when data must not leave your VNet, or when you need to stay portable across clouds.
  • Always keep it local
  • Only for demos
  • Never — Foundry is always better

Q5. Best deployment pattern for a MuleSoft customer running Foundry Agents?

  • Rewrite everything in Foundry
  • Only use Bing grounding
  • Foundry drives the run; your tool functions call Anypoint APIs. Foundry gets managed identity + evaluations + content safety. Anypoint gets connector library + DataWeave + governance. Clean separation, no rebuilding either side.
  • Skip MuleSoft entirely
