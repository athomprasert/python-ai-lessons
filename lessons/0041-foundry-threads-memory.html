Foundry Threads & memory: state that survives the process

L28 was ADK Session.state. L35 was AgentCore Memory. This is the Azure story — Foundry Threads for per-conversation state, plus explicit "user memory" for cross-conversation preferences.

L40 ✓ L41 — Threads / memory L42 — Multi-agent L43 — Deploy capstone

The mission moment

In L40 you created a thread, sent one message, deleted the agent. The thread was disposable. Real agents need memory: "the same customer coming back tomorrow should not have to re-explain their org." L41 makes threads persistent, adds a second turn, and shows the split between short-term (thread history) and long-term (user preferences that outlive threads).

The two-level memory pattern (universal)
Every managed agent platform ends up with the same split:
  • Short-term / session: a thread's messages — what was said, what tools were called, in this conversation.
  • Long-term / user: facts about the user or org that persist across threads (preferences, past decisions, extracted summaries).
L28 called them Session.state and (implicitly) whatever store you chose. L35 called them "events" and "strategies." Foundry calls them "threads" and "user-scoped context" — you'll typically wire your own persistence for the latter (e.g. Azure Cosmos DB), or use a Foundry-managed memory feature when available.
Source honesty
Foundry's "managed memory" feature area is evolving. As of 2025, threads are the stable primitive for short-term state, and long-term memory is often implemented by the customer (Cosmos DB, Table Storage, or a vector index). Some Foundry SDK preview versions expose managed memory APIs — verify the current shape against Microsoft Learn before wiring anything permanent.

The two new syntax constructs

New syntax · reusing a thread
A thread is a first-class Foundry resource with an ID. Save the ID, look it up next session: 
# first time
thread = project.agents.threads.create()
save_to_db(user_id="alice", thread_id=thread.id)

# next session
thread_id = load_from_db(user_id="alice")
project.agents.messages.create(thread_id=thread_id, role="user",
                               content="What did we decide about org 12345 yesterday?")
run = project.agents.runs.create_and_process(thread_id=thread_id, agent_id=agent.id)
Foundry replays the full thread transcript into the model's context on every run — you don't have to. Watch context-window cost: after 100 turns the thread doesn't shrink itself.
New syntax · user memory injected into instructions
For cross-conversation facts, the practical pattern is: store user context in your own DB, then inject it into instructions or as the first user message at run creation: 
context = load_user_context("alice")  # your Cosmos/Table Storage lookup

run = project.agents.runs.create(
    thread_id=thread.id,
    agent_id=agent.id,
    additional_instructions=f"User context: {context}",  # per-run override
)
additional_instructions stacks on top of the agent's base instructions for this run only — cleaner than editing the agent every time. This is the deliberate boundary between short-term (thread) and long-term (user profile) memory.

Verification

Four checks
  1. Turn 1 for Alice returns the org status. Foundry portal shows one thread.
  2. Turn 2 for Alice, same thread, answers "12345" without the number being resent. This proves short-term memory (thread history) works.
  3. Turn 3 for Bob, different thread, does NOT know the number. Proves per-user thread isolation.
  4. Kill the script, re-run it. Alice's thread ID is loaded from thread_store.json — the conversation still remembers. Proves persistence beyond the Python process.
Watch the context bill
Every run replays the full thread. After 200 turns that's expensive. Real production adds either (a) periodic summarisation into additional_instructions and thread reset, or (b) a moving window. Foundry doesn't do this for you.

MuleSoft angle
Where does the system of record for user preferences live? In an enterprise, it's usually Salesforce, a CMDB, or a customer profile in a data mesh. That means load_user_context in real code is a MuleSoft System API call. Not a nice-to-have — the moment your Foundry agent has a memory feature, "where does that memory come from" is a MuleSoft conversation.

Where the abstraction leaks

  • Foundry doesn't summarise threads for you. Long threads become expensive. Instrument turn count and cost, plan a compaction policy.
  • Threads persist until you delete. No TTL. Legal / GDPR right-to-delete lands on you, not on Foundry.
  • Cross-user leakage is your job to prevent. If your {user_id → thread_id} map is wrong, one user gets another user's memory. Test the isolation.
  • additional_instructions ≠ system prompt. It appends per-run; agents can still be steered by user messages against it. Treat it as guidance, not a security boundary.
  • Long-term memory is usually your problem. Managed memory features in Foundry are still evolving — assume you'll wire persistence yourself in prod, and treat any managed feature as a bonus.

Quick check

Q1. Why can Turn 2 for Alice answer "12345" without you resending it?

  • Foundry has a shared cache
  • The thread ID is the same for both turns. Foundry stores the message history on the thread and replays it into the model's context on every run — the model sees Turn 1's messages when responding to Turn 2. Thread ≈ conversation state.
  • The model memorises it
  • additional_instructions carries it

Q2. Bob asks "what number again?" and gets confused. Why is that correct?

  • Bug in the model
  • Foundry outage
  • Bob has his own thread. Threads are the isolation boundary — Alice's history is not visible to Bob's runs. This is exactly the property you want; if Bob saw Alice's history you'd have a data-leak bug.
  • Bob doesn't have permission

Q3. After 300 turns on Alice's thread, response latency and cost climb. Best fix?

  • Periodically summarise older turns into additional_instructions (or a stored user context) and reset the thread. Foundry doesn't auto-compact — replay of the full transcript is your bill.
  • Delete the whole thread
  • Switch models
  • Use a larger context window

Q4. Where should long-term user preferences live in a MuleSoft-integrated Foundry deployment?

  • In Foundry thread metadata
  • In the model's context permanently
  • In a per-agent global
  • In the actual system of record (Salesforce / CMDB / customer profile), exposed via an Anypoint System API. load_user_context is a wrapper that calls that API. Keeps the source of truth in the right place and avoids syncing state into Foundry.

Q5. A user exercises GDPR right-to-delete. What must happen?

  • Nothing — Azure handles it
  • Delete the model deployment
  • Explicitly delete their thread(s), any long-term memory you stored elsewhere (Cosmos/Table), and any linked traces retained in Application Insights per your policy. Foundry doesn't auto-expire threads — deletion is your responsibility.
  • Delete the whole Foundry project
