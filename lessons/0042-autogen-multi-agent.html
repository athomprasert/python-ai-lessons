AutoGen multi-agent: RoundRobin, Selector, Swarm

Three coordination patterns, one problem. Same shape as L25 (LangGraph), L29–30 (ADK), L36 (Strands). This is Azure's dialect of the multi-agent story.

The mission moment

In L38–41 you had one agent doing everything. Real MuleSoft assistants split into specialists: a docs expert, a platform ops expert, a solution architect. Multi-agent is how you make those specialists collaborate on the same conversation without you writing router code.

AutoGen v0.4's autogen_agentchat.teams module gives you three built-in team types. You pick the one that fits your coordination pattern.

Every framework has the same three patterns under different names: LangGraph supervisor / swarm / hierarchical; ADK SequentialAgent / ParallelAgent / sub_agents; Strands agent-as-tool / Swarm / GraphBuilder; AutoGen RoundRobin / Selector / Swarm. Learn once, translate.

The three new syntax constructs

New syntax · RoundRobinGroupChat
Simplest. Pass a list of agents and a termination condition. Agents speak in order until the condition fires. 
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_agentchat.conditions import TextMentionTermination, MaxMessageTermination

team = RoundRobinGroupChat(
    participants=[docs_agent, ops_agent, architect_agent],
    termination_condition=TextMentionTermination("APPROVED")
                          | MaxMessageTermination(10),
)
result = await team.run(task="How do I rotate a Connected App secret?")
Termination conditions compose with |. New syntax: bitwise-or overloaded on condition objects — same trick as SQLAlchemy filters. Also note & for AND.
New syntax · SelectorGroupChat
The supervisor pattern. An LLM inspects the conversation and picks the next agent to speak. Give it a selector prompt so it knows how to choose. 
from autogen_agentchat.teams import SelectorGroupChat

team = SelectorGroupChat(
    participants=[docs_agent, ops_agent, architect_agent],
    model_client=client,   # the model that picks the speaker
    selector_prompt="Read the conversation and pick the best agent for the "
                    "next turn. docs_agent answers documentation questions. "
                    "ops_agent handles live Anypoint status. architect_agent "
                    "does solution design. Only respond with the agent name.",
    termination_condition=MaxMessageTermination(6),
)
The selector calls the model on every turn — count it in your cost model.
New syntax · Swarm with handoffs
Explicit transfer. Each agent declares which agents it can hand to; the model can emit a HandoffMessage and the team routes it. 
from autogen_agentchat.teams import Swarm

triage = AssistantAgent(
    name="triage",
    model_client=client,
    handoffs=["docs_agent", "ops_agent"],   # who this agent can hand to
    system_message="Classify the request. Hand off to docs_agent for "
                   "documentation questions, or ops_agent for live status.",
)

team = Swarm(participants=[triage, docs_agent, ops_agent],
             termination_condition=MaxMessageTermination(6))
The handoffs=[...] parameter converts each name into an auto-generated tool (transfer_to_docs_agent) that the agent can call. Same idea as ADK's sub_agents + transfer_to_agent and Strands' Swarm.

Verification

The apples-to-apples test
Run all three scripts with the same task: "Give me the runbook AND the live status of org 12345." — the composed question that requires both tools.
  1. RoundRobin: docs_agent speaks, then ops_agent speaks. Fixed order. Both fire regardless of relevance.
  2. Selector: LLM decides. Should pick both, in whichever order it deems best. If you ask a docs-only question, only docs_agent should be chosen.
  3. Swarm: triage speaks first, chooses a specialist via handoff tool call, that specialist answers and hands to user. Deterministic transfer, not fixed rotation.
Compare the message logs. Three different coordination shapes for the same task.

Docs / Ops / Architect isn't a whiteboard pattern — it maps directly to Experience API / Process API / System API. Each agent has its own responsibility and its own set of Anypoint tools. When customers ask "how do we scale one AI helper into a real assistant?", multi-agent + API-led connectivity is the same answer in two languages: small components with clear responsibilities, coordinated at the top.

Where the abstraction leaks

  • Selector cost stacks up. Every turn calls the model to pick the next speaker AND then calls the picked agent. If your selector prompt is verbose, that's twice the tokens per turn. Keep it tight.
  • Swarm handoff needs discipline. If specialists don't hand back to user, they can hand to each other forever until termination fires. Design your handoff graph acyclic or bounded.
  • RoundRobin isn't intelligent. It runs the loop regardless of relevance — every agent will produce an answer to every task. Wasteful unless every agent genuinely contributes.
  • Termination conditions are AND/OR — not smart. There's no "terminate when a good answer exists." Add your own signal (e.g. an APPROVED string, or a HandoffTermination) explicitly.
  • Debugging multi-agent is hard. Turn on message streaming (run_stream) so you see every message rather than just the final. If you skip this you'll never understand why the wrong agent won.

Quick check

Q1. User asks a docs-only question. RoundRobin runs docs_agent AND ops_agent. Why?

  • Bug
  • RoundRobin has no smarts — it rotates through all participants until termination. Ops_agent runs even though its expertise isn't needed. Pick a different team type (Selector or Swarm) when relevance-aware routing matters.
  • Wrong model
  • The question is ambiguous

Q2. Selector picks the wrong agent on 30% of queries. Best first fix?

  • Switch to a bigger model
  • Add more agents
  • Rewrite the selector prompt with sharper role descriptions and concrete examples. The selector is an LLM classifier — its accuracy depends almost entirely on how clearly you describe each agent's remit. Bigger models help; a clearer prompt helps more.
  • Use RoundRobin

Q3. Swarm never terminates — agents keep handing off to each other. Cause?

  • Wrong model
  • Bug in AutoGen
  • Missing tools
  • No agent hands to "user", and MaxMessageTermination isn't set. Handoff cycles are the classic Swarm failure. Add HandoffTermination("user") and require specialists to hand back to user after answering.

Q4. Which pattern maps closest to LangGraph supervisor / ADK sub_agents / Strands agent-as-tool?

  • RoundRobinGroupChat
  • SelectorGroupChat — the LLM inspects the conversation and picks who speaks next. Supervisor pattern by another name.
  • Swarm
  • None

Q5. When would you actually pick RoundRobin in production?

  • Never
  • Always — simplest wins
  • Review / debate pipelines where every agent MUST contribute in order — e.g. "draft → critique → revise" or "SE proposes → architect reviews → security signs off." Fixed rotation is exactly what you want when the workflow is a checklist, not a routing decision.
  • Only for demos
