The three new syntax constructs

New syntax · Entra ID auth via DefaultAzureCredential + FastAPI JWT dependency
Two layers. Foundry SDK uses DefaultAzureCredential (which prefers the App Service managed identity in prod). Your FastAPI service validates the caller's Entra ID JWT before letting them touch Foundry. 
from fastapi import FastAPI, Depends, HTTPException, Header
import jwt   # PyJWT
from jwt import PyJWKClient

_jwks = PyJWKClient(f"https://login.microsoftonline.com/{TENANT}/discovery/v2.0/keys")

def verified_user(authorization: str = Header(...)) -> dict:
    if not authorization.startswith("Bearer "):
        raise HTTPException(401, "missing bearer")
    token = authorization.split()[1]
    signing_key = _jwks.get_signing_key_from_jwt(token).key
    return jwt.decode(token, signing_key, algorithms=["RS256"],
                      audience=APP_CLIENT_ID, issuer=ISSUER)
Only verified_user(...)'s sub (or oid) claim becomes the user ID for thread routing. Never trust a header the client controls.
New syntax · Content safety on the Foundry side
Foundry has managed content-safety filters. You enable them in the portal or via the SDK on the agent or the run. Blocked inputs return a specific error you can catch. 
# In your project, enable "Prompt Shields" and "Groundedness detection"
# under Content Safety.
try:
    run = project.agents.runs.create_and_process(
        thread_id=thread.id, agent_id=agent.id
    )
except HttpResponseError as e:
    if getattr(e, "error", None) and e.error.code == "content_filter":
        return {"error": "Message blocked by content safety policy."}
    raise
Content safety is not a substitute for authz — it's the "did the LLM see something dangerous or say something unsafe?" layer.
New syntax · Application Insights via connection string
Foundry writes traces to your project's linked Application Insights automatically. To trace your own code (FastAPI, MuleSoft calls) alongside, add: 
from azure.monitor.opentelemetry import configure_azure_monitor
configure_azure_monitor(connection_string=os.environ["APPINSIGHTS_CONN"])
One line. Your logs, spans, and Foundry's server-side traces land in the same Application Insights workspace, correlatable by trace ID.

The shipping check

Five things to prove before you call it done
  1. Unauthenticated call is rejected. Same curl without Authorization → 401. If not, your JWT dependency is not wired.
  2. Two users get isolated threads. Alice's token gets Alice's history; Bob's token gets Bob's. Check in Foundry portal's Threads view.
  3. Content-safety block returns a clean 400. Send a message that trips the filter (Foundry's Prompt Shields have example prompts). You should see your JSON error, not a stack trace.
  4. Trace correlation works. One request in Application Insights, expanded → shows your FastAPI span, the Foundry run span, tool-call span, model call. If it doesn't, the connection string didn't take.
  5. Restart survives. Kill the process, restart. Alice's next message still lands in her existing thread. Not a soft-restart — a real process crash test.

Quick check

Q1. A colleague suggests reading the user_id from a header your app trusts. Why is that wrong?

  • Headers aren't reliable
  • The client controls headers. Any attacker who reaches your endpoint can spoof another user's ID and read their thread history. The user identity must come from a cryptographically verified token — sub/oid claim in the Entra ID JWT, not a header the client sets.
  • Azure blocks custom headers
  • It's fine

Q2. Content-safety filter blocks a message. What should you do in code?

  • Retry
  • Ignore the error
  • Catch the specific error code (content_filter), return a clean 400 to the caller, and log the incident with the run/request ID so someone can review. Don't retry — retrying doesn't change the filter's decision, and repeated hits may raise your abuse profile.
  • Escalate to Foundry support

Q3. Application Insights shows a Foundry run span but nothing from your FastAPI code. Why?

  • You need a different plan
  • Foundry doesn't expose traces
  • Wrong region
  • You forgot configure_azure_monitor(connection_string=...), or the connection string points at a different resource than the Foundry project. Foundry auto-instruments itself; your app has to opt in explicitly.

Q4. Which single piece of this stack is not Foundry's job?

  • Long-term user profile from your systems of record. Foundry manages threads (conversation memory) but knowing "Alice manages org 12345" comes from Salesforce / CMDB / your data — reached via MuleSoft in most enterprises. Foundry never sees that unless you inject it into additional_instructions.
  • Thread persistence
  • Content safety
  • Model inference

Q5. Foundry Agent Service vs AutoGen local — for this shippable, which did we pick and why?

  • AutoGen local — cheaper
  • Foundry — because it's Microsoft
  • Foundry — because managed identity, content safety, threads, and tracing are wired in without extra code. The tradeoff is Azure-only. If portability across clouds mattered more than these managed properties, keep AutoGen local and add each piece yourself.
  • Doesn't matter
