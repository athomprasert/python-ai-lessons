Lesson 44 · Anypoint Studio · MuleSoft AI Chain

Build an agent inside Anypoint Studio with MuleSoft AI Chain

Same tool-using agent as L18/L20/L26/L32/L38 — but built as a Mule flow in Anypoint Studio using the MuleSoft AI Chain (MAC) connectors. Studio-first walkthrough with 8 screenshot slots for you to capture.

What you'll build
A running Mule application in Anypoint Studio that answers questions using an LLM, calls an external API as a "tool", and returns a structured response over HTTP. Same job as your Python framework lessons — different tool. By the end, you'll see when MAC beats a code-first framework and when it doesn't.
Source honesty — read before you install
MuleSoft AI Chain (MAC) connectors are actively evolving. Names, operations, and package coordinates have shifted between releases (e.g. mac-agents, ms-inference, mulechain-ai). This lesson uses the current MAC connector family circa 2026 but verify the exact operation names on Anypoint Exchange before you drop them into your flow. Screenshots you capture will be authoritative for the version you have installed.

Prerequisites

  • Anypoint Studio installed (Windows/Mac/Linux) — the Eclipse-based IDE from MuleSoft
  • An Anypoint Platform account so Studio can pull connectors from Exchange
  • An LLM API key — this lesson uses Anthropic Claude (any provider MAC supports also works: OpenAI, Azure OpenAI, Ollama)
  • Basic Studio comfort: dragging components onto the canvas, editing the XML view, running a project locally

Part 1 · The agent we're building

To keep this comparable to your Python lessons, we're building the same little agent from L18, L20, L26, L32, and L38:

# The task — a tool-using assistant User # "What is the weather in Bangkok, and search our internal docs for 'invoice policy'" ↓ HTTP POST /ask MAC Agent Flow # decide which tool(s) to call ├─ tool 1 → weather API └─ tool 2 → doc search (mock or vector store) ↓ LLM final answer # stitches results into natural language ↓ HTTP 200 JSON Response

The same agent shape you built five times in Python — this time as a Mule flow that Anypoint runs, monitors, and can deploy to CloudHub or Runtime Fabric with no extra code.

Part 2 · Install the MAC connectors

Steps:
  1. Open Anypoint Studio → File → New → Mule Project. Name it mac-agent-demo. Click Finish.
  2. In the Mule Palette (right panel), click Search in Exchange.
  3. Log in to Anypoint if prompted.
  4. Search: MuleSoft AI Chain. You'll see several connectors — MuleSoft AI Chain, MuleSoft Inference, MuleSoft Vectors, MuleSoft Embeddings are the common ones.
  5. Add MuleSoft AI Chain and MuleSoft Inference to your selections. Click Finish.
  6. Studio downloads the connectors and adds them to the palette.

Part 3 · Configure the LLM connector

New concept · Connector configuration
In Mule, a configuration is a reusable block of settings (API keys, URLs, model names) that any operation in the flow can point to. You create it once per project. Think of it as the Mule equivalent of the OpenAIChatCompletionClient(...) object in L38 — one config, many operation calls.
Steps:
  1. In the flow canvas, drag MuleSoft AI Chain → Chat answer prompt (or the equivalent inference operation for your MAC version) onto the canvas.
  2. With the operation selected, look at the Configuration field (bottom-right). Click the green +.
  3. In the config dialog:
    • LLM Type: ANTHROPIC (or OPENAI, OLLAMA, etc.)
    • Model Name: claude-haiku-4-5
    • API Key: ${anthropic.api.key} — reference a property, don't paste the raw key
    • Temperature: 0.2
  4. Create a src/main/resources/config.yaml with the key. Add Configuration Properties in the Global Elements pointing at it.
  5. Click Test Connection. Green tick = ready.
Screenshot 03 · LLM connector configuration dialog
Capture instructions: Screenshot the LLM config dialog with LLM Type, Model Name, and property-reference API key filled in — the moment of clicking Test Connection is ideal. Save as lessons/img/l44/03-llm-config.png.
Never paste the raw key
Even in a demo project. Studio's config dialog will happily accept the raw string — but the moment you commit or share the project the key is out. Use ${property.name} and put the actual value in config.yaml which is .gitignored. Same rule as L06 System & environment.

Part 4 · Build the agent flow

Now assemble the flow — HTTP entry, LLM call with tools, response.

Steps:
  1. Drag HTTP → Listener onto a new flow. Path: /ask, method POST. Create a new HTTP Listener config on 0.0.0.0:8081.
  2. After the listener, drag MuleSoft AI Chain → Agent (tool-calling). Point it at the LLM config from Part 3.
  3. In the operation config, add two tools (the exact UI varies by MAC version — this is the concept):
    • Tool 1: get_weather — description "Get current weather for a city", takes city string. Points at an HTTP Request to a weather API.
    • Tool 2: search_docs — description "Search internal docs", takes query string. Points at a Choice / static mock for now (returns "Invoices due within 30 days per policy X-123").
  4. Set the Prompt input to #[payload.question] so the flow reads the incoming question.
  5. After the agent operation, add a Transform Message to shape the response: { answer: payload.response }.
  6. Save. The XML view (bottom tab of the canvas) shows the underlying flow definition — read it once to build intuition.

Part 5 · The XML Studio generates (for reference)

Studio is a designer over Mule XML. Once you're comfortable, you can read and edit the XML directly. Here's the approximate shape of what Parts 3-4 produce:

<!-- src/main/mule/agent-flow.xml — abridged for readability -->
<mule xmlns:mac="http://www.mulesoft.org/schema/mule/mulesoft-ai-chain"
      xmlns:http="http://www.mulesoft.org/schema/mule/http">

  <mac:config name="AI_Chain_Config"
              llmType="ANTHROPIC"
              modelName="claude-haiku-4-5"
              apiKey="${anthropic.api.key}" />

  <http:listener-config name="HTTP_Listener_Config">
    <http:listener-connection host="0.0.0.0" port="8081"/>
  </http:listener-config>

  <flow name="ask-agent">
    <http:listener config-ref="HTTP_Listener_Config" path="/ask"/>

    <mac:agent-tool-calling config-ref="AI_Chain_Config"
                             prompt="#[payload.question]">
      <mac:tools>
        <mac:tool name="get_weather"
                  description="Get current weather for a city">
          <!-- flow reference or inline HTTP request -->
        </mac:tool>
        <mac:tool name="search_docs"
                  description="Search internal docs">
          <!-- static or vector-store-backed lookup -->
        </mac:tool>
      </mac:tools>
    </mac:agent-tool-calling>

    <ee:transform>
      <ee:message>
        <ee:set-payload>%dw 2.0
output application/json
---
{ answer: payload.response }</ee:set-payload>
      </ee:message>
    </ee:transform>
  </flow>
</mule>
New concept · DataWeave in one line
%dw 2.0 / output application/json / --- / { answer: payload.response } is DataWeave — Mule's transformation language. Every Mule flow ends up using it somewhere. Think of it as a purpose-built alternative to Python dict comprehensions, but for shape transformation across formats (JSON, XML, CSV, Java).
Element names may vary
mac:agent-tool-calling is illustrative — your installed MAC connector version may use mac:chat-answer-prompt-with-tools, mulechain-ai:agent, or a different name. Inspect the actual XML your Studio generates. The shape (config + operation + tools) is stable across versions.

Part 7 · Same task, six implementations

You've now built this agent six times. Here's the honest comparison:

The honest decision tree
  • Single agent + many enterprise systems (SAP, mainframe, Workday, ServiceNow) → MAC on Anypoint. You inherit 300+ connectors and enterprise-grade governance for free.
  • Complex reasoning loop, HITL, replay/time-travel → LangGraph. Visual flows are awkward for stateful graphs.
  • Multi-agent orchestration (supervisor / swarm / handoff) → code-first framework (ADK / Strands / AutoGen) on its cloud runtime.
  • Rapid prompt experimentation → Python framework. Faster iteration cycle than a flow editor.
  • Integration developers own the agent, Python team doesn't exist → MAC. Same team, same tool.
"The customer question isn't 'MAC or LangGraph?' — it's 'where does the value of this agent come from?' If the value is which systems it can reach, MAC on Anypoint is the shortest path because the connectors are already there. If the value is how sophisticated its reasoning is, a code-first framework wins. Most enterprise agents I see are the first kind — service desk, order status, HR self-service. That's why MAC is a real answer, not marketing filler."

Quiz

Test your understanding

  1. Why do we put the API key in config.yaml and reference it as ${anthropic.api.key} instead of pasting it into the connector dialog?
    Show answer
    So it doesn't get committed to source control. Studio will happily accept the raw string, but the moment you push the project, the key is out. Same rule as L06 — never commit secrets.
  2. Which layer of the MuleSoft agent stack (from the positioning reference) does this lesson build on?
    Show answer
    Anypoint Platform — the integration layer. MAC is a connector family that runs inside Anypoint's Mule runtime. Omni Gateway and Agent Fabric sit above it.
  3. Name one scenario where you'd pick LangGraph over MAC even though the agent needs to hit SAP.
    Show answer
    A stateful multi-turn workflow with human-in-the-loop approvals and checkpoint/replay — LangGraph's interrupt() and checkpointing are much cleaner in code than as a Mule flow. You'd still put Anypoint underneath as the SAP integration layer — LangGraph calls the Anypoint API as a tool.
  4. What is DataWeave?
    Show answer
    Mule's transformation language for shape-shifting data between formats (JSON, XML, CSV, Java, YAML). Every non-trivial Mule flow uses it. Roughly analogous to writing dict comprehensions and format converters in Python — but purpose-built for integration.
  5. The comparison table shows MAC at "~0 lines of code". Is that literally true?
    Show answer
    No — it's shorthand. You still write DataWeave transformations and configure XML behind the scenes. But you don't write imperative agent-loop code; the connector does that. The real trade-off is code-lines-saved vs flow-editor-clicks. Non-Python teams pick the second every time.
