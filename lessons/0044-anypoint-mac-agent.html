Lesson 44 · Anypoint Studio · MuleSoft Inference connector

Build a tool-using agent inside Anypoint Studio with the MuleSoft Inference connector

Same tool-using agent as L18/L20/L26/L32/L38 — but built as a Mule flow in Anypoint Studio using a single connector: MuleSoft Inference. Studio-first walkthrough with 9 screenshot slots for you to capture.

What you'll build
A running Mule application in Anypoint Studio that answers questions using an LLM, decides which "tools" to call, and returns a structured response over HTTP. Same job as your Python framework lessons — different tool, and built with just one connector: MuleSoft Inference. By the end, you'll see when this approach beats a code-first framework and when it doesn't.
Source honesty — read before you install
The MuleSoft Inference connector is part of the actively-evolving MuleSoft AI Chain (MAC) project. Names, operations, and package coordinates shift between releases. This lesson matches Inference connector v1.2.0 (Exchange asset com.mulesoft.connectors:mule4-inference-connector, XML namespace ms-inference) but verify the exact operation names on Anypoint Exchange before you drop them into your flow. Screenshots you capture will be authoritative for the version you have installed.

Prerequisites

  • Anypoint Studio installed (Windows/Mac/Linux) — the Eclipse-based IDE from MuleSoft
  • An Anypoint Platform account so Studio can pull connectors from Exchange
  • An OpenAI API key — this lesson uses OpenAI with gpt-4o-mini (the Inference connector also supports Anthropic, Azure OpenAI, Ollama, Groq, and more)
  • Basic Studio comfort: dragging components onto the canvas, editing the XML view, running a project locally

Part 1 · The agent we're building

To keep this comparable to your Python lessons, we're building the same little agent from L18, L20, L26, L32, and L38:

# The task — a tool-using assistant User # "What is the weather in Bangkok, and search our internal docs for 'invoice policy'" ↓ HTTP POST /ask Inference [Tools] operation # LLM decides which tool(s) to call ├─ tool 1 → weather API # you execute this in the flow └─ tool 2 → doc search # mock or vector store — you execute it ↓ feed tool results back to the LLM LLM final answer # stitches results into natural language ↓ HTTP 200 JSON Response

The same agent shape you built five times in Python — this time as a Mule flow that Anypoint runs, monitors, and can deploy to CloudHub or Runtime Fabric.

Important · "Reasoning only" — you run the tools
The Inference connector's tool operation is called [Tools] Native Template (Reasoning only). That name matters. Unlike a fully-managed agent that runs the whole loop for you, this operation does one thing: it sends your prompt plus the tool definitions to the LLM and returns which tool the model wants to call, with what arguments. You then wire the actual call (an HTTP Request to the weather API, a database lookup, etc.) in the Mule flow and feed the result back. More explicit, more flow-building — but you see every step, which is exactly the tool-execution the Python frameworks hid from you in L20–L43.

Part 3 · Configure the Inference connector

New concept · Connector configuration
In Mule, a configuration is a reusable block of settings (API keys, URLs, model names) that any operation in the flow can point to. You create it once per project. Think of it as the Mule equivalent of the OpenAIChatCompletionClient(...) object in L38 — one config, many operation calls. For the Inference connector this block is called a Text Generation config, and inside it you pick a connection (OpenAI, Anthropic, Ollama, …).
Steps:
  1. In the flow canvas, drag MuleSoft Inference → [Tools] Native Template (Reasoning only) onto the canvas.
  2. With the operation selected, look at the Configuration field (bottom-right). Click the green + to create a MuleSoft Inference Text Generation config.
  3. In the config dialog, set the Connection to OpenAI Connection, then fill in:
    • OpenAI Model Name: gpt-4o-mini
    • API Key: ${config.apiKey} — reference a property, don't paste the raw key
    • Timeout: 60000 (ms)
  4. Create a src/main/resources/config.yaml with the key (apiKey: sk-...). Add Configuration Properties in the Global Elements pointing at it.
  5. Click Test Connection. Green tick = ready.

Never paste the raw key
Even in a demo project. Studio's config dialog will happily accept the raw string — but the moment you commit or share the project the key is out. Use ${property.name} and put the actual value in config.yaml which is .gitignored. Same rule as L06 System & environment.

Part 4 · Build the agent flow

Now assemble the flow — HTTP entry, the LLM tool-reasoning call, your tool execution, then a second LLM call to compose the answer. Because the operation is "reasoning only", the tool-running step is yours to build.

Shortcut · Copy-paste the finished flow
Building this click-by-click is fiddly. If you'd rather see it working first and read it after, grab the complete, runnable flow: examples/l44/agent-flow.xml plus the key template config.yaml.example. Both tools are mocks, so it runs with just an OpenAI key — no external APIs. In Studio: paste the XML into src/main/mule/agent-flow.xml; then copy config.yaml.example to src/main/resources/config.yaml, paste your key into it, and run. Copying to the un-suffixed name matters — config.yaml is .gitignored so your real key never gets committed.
Steps:
  1. Drag HTTP → Listener onto a new flow. Path: /ask, method POST. Create a new HTTP Listener config on localhost:8081. (Use localhost, not 0.0.0.0 — on Mule 4.9+ a plain-HTTP listener bound to all interfaces fails deployment with "Connection is not secure. Should use HTTPS". Binding to localhost keeps it local-only and passes.)
  2. Add a Transform Message (or Set Variable) that builds the tool-reasoning inputs: a template (system role), instructions, the user's data (#[payload.question]), and a tools array in OpenAI function-calling JSON. Define two tools:
    • get_weather — description "Get current weather for a city", parameter city (string).
    • search_docs — description "Search internal docs", parameter query (string).
  3. Drag MuleSoft Inference → [Tools] Native Template onto the flow, pointing at the Text Generation config from Part 3. Wire template, instructions, data, and tools from the variable above.
  4. The operation returns the model's tool-call decision as an array (which tool(s), what arguments). Add a Transform that walks the array with map and, for each requested tool, actually runs it:
    • get_weather → a static mock for now (returns "It is 33 C and sunny"); swap for an HTTP Request to a real weather API later.
    • search_docs → a static mock for now (returns "Invoices due within 30 days per policy X-123"); swap for the Vectors connector later.
    Why a Transform and not a Choice router? The decision is an array — the model can ask for several tools at once — and map handles 0, 1, or many in a single pass. For real side-effecting tools (an HTTP call vs. a DB query vs. a Salesforce op), a Choice per tool inside a foreach is the cleaner production pattern; the mock keeps it to one component.
  5. Collect the tool results and make a second call — [Chat] Answer Prompt — feeding the results back so the LLM writes the final natural-language answer.
  6. End with a Transform Message to shape the response: { answer: payload.response }.
  7. Save. The XML view (bottom tab of the canvas) shows the underlying flow definition — read it once to build intuition.
Why the extra steps vs. Python
In L38 AutoGen ran the tool loop invisibly. Here you can see it: reason → route → execute → answer. That transparency is the point — every hop is a Mule component you can log, retry, secure, and monitor in Anypoint. It's more clicks, but nothing is hidden.

Part 5 · The XML Studio generates (for reference)

Studio is a designer over Mule XML. Once you're comfortable, you can read and edit the XML directly. Here's the approximate shape of what Parts 3-4 produce:

<!-- src/main/mule/agent-flow.xml — abridged for readability -->
<mule xmlns:ms-inference="http://www.mulesoft.org/schema/mule/ms-inference"
      xmlns:http="http://www.mulesoft.org/schema/mule/http">

  <!-- One config, one connection: OpenAI -->
  <ms-inference:text-generation-config name="OpenAI_Config">
    <ms-inference:openai-connection openAIModelName="gpt-4o-mini"
                                    apiKey="${config.apiKey}"
                                    timeout="60000"/>
  </ms-inference:text-generation-config>

  <http:listener-config name="HTTP_Listener_Config">
    <http:listener-connection host="localhost" port="8081"/>
  </http:listener-config>

  <flow name="ask-agent">
    <http:listener config-ref="HTTP_Listener_Config" path="/ask"/>

    <!-- 1. Ask the LLM WHICH tool to call (reasoning only) -->
    <ms-inference:tools-native-template config-ref="OpenAI_Config">
      <ms-inference:template>You are a helpful assistant</ms-inference:template>
      <ms-inference:instructions>Answer the request politely</ms-inference:instructions>
      <ms-inference:data>#[payload.question]</ms-inference:data>
      <ms-inference:tools>#[vars.tools]</ms-inference:tools>  <!-- OpenAI function-calling JSON -->
    </ms-inference:tools-native-template>

    <!-- 2. YOU execute the chosen tool(s) — e.g. route on the returned tool name -->
    <!--    <choice> → <http:request> for get_weather, static reply for search_docs -->

    <!-- 3. Feed results back so the LLM writes the final answer -->
    <ms-inference:chat-answer-prompt config-ref="OpenAI_Config">
      <ms-inference:prompt>#[/* question + tool results */]</ms-inference:prompt>
    </ms-inference:chat-answer-prompt>

    <ee:transform>
      <ee:message>
        <ee:set-payload>%dw 2.0
output application/json
---
{ answer: payload.response }</ee:set-payload>
      </ee:message>
    </ee:transform>
  </flow>
</mule>
New concept · DataWeave in one line
%dw 2.0 / output application/json / --- / { answer: payload.response } is DataWeave — Mule's transformation language. Every Mule flow ends up using it somewhere. Think of it as a purpose-built alternative to Python dict comprehensions, but for shape transformation across formats (JSON, XML, CSV, Java).
Element names may vary
These element names match Inference connector v1.2.0 (ms-inference:tools-native-template, ms-inference:chat-answer-prompt, ms-inference:openai-connection). A newer version may rename an operation or add a fully-managed agent op. Inspect the actual XML your Studio generates. The shape (config + connection + reasoning op + your tool execution) is what's stable.

Part 5b · Level 2 — skip the tool JSON with MCP

Parts 4-5 taught you the mechanism: hand-write the tools JSON, then route and execute each tool yourself. That's great for understanding — but in real projects you don't want to author tool definitions by hand, especially when the "tool" is just an existing REST API you already run on Anypoint. There's a shortcut: MCP.

New concept · MCP (Model Context Protocol)
MCP is an open standard that lets an LLM discover and call tools exposed by a server, without anyone hand-writing tool schemas into the agent. Think of it as USB-C for AI tools: expose a tool once on an MCP server, and any MCP-aware agent can plug in and use it. MuleSoft ships two pieces that make this a drop-in upgrade to the flow you just built.

The two connectors involved

  • MCP Connector (com.mulesoft.connectors:mule-mcp-connector, v1.6.1, namespace mcp) — turns a Mule flow into an MCP server. Each mcp:tool-listener is one tool. Wrap an existing REST API (an http:request, or an APIkit flow generated from your RAML/OpenAPI spec) and it becomes a tool automatically — the spec is the tool definition.
  • Inference Connector's [MCP] Tooling operation — the counterpart to [Tools] Native Template. Instead of a hand-written tools array, you point it at an MCP client config. It auto-discovers the tools the server exposes, lets the LLM pick them, and runs them for you. The manual reason→route→execute plumbing from Part 4 disappears.
The upgrade in one line
Part 4 = you define tools (JSON) and you execute them (Transform, or Choice + HTTP for real tools). Part 5b = an MCP server defines tools from your existing REST APIs, and [MCP] Tooling executes them for you. Same agent, far less hand-authoring.

Shape of the two flows

First, an MCP server that exposes an existing REST API (here, a weather API) as a tool:

<!-- MCP server: expose an existing REST API as a tool -->
<mule xmlns:mcp="http://www.mulesoft.org/schema/mule/mcp"
      xmlns:http="http://www.mulesoft.org/schema/mule/http">

  <mcp:server-config name="MCP_Server" serverName="tools-server" serverVersion="1.0.0">
    <!-- SSE transport; give the HTTP listener a high readTimeout so the
         long-lived stream isn't killed mid-handshake -->
    <mcp:sse-server-connection listenerConfig="HTTP_Listener_Config"/>
  </mcp:server-config>

  <flow name="weather-tool">
    <mcp:tool-listener config-ref="MCP_Server" name="get_weather">
      <mcp:description>Get current weather for a city</mcp:description>
      <mcp:parameters-schema>{ "type":"object", "properties": { "city": { "type":"string" } }, "required":["city"] }</mcp:parameters-schema>
    </mcp:tool-listener>
    <!-- call the existing REST API — this IS the tool's work -->
    <http:request config-ref="Weather_API_Config" path="/current"/>
  </flow>
</mule>

Then the agent flow discovers the server's tools and invokes them over MCP — you never hand-write the tools JSON, and each tool runs via [MCP] Call Tool instead of your own HTTP/Transform:

<!-- Agent flow: list tools → LLM reasons → call tool over MCP → answer -->
<mcp:client-config name="MCP_Client">
  <!-- base URL only; generous requestTimeout for the LLM+tool round-trip -->
  <mcp:sse-client-connection serverUrl="http://localhost:8082" requestTimeout="600"/>
</mcp:client-config>

<!-- 1) Discover what the server offers -->
<mcp:list-tools config-ref="MCP_Client"/>
<!-- 2) (Transform the tool list into OpenAI function JSON — see the runnable file) -->
<!-- 3) LLM decides which tool(s) to call (reasoning only) -->
<ms-inference:tools-native-template config-ref="OpenAI_Config" target="plan">
  <ms-inference:data>#[vars.question]</ms-inference:data>
  <ms-inference:tools>#[vars.tools.tools]</ms-inference:tools>
</ms-inference:tools-native-template>
<!-- 4) Actually run each chosen tool THROUGH MCP -->
<mcp:call-tool config-ref="MCP_Client" toolName="#[vars.currentTool.function.name]">
  <mcp:arguments>#[read(vars.currentTool.function.arguments, "application/json")]</mcp:arguments>
</mcp:call-tool>
Element names & versions
Namespaces and elements above match MCP Connector v1.6.1 and Inference v1.2.0. A full copy-paste-runnable version of this MCP flow is in examples/l44/agent-flow-mcp.xml — it mirrors MuleSoft's own same-app MCP example (example-mule-apps → mcp_demo): SSE transport, long timeouts, and explicit list-tools → reason → call-tool. A connector also ships a one-step [MCP] Tooling operation (ms-inference:mcp-tools-native-template); it's neat, but the discover-reason-call shape above is the one proven to run when the server and client share an app.
Demo limitation — one request per deploy
Because agent-flow-mcp.xml runs the MCP server and client in the same app, the long-lived SSE stream isn't re-established between requests. The first POST /ask works fully; a second one fails with Trying to write in a closed buffer — just redeploy to run it again. In a real deployment the server and client are two separate apps, each with its own connection lifecycle, so this doesn't happen. The point of this file is to prove the MCP discover-reason-call pattern end-to-end, which it does.
Why this is the better idea for your day job
You already build REST APIs on Anypoint every week. With MCP you don't rebuild them as "AI tools" — you expose the ones you have. One API spec → one MCP server → usable by this Mule agent, by Claude Desktop, by Agentforce, by any MCP-aware client. That reuse story is exactly the MuleSoft value proposition, now pointed at AI.

Deployment fails? Two common errors on Mule 4.9+
Ignore any "Already in lifecycle phase 'dispose'" line — that's just cleanup noise after a failed start. Look higher in the log for the real cause:
  • "Connection is not secure. Should use HTTPS" — your HTTP listener is bound to 0.0.0.0. Change the host to localhost (local dev) or add a TLS context (production).
  • "The name type is a reserved word" — a DataWeave object key uses a reserved word. Wrap it in single quotes: 'type':. Same applies to input, output, default, is, as, and others.

Part 7 · Same task, six implementations

You've now built this agent six times. Here's the honest comparison:

LessonFrameworkRuntimeLines of codeBest-fit use case

L20-25LangGraphPython + LangGraph Cloud / self-host~120Complex reasoning loops, HITL, checkpointing
L26-31Google ADKVertex Agent Engine~60Google-native shops, declarative multi-agent

The honest decision tree
  • Single agent + many enterprise systems (SAP, mainframe, Workday, ServiceNow) → MAC on Anypoint. You inherit 300+ connectors and enterprise-grade governance for free.
  • Complex reasoning loop, HITL, replay/time-travel → LangGraph. Visual flows are awkward for stateful graphs.
  • Multi-agent orchestration (supervisor / swarm / handoff) → code-first framework (ADK / Strands / AutoGen) on its cloud runtime.
  • Rapid prompt experimentation → Python framework. Faster iteration cycle than a flow editor.
  • Integration developers own the agent, Python team doesn't exist → MAC. Same team, same tool.
"The customer question isn't 'MAC or LangGraph?' — it's 'where does the value of this agent come from?' If the value is which systems it can reach, MAC on Anypoint is the shortest path because the connectors are already there. If the value is how sophisticated its reasoning is, a code-first framework wins. Most enterprise agents I see are the first kind — service desk, order status, HR self-service. That's why MAC is a real answer, not marketing filler."

Quiz

Test your understanding

  1. Why do we put the API key in config.yaml and reference it as ${openai.api.key} instead of pasting it into the connector dialog?
    Show answer
    So it doesn't get committed to source control. Studio will happily accept the raw string, but the moment you push the project, the key is out. Same rule as L06 — never commit secrets.
  2. Which layer of the MuleSoft agent stack (from the positioning reference) does this lesson build on?
    Show answer
    Anypoint Platform — the integration layer. MAC is a connector family that runs inside Anypoint's Mule runtime. Omni Gateway and Agent Fabric sit above it.
  3. Name one scenario where you'd pick LangGraph over MAC even though the agent needs to hit SAP.
    Show answer
    A stateful multi-turn workflow with human-in-the-loop approvals and checkpoint/replay — LangGraph's interrupt() and checkpointing are much cleaner in code than as a Mule flow. You'd still put Anypoint underneath as the SAP integration layer — LangGraph calls the Anypoint API as a tool.
  4. What is DataWeave?
    Show answer
    Mule's transformation language for shape-shifting data between formats (JSON, XML, CSV, Java, YAML). Every non-trivial Mule flow uses it. Roughly analogous to writing dict comprehensions and format converters in Python — but purpose-built for integration.
  5. The comparison table shows MAC Inference at "~0 imperative lines". Is that literally true?
    Show answer
    No — it's shorthand. You still write DataWeave transformations, define the tools JSON, and wire the tool-execution routing as Mule components. What you avoid is imperative glue code — HTTP clients, retry loops, JSON parsing — because those are visual components. The real trade-off is code-lines-saved vs flow-editor-clicks. Non-Python teams pick the second every time.
