Lesson 45 · Agent interoperability · MCP & A2A

MCP & A2A: how agents use tools, and how agents talk to each other

Two open protocols, one clean split. You already used MCP in L27 and L44 — now see what it actually is under the hood, meet its sibling A2A, and finish a tiny Python MCP client that discovers and calls a tool.

TL;DR
MCP connects one agent to its tools and data (a database, an API, a file). A2A connects one agent to other agents so they can delegate work to each other. They are complementary layers, not rivals — an agent uses MCP to reach its own tools, and A2A to collaborate with peers. In the hands-on section you complete a Python MCP client and watch it run the exact three-step lifecycle — initialize → list tools → call tool — you saw fly past in the L44 logs.

Why this matters to you

As a MuleSoft SE, your whole world is systems talking to systems through contracts. These two protocols are that same idea aimed at AI:

  • MCP is like a connector spec. It standardises how an agent plugs into a tool or data source — so any MCP-aware agent can use any MCP server, the way any HTTP client can call any REST API.
  • A2A is like an API contract between services — except the "services" are autonomous agents that reason and delegate. It standardises how one agent discovers and hands a task to another, even across vendors and frameworks.

MuleSoft has already staked out the MCP layer (an MCP Bridge on the Omni/Flex Gateway that turns any API into agent-ready tools — mulesoft.com/platform/ai/model-context-protocol). A2A is younger, and Salesforce (MuleSoft's parent) sits on its steering committee. Knowing both lets you speak credibly to the "how will our agents interoperate?" question customers are starting to ask.

Part 1 · MCP in one picture

The Model Context Protocol (MCP) is an open standard (originated by Anthropic) for connecting an AI application to external tools and data. It has three roles:

# MCP roles HOST # the AI app the user talks to (Claude Desktop, an IDE, your agent) └─ CLIENT # a connector inside the host — one per server — speaks the protocol │ JSON-RPC 2.0 (initialize → list → call) ▼ SERVER # exposes TOOLS, RESOURCES, PROMPTS (your Python file, or a Mule flow)

An MCP server exposes three kinds of things:

  • Tool — a function the model can call to take an action (think a POST endpoint). Example: get_order_status(order_id).
  • Resource — read-only context the model or user can load (think a GET endpoint), addressed by a URI.
  • Prompt — a reusable, templated message/workflow.
New idea · JSON-RPC lifecycle
Every MCP conversation follows the same three steps: 1) initialize — client and server agree on a protocol version and exchange capabilities; 2) list — client asks tools/list to discover what's available; 3) call — client sends tools/call with arguments and gets a result. This is exactly the handshake you watched succeed in the L44 MCP logs (Client initialize request → Server response → list-tools → call-tool). Transport is stdio (local subprocess) or Streamable HTTP. (The old standalone SSE transport is now deprecated — see sources.)

Part 2 · A2A in one picture

The Agent2Agent Protocol (A2A) was introduced by Google in 2025 and donated to the Linux Foundation; its steering committee includes Salesforce, Microsoft, AWS, SAP and others (a2a-protocol.org). Where MCP points an agent down at its tools, A2A points an agent sideways at other agents.

# A2A: two autonomous agents collaborating CLIENT AGENT ──① GET /.well-known/agent-card.json ─▶ REMOTE AGENT │ │ (publishes an Agent Card: ──② send Message (a Task) ───────────▶ its skills + endpoint) │ │ ◀─③ Task: working → completed (Artifact) (does the work, returns result)

The core A2A vocabulary (taught concept-first — the exact field names are still settling, so verify against the live spec before you quote them to a customer):

  • Agent Card — a small JSON "business card" a remote agent publishes at /.well-known/agent-card.json. It lists the agent's skills, its endpoint URL, and how to authenticate. This is how one agent discovers another.
  • Task — the unit of work. It moves through a lifecycle: submitted → working → completed (or failed / canceled / input-required).
  • Message & Parts — a message is one exchange; it carries parts (text, a file, or a structured data blob).
  • Artifact — the task's output, kept separate from the back-and-forth conversation.
Source honesty — verify before customer commitments
A2A hit its first stable release (spec v1.0.0) only in April 2026, with breaking changes from the 0.3 line. Tutorials, blog posts, and any model's memory older than mid-2026 likely describe the old terminology (e.g. the card was once agent.json, now agent-card.json). The durable idea — discover an agent via a card, delegate a task, get an artifact back — is stable; the exact field names and SDK class names are not. Always confirm against the live spec before a demo. This lesson deliberately teaches the framing, not the fine print.

Part 3 · MCP vs A2A — the one table to remember

The official A2A docs put it best: "A2A is about agents partnering on tasks, while MCP is more about agents using capabilities" (a2a-protocol.org/topics/a2a-and-mcp).

The mental model that ties it together
They stack. A single agent can be an A2A server to its peers (offering a skill via an Agent Card) and an MCP client internally (reaching its own tools/data to actually do the work). MCP is the agent's hands; A2A is the agent's phone line to colleagues.

Part 4 · Hands-on — finish a Python MCP client

Two tiny files live in lessons/examples/l45/: a finished server (mcp_server.py) that exposes a get_order_status tool, and a client (mcp_client.py) that's done except for the final tool call. Your job is to write that call — the payoff step of the whole protocol.

Setup & run
  1. This project's virtual environment already has the mcp SDK installed — no install step needed. (On this Mac the command is python3, not python, so the easiest thing is to use the venv's own interpreter.)
  2. From the lessons/examples/l45/ folder, run:
    ~/ai/claude/projects/python/.venv/bin/python mcp_client.py
    …or source ~/ai/claude/projects/python/.venv/bin/activate first, then just python mcp_client.py.
  3. Right now it prints the tool list, then a placeholder line. You'll replace the placeholder.
Fresh machine with no venv? Install once with pip3 install "mcp[cli]>=1.27,<2" (note pip3, not pip).

Here is the section of mcp_client.py you complete — the initialize and list steps are done for you:

# STEP 1 — initialize: negotiate protocol version + capabilities. await session.initialize() # STEP 2 — discover: ask the server what tools it exposes. tools = await session.list_tools() print("Available tools:", [t.name for t in tools.tools]) # STEP 3 — call a tool. <<< YOUR TASK >>> # TODO: call "get_order_status" with order_id "A100". # result = await session.call_tool(<name>, arguments=<dict>) # the text is at: result.content[0].text # then: print("Order A100 status:", <that text>)
New syntax · async / await
MCP's Python API is asynchronous. await means "pause here until this network call finishes, without blocking the whole program." You can only use await inside a function defined with async def, and you kick the whole thing off with asyncio.run(main()). For now, just follow the pattern — treat await as "wait for the reply." We'll go deeper on async in a later lesson if you want.
Success looks like
When your STEP 3 is right, running python mcp_client.py prints: 
Available tools: ['get_order_status']
Order A100 status: shipped — arriving tomorrow
(A couple of grey INFO lines from the server may appear too — that's normal logging, not an error.)

Check yourself

  1. Your teammate says "let's use A2A so our agent can query the orders database." Right protocol?
    Show answer
    No — a database is a tool/data source, so that's an MCP job. A2A would be right if the agent needed to hand a task to another agent (say, a separate fulfilment agent).
  2. In the demo, which of the three lifecycle steps actually runs the business logic on the server?
    Show answer
    Step 3, call_tool. initialize just does the handshake; list_tools only discovers what's available; call_tool is what invokes get_order_status and returns its result.
  3. Where would another agent look to discover what your A2A agent can do?
    Show answer
    At its Agent Card, served at /.well-known/agent-card.json — it lists the agent's skills, endpoint, and auth.
