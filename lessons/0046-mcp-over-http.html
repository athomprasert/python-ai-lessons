Lesson 46 · Agent interoperability · MCP transports

Run an MCP server over HTTP — the transport your MuleSoft server used

L45 used stdio (the client launched the server). Now flip to Streamable HTTP: the server runs on its own URL, and clients connect to it — exactly how the MuleSoft MCP server behaved in L44. Two terminals, one new gotcha, one tool you add yourself.

TL;DR
Same MCP protocol as L45, different transport. You'll start a Python MCP server with mcp.run(transport="streamable-http") so it listens on http://127.0.0.1:8000/mcp, then connect a client by URL from a second terminal. Along the way you fix the one real difference — the HTTP client hands back a 3-tuple, not a 2-tuple — and add a second tool (cancel_order) to the server.

Part 1 · Two transports, one protocol

In L45's last question you asked "how did the client find the server without running it first?" — the answer was stdio: the client launched the server as a subprocess and spoke over pipes. That's great for a local tool, but it can't work when the server is a shared service on another machine. For that you use Streamable HTTP: the server is always-on at a URL, and any number of clients connect to it.

Part 2 · The server (and your first task)

Open lessons/examples/l46/mcp_http_server.py. It's the same idea as L45 — functions become tools via @mcp.tool() — with two changes: the constructor sets a host/port, and it runs with the HTTP transport: 

from mcp.server.fastmcp import FastMCP

# host/port are set here; the HTTP endpoint defaults to "/mcp"
mcp = FastMCP("l46-http-server", host="127.0.0.1", port=8000)

@mcp.tool()
def get_order_status(order_id: str) -> str:
    """Look up the delivery status of a customer order by its ID."""
    ...

if __name__ == "__main__":
    mcp.run(transport="streamable-http")   # <- the key line

Reminder · the decorator IS the registration
You don't wire the new tool up anywhere else. The @mcp.tool() line is what makes cancel_order discoverable — its name, type hints, and docstring become the tool's schema and description automatically. Same pattern you learned in L45; you're just repeating it.

Part 3 · The client (and the one real gotcha)

Open lessons/examples/l46/mcp_http_client.py. It connects by URL instead of launching a subprocess. Everything after connecting — initialize → list → call — is identical to L45. But the connect line has a deliberate bug for you to fix.

New idea · the HTTP client returns THREE values
L45's stdio_client(...) gave you two streams: (read, write). The HTTP transport also tracks a session id (HTTP is stateless, so MCP threads a session through headers), so streamablehttp_client(url) hands back three values: (read, write, get_session_id). You don't need the id here — but Python forces you to unpack all three, or you get a ValueError: too many values to unpack.

Your task (part B): fix the unpacking, then uncomment the call to your new tool.

# TODO (part B1): the line below unpacks only TWO values but gets THREE. # change: as (read, write): # to: as (read, write, _get_session_id): async with streamablehttp_client(SERVER_URL) as (read, write): async with ClientSession(read, write) as session: await session.initialize() tools = await session.list_tools() print("Available tools:", [t.name for t in tools.tools]) result = await session.call_tool("get_order_status", arguments={"order_id": "A100"}) print("Order A100 status:", result.content[0].text) # TODO (part B2): after adding cancel_order to the server, uncomment: # cancelled = await session.call_tool("cancel_order", arguments={"order_id": "A200"}) # print("Cancel A200:", cancelled.content[0].text)
Why "_" for the third value?
Naming a variable _ (or prefixing with _) is the Python convention for "I have to receive this value, but I'm not going to use it." It signals intent to the next reader — nothing magic, just a name.

The bridge back to L44
This is exactly what your MuleSoft MCP server did. In L44 the Mule app listened on http://localhost:8082 and your client used a base URL — Streamable HTTP, same as here. The lesson: an MCP server is just "a thing at a URL that speaks the protocol." Whether it's 20 lines of Python or a Mule flow with an MCP Bridge on the gateway, any MCP client can talk to it. That interchangeability is the whole point of a protocol.

Check yourself

  1. You edit the server to add a third tool, but the client still lists only two. What did you forget?
    Show answer
    To restart the server. A running Python process holds the old code — Ctrl-C in terminal 1 and rerun python mcp_http_server.py so it picks up your change.
  2. Why does the HTTP client return three values when the stdio client returned two?
    Show answer
    HTTP is stateless, so MCP tracks a session id across requests; the transport gives you a get_session_id helper as the third value. stdio is a single persistent pipe, so it needs no session id.
  3. Your MuleSoft MCP server from L44 and this Python server both listen on an HTTP URL. Could the L46 Python client call the Mule server?
    Show answer
    Yes — that's the point of a shared protocol. Point SERVER_URL at the Mule server's base URL + its MCP path and the same initialize → list → call flow works. (Mind L44's same-app SSE quirk, but a standalone Mule MCP server is fair game.)
