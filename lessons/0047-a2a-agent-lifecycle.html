Lesson 47 · Agent interoperability · A2A

Build an A2A agent — discovery and the task lifecycle

L45–46 were the agent → tools half (MCP). This is the agent → agent half: you'll publish an agent that advertises itself with an Agent Card, then a client that discovers it and delegates a task — and watch that task move through its lifecycle, streamed back live.

TL;DR
An A2A agent is an HTTP service that (1) publishes a JSON Agent Card at /.well-known/agent-card.json so others can discover it, and (2) accepts a delegated Task and drives it through states: submitted → working → completed, streaming each change plus the result Artifact back. You'll finish a working a2a-sdk agent + client, two terminals, and see all five lifecycle events print.
Version honesty — read this before you Google
This lesson targets a2a-sdk 1.1.2 (what's installed in your venv, verified end-to-end). The A2A SDK changed a lot across versions — most tutorials online use the older API (a single url field, A2AStarletteApplication, pydantic models). The 1.1.2 code here uses proto-based types, supported_interfaces, a Role enum, and route-builder functions instead. If a snippet online doesn't match this lesson, it's almost certainly a version gap — trust the running code.

Part 2 · The Agent Card (how discovery works)

An agent's first job is to be findable. The Agent Card is a small JSON document served at a well-known path — the agent's public "business card": its name, what it can do (skills), and how to reach it (interfaces). A client fetches it and now knows how to talk to the agent, with zero prior wiring. 

from a2a.types import AgentCard, AgentSkill, AgentCapabilities, AgentInterface

CARD = AgentCard(
    name="fx-agent",
    description="Converts currency amounts between currencies.",
    version="1.0.0",
    # supported_interfaces = how to reach me (URL + which protocol to speak)
    supported_interfaces=[AgentInterface(url="http://127.0.0.1:9100/",
                                         protocol_binding="JSONRPC")],
    capabilities=AgentCapabilities(streaming=True),
    default_input_modes=["text/plain"], default_output_modes=["text/plain"],
    skills=[AgentSkill(id="fx", name="Currency convert",
                       description="Convert an amount between currencies.", tags=["finance"])],
)
See it for real
Once the server is running (Part 4), open http://127.0.0.1:9100/.well-known/agent-card.json in a browser. That JSON is the discovery mechanism — any A2A client anywhere can read it and know how to delegate work to your agent.

Part 3 · The task lifecycle (your main task)

When a client delegates work, the agent doesn't just return a value — it creates a Task and moves it through states. This is the heart of A2A: tasks can be quick or long-running, and the caller watches progress the whole way. In a2a_server.py, the FxExecutor.execute() method is where you drive it, using a TaskUpdater helper:

# In FxExecutor.execute() — <<< YOUR TASK >>> uncomment + complete all five: await updater.submit() # -> TASK_STATE_SUBMITTED ("got it") await updater.start_work() # -> TASK_STATE_WORKING ("on it") result = f"{user_text} -> 135.20 SGD (mock rate)" # add_artifact publishes the actual RESULT of the task await updater.add_artifact([new_text_part(result)], name="conversion") await updater.complete() # -> TASK_STATE_COMPLETED ("done")
One rule the SDK enforces
The Task object must be enqueued before any status update — otherwise you get InvalidAgentResponseError: Agent should enqueue Task before TaskStatusUpdateEvent. The scaffold already does this for you (it calls new_task_from_user_message(...) and enqueues it right before creating the TaskUpdater). Your job is just the five lifecycle lines above.

create_from_url() is the discovery step — you give it a base URL, it reads the card, and hands back a client that already knows the right protocol. Same "just point it at a URL" idea as the MCP HTTP client in L46.

Part 5 · The whole mini-series, connected

MCP + A2A together — the real architecture
You've now built both halves. In production they compose: an A2A agent (like this one) receives a delegated task, and to fulfil it, calls MCP tools/servers (like L46's) for data and actions. So fx-agent could, inside execute(), call an MCP server for a live exchange rate — A2A on the outside (agent-to-agent), MCP on the inside (agent-to-tools). That's the shape of a multi-agent system: A2A for delegation between agents, MCP for each agent's capabilities.
"Your customer won't build one giant agent — they'll build several, owned by different teams, maybe on different stacks. A2A is how those agents delegate to each other without bespoke point-to-point integration: publish an Agent Card, and any other agent can discover and call it. MuleSoft's role is the same story as always — govern, secure, and expose these agent endpoints on the platform, so the mesh of agents is managed, not a free-for-all. MCP exposes your APIs as tools; A2A lets agents cooperate. Same reuse pitch, one layer up."

Check yourself

  1. What's the one-line difference in purpose between MCP and A2A?
    Show answer
    MCP connects an agent to tools & data (a function call); A2A connects an agent to other agents (delegate a task with a lifecycle). Tools vs teammates.
  2. Your client runs but the run just hangs after TASK_STATE_WORKING. What did you forget in the server?
    Show answer
    await updater.complete(). Without a terminal state (completed/failed/canceled), the streaming client keeps waiting for the task to finish.
  3. How does the client know how to talk to the agent without any hardcoded config beyond a URL?
    Show answer
    It fetches the Agent Card from <url>/.well-known/agent-card.json (via create_from_url). The card's supported_interfaces tell the client which protocol to speak. That's A2A discovery.
