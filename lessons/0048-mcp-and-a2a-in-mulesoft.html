MCP server & A2A agent — the MuleSoft side

You built both protocols by hand in Python (L45–47). Now the mirror image: expose an MCP server and an A2A agent as Mule flows in Anypoint Studio — using the MCP Connector and A2A Connector — then call each from the same Python clients you already wrote. Studio-first walkthrough with screenshot slots you capture.

TL;DR
Two Anypoint connectors turn a Mule flow into an agent endpoint. The MCP Connector's <mcp:tool-listener> exposes a flow as an MCP tool (call it from your L46 client). The A2A Connector's <a2a:task-listener> exposes a flow as an A2A agent with an Agent Card (call it from your L47 client). Same primitives you coded in Python — now declared in a flow, with Anypoint auth/governance for free.

Part 2 · Expose a flow as an MCP server

The MCP Connector runs in two modes — server (expose your flow as MCP tools) and client (call a remote MCP server). We want server mode, over Streamable HTTP — exactly the transport your L46 client speaks.

Add the connector & build the tool flow:
  1. File → New → Mule Project, name it agent-endpoints-demo.
  2. In Mule Palette → Search in Exchange, type mcp, add MCP Connector, Finish.
  3. Drag an MCP → Tool Listener source onto a new flow. This is your <mcp:tool-listener> — the flow's logic becomes the tool body.
  4. On the listener, set the tool name, description, and input schema — this is the metadata your L45/L46 client saw via list_tools().
  5. Configure the global MCP config for Streamable HTTP (bind it to an HTTP Listener config, e.g. port 8081, path /mcp).
  6. In the flow body, add a Transform Message that returns the tool result (start hardcoded, like L46's mock rate).

Part 3 · Expose a flow as an A2A agent

Now the agent-to-agent half. The A2A Connector also has server and client connection types. Server mode makes your flow a discoverable agent — the direct analog of your L47 a2a_server.py.

Add the connector & build the agent flow:
  1. In Mule Palette → Search in Exchange, type a2a, add A2A Connector, Finish.
  2. Drag an A2A → Task Listener source onto a new flow. This is <a2a:task-listener> — incoming delegated tasks land here (like FxExecutor.execute() receiving a task).
  3. Open the global A2A Server config. Set the Agent Card via either Card Json (paste the JSON inline) or Card Path (point to a file). This is your L47 AgentCard(...) — name, description, skills, URL — as declared config.
  4. Bind the server config to an HTTP Listener (e.g. port 8082). The card is then served for discovery, the connector's equivalent of L47's /.well-known/agent-card.json.
  5. In the flow body, produce the task result and let the connector drive the lifecycle back to the caller (submitted → working → completed — the states you drove by hand with TaskUpdater).

Part 4 · The client side (Mule calling agents)

You've used Python as the client so far, but a Mule flow can be the caller too — one Mule agent delegating to another. That's the A2A Connector's client operations:

OperationWhat it doesL47 Python analog
<a2a:get-card>Fetch a remote agent's card (discovery)create_from_url()
<a2a:send-message>Send a message/task, get the responsesend_message()
<a2a:send-stream-message>Send + stream status/artifacts over SSEthe streaming async for loop
<a2a:task-resubscribe>Reconnect to an in-flight task's stream after a drop(no L47 equivalent — production nicety)
Get Task / Cancel TaskLifecycle control on an existing taskTaskState checks / cancel

So a MuleSoft agent can be discovered and discover others — full participant in the A2A mesh, not just a server.

"Your customer already has the integration estate — APIs, connectors, governance. These two connectors mean they don't stand up a separate Python service to join the agent world: an existing Mule flow becomes an MCP tool or an A2A agent with a listener and a config, inheriting Anypoint's auth, policies, and monitoring. And because it's the open protocols, their Mule agents interoperate with Agentforce, LangGraph, or anyone else's agents. That's the MuleSoft pitch in one line — you don't rebuild for agents, you expose what you already have, governed."
Where this sits in the bigger picture
These connectors are the build layer. Above them, Omni Gateway / Agent Fabric govern the MCP and A2A endpoints at scale — discovery via the Agent Registry, routing via the Agent Broker, policy at the gateway. See the MuleSoft agentic reference architecture reference doc for how the flow you just built plugs into that fabric.

Check yourself

  1. Which listener turns a Mule flow into an MCP tool, and which into an A2A agent?
    Show answer
    <mcp:tool-listener> → MCP tool; <a2a:task-listener> → A2A agent. Different connectors, same "flow-as-endpoint" idea.
  2. In L47 you wrote AgentCard(...) in Python. Where does that live in the MuleSoft A2A agent?
    Show answer
    In the A2A Server config, as Card Json (inline) or Card Path (file). Same card content — declared instead of coded — and served for discovery like L47's well-known path.
  3. Why can your unchanged L46/L47 Python clients call these Mule endpoints?
    Show answer
    Because MCP and A2A are open protocols. The client only speaks the wire protocol; it doesn't care whether the server is Python or a Mule flow. That interoperability is the entire point.
