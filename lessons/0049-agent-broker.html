The Agent Broker & Agent Script orchestration

What the Agent Broker is, why it exists when a consuming agent can already route to peers over A2A, and how Agent Script turns broker orchestration into an explicit graph of nodes, edges, and triggers. Grounded entirely in MuleSoft's public docs and release notes.

TL;DR
The Agent Broker is a context-aware routing service that "plans and delegates work based on context" — request intent, identity, policies, and runtime state. It isn't a capability a consumer lacks (a consumer can route via A2A); it centralizes routing so the logic lives in one governed place instead of hardcoded in every agent. As of the July 2026 release, you author that orchestration in Agent Script — "a graph-based language that gives you explicit control over how brokers coordinate agents, tools, and LLMs."
Source & version honesty
Agent Fabric is new and moving fast. Everything here is drawn from public MuleSoft sources — the Agent Fabric docs, the Agent Fabric release notes (broker: 3 Oct 2025; Agent Script: 14 Jul 2026), and the Agent Script guide. Exact syntax and capabilities shift between releases — verify against those docs before a customer commitment, and don't quote a field name from memory.

Start with the honest objection

A fair challenge: in A2A, a consuming agent can already discover peers (via Agent Cards) and route to them directly. So a broker is not a capability the consumer lacks. If that's true, why introduce one?

The answer is about where routing logic lives, not whether routing is possible. Without a broker, every consumer embeds its own routing:

  • Each agent hardcodes "refunds → billing-agent" — duplicated across N consumers.
  • Change a downstream agent and every consumer that knew about it must be updated.
  • Routing policy (identity, "prod agents only", fallbacks) is reimplemented per agent, inconsistently.
  • It's the point-to-point API mesh again — one level up, between agents.

A broker makes routing a decision made once, in one governed place. The honest analogy: it's an orchestration layer for agents, the way a Process API is an orchestration layer for services. You don't need one for three agents wired directly — you need it when direct wiring stops scaling. So the broker is genuinely optional for small, static topologies; it earns its place with many agents, dynamic routing, and a team that wants routing rules centralized rather than scattered.

What the broker routes on

The docs describe the broker as an "intelligent routing agent that plans and delegates work based on context." That context has four public inputs — everything else about routing hangs off these:

Routing inputMeaningCould a consumer do it alone?
Request intentMatch the task to an agent's advertised skills (from its Agent Card)Yes — the broker just centralizes it
IdentityRoute based on who is calling and their scopesPainful & inconsistent per-consumer — the broker's real value
PoliciesGovernance rules applied to the routing decisionHard to keep consistent across consumers
Runtime stateDelegate based on current conditions and track task stateA hardcoded consumer just fails when a target is down

Broker decides · Omni Gateway enforces

Keep these two boxes distinct: the broker makes the routing/orchestration decision; the Omni Gateway governs the traffic. The gateway applies policies and controls across agents, brokers, LLM proxies, and MCP servers, with support for both MCP and A2A. So the broker decides which agent; the gateway enforces auth, rate limits, and policy on the calls.

# Two distinct responsibilities consuming agent │ ═══ OMNI GATEWAY ═══ # governance: auth, rate-limit, policy (MCP + A2A) │ AGENT BROKER # decision: which agent handles this? plan & delegate │ target agent / MCP tool / LLM

Watch the common mistake: don't attribute rate-limiting and auth to the broker. Those are gateway functions. The broker's job is the routing decision and task delegation.

How broker orchestration evolved (per public release notes)

The public release notes show the broker maturing in two visible steps:

Release noteWhat shipped
3 Oct 2025"Orchestrate agents across ecosystems with agent brokers, a context-aware routing service for agents and assets." — the broker as a routing service.
14 Jul 2026"Build agent networks with predictable, governed orchestration using Agent Script — a graph-based language that gives you explicit control over how brokers coordinate agents, tools, and LLMs." — orchestration becomes an authored graph.

The shift worth understanding: from "a routing service that plans for you" toward "orchestration you define explicitly as a graph." Which brings us to Agent Script.

Agent Script — orchestration as a graph

Per the July 2026 release note, you define broker orchestration as a graph of nodes, edges, and triggers in Agent Script (.agent) files. The key design idea is mixing two kinds of node:

  • "Use probabilistic nodes for complex, open-ended decisions" — where an LLM should reason.
  • "Use deterministic nodes to enforce fixed execution paths" — where the flow must be predictable.

That combination is the point: a graph lets you pin the order where the business needs guarantees, and let an LLM reason only where judgement is genuinely required. The Agent Script guide frames the language as combining the flexibility of natural-language instructions with the reliability of programmatic expressions for handling business rules, including conditional expressions that determine the agent's execution path.

# The mental model — a broker graph in Agent Script trigger ─▶ [deterministic node] ─▶ [probabilistic node] ─▶ [deterministic node] ─▶ result (fixed path) (LLM reasoning) (fixed path) edges connect nodes and guarantee the order you declared
Why graph > free-form loop
A purely LLM-driven orchestrator is flexible but hard to make predictable — the same request can be handled differently run to run, which is a problem for enterprise workflows that must be auditable and repeatable. An explicit graph gives "predictable, governed orchestration": determinism where you need it, reasoning where you don't.

Where the broker sits in Agent Fabric

The broker is one box in the wider fabric. Around it:

  • Omni Gateway — governs MCP + A2A traffic (auth, rate-limit, policy) for agents, brokers, LLM proxies, and MCP servers.
  • Agent Registry (Anypoint Exchange) — a centralized catalog for AI assets; how agents are discovered and reused.
  • Agent Visualizer + Anypoint Monitoring — see the network and its metrics/traces.
"Don't sell the broker as 'you can't route without it' — a sharp customer will call that out, because in A2A a consumer can route to peers directly. Sell it as where your routing rules live. With twenty agents across five teams, do you want routing logic hardcoded and duplicated in every agent, or declared once as a governed graph? That's the broker: context-aware routing on intent, identity, policy and runtime state, authored explicitly in Agent Script — while the Omni Gateway enforces auth and rate limits around it. Predictable where it must be, intelligent where it should be."

Check yourself

  1. A customer says "our consuming agents can already route to each other — why add a broker?" What's the honest answer?
    Show answer
    They're right that routing is possible without one. The broker's value is centralizing where routing logic lives — declared once instead of hardcoded and duplicated across every consumer — plus consistent identity/policy handling. It's optional for small/static topologies; it pays off at scale.
  2. What does the Broker decide, and what does the Omni Gateway enforce?
    Show answer
    Broker = which agent handles the task, plus planning and delegation. Gateway = governance on the traffic (auth, rate limits, policy) across agents, brokers, LLM proxies, and MCP servers. Don't attribute rate-limiting/auth to the broker.
  3. In Agent Script, what's the difference between a probabilistic and a deterministic node?
    Show answer
    Probabilistic nodes are "for complex, open-ended decisions" (LLM reasoning); deterministic nodes "enforce fixed execution paths." Mixing them in one graph gives predictable orchestration with reasoning only where needed.
