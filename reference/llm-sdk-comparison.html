LLM SDK Comparison

Gemini vs OpenAI vs Anthropic — same task in each SDK, so you can switch providers without re-learning the world.

The one thing to remember

Gemini's chat is the only one of the three that auto-manages history for you. OpenAI's core API and Anthropic's API are stateless — every call, you pass the full conversation as a messages=[...] list. Everything else (tool use, streaming, structured output) follows similar shapes across all three, but with different field names.

Install & init

Gemini

# pip install google-genai
from google import genai

client = genai.Client(
    api_key=os.environ["GEMINI_API_KEY"]
)

OpenAI

# pip install openai
from openai import OpenAI

client = OpenAI(
    api_key=os.environ["OPENAI_API_KEY"]
)

Anthropic

# pip install anthropic
from anthropic import Anthropic

client = Anthropic(
    api_key=os.environ["ANTHROPIC_API_KEY"]
)

Single-turn call

Gemini

r = client.models.generate_content(
    model="gemini-2.0-flash",
    contents="Hello",
)
print(r.text)

OpenAI

r = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user",
                "content": "Hello"}],
)
print(r.choices[0].message.content)

Anthropic

r = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    messages=[{"role": "user",
                "content": "Hello"}],
)
print(r.content[0].text)
Watch the differences. OpenAI requires messages=[...] even for a one-liner. Anthropic requires an explicit max_tokens — there's no default cap. Gemini accepts a plain string for contents.

Multi-turn conversation (state)

This is the biggest divergence. Three different mental models:

SDKState helperWhat you do
Gemini client.chats.create() SDK stores history in a Python object. Call chat.send_message(text) repeatedly — history auto-prepended.
OpenAI Partial — see notes Chat Completions API: fully stateless, you maintain a messages list. Responses API (newer): pass previous_response_id for server-side state. Assistants API: threads hold history.
Anthropic None Always stateless. You append the assistant's reply to your messages list yourself before the next call.

Gemini

chat = client.chats.create(
    model="gemini-2.0-flash",
)
r1 = chat.send_message("My name is Att.")
r2 = chat.send_message("What's my name?")
# Knows: "Att"

OpenAI

msgs = [{"role": "user",
         "content": "My name is Att."}]
r1 = client.chat.completions.create(
    model="gpt-4o-mini", messages=msgs)

msgs.append({"role": "assistant",
   "content": r1.choices[0].message.content})
msgs.append({"role": "user",
             "content": "What's my name?"})
r2 = client.chat.completions.create(
    model="gpt-4o-mini", messages=msgs)

Anthropic

msgs = [{"role": "user",
         "content": "My name is Att."}]
r1 = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024, messages=msgs)

msgs.append({"role": "assistant",
             "content": r1.content[0].text})
msgs.append({"role": "user",
             "content": "What's my name?"})
r2 = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024, messages=msgs)
The sharp edge with Gemini's chat: history lives in the Python process only. Restart your script and it's gone. For multi-request servers, multi-user systems, or anything beyond a single CLI run, you need to persist history yourself (DB) or use a framework with built-in memory (LangGraph checkpointer, ADK sessions). See L18 productionising checklist.

System prompt

Gemini

from google.genai import types

cfg = types.GenerateContentConfig(
    system_instruction="You are terse."
)
r = client.models.generate_content(
    model="gemini-2.0-flash",
    contents="Hello",
    config=cfg,
)

OpenAI

r = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
      {"role": "system",
       "content": "You are terse."},
      {"role": "user",
       "content": "Hello"},
    ],
)

Anthropic

r = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system="You are terse.",
    messages=[{"role": "user",
                "content": "Hello"}],
)

Three different shapes. Gemini puts it in a config object. OpenAI uses a {"role": "system"} message in the list. Anthropic has a dedicated top-level system= parameter (and refuses to mix it into messages).

Tool use (function calling)

All three share the same loop: LLM returns a tool call → your code runs the tool → you send the result back → LLM continues. The field names and result-message shapes differ.

ConceptGeminiOpenAIAnthropic
Pass tools config=types.GenerateContentConfig(tools=[fn])
(SDK auto-extracts schema from the Python function)		 tools=[{"type":"function","function":{...}}]
(explicit JSON schema)		 tools=[{"name":..., "input_schema":{...}}]
(explicit JSON schema)
Read tool call response.function_calls response.choices[0].message.tool_calls response.content blocks where block.type == "tool_use"
Send tool result types.Part.from_function_response(name, response={"result": ...}) {"role":"tool","tool_call_id":id,"content":...} {"type":"tool_result","tool_use_id":id,"content":...} inside user message
Stop reason No function_calls → final answer finish_reason == "stop" stop_reason == "end_turn"
Gemini's biggest ergonomic win: you can pass a plain Python function to tools=[my_fn] and the SDK reads the docstring and type hints to generate the schema. With OpenAI and Anthropic, you write the JSON schema by hand (or use a helper like pydantic or instructor). This is the win L18 leans on.

Streaming

Gemini

for chunk in client.models.generate_content_stream(
    model="gemini-2.0-flash",
    contents="Write a haiku",
):
    print(chunk.text, end="")

OpenAI

stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user",
        "content": "Write a haiku"}],
    stream=True,
)
for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta: print(delta, end="")

Anthropic

with client.messages.stream(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    messages=[{"role": "user",
        "content": "Write a haiku"}],
) as stream:
    for text in stream.text_stream:
        print(text, end="")

Structured output (JSON)

All three support enforcing a JSON schema on the response. Gemini and OpenAI accept a Pydantic model directly; Anthropic uses a "tool" trick or explicit schema.

Gemini

class Recipe(BaseModel):
    name: str
    minutes: int

r = client.models.generate_content(
    model="gemini-2.0-flash",
    contents="Give me pasta",
    config={
      "response_mime_type": "application/json",
      "response_schema": Recipe,
    },
)
recipe: Recipe = r.parsed

OpenAI

class Recipe(BaseModel):
    name: str
    minutes: int

r = client.beta.chat.completions.parse(
    model="gpt-4o-mini",
    messages=[{"role": "user",
        "content": "Give me pasta"}],
    response_format=Recipe,
)
recipe: Recipe = r.choices[0] \
    .message.parsed

Anthropic

# Use "tool use" as JSON forcing
schema_tool = {
    "name": "return_recipe",
    "input_schema": Recipe.model_json_schema(),
}
r = client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    tools=[schema_tool],
    tool_choice={"type": "tool",
                 "name": "return_recipe"},
    messages=[{"role": "user",
        "content": "Give me pasta"}],
)
recipe = Recipe(**r.content[0].input)

Async

SDKHow
Gemini client.aio.models.generate_content(...) — same client, .aio namespace gives async equivalents.
OpenAI Separate client class: AsyncOpenAI(). Same method names, all await-able.
Anthropic Separate client class: AsyncAnthropic(). Same method names, all await-able.

Vision (image input)

All three accept images; the input shapes are different.

SDKHow to pass an image
Gemini contents=[PIL.Image.open("x.jpg"), "Describe this"] or types.Part.from_bytes(data=..., mime_type="image/jpeg")
OpenAI In content array: {"type":"image_url","image_url":{"url":"data:image/..."}}
Anthropic In content array: {"type":"image","source":{"type":"base64","media_type":"image/jpeg","data":...}}

Common surprises when switching

  • Anthropic requires max_tokens on every call. No default. Easy to forget when porting from OpenAI/Gemini.
  • OpenAI's "tool" role is a real role. Don't put tool results under "role": "user" — Anthropic does the opposite.
  • Gemini chat won't survive a process restart. It is not persistent storage.
  • Model names are wildly different. gemini-2.0-flash, gpt-4o-mini, claude-sonnet-4-5. Centralise the string.
  • Anthropic "structured output" via tool-use means the result lives in content[0].input, not in a normal text field. Different parsing path.
  • OpenAI has three APIs that all do "chat": Chat Completions (the classic, stateless), Responses (newer, supports server-side state), Assistants (threads, runs, files). Don't conflate them.

If you must support more than one provider

Don't roll your own abstraction layer. Use one of:

  • LiteLLM — drop-in wrapper that exposes an OpenAI-compatible API for ~100 providers including Gemini and Anthropic.
  • LangChain chat models — common ChatModel interface across providers (heavier dependency).
  • aisuite — minimal OpenAI-shaped wrapper across providers.

Note: for tool use, even with these wrappers you'll often still hit provider-specific edges (especially Anthropic's tool_result shape). Test each provider end-to-end before trusting the abstraction.

Mental models

Gemini = "framework-lite". The SDK does a lot for you (chat state, schema from functions). Less code, more magic.
OpenAI = "the de facto shape". Most third-party tools speak OpenAI's messages + tool_calls format. If you only know one, know this one.
Anthropic = "explicit, no magic". Stateless, forces max_tokens, makes tool-result shapes structural. More verbose, fewer surprises.
