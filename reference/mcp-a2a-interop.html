MCP + A2A: the two agent-interop protocols, composed

The wrap-up for the L45–47 mini-series. One architecture diagram, the primitives side-by-side, a "tool or agent?" decision guide, and the SE talk track — designed to re-read after the hands-on lessons have faded.

One-line summary
MCP connects an agent to tools & data (a function call). A2A connects an agent to other agents (delegate a task with a lifecycle). They are not competitors — they stack: A2A on the outside (agents delegating to agents), MCP on the inside (each agent reaching its tools). Almost every real multi-agent system uses both.
Source honesty — read before quoting anything
Both protocols and their SDKs are moving fast. The a2a-sdk API changed sharply across versions — the L47 code targets 1.1.2 specifically (proto types, supported_interfaces, route-builder functions); older tutorials won't match. On the MuleSoft side: Agent Fabric / Omni Gateway does govern A2A and MCP endpoints (confirmed on docs.mulesoft.com/agent-fabric), but a specific supported A2A spec version number is not stated in the public docs. Verify current product names, SDK versions, and spec revisions before a customer call — the architectural framings below are the durable part.

The one diagram to memorize

L47's Part 5 gestured at this — here it is drawn properly. This is the shape of a real multi-agent system: A2A for delegation between agents, MCP for each agent's own capabilities, an integration platform underneath doing the actual system work.

# A2A on the outside, MCP on the inside, integration underneath Orchestrator agent ↓ A2A: discover Agent Card, delegate a Task (submitted→working→completed) Specialist agent (e.g. fx-agent from L47) ↓ MCP: call a tool to fulfil the task (L46's HTTP MCP server) MCP server / tool layer ↓ REST call Anypoint System / Process APIs # DataWeave, orchestration, connectors ↓ connector protocols Salesforce · SAP · ServiceNow · DB · Mainframe · Files

Read it top-down: an orchestrator delegates to a specialist agent over A2A; that agent, inside its own execute(), calls tools over MCP to get the data it needs; those tools front real systems of record. Delete the A2A layer and agents can't cooperate without bespoke point-to-point wiring. Delete the MCP layer and each agent hardcodes its integrations. Same "delete a layer, rebuild what it gave you" logic as the rest of the stack.

Decision guide — should a capability be an MCP tool or an A2A agent?

The most common design question once you have both in your toolbox. The rule of thumb: expose a tool when the work is a stateless function; expose an agent when the work reasons, decides, or runs long.

Make it an MCP tool when:
  • The work is a single, well-defined operation — "get the exchange rate", "create the ticket", "run this query"
  • It returns quickly and doesn't need to report progress
  • There's no independent decision-making — the caller decides, the tool executes
  • You'd describe it as a function, not a worker
Make it an A2A agent when:
  • The work involves reasoning or multiple steps the caller shouldn't have to orchestrate
  • It can be long-running and the caller wants to watch progress (the task lifecycle earns its keep here)
  • It's owned by a different team, or built on a different stack, and you want a clean contract between them
  • You'd describe it as a teammate you hand a job to, not a function you call
Often it's both, nested:
  • A specialist A2A agent internally calls several MCP tools to do its job (the diagram above)
  • A capability starts life as an MCP tool, then grows reasoning and becomes an agent — the tool it used becomes one of the agent's tools
  • Rule of thumb: if you keep adding "and then decide…" logic on the caller side of a tool, that logic wants to live inside an agent instead

Where MuleSoft sits

The positioning is the same one layer up from the MCP story (see the MCP integration positioning doc). MCP exposes your APIs as tools; A2A lets agents cooperate. MuleSoft's job in both is governance: expose, secure, version, and monitor these endpoints so the mesh of agents is managed rather than a free-for-all.

Concretely, two named products do the MCP-server side today: MCP Bridge (GA 6 Mar 2026) turns existing APIs into MCP servers from Anypoint API Manager with no code and no redeploy — it runs on Omni Gateway and inherits your existing API policies; the MCP Connector builds an MCP server into a Mule flow when you need custom logic. Bridge is the "meet agents where they are" fast path; the Connector is the flexibility path. A2A endpoints are governed the same way through Agent Fabric / Omni Gateway.

"Your customer won't build one giant agent — they'll build several, owned by different teams, maybe on different stacks. A2A is how those agents delegate to each other without bespoke point-to-point integration: publish an Agent Card, and any other agent can discover and call it. MCP is how each of those agents reaches your actual systems — Salesforce, SAP, the mainframe. MuleSoft's role is the constant across both: Agent Fabric and Omni Gateway govern the A2A and MCP endpoints so the whole mesh lives under the same policies, auth, and catalog as your REST APIs. Same reuse-and-govern pitch you've always made — now covering agent-to-agent and agent-to-tool, not just app-to-app."
The strategic frame
Every team standing up an A2A agent or an MCP server is creating a new endpoint that needs auth, versioning, observability, and a catalog entry. That's the integration-governance conversation MuleSoft has always owned — the protocols are new, the problem isn't. Lead with "these are open, complementary protocols" (true, and it builds trust), then the governance follow-up writes itself.
