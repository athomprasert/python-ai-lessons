MCP integration: AgentCore Gateway vs Anypoint vs MuleSoft MCP

A pre-customer-call reference. Two comparisons, two SE talk tracks, one architecture diagram, one decision tree — designed to open on your phone five minutes before a meeting.

One-line summary
AgentCore Gateway is a lightweight MCP wrapper for clean REST/Lambda backends. Anypoint Platform is an integration platform — it exists whether or not you have agents. MuleSoft MCP exposes existing Anypoint APIs as MCP tools. The right answer for most enterprises is both: Gateway (or MuleSoft MCP) as the agent-facing layer, Anypoint underneath doing the actual integration work.
Source honesty — read before customer call
MuleSoft's MCP story has two named products: MCP Bridge (API-Manager-guided, config-only, GA 6 Mar 2026) and the MCP Connector (build-into-a-flow). AgentCore Gateway is AWS-managed and GA'd late 2025 but the docs churn. Product names still evolve — verify current features before quoting anything specific. The architectural framings in this doc are stable.

The three-layer picture

The diagram worth memorising. Every conversation returns to this.

# The natural stack — three layers, three jobs Agent # Strands / LangGraph / Pydantic AI / ADK ↓ MCP tool call AgentCore Gateway OR MuleSoft MCP # Protocol façade, auth, tool discovery ↓ REST call Anypoint System / Process APIs # DataWeave, orchestration, connectors ↓ connector protocols Salesforce · SAP · ServiceNow · DB · Mainframe · Files

Delete a layer and you rebuild what it gave you. Delete Gateway/MCP → you write an MCP server. Delete Anypoint → you rewrite the same SAP/SFDC connectors in Lambda for the tenth time. That's the whole positioning story compressed.

The critical insight: AgentCore Gateway is a lightweight MCP wrapper. MuleSoft MCP is a full integration platform that happens to speak MCP. That framing determines the choice — is MCP a protocol translation problem (Gateway) or the new API contract on top of your existing estate (MuleSoft)?

They coexist: Bridge to govern many APIs uniformly with no code; Connector when a single integration needs custom flow logic (reshape with DataWeave, orchestrate across systems). Both put MuleSoft on the MCP path — Bridge is the "meet agents where they are, no dev project" option.

"AgentCore Gateway is great when you're wrapping already-clean AWS-hosted endpoints — think of it as 'API Gateway for MCP.' Where MuleSoft's MCP pulls ahead is on everything an integration platform gives you that a proxy can't: connectors to SAP and mainframe, DataWeave to reshape ugly responses into what an LLM can actually use, orchestration when one agent call needs to touch three systems, and Anypoint governance so MCP tools live in the same catalog as your REST APIs — same versioning, same SLAs, same monetization. If your customer is 100% AWS and their backends are already Lambda-clean, Gateway is fine. Everyone with an SAP, a mainframe, or a central IT function ends up needing MuleSoft on the path — regardless of which cloud hosts the agent."

Decision tree — when to pick just one

Just AgentCore Gateway, no Anypoint / MuleSoft MCP:
  • Startup or greenfield build
  • Backends are already clean REST APIs / Lambda
  • Nothing needs transforming or orchestrating
  • Small integration footprint (few backends, no legacy)
  • Buyer is an AWS-committed engineering team, not central IT
Just MuleSoft (MCP + Anypoint), no Gateway:
  • Multi-cloud, on-prem, or data-residency constraints rule out AWS-only
  • Central IT owns the integration story and wants MCP under the same governance as REST
  • Backends are messy (SAP, mainframe) — DataWeave + connectors save weeks
  • You want the same flow callable by agents (MCP) and traditional apps (REST)
  • Anypoint is already in place — no reason to introduce a parallel layer
Both, stacked (the most common enterprise pattern):
  • AWS-committed for the AI layer, but the integration estate is broader
  • Gateway handles agent-facing concerns (semantic tool selection, per-user OAuth to backend, MCP protocol)
  • Anypoint (with or without MCP) handles the actual integration under the hood
  • Bonus: keeps AWS agent code portable — swap Gateway for MuleSoft MCP later without changing the agent

Common objections and how to handle them

ObjectionResponse
"We're 100% AWS — why do we need MuleSoft?" You don't, if your backends are already REST and clean. The moment SAP, mainframe, or complex SFDC enters the picture, Gateway alone forces you to build the transformation and connector work in Lambda. That's rebuilding Anypoint, badly.
"MuleSoft is too heavy for a simple agent." Correct — for a simple agent. If the agent's job is to call three clean endpoints, use Gateway. If the agent's job is to reason across your systems of record, you need what Anypoint does regardless of whether MCP is involved.
"AWS said Gateway is all we need." Gateway is all you need for the MCP layer. It's not an integration platform, and AWS docs don't claim otherwise. The question isn't Gateway vs Anypoint — it's what sits under Gateway.
"We'll write our own MCP server." Fair — many teams do. Track the cost: auth, per-user OAuth flows to backends, tool discovery, semantic selection over N tools, observability, versioning, catalog. That's the shopping list Gateway or MuleSoft MCP gives you managed. Build vs buy applies here like anywhere else.
"Doesn't MCP replace REST APIs entirely?" No — MCP is agent-to-tool. Traditional apps, integrations, and B2B partners still speak REST/SOAP/events. Building MCP-only means the same flow exists twice. MuleSoft MCP lets one flow serve both callers.
The strategic MuleSoft SE frame
Gateway isn't a threat — it expands your addressable conversation. Every AgentCore Gateway deployment is a candidate customer for MuleSoft underneath it. Lead with honest positioning ("Gateway is great for the MCP layer") and the follow-up on integration depth writes itself. Trying to win by saying "MuleSoft has MCP too, skip Gateway" is a losing argument on AWS-committed accounts — you'd be defending a layer, not expanding the deal.
