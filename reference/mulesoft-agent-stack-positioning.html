MuleSoft Agent Stack: Omni Gateway, Agent Fabric, Anypoint

A pre-customer-call reference. How MuleSoft's evolving agent-era stack — Omni Gateway, Agent Fabric, Anypoint Platform — positions against code-first frameworks and Agentforce. One diagram, five pieces, and the SE talk tracks you'll actually use.

One-line summary
Anypoint Platform integrates enterprise systems (stable, 15 years). Omni Gateway is the unified API + MCP + model gateway in front of it (recent). Agent Fabric is the agent-to-agent broker across vendors (newest). Together, they form Salesforce/MuleSoft's answer to "how do agents — Agentforce or code-first — safely reach enterprise systems and each other, with governance." Frameworks and Agentforce sit above this stack; systems of record sit below.
Source honesty — read before customer call
Omni Gateway and Agent Fabric are recent Salesforce/MuleSoft announcements and the products are shipping in pieces. Names, feature scope, and packaging keep evolving. Anypoint Platform positioning is stable and well-documented. Verify current product names, feature availability, and pricing against mulesoft.com and salesforce.com/agentforce before quoting specifics. The architectural framings on this page are the durable part.

The five-layer picture

This is the whiteboard diagram. Every customer conversation returns to it.

# Five layers, five jobs — MuleSoft owns the middle three Caller # Human user, another agent, a channel (Slack, Teams, mobile, voice) ↓ Agent # Agentforce OR LangGraph/ADK/Strands/AutoGen on Foundry/AgentCore/Vertex ↓ A2A protocol ────────────────────────────────┐ Agent Fabric # Multi-vendor agent broker: A2A routing, discovery, trust, audit ↓ tool call / MCP / REST Omni Gateway # Unified API+MCP+model gateway: auth, rate limit, redaction, cost, catalog ↓ REST / SOAP / connector Anypoint Platform # Integration: Experience/Process/System APIs, DataWeave, connectors, governance ↓ connector protocols Salesforce · SAP · ServiceNow · Workday · Snowflake · Mainframe

Delete a layer and you rebuild what it gave you. Delete Anypoint → you write connectors and transformations in the gateway. Delete Omni Gateway → agents wire directly to every API with per-agent auth. Delete Agent Fabric → cross-vendor agent-to-agent traffic becomes point-to-point spaghetti. Each layer earns its keep by the mess you'd have to build without it.

The three pieces, one paragraph each

1. Anypoint Platform (stable, well-documented)

The integration platform. Design → build → deploy → govern → deprecate for APIs across REST, SOAP, GraphQL, JMS, files, DB, and 300+ prebuilt connectors (SFDC, SAP, ServiceNow, Workday, Snowflake, mainframe). DataWeave for transformation. API-led connectivity (Experience / Process / System APIs) for reuse. Runs anywhere Mule runs — CloudHub, Runtime Fabric, Flex Gateway, self-hosted. This is the layer that has always existed — everything else in the agent stack builds on top.

2. Omni Gateway (the unified agent-era gateway)

A single gateway that speaks API + MCP + model traffic — the three surfaces an agent touches. In front of Anypoint (and increasingly in front of your model providers), it does policy enforcement (auth, rate limit, quota, per-caller entitlements), redaction and PII masking, cost tracking / chargeback per user or agent, and a governed catalog of tools/models available to agents. Think of it as API Gateway extended for the agent era — the same governance you already have for REST, now covering MCP tool discovery and model call metering too. Because it fronts Anypoint, one flow can be exposed as REST, MCP, and a model-aware endpoint with the same policies applied.

3. Agent Fabric (multi-vendor agent broker)

Cross-vendor agent-to-agent (A2A) coordination. When an Agentforce agent needs a specialised code-first agent (or vice versa), when Slack's agent needs to hand off to a partner's agent, when your enterprise has three or four agent stacks in production — Agent Fabric is the broker that lets them find, invoke, and audit each other with consistent identity and trust. Think "service mesh for agents" — discovery, routing, audit, quota — for a world where no single vendor owns every agent your enterprise runs. Not a runtime; a coordination layer sitting alongside agents.

"Agentforce doesn't compete with the MuleSoft agent stack — it consumes it. When an Agentforce action needs to update SAP, look up Workday, or write to a mainframe, the shortest path is a MuleSoft-backed Agentforce action calling Anypoint. When enterprise-grade governance matters — per-user auth to backends, redaction, chargeback — that lives naturally in Omni Gateway. When Agentforce needs to hand off to a specialised code-first agent (or a partner's agent, or a Slack agent), Agent Fabric is what makes that multi-vendor coordination governable. The mental model is simple: Agentforce owns the CRM-facing conversation, MuleSoft owns the integration and governance layer underneath."

Decision tree — three enterprise patterns

Framework-only + Anypoint (small-to-mid, code-first shop):
  • Building agents in LangGraph / Strands / AutoGen; no Agentforce yet
  • Anypoint handles the integration; agents call Anypoint APIs as tools
  • Omni Gateway / Agent Fabric can be added later — start with the layer that pays back fastest
  • Simplest to reason about; two products, clear responsibilities
Agentforce + Anypoint (Salesforce-anchored enterprise):
  • Service, sales, marketing conversations live in Agentforce
  • Every action that touches non-Salesforce data flows through Anypoint
  • Omni Gateway earns its keep when redaction, per-user OAuth to backend, or chargeback matter
  • Agent Fabric optional until multi-vendor agent coordination becomes real
Hybrid at scale (the full stack — large, multi-vendor enterprises):
  • Agentforce for CRM conversations; code-first framework agents for non-CRM specialists (dev, data, security, ops)
  • Omni Gateway fronts everything — API, MCP, and model traffic — with unified policy
  • Agent Fabric brokers agent-to-agent across Agentforce, in-house frameworks, and partner agents
  • Anypoint underneath, doing the integration work it has always done
  • One governance model, one audit surface, one chargeback story — regardless of which agent stack the request came from

Common objections and how to handle them

ObjectionResponse
"That's three MuleSoft products for one job." Three products, three jobs. Anypoint integrates systems. Omni Gateway governs traffic (API + MCP + model). Agent Fabric brokers across agent vendors. Each addresses a distinct enterprise problem — you'd rebuild each one badly if you skipped it. Start with Anypoint (foundational) and add the others when the specific pain shows up.
"Why do we need Omni Gateway if we already have Anypoint API Manager?" API Manager governs API traffic. Omni Gateway extends the same governance model to MCP traffic and model traffic — the two surfaces agents introduced that didn't exist before. If your agents call only REST, API Manager is fine. Once they call MCP tools or LLM providers directly, you want one governance layer, not three.
"How is Omni Gateway different from AgentCore Gateway or Azure API Management?" AgentCore Gateway is an AWS-managed MCP wrapper for AWS-hosted endpoints. API Management gateways don't natively govern MCP or model traffic. Omni Gateway is the cross-cloud, integration-platform-native gateway that speaks all three (API + MCP + model). See also the MCP integration positioning reference.
"Agent Fabric sounds like an agent framework. Is it competing with LangGraph / AutoGen?" No — it's above them. Agent Fabric doesn't reason, doesn't call LLMs, doesn't hold prompts. It brokers cross-vendor agent-to-agent calls with discovery, trust, and audit. Frameworks build individual agents; Agent Fabric coordinates agents built in different frameworks by different teams (or vendors).
"We're 100% Agentforce — do we need any of this?" Yes if your Agentforce agents touch non-Salesforce systems, need enterprise-grade governance, or ever need to coordinate with an agent your customer or partner runs. Anypoint is almost certainly involved on day one. Omni Gateway when governance matters. Agent Fabric when multi-vendor becomes real.
"Isn't this just marketing — bundling old products with an 'agent' label?" Anypoint is the mature product; that's fair. Omni Gateway and Agent Fabric are genuinely new categories because MCP traffic, model traffic, and cross-vendor A2A didn't exist as governance concerns two years ago. What's not new is the pattern: an integration platform is the durable middle layer, and this stack is what that middle layer looks like in the agent era.
The strategic MuleSoft SE frame
Every agent conversation — Agentforce, LangGraph, AutoGen, Strands, ADK — ends at "how does this thing safely reach our systems and interoperate with our other agents?" That is the MuleSoft agent stack's home turf. You don't have to pick agent stacks for the customer. You just have to be present in the layer underneath whichever one they choose. Position:
  • Anypoint — the durable middle layer between agents and systems of record
  • Omni Gateway — one governance surface for API + MCP + model traffic
  • Agent Fabric — the multi-vendor coordination layer no single agent stack can solve alone
That framing works whether the customer is 100% Agentforce, 100% AWS-native, 100% Azure-native, or hybrid. MuleSoft doesn't have to win the "which agent stack" argument — it wins by being the layer they all depend on.

Where the abstraction leaks

  • Product names will drift. Omni Gateway / Agent Fabric are new-enough that packaging and naming may change. Anchor your customer conversation on the job each layer does, not the current SKU.
  • Feature completeness varies. Not every capability described here is GA today. Confirm before promising specific policies, metering breakdowns, or A2A protocol coverage.
  • Overlap with cloud-native gateways. Omni Gateway overlaps in scope with AgentCore Gateway, Azure API Management, Apigee. Position by what MuleSoft uniquely brings — integration platform native, cross-cloud, MCP + API + model in one governance model.
  • Agent Fabric is early. Cross-vendor A2A standards are still solidifying. Expect the product to evolve as A2A protocols mature.
  • You still need agents. This stack doesn't build agents — it connects them. Every conversation still needs an Agentforce agent or a framework agent above it.
One customer conversation, three checkpoints
When walking a customer through the stack, hit these in order:
  1. "Where does your agent's tool call go today?" → surfaces Anypoint (or the absence of it)
  2. "How do you govern who can call what, with what quotas, redacting what?" → surfaces Omni Gateway
  3. "When Agentforce needs to talk to your dev-copilot agent — or your partner's agent — how does that happen?" → surfaces Agent Fabric
Three questions. Three problems. Three MuleSoft answers.
