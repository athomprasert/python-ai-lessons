MuleSoft in the Agentic Enterprise Reference Architecture

Where Agent Fabric, Omni Gateway, and MuleSoft's MCP server pieces sit when an enterprise runs many agents. One layered diagram, a component-by-component table, a worked request trace, and the SE talk track — grounded in the L45–47 protocol lessons.

One-line summary
Agent Fabric is MuleSoft's umbrella for governing and orchestrating an enterprise's agents. Inside it: the Agent Broker plans and routes work, Omni Gateway is the policy/control plane for MCP and A2A traffic, the Agent Registry (in Anypoint Exchange) is discovery/reuse, and MCP Bridge / MCP Connector turn your APIs into agent-callable tools. It's API-led architecture applied one layer up: instead of governing app-to-app APIs, you govern agent-to-agent (A2A) and agent-to-tool (MCP) traffic.

The problem it solves — "agent sprawl"

Once every team can spin up an agent, you get the same mess APIs created a decade ago: dozens of agents and MCP servers, built on different stacks, with no shared discovery, no consistent auth, and no visibility into who's calling whom. Agent Fabric is MuleSoft's answer — a governed, coordinated network of agents rather than a free-for-all. If you lived through "API sprawl → API-led connectivity," this is the exact same story, one abstraction level higher.

The one diagram to memorize

Read it top-down. Consuming agents delegate work; the broker decides who does it; the gateway governs every hop; the registry makes things findable; MCP turns your existing API estate into tools. Observability wraps the whole thing.

# MuleSoft Agentic Enterprise Reference Architecture CONSUMING AGENTS / APPS # Agentforce, custom agents, 3rd-party agents, copilots │ A2A: delegate a task (Agent Card discovery + task lifecycle) ▼ AGENT BROKER # orchestration: plan & route by intent, policy, identity, runtime state │ ┌──────────────────────────────────────────────────────────────┐ │ OMNI GATEWAY # governance plane for MCP + A2A: auth, rate-limit, │ │ # routing, observability — managed ingress/egress │ └──────────────────────────────────────────────────────────────┘ │ MCP: call a tool │ A2A: call another agent ▼ ▼ MCP SERVERS # from your APIs, via… SPECIALIST AGENTS # internal / external ├─ MCP Bridge # API Manager, no code (GA 6 Mar 2026) └─ MCP Connector # built into a Mule flow, custom logic │ REST / SOAP / events ▼ ANYPOINT SYSTEM / PROCESS APIs # DataWeave, orchestration, 300+ connectors ▼ Salesforce · SAP · ServiceNow · DB · Mainframe · Files # ── cross-cutting, spanning every layer above ────────────────── AGENT REGISTRY (Anypoint Exchange) # discover & reuse agents, MCP servers, LLMs AGENT SCANNERS # auto-discover & catalog external agents AGENT VISUALIZER + ANYPOINT MONITORING # map the network; metrics, logs, traces

Delete a layer and you rebuild what it gave you — the same logic as the classic API-led stack. No broker → every agent hardcodes who to call. No Omni Gateway → auth and rate-limiting get reinvented per agent. No registry → nobody can find what already exists, so it's rebuilt. No MCP layer → agents can't reach your systems of record without bespoke integration.

How one request flows

A worked trace makes the layering concrete. Say an Agentforce agent needs "the latest order status for customer 12345, then draft a follow-up":

  1. Delegation (A2A). The consuming agent doesn't know the internals — it delegates the task to the fabric. Discovery happens via the Agent Registry (Agent Cards live here, same mechanism as L47's /.well-known/agent-card.json).
  2. Routing (Agent Broker). The broker reads intent + identity + policy and decides which agent or tool handles each part — order lookup vs. drafting.
  3. Governance (Omni Gateway). Every hop — agent-to-agent or agent-to-tool — passes the gateway, which enforces auth, rate limits, and emits telemetry. Nothing bypasses it.
  4. Tool call (MCP). The order-lookup step calls an MCP server that MCP Bridge generated from your existing "Orders" API — no new code, existing API Manager policies apply.
  5. Integration (Anypoint). That API fronts the real system of record (SAP, Salesforce, a DB) through Anypoint connectors and DataWeave.
  6. Observability (Visualizer + Monitoring). The whole path shows up as a traced, visualized flow — you can see the agent-to-agent-to-tool chain end to end.

The SE talk track

"Your agents are about to sprawl the same way your APIs did. Every team will build one, on a different stack, calling each other point-to-point — no discovery, no shared auth, no idea who's calling whom. Agent Fabric is the same play we ran for API sprawl, one level up: a registry so agents are discoverable and reusable, a broker so work is routed by policy and intent instead of hardcoded, and Omni Gateway as the single control plane so every agent-to-agent and agent-to-tool call is authenticated, rate-limited, and observable. And because it speaks the open protocols — A2A between agents, MCP to tools — you're not locked in: your Agentforce agents, your custom LangGraph agents, and third-party agents all live in the same governed fabric. MCP Bridge means you expose your existing Anypoint APIs as agent tools with no new code. The pitch hasn't changed in fifteen years — reuse and govern — the endpoints just got smarter."
The strategic frame
Don't sell this as a brand-new product category — sell it as API-led connectivity for agents. Every customer who bought the "govern your API sprawl" story already understands the "govern your agent sprawl" story; you're extending an architecture they've already committed to, not asking them to learn a new one. Omni Gateway is API Manager for agents; the Agent Registry is Exchange for agents; the Agent Broker is the orchestration layer for agents. Familiar shapes, new traffic.
