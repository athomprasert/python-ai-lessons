OpenAI-compatible API

The de facto lingua franca for LLM HTTP — a JSON shape that ~100 providers speak. Like USB-C, but for chat.

One-line summary

"OpenAI-compatible" means the HTTP server accepts the exact same JSON request shape as OpenAI's /v1/chat/completions endpoint and returns the same response shape. Existing OpenAI client code works against it by changing only base_url and api_key.

What it actually is — and is not

It isIt is not
A de facto JSON wire formatA formal standard or RFC
Free to implement — anyone canOwned, licensed, or partnered
The most-supported shape across LLM providersA guarantee that features match — most providers cover 70–90% of the spec
Convenient for switching providers via base_urlA replacement for testing each provider end-to-end

The request shape

POST /v1/chat/completions
Authorization: Bearer YOUR_KEY
Content-Type: application/json

{
  "model": "gpt-4o-mini",
  "messages": [
    {"role": "system",    "content": "You are terse."},
    {"role": "user",      "content": "Hello"}
  ],
  "temperature": 0.7,
  "max_tokens": 200,
  "stream": false,
  "tools": [/* optional */],
  "tool_choice": "auto",
  "response_format": { "type": "json_object" }
}

Common fields you'll send

FieldTypeWhat it does
modelstringProvider-specific model name. The only field that's not portable.
messagesarrayThe conversation. See "the messages array" below.
temperature0.0–2.0Randomness. Lower = more deterministic.
max_tokensintCap on output length.
top_p0.0–1.0Nucleus sampling. Use or temperature, not both.
streamboolIf true, server sends SSE chunks instead of one response.
toolsarrayFunction-calling schema definitions.
tool_choicestring / obj"auto", "none", or force a specific tool.
response_formatobj{"type":"json_object"} or full JSON schema.
seedintBest-effort reproducibility (OpenAI only fully honours it).
stopstr / arrayStop sequences.

The messages array

Each message is an object with role and content:

RoleUsed forNotes
"system"Instructions to the modelUsually only one, at index 0. Some providers append it to the first user message internally.
"user"Human inputThe thing the user typed.
"assistant"Model's prior repliesYou append these so the model has context for the next turn.
"tool"Result of a tool callMust include tool_call_id matching the call that produced this result.

content is usually a string, but for multimodal input it's an array of typed parts:

{
  "role": "user",
  "content": [
    {"type": "text",      "text": "What's in this image?"},
    {"type": "image_url", "image_url": {"url": "data:image/jpeg;base64,..."}}
  ]
}

The response shape

{
  "id": "chatcmpl-abc123",
  "object": "chat.completion",
  "created": 1718900000,
  "model": "gpt-4o-mini",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "Hi.",
        "tool_calls": [/* optional */]
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 12,
    "completion_tokens": 4,
    "total_tokens": 16
  }
}

What you actually read

FieldMeaning
choices[0].message.contentThe assistant's text answer. May be null if the model called a tool instead of replying.
choices[0].message.tool_callsArray of tool-call requests when the model wants to use a function.
choices[0].finish_reason"stop" (done), "length" (hit max_tokens), "tool_calls" (wants tools), "content_filter" (refused).
usageToken counts for billing / observability.

Tool calling — the exact shape

You send (in the request):

"tools": [
  {
    "type": "function",
    "function": {
      "name": "get_weather",
      "description": "Get current weather for a city",
      "parameters": {
        "type": "object",
        "properties": {
          "city": {"type": "string"}
        },
        "required": ["city"]
      }
    }
  }
]

The model returns (in choices[0].message):

{
  "role": "assistant",
  "content": null,
  "tool_calls": [
    {
      "id": "call_xyz",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\":\"Singapore\"}"   # string, you JSON.parse it
      }
    }
  ]
}

You send back (next request, in messages):

{
  "role": "tool",
  "tool_call_id": "call_xyz",
  "content": "{\"temp_c\": 31, \"condition\": \"humid\"}"
}
Compare to the native Gemini SDK (used in L18): Gemini returns response.function_calls and tool results are types.Part.from_function_response(...). Different field names, same loop. See SDK comparison for the side-by-side.

Streaming output (SSE)

With "stream": true, the server sends Server-Sent Events. Each event is a partial chunk:

data: {"choices":[{"delta":{"role":"assistant"},"index":0}]}
data: {"choices":[{"delta":{"content":"Hi"},"index":0}]}
data: {"choices":[{"delta":{"content":" there"},"index":0}]}
data: {"choices":[{"delta":{},"finish_reason":"stop","index":0}]}
data: [DONE]

Your code accumulates the delta.content strings to reconstruct the full answer. The OpenAI SDK does this for you when you iterate the stream object.

The "switch in one line" trick

The whole point. Same client, three different backends:

from openai import OpenAI

# Local Ollama (no API key needed but the SDK requires one)
ollama = OpenAI(base_url="http://localhost:11434/v1", api_key="ollama")

# Groq — same code, very fast inference
groq = OpenAI(base_url="https://api.groq.com/openai/v1",
              api_key=os.environ["GROQ_API_KEY"])

# Gemini via its OpenAI-compat endpoint
gemini = OpenAI(
    base_url="https://generativelanguage.googleapis.com/v1beta/openai/",
    api_key=os.environ["GEMINI_API_KEY"],
)

# Calling code below is identical for all three:
r = gemini.chat.completions.create(
    model="gemini-2.0-flash",
    messages=[{"role": "user", "content": "Hello"}],
)

Where it leaks — "compatible" ≠ "identical"

Most providers cover the basics. Edges where you will get bitten:

  • Tool calling. Many providers — especially small local models — produce malformed tool_calls (invalid JSON in arguments, mismatched tool_call_id). Test before trusting.
  • JSON mode / response_format. OpenAI honours strict JSON schema. Most others honour {"type":"json_object"} as a hint only.
  • Streaming edge cases. Empty deltas, finish reasons during tool calls, and the [DONE] sentinel are handled differently by some servers.
  • System prompts. Some models silently concatenate the system message into the first user message; behaviour drifts.
  • Parallel tool calls. Spec allows tool_calls to be an array of multiple calls. Some providers return only one per turn.
  • Vision / audio parts. The typed content array is least portable — image URLs, base64 prefixes, and MIME types vary.
  • seed reproducibility. Only OpenAI takes it seriously. Others ignore it.
  • Model names. The one field that's never portable. Centralise it behind a config var.

When to use it vs the native SDK

SituationUse
You're committed to one provider, want best ergonomicsNative SDK (e.g. google-genai for Gemini)
You want to swap models for cost/latency comparisonOpenAI client + base_url per provider
You want to test local models alongside cloud onesOpenAI client → Ollama / vLLM
You're building a multi-provider production appLiteLLM / OpenRouter as a gateway, expose OpenAI shape internally
You rely on a provider-specific feature (Gemini's auto-schema-from-Python-fn, Anthropic's cache_control)Native SDK — those features don't translate

The MuleSoft analogue

This is the same pattern as exposing a System API and switching backends. Define one stable contract (OpenAI shape) and put a routing layer in front. Callers don't change when the backend changes. LiteLLM / OpenRouter are essentially a "System API for LLMs" — the same role you'd build a Mule app to play if you wanted vendor-portable AI inside an enterprise integration stack.

Mental models

USB-C for LLMs. Not owned by anyone, but everyone agreed to make their plug fit the same hole. The cable works; the device behind it might still have quirks.
SQL standard, in spirit. Like ANSI SQL — every database "supports" it, but Postgres, MySQL, and SQLite all have dialect-level differences. You can write portable queries if you stay on the common subset.

Quick checklist before relying on compatibility

  1. Does the provider's docs explicitly say "OpenAI-compatible"?
  2. Which version of the OpenAI spec do they target? (most cover Chat Completions, fewer cover Responses or Assistants)
  3. Does tool calling work end-to-end? Send a real tool call and parse the response.
  4. Does response_format: {"type": "json_object"} actually produce parseable JSON?
  5. Does streaming finish cleanly? Watch for hangs at [DONE].
  6. Are token-count fields populated in usage? Some local servers leave this empty.
