Reranker

A second pass that reorders retrieved chunks using a slower, sharper model — so the most relevant ones float to the top.

One-line summary

Retrieval (fast, recall-oriented) gets you the top 50; reranker (slow, precision-oriented) narrows that to the top 5.

Two-stage pipeline

USER QUESTION │ ▼ ┌────────────────────────────────┐ │ STAGE 1 — Retrieval (FAST) │ │ Vector / BM25 / hybrid │ │ Searches all 10,000 chunks │ │ Returns: top 50 candidates │ └──────────────┬─────────────────┘ │ 50 chunks ▼ ┌────────────────────────────────┐ │ STAGE 2 — Reranker (SHARP) │ │ Cross-encoder scores each │ │ (query, chunk) pair directly │ │ Returns: top 5 │ └──────────────┬─────────────────┘ │ 5 chunks ▼ Pass to the LLM

Why two stages? Running the slow model on all 10,000 chunks would take ~100s. On 50 candidates it's ~500ms. Cheap filter first, sharp tool second.

Bi-encoder vs cross-encoder

The single most important distinction.

Bi-encoder (stage 1)

Query ─→ [encoder] ─→ q_vec │ ▼ Chunk ─→ [encoder] ─→ c_vec │ ▼ cosine(q_vec, c_vec)

Encoded separately. Vectors pre-computed at index time. Fast over millions of chunks, but blind to query↔chunk interactions.

Cross-encoder (stage 2)

Query ─┐ ├─→ [reads BOTH] ─→ score Chunk ─┘

Encoded together. Attention compares phrases between query and chunk. Slow per pair, but understands "how often" → "quarterly".

Worked example

Question: "How often should I rotate Connected App secrets?"

ChunkBi-encoderCross-encoder
A: "Connected Apps use OAuth tokens."0.78 ← misranked top0.31
B: "Rotate Connected App secrets quarterly per security policy."0.720.95 ← actual best
C: "Connected Apps integrate with external clients."0.690.18

The bi-encoder ranks A on top because all the words match the query. The cross-encoder spots that B literally answers the question.

Code — drop-in for any RAG

from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-12-v2")

def rerank(question, candidate_chunks, top_n=5):
    pairs = [(question, c) for c in candidate_chunks]
    scores = reranker.predict(pairs)
    ranked = sorted(zip(scores, candidate_chunks), reverse=True)
    return [(float(s), c) for s, c in ranked[:top_n]]

Wire into a hybrid pipeline as a final stage:

def hybrid_with_rerank(q, coll, bm25, chunk_ids, top_n=5):
    # Wide net first
    candidates = hybrid_search(q, coll, bm25, chunk_ids, k_each=25, top_k=25)
    chunks = [c for _, c in candidates]
    # Sharp narrow
    return rerank(q, chunks, top_n=top_n)

Metrics it moves

MetricEffect
Precision@KBig improvement — the headline win
MRRBig improvement — pushes the best chunk to rank 1
Context PrecisionImproves — same mechanism
Faithfulness / Hallucination RateModest improvement (cleaner context = fewer hallucinations)
Hit@KFlat — reranker can't find chunks not already in the candidate set
Recall@KMostly flat; sometimes +1–3pp
Latency (p95/p99)Worse by typically 100–500ms

When to add it

  • Eval shows MRR is low while Hit@K is fine
  • Eval shows Precision@K is low while Recall@K is OK
  • Generation cost dominated by junk chunks polluting the prompt
  • You're already using high K (≥ 20) for recall and want focus back

When NOT to bother

  • Hit@K is low — the right chunk isn't in the candidate set; reranking can't conjure it. Fix retrieval first.
  • Latency-bound and SLO already tight — +200ms might break it
  • Tiny corpus (< 100 chunks) — nothing to rerank
  • You haven't measured. Don't add it speculatively.

Mental models

Search vs interview. Bi-encoder retrieval is a resume screen — fast, looks at each candidate independently. Cross-encoder rerank is the actual interview — slow, sees the candidate face-to-face. You can't interview 10,000 candidates and you can't hire from resume scores alone.
MuleSoft analogue. Same shape as a two-tier filter in a Mule flow: cheap gateway check (API key, rate limit) drops 95% of bad traffic; deeper validation runs only on survivors.

Quick gotchas

  • Reranker scores aren't probabilities. They're raw model logits. Don't compare across different reranker models.
  • Stage 1 limits stage 2. If hybrid retrieval misses the right chunk in top 25, no reranker can save you. Tune retrieval recall before adding rerank.
  • First call is slow. The model loads on first predict() — warm it at startup.
  • CPU is fine for L-12 MiniLM at ≤ 50 candidates. For larger or BGE-large, use GPU.
